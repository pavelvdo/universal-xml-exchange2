# Глава 19. Механизм заданий

> Agent ref: Механизм заданий: фоновые задания, регламентные задания, планировщик; выполнение в файловом, клиент-серверном варианте и в мобильном приложении.
> Ключевые разделы: 19.1 Общая информация, 19.2 Фоновые задания, 19.3 Регламентные задания, 19.4 Окружения, 19.5 Метаданные регламентного задания
> ---

## 19.1. Общая информация

Механизм заданий предназначен для выполнения прикладной функциональности по расписанию или асинхронно.

**Задачи механизма:**
- определение регламентных процедур на этапе конфигурирования;
- выполнение действий по расписанию;
- асинхронный вызов процедуры или функции без ожидания завершения;
- отслеживание хода выполнения и статуса завершения;
- получение списка текущих заданий;
- ожидание завершения одного или нескольких заданий;
- управление заданиями (отмена, блокировка и др.).

**Компоненты:** фоновые задания, метаданные регламентных заданий, регламентные задания, планировщик заданий.

**Фоновые задания** — выполнение прикладных задач асинхронно средствами встроенного языка.

**Регламентные задания** — выполнение по расписанию. Хранятся в информационной базе, создаются на основе метаданных конфигурации (наименование, метод, использование и т.д.). Фактическое выполнение — через фоновое задание, запускаемое планировщиком.

**Расписание** регламентного задания задаётся, как правило, в информационной базе; для предопределённых заданий может быть задано в конфигурации.

**Планировщик заданий** периодически проверяет соответствие текущего времени расписанию регламентного задания и при совпадении создаёт фоновое задание для выполнения.

**Ключ задания** — прикладное значение для обеспечения выполнения в один момент времени не более одного фонового задания. Уникальность учитывается раздельно: для фоновых заданий, запускаемых из встроенного языка («самостоятельные»), и для заданий, запускаемых через регламентные («регламентные»). Для самостоятельных фоновых заданий ключ гарантирует одно задание с данным значением ключа и именем метода (имя модуля + имя процедуры/функции). Для регламентных — одно фоновое задание, порождённое данным регламентным заданием с данным значением ключа. Ключ позволяет группировать задания по прикладному признаку так, чтобы в рамках группы выполнялось не более одного задания.

## 19.2. Фоновые задания

### 19.2.1. Общие сведения

Фоновые задания удобны для длительных вычислений с асинхронным получением результата.

**Метод задания** — любая процедура или функция неглобального общего модуля, вызываемая на сервере. Параметры должны в точности соответствовать параметрам метода; допустимы любые значения, передаваемые на сервер. Сериализованные параметры не более 1 Гбайт; при превышении — исключение. Если метод — функция, возвращаемое значение игнорируется.

**Права:** создавать задание может любой пользователь; выполняется от имени создателя. Получать задания и ожидать завершения — пользователь с административными правами или создатель заданий.

**Сеанс:** фоновое задание — сеансовый объект, не привязанный к пользовательскому сеансу; для каждого задания создаётся системный сеанс от имени пользователя, выполнившего вызов. Сохраняемого состояния нет.

**Вложенность:** фоновое задание может порождать другие фоновые задания. В клиент-серверном варианте это даёт распараллеливание по рабочим процессам кластера (несколько дочерних заданий с ожиданием завершения в основном).

**Данные:** задание может помещать данные во временное хранилище вызывающего сеанса (например, отчёт). Передача из вызывающего сеанса в сеанс задания невозможна.

**Порядок запуска:**
- служебные действия (фиксация в журнале регистрации, установка параметров соединения с внешними источниками данных, режим интерфейса и т.д.);
- обработчик УстановкаПараметровСеанса в модуле сеанса;
- выполнение метода задания.

**Хранение:** завершённые (успешно или аварийно) фоновые задания хранятся до суток, затем удаляются.

**Лимиты истории выполнения:**
- фоновых заданий — не более 1 000;
- регламентных заданий — не более 1 000 запусков (система хранит не менее 3 последних запусков на задание; при превышении лимита — 2, затем 1);
- системных фоновых заданий (отчёты, поиск и т.д.) — не более 1 000 запусков.

### 19.2.2. Работа из встроенного языка

Управление — через менеджер ФоновыеЗадания:
- **ФоновыеЗадания.Выполнить(** метод, массив параметров, ключ, описание **)** — запуск; возвращает ФоновоеЗадание.
- **ФоновыеЗадания.ПолучитьФоновыеЗадания(** отбор **)** — список по отбору.
- **ФоновыеЗадания.НайтиПоУникальномуИдентификатору(** идентификатор **)** — поиск по УникальныйИдентификатор.
- **ФоновоеЗадание.ОжидатьЗавершенияВыполнения(** таймаут **)** — ожидание одного задания; возвращает объект с текущим состоянием.
- **ФоновыеЗадания.ОжидатьЗавершенияВыполнения(** массив заданий, таймаут **)** — ожидание нескольких заданий.

Метод задаётся строкой «ИмяОбщегоМодуля.ИмяПроцедурыИлиФункции». Параметры — массив; порядок элементов соответствует порядку параметров метода (индекс 0 — первый параметр). Значения должны поддерживать сериализацию. Ключ обеспечивает уникальность запуска.

Пример запуска (серверный общий модуль ОбщийМодульФоновые, метод ВыгрузитьДанные(ОписаниеДанных, ПутьКФайлу)):

```bsl
ПараметрыВызова = Новый Массив;
ПараметрыВызова.Добавить(ВыгружаемыеДанные);
ПараметрыВызова.Добавить("c:\temp\data.dat");
ЗаданиеВыгрузки = ФоновыеЗадания.Выполнить("ОбщийМодульФоновые.ВыгрузитьДанные", ПараметрыВызова, "ВыгрузкаДанных", "Выгрузка данных в файл на диске");
```

Состояние (Свойство Состояние) фиксируется на момент создания объекта ФоновоеЗадание и не обновляется; для актуальной проверки нужно заново получать описание задания (НайтиПоУникальномуИдентификатору или ОжидатьЗавершенияВыполнения). Отмена — ФоновоеЗадание.Отменить(); после отмены необходимо выполнить «сборку мусора» (удаление временных файлов и т.п.).

Пример ожидания завершения (таймаут 10 с):

```bsl
ЗаданиеВыгрузки = ФоновыеЗадания.НайтиПоУникальномуИдентификатору(Идентификатор);
Пока Истина Цикл
    ЗаданиеВыгрузки = ЗаданиеВыгрузки.ОжидатьЗавершенияВыполнения(10);
    Если ЗаданиеВыгрузки.Состояние = СостояниеФоновогоЗадания.Активно Тогда
        Продолжить;
    Иначе
        Прервать;
    КонецЕсли;
КонецЦикла;
```

Сообщения пользователю из задания (СообщениеПользователю) — метод ФоновоеЗадание.ПолучитьСообщенияПользователю(); доступно во время работы и после завершения.

### 19.2.3. Расширения

При старте фонового задания в его сеансе подключаются расширения родительского сеанса. Запуск без расширений родителя, но со всеми расширениями, зарегистрированными для данной информационной базы и набора разделителей: **РасширенияКонфигурации.ВыполнитьФоновоеЗаданиеБезРасширений()**.

## 19.3. Регламентные задания

Используются для периодических или однократных действий по расписанию. Хранятся в информационной базе, создаются на основе метаданных (вызываемый метод, наименование, ключ, использование и т.д.). Дополнительно при создании можно указать расписание (если не задано в метаданных), параметры метода, пользователя от имени которого выполняется задание.

**Права:** создание и управление — только пользователи с административными правами. В файловом варианте создавать и редактировать регламентные задания можно без запуска планировщика.

**Метод** — процедура или функция неглобального общего модуля, вызываемая на сервере. Параметры — как у метода; если метод — функция, возвращаемое значение игнорируется.

**Предопределённые регламентные задания** задаются в конфигурации, не создаются и не удаляются явно. При обновлении конфигурации с установленным признаком предопределённости задание создаётся в базе; при снятии — удаляется. Начальные свойства (в т.ч. расписание) — из метаданных; в дальнейшем меняются через встроенный язык. Изменение свойств в метаданных уже созданного предопределённого задания не меняет его в базе. Параметров у предопределённых заданий нет.

**Расписание** (РасписаниеРегламентныхЗаданий): дата/время начала и окончания, период, дни недели и месяцы и др. Проверка выполнения на дату — метод ТребуетсяВыполнение(). Выполнение — от имени указанного пользователя; если не указан — по набору ролей из свойства конфигурации ОсновныеРоли; при пустом списке ролей — без ограничения прав.

**Выполнение:** планировщик создаёт фоновое задание; если задание уже выполняется, повторный запуск по расписанию не выполняется.

**Перезапуск при аварии:** при аварийном завершении задания или рабочего/клиентского процесса задание может перезапускаться. Задаются количество повторов и интервал между ними (свойства метаданных/задания). После исчерпания повторов новые попытки — только по следующему срабатыванию расписания; при наступлении следующего времени по расписанию процесс перезапуска начинается заново. При перезапуске выполнение начинается с начала метода, а не с места аварии.

**Примеры расписаний (значения параметров):**

| Расписание | Параметры |
|------------|-----------|
| Каждый час, один день | ПериодПовтораДней = 0, ПериодПовтораВТечениеДня = 3600 |
| Каждый день один раз | ПериодПовтораДней = 1, ПериодПовтораВТечениеДня = 0 |
| Через день один раз в день | ПериодПовтораДней = 2 |
| Каждый час с 01:00 до 07:00 | ПериодПовтораДней = 1, ПериодПовтораВТечениеДня = 3600, ВремяНачала = 01:00, ВремяКонца = 07:00 |
| Суббота и воскресенье в 09:00 | ПериодПовтораДней = 1, ДниНедели = 6,7, ВремяНачала = 09:00 |
| Каждый день, неделя через неделю | ПериодПовтораДней = 1, ПериодНедель = 2 |
| Один раз в 01:00 | ВремяНачала = 01:00 |
| Последнее число месяца в 09:00 | ПериодПовтораДней = 1, ДеньВМесяце = -1, ВремяНачала = 09:00 |
| 5-е число месяца в 09:00 | ПериодПовтораДней = 1, ДеньВМесяце = 5, ВремяНачала = 09:00 |
| Вторая среда месяца в 09:00 | ПериодПовтораДней = 1, ДеньНеделиВМесяце = 2, ДниНедели = 3, ВремяНачала = 09:00 |

## 19.4. Особенности выполнения в разных окружениях

### 19.4.1. Файловый вариант

Фоновые и регламентные задания выполняются клиентскими приложениями или расширением веб-сервера. Фоновые задания выполняются на том клиенте, который их запустил; одновременно на одном клиенте выполняется не более одного фонового задания. Для каждой базы, доступной через веб-сервер, выполнение фоновых и регламентных заданий последовательное.

- **Самостоятельные фоновые задания** (запуск из кода или отчёты): информация доступна только в том клиенте/расширении, где выполнялись; уникальность ключа — в рамках клиента; после завершения клиента история не сохраняется.
- **Регламентные фоновые задания:** информация доступна на всех клиентах и сохраняется между запусками.
- **Исполнитель регламентных заданий:** одно клиентское приложение (первое по порядку запуска без запрета). Управление: параметр командной строки /AllowExecuteScheduledJobs (толстый/тонкий клиент); атрибут allowexecutescheduledjobs элемента point в default.vrd для веб-публикации. При явном указании исполнения регламентных заданий это приложение выполняет их независимо от остальных. Регламентные задания выполняются расширением веб-сервера, пока оно обслуживает хотя бы один сеанс.
- Если в момент по расписанию не работает ни один клиент — задание не выполнится.
- Регламентные задания обрабатываются раз в 60 секунд.
- Планировщик фоновых заданий инициализируется через 60 с после старта клиента или после первой попытки запуска фонового задания. До инициализации в синхронном режиме выполняются: поиск в динамическом списке, построение отчётов, поиск для ввода по строке, глобальный поиск.
- Планировщик регламентных заданий — через 60 с после планировщика фоновых; прикладные регламентные задания начинают выполняться не ранее чем через 120 с после старта клиента.
- Фоновые задания, не завершившиеся за 30 с после завершения клиента, принудительно завершаются (в журнале регистрации: Фоновое задание. Принудительное завершение (_$Job$_.Terminate)).
- В сеансе, полученном через automation-сервер «1С:Предприятие» или COM-соединение, регламентные задания не поддерживаются; созданные фоновые задания попадают в очередь, но не запускаются.

### 19.4.2. Клиент-серверный вариант

Планировщик заданий расположен в менеджере кластера. Он выбирает наименее загруженный рабочий процесс для каждого фонового задания из очереди; рабочий процесс выполняет задание и сообщает результат планировщику.

**Блокировка регламентных заданий** при: явной блокировке регламентных заданий на базу (консоль кластера); блокировке начала сеансов (консоль кластера); вызове УстановитьМонопольныйРежим(Истина); обновлении конфигурации и в ряде других случаев.

### 19.4.3. Мобильное приложение

Фоновые задания выполняются мобильным приложением последовательно (одновременно не более одного). Регламентные задания на мобильном устройстве не поддерживаются. Планировщик фоновых заданий инициализируется после первой попытки запуска фонового задания; до этого в синхронном режиме выполняются поиск в динамическом списке, построение отчётов, поиск для ввода по строке, глобальный поиск. Принудительное завершение заданий, не завершившихся за 30 с после завершения приложения.

## 19.5. Создание метаданных регламентного задания

Объект конфигурации создаётся в ветке Общие → Регламентные задания (команда Добавить).

**Свойства метаданных:** Ключ (строка), Расписание, Использование, Предопределённое, Количество повторов при аварийном завершении, Интервал повтора при аварийном завершении.

**Расписание** (форма): Общее — дата начала/завершения, режим повтора; Дневное — время начала/окончания запуска, время принудительного завершения, период и пауза повтора, продолжительность выполнения; Недельное — дни недели, интервал в неделях (при повторе каждые N недель учитывается Первый день недели из региональных настроек базы); Месячное — месяцы, день месяца или день недели в месяце (с начала/конца).

При записи регламентного задания проверяется возможность запуска по расписанию. Пример: при сохранении 12.09.2021 12:33 расписание «каждый день; с 1:00 один раз в день» — задание выполнится сразу после записи и далее ежедневно в 1:00. Расписание «каждый день; с 1:00 по 1:01 один раз в день» — при записи в 12:33 не выполнится (время вне интервала), далее — каждый день в 1:00.

**Пример создания регламентного задания с расписанием «каждый день один раз»:**

```bsl
Расписание = Новый РасписаниеРегламентногоЗадания;
Расписание.ПериодПовтораДней = 1;
Расписание.ПериодПовтораВТечениеДня = 0;
Задание.Расписание = Расписание;
Задание.Записать();
```

**Пример запуска фонового задания (обновление индекса полнотекстового поиска):**

```bsl
ФоновыеЗадания.Выполнить("ОбновлениеИндексаПолнотекстовогоПоиска");
```
