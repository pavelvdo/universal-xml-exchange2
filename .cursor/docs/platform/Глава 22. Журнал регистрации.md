# Глава 22. Журнал регистрации

> **Agent ref:** Журнал регистрации: события, уровни детализации, запись (ЗаписьЖурналаРегистрации), управление (ПолучитьИспользованиеЖурналаРегистрации, УстановитьИспользованиеЖурналаРегистрации; ПолучитьИспользованиеСобытияЖурналаРегистрации, УстановитьИспользованиеСобытияЖурналаРегистрации), получение записей (ВыгрузитьЖурналРегистрации), фильтр по полю Данные, события Доступ и Отказ в доступе, аудит прав доступа, дополнительные методы (СкопироватьЖурналРегистрации, ПолучитьПериодРазделенияХраненияДанныхЖурналаРегистрации и др.).
> **Ключевые разделы:** 22.1 Общая информация, 22.2 Управление детализацией, 22.3 Запись событий, 22.4 Управление регистрацией (22.4.2 Доступ, 22.4.3 Отказ в доступе), 22.5 События аудита прав доступа, 22.6 Получение записей, 22.7 Дополнительные методы.
> ---

## 22.1. Общая информация

Журнал регистрации предназначен для выяснения, какие события происходили в определённый момент или какие действия выполнял пользователь. В нём фиксируются различные события; администратор получает историю работы пользователей с системой.

Журнал регистрации **не хранится в базе данных** и не сохраняется при загрузке/выгрузке информационной базы.

Платформа фиксирует в журнале основные действия: модификация данных, доступ и отказ в доступе к данным, регламентные операции, подключение и отключение от системы. Доступна программная работа с журналом (в главе описываются средства программной работы).

## 22.2. Управление детализацией

Уровень детализации сообщений задаётся методами глобального контекста **ПолучитьИспользованиеЖурналаРегистрации()** и **УстановитьИспользованиеЖурналаРегистрации()**.

Пример: включить регистрацию всех уровней (Ошибка, Информация, Предупреждение, Примечание):

```bsl
Уровни = Новый Массив;
Уровни.Добавить(УровеньЖурналаРегистрации.Ошибка);
Уровни.Добавить(УровеньЖурналаРегистрации.Информация);
Уровни.Добавить(УровеньЖурналаРегистрации.Предупреждение);
Уровни.Добавить(УровеньЖурналаРегистрации.Примечание);
УстановитьИспользованиеЖурналаРегистрации(Уровни);
```

## 22.3. Запись событий

Запись событий в журнал выполняется методом глобального контекста **ЗаписьЖурналаРегистрации()**. Системные события этим методом записать нельзя. Имена пользовательских событий могут содержать точку (образуют группы).

Включение/выключение регистрации конкретного события: **ПолучитьИспользованиеСобытияЖурналаРегистрации()**, **УстановитьИспользованиеСобытияЖурналаРегистрации()**, объект **ИспользованиеСобытияЖурналаРегистрации**. Пример отключения регистрации события «Ошибка выполнения» (_$PerformError$_):

```bsl
ИспользованиеСобытия.Использование = Ложь;
УстановитьИспользованиеСобытияЖурналаРегистрации("_$PerformError$_", ИспользованиеСобытия);
```

**Примечание.** События транзакций (_$Transaction$_.Begin, _$Transaction$_.Commit, _$Transaction$_.Rollback) отключить через УстановитьИспользованиеСобытияЖурналаРегистрации() нельзя.

## 22.4. Управление регистрацией событий

### 22.4.1. Общая информация

Два события имеют расширенную настройку: **Доступ** (_$Access$_.Access) и **Отказ в доступе** (_$Access$_.AccessDenied). Особенности настройки:

- Менять настройки регистрации только при включённом монопольном режиме.
- Изменение, выполненное в текущем сеансе, гарантированно применяется после перезапуска всех активных сеансов данной информационной базы.

### 22.4.2. Настройка параметров события «Доступ»

Событие **Доступ** регистрирует факты доступа пользователей к данным. Настраиваются: необходимость регистрации; перечень объектов метаданных; поля доступа (чтение которых регистрируется); поля регистрации (значения которых попадают в журнал). Фиксация происходит при любом доступе к данным из сеанса, в том числе из кода на сервере.

Зависимость результата от настроек:

| Поля доступа | Поля регистрации | Результат |
|--------------|------------------|-----------|
| Не задано    | —                | Событие не регистрируется |
| Заданы       | Не заданы        | Событие регистрируется без детализации |
| Заданы       | Заданы           | Событие регистрируется с полями регистрации |

Событие формируется только при успешном чтении данных.

Пример настройки для справочника и регистра сведений:

```bsl
НастройкаСправочника = Новый ОписаниеИспользованияСобытияДоступЖурналаРегистрации();
НастройкаСправочника.Объект = "Справочник.ФизическиеЛица";
НастройкаСправочника.ПоляДоступа.Добавить("ПаспортныеДанные");
НастройкаСправочника.ПоляДоступа.Добавить("Дети.СвидетельствоОРождении");
НастройкаСправочника.ПоляРегистрации.Добавить("Дети.ИмяРебенка");
АльтернативыПолей = Новый Массив();
АльтернативыПолей.Добавить("Фамилия");
АльтернативыПолей.Добавить("Имя");
НастройкаСправочника.ПоляРегистрации.Добавить(АльтернативыПолей);
НастройкаОбъектовМетаданных = Новый Массив();
НастройкаОбъектовМетаданных.Добавить(НастройкаСправочника);

НастройкаРегистраСведений = Новый ОписаниеИспользованияСобытияДоступЖурналаРегистрации();
НастройкаРегистраСведений.Объект = "РегистрСведений.ОкладыСотрудников";
НастройкаРегистраСведений.ПоляДоступа.Добавить("Оклад");
НастройкаРегистраСведений.ПоляРегистрации.Добавить("Сотрудник");
НастройкаОбъектовМетаданных.Добавить(НастройкаРегистраСведений);
```

В этом примере: регистрация доступа к справочнику ФизическиеЛица при участии в запросе полей ПаспортныеДанные или СвидетельствоОРождении (ТЧ Дети); в журнал записываются ПаспортныеДанные, ИмяРебенка (Дети), а также одно из полей Фамилия/Имя (по наличию в запросе, при обоих — с меньшим индексом). Для регистра сведений ОкладыСотрудников при обращении к полю Оклад в событие попадают данные поля Сотрудник; если у справочника ФизическиеЛица в полях регистрации не задана Ссылка, детализация по ссылке в поле Данные журнала не заполняется.

### 22.4.3. Настройка параметров события «Отказ в доступе»

Событие **Отказ в доступе** регистрирует факты отказа в доступе. Настраиваются: необходимость регистрации; перечень объектов метаданных; поля регистрации при отказе.

| Использование | Поля регистрации | Результат |
|---------------|------------------|-----------|
| Не задано     | —                | Событие не регистрируется |
| Задано        | Не заданы        | Событие без детализации (поле Данные пустое) |
| Задано        | Заданы           | Событие с полями регистрации в поле Данные |

Событие формируется: при проверке доступа к объекту целиком (нарушение прав при обращении к методам/свойствам, отказы при проверке прав на конфигурацию) — в Данные помещается структура с свойством **Право**; при проверке ограничений доступа к данным (нарушение RLS) — в Данные структура с **Действие** и **Данные** (поля регистрации при настройке). При отказе в операции Чтение запись полей регистрации выполняется по тем же правилам, что и для события Доступ; для Изменение/Удаление/Добавление событие регистрируется, но запись данных не выполняется.

Пример настройки:

```bsl
НастройкаСправочника = Новый ОписаниеИспользованияСобытияОтказВДоступеЖурналаРегистрации();
НастройкаСправочника.Объект = "Справочник.ФизическиеЛица";
НастройкаСправочника.ПоляРегистрации.Добавить("ПаспортныеДанные");
НастройкаСправочника.ПоляРегистрации.Добавить("Дети.ИмяРебенка");
АльтернативыПолей = Новый Массив();
АльтернативыПолей.Добавить("Фамилия");
АльтернативыПолей.Добавить("Имя");
НастройкаСправочника.ПоляРегистрации.Добавить(АльтернативыПолей);
```

## 22.5. События аудита прав доступа

Доступно только для лицензии КОРП. События фиксируют, кто изменял роли, права доступа пользователей, профили безопасности, административные параметры. Включение: в клиент-серверном варианте — флажок «Разрешать запись событий аудита прав доступа» в настройках кластера (свойство АдминистрированиеКластер.РазрешитьЗаписьСобытийАудитаПравДоступа); в файловом — параметр EnableAccessRightAuditEventsWrite в conf.cfg. Подготовка данных для этих событий ресурсоёмка, включать их стоит только при обоснованной необходимости.

События (имена в виде XML выгрузки конфигурации):

- **\_$InfoBase$_.RoleUpdate** (изменение роли): при сохранении конфигурации изменён один или несколько объектов Роль. Данные — структура с ключами Права (ДоступВключен, ДоступВыключен), Ограничения (Добавлены, Изменены, Удалены), ШаблоныОграничений (Добавлены, Изменены, Удалены).
- **\_$User$_.PermissionChange** (изменение прав пользователей): при изменении ролей изменился набор доступных объектов для части пользователей. Данные: Пользователи (массив имён), РазделениеДанных, ПрофильБезопасности.
- **\_$InfoBase$_.SecurityProfileChange** (изменение профиля безопасности): изменены настройки профиля безопасности ИБ. Данные: АдминистраторКластера, ПрофильБезопасности (структура по свойствам АдминистрированиеПрофильБезопасности). Запись отложенная — системное фоновое задание InfoBaseSecurityProfileChange.
- **\_$InfoBase$_.AdministrationParametersChange** (изменение административных параметров ИБ): данные — ФиксированнаяСтруктура по свойствам АдминистрированиеИнформационнаяБаза. Запись отложенная — задание InfoBaseAdministrationParametersChange.

## 22.6. Получение записей журнала регистрации

Метод глобального контекста **ВыгрузитьЖурналРегистрации()** — программное получение записей. Можно выгружать журнал текущей ИБ или другой (параметр ИмяВходногоФайла); чтение другой ИБ рекомендуется только при локальной доступности файлов журнала.

Ограничения и рекомендации:

1. **Клиент-сервер:** ограничение количества сеансов, одновременно получающих данные (не более 4 на одну ИБ, при разделении данных — 1 сеанс на область). В файловом варианте ограничений на объём нет.
2. Хранение по умолчанию — разбиение по неделям. Рекомендуется выбирать интервал так, чтобы размер одного .lgp не превышал 100 МБ.
3. **Индексация** ускоряет отбор при использовании отбора по индексируемым полям: Дата, Важность, Пользователь, Компьютер, ИмяПриложения, Событие, Данные (только ссылочный тип), Метаданные, РазделениеДанныхСеанса.

Колонка **ПредставлениеДанных** заполняется только если у пользователя есть право на просмотр представления или право Администрирование. Иначе (в конфигураторе без Администрирование; в 1С:Предприятие без привилегированного режима при недоступности представления или отсутствии прав Просмотр/RLS) — пустая.

**Фильтр** задаётся параметром **Фильтр** метода ВыгрузитьЖурналРегистрации(): одна структура — условия по «И»; массив структур — по «ИЛИ». Отбор по полю **Данные**: допустимы Строка, Число, Дата, Ссылка, Структура (для событий Доступ, Отказ в доступе, Аутентификация и др.), массив структур. Значение элемента фильтра: конкретное значение, структура или массив. Правила сопоставления: конкретное значение — сравнение; массив — запись подходит, если значение из записи совпадает с одним из элементов массива; структура — подходит, если в данных записи есть все ключи фильтра и значения совпадают. Для вложенных структур/таблиц поиск соответствия структуре фильтра выполняется только на том же уровне вложенности. При дублировании имён колонок в таблице в поле Данные они объединяются в одну; строки дублируются так, чтобы в результирующей колонке были все уникальные значения.

**ПолучитьЗначенияОтбораЖурналаРегистрации()** — доступные значения отбора для параметров Пользователь, Компьютер, ИмяПриложения, Событие, Метаданные, РабочийСервер, ОсновнойIPПорт, ВспомогательныйIPПорт. **ПредставлениеСобытияЖурналаРегистрации()** — представление события на языке интерфейса. Для поля Метаданные запись соответствует отбору, если хотя бы один элемент массива отбора равен хотя бы одному элементу массива записи.

## 22.7. Дополнительные методы

- **СкопироватьЖурналРегистрации()** — копирование фрагмента журнала с заданным фильтром.
- **ПолучитьПериодРазделенияХраненияДанныхЖурналаРегистрации()**, **УстановитьПериодРазделенияХраненияДанныхЖурналаРегистрации()** — период разделения (перечисление ПериодРазделенияХраненияДанныхЖурналаРегистрации). Установка поддерживается только для формата .lgf.
- **ПолучитьПериодЖурналаРегистрации()** — возвращает тип Диапазон с датами хранения.
- **СократитьЖурналРегистрации()** — сокращение журнала; поддерживается для обоих форматов журнала.
