# Профиль `explore-bug` (дефект)

Канва для брифа и финального блока `## Постановка ЗНИ`. Результат — **ответ в чате**, не файл-отчёт. Полный отчёт каждого `Task` сохраняется в `temp/reports/` (см. `preserve-subagent-reports.mdc`).

## Подсказки для брифа (слоты **Симптом** / **Вопрос**; в промптах — `user-goal`; форма — `brief-card.md`)

| Типичная цель заказчика | Критерий «решено» |
|-------------------------|-------------------|
| Убедиться, что не баг / не регресс | Знаю: чинить или не чинить |
| Починить verify / поведение | Воспроизведён сценарий и есть рецепт |
| Ускорить (PERF) | Знаю: оптимизировать или оставить как есть |

## Маркеры verified / hypothesis

- **`[verified]`** — есть код:строка, трасса, лог или воспроизведённый сценарий.
- **`[hypothesis: <план верификации>]`** — план обязателен в той же строке.
- В финальном блоке `## Постановка ЗНИ` (поле «Корневая причина») — обязателен один из маркеров. Hypothesis без плана **запрещена** (`verified-cause-gate.mdc`).

## Типичные предметы исследования

- Трасса / стек / PERF — `Task` → `onec-trace-analyst`; полный отчёт → `temp/reports/trace-analysis-YYYY-MM-DD-<slug>.md`.
- Симптом и воспроизведение.
- Цепочка вызовов до точки отказа.
- Точка отказа (корневая причина).
- Регрессы смежных сценариев.

## Trace-analyst: обязательные выводы

При наличии трассы отчёт обязан ответить (в чате — превью, в файле — полностью):

1. Совпадает ли наблюдение пользователя с цепочкой в трассе?
2. Для «двух вариантов / двух снимков байтов»: отделить **подготовку verify** от **клиентского retry** от **фильтра расширения после подготовки**.
3. Явно: **вызван ли** перехват `cfe` (по имени процедуры в трассе); **сколько адресов** остаётся после него (если в трассе не видно — сказать «не видно в coverage», не «норма»).

Промпт `Task` должен требовать в отчёт секцию `## Для заказчика` по шаблону ниже + `## Recommended next steps` + `## Anomalies / concerns` (если есть).

## Шаблон `## Для заказчика` (в полном отчёте `Task`)

Каждый пункт — отдельная строка или короткий абзац:

1. **Что наблюдаешь у себя** — дословно **Вопрос** / `user-goal` из брифа (симптом пользователя, **не** симптом трассы/отладчика).
2. **Почему так устроено** — one-liner про механизм найденной находки.
3. **Связано ли это с симптомом** — одно из: `объясняет полностью` | `объясняет частично, есть ещё причина` | `не объясняет, симптом из другой ветки`.
4. **Что чинить и где** — модуль/расширение + конкретное поле/процедура. Если п.3 = «не объясняет» — здесь указывается **другая** причина (`[verified]` или `[hypothesis: …]`), а не «допустимое поведение, дальше сами».

## Пример маршрута (с трассой)

1. Архивный контекст (если ссылок нет — пропустить).
2. `Task` → `onec-trace-analyst` → превью в чате + Symptom Relevance Check + «Итог / Вердикт / Дальше».
3. (Опционально) `Task` → `onec-code-explorer` для подтверждения гипотезы из трассы.
4. Финал: блок `## Постановка ЗНИ` в чате.

Лимит шагов — ≤ 3 `Task` (architect — отдельно при срабатывании gate).

## Symptom Lock (HALT для «Вердикта»)

В bug-профиле **запрещён** финальный вердикт «штатно» / «норма» / «допустимое поведение», пока якорь **Вопрос** / `user-goal` не снят или явно не пересогласован в чате. Если найденное **не объясняет** симптом — обязан назвать **другую** причину или явный «не нашли, нужен шаг X».

## HALT для слота «Дальше»

После шага с отчётом в слоте **Дальше** (синтез «Итог / Вердикт / Дальше») **запрещено**:

- абстрактные «собрать отчёт», «продолжить расследование», «оптимизация»;
- бинарные меню («хватит / копаем дальше»);
- описания шагов оркестратора («запустить explorer», «делегировать архитектору»).

**Разрешено** только:

- конкретные **гипотезы устранения симптома** со ссылкой на модули/поля;
- разные **сценарии воспроизведения**, если симптом не локализован;
- «симптом объяснён, чинить нечего» vs «симптом не объяснён, гипотеза N» — формулировать именно так.

## Финал bug — блок `## Постановка ЗНИ`

Каркас, правила сборки и композиция финального сообщения — **только** в [`../templates/handoff-block.md`](../templates/handoff-block.md) (SSOT, здесь не дублируется).

Это **последний блок в чате**, не файл-отчёт. `/opsx:new` подхватывает его прямо из истории чата. Запрещены однострочные пункты-телеграммы для Симптома, Причины и Что менять; «Architect / verify» — машиночитаемо (`required | not-required | report: <путь>`), не «да/нет» прозой.

## Handoff после ответа

- Новая ЗНИ: `/opsx:new <name>` — берёт блок `## Постановка ЗНИ` из чата (приоритет 1) или handoff-файл (приоритет 2).
- Существующая ЗНИ: `/opsx:extend <change> --from-report temp/reports/<тип>-YYYY-MM-DD-<slug>.md` (отчёт `Task`) или handoff-файл.
- Handoff-файл (`temp/explore-handoff-YYYY-MM-DD-<slug>.md`) создаётся **только** по словесной просьбе пользователя.
