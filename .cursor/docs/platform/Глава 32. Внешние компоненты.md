# Глава 32. Внешние компоненты

> **Agent ref:** Внешние компоненты, Native API, COM, подключение (изолированно / не изолированно), УстановитьВнешнююКомпоненту, ПодключитьВнешнююКомпоненту, хост-процесс, кеширование, асинхронные методы (НачатьПодключениеВнешнююКомпоненты, обещания). Клиент (тонкий/толстый), веб-клиент, сервер, мобильное приложение.
> **Ключевые разделы:** 32.1 Общая информация, 32.2–32.5 Особенности по средам, 32.6 Асинхронная работа, 32.7 Примеры.

## 32.1. Общая информация

Система «1С:Предприятие» расширяется внешними компонентами. Используются две технологии:
- **Native API** — кроссплатформенно (Linux, macOS, Windows), веб-клиент, мобильное приложение (Android, iOS, Windows).
- **COM** — только Windows.

Компоненты для веб-клиента или мобильного приложения упаковываются в zip-архив особой структуры; такой архив можно использовать и на сервере, и в других клиентах.

### Типы подключения

Параметр **ТипПодключения** метода `ПодключитьВнешнююКомпоненту()` (тип `ТипПодключенияВнешнейКомпоненты`):

- **Не изолированно** — компонента загружается в тот же процесс, что и код встроенного языка. Аварийное завершение компоненты приводит к завершению процесса и всех его сеансов.
- **Изолированно** — компонента загружается в отдельный хост-процесс. При аварии компоненты клиентский сеанс не страдает; завершается только хост-процесс.

Если хост-процесс создать не удалось (например, нет прав), компонента подключается не изолированно. Если компонента не поддерживает не изолированное подключение — подключение не выполняется, формируется исключение. Подключение по одному параметру (только идентификатор) всегда не изолированное.

Успешность подключения зависит от поддерживаемых типов компоненты и режима совместимости конфигурации:
- Режим совместимости 8.3.20 и ранее — всегда не изолированно.
- Иначе: на клиенте — не изолированно, на сервере — изолированно.
- Если указан тип подключения и компонента его не поддерживает — исключение.

### Обработка аварийного завершения

Для изолированных компонент в конфигурации нужны обработчики:
- `ОбработкаОтключенияВнешнейКомпонентыПриОшибке()` — модуль приложения;
- `ОтключениеВнешнейКомпонентыПриОшибке()` — модуль формы.

Управление передаётся после аварии и после завершения выполнения встроенного языка. В обработчике можно выполнить повторное подключение и инициализацию. На сервере авария изолированной компоненты не обрабатывается — только на клиенте через эти обработчики.

### Кеширование и манифест

Компоненты из конфигурации или ИБ сохраняются и при последующем подключении (`УстановитьВнешнююКомпоненту()`, `ПодключитьВнешнююКомпоненту()`, `ПодключитьВнешнююКомпонентуАсинх()`) используются без повторного получения. На сервере кеш действует в рамках рабочего процесса. Для компонент из ИБ отслеживается изменение данных при подключении (не чаще 1 раза в секунду); при изменении кеш очищается. Для компонент в zip параметры задаются файлом-манифестом.

### Схема работы (Native API)

1. Установка на компьютер пользователя — `УстановитьВнешнююКомпоненту()` (обязательно для тонкого и веб-клиента).
2. Подключение — `ПодключитьВнешнююКомпоненту()`; неустановленная компонента даёт ошибку.
3. Создание объекта — конструктор `Новый()` со строковым идентификатором вида `Addin.<Имя>.ИмяОбъекта`, где *Имя* — второй параметр `ПодключитьВнешнююКомпоненту()`, *ИмяОбъекта* — имя объекта в компоненте. Пример: `ПодключитьВнешнююКомпоненту(…, "MyName");` → `Новый("Addin.MyName.Таймер")`.

Не объединять установку и подключение в один код: установка — однократное событие, повторная установка вызывает интерактивное исключение.

COM: регистрация и загрузка через `ЗагрузитьВнешнююКомпоненту()` / `ПодключитьВнешнююКомпоненту()` выполняются «для пользователя»; при неудаче выполняется попытка регистрации «для компьютера».

## 32.2. Тонкий и толстый клиент

Допустимы компоненты на COM и Native API (файлы на диске или zip). Обновление — повторный вызов `УстановитьВнешнююКомпоненту()`.

Тонкий клиент: установка обязательна; компоненты попадают в `%APPDATA%\1C\1Cv8\ExtCompT`. Этот каталог не очищается по `/ClearCache`.

---

## 32.3. Веб-клиент

Допустимы компоненты Native API или COM (только Windows), упакованные в zip особого вида. Установка запрашивает разрешение пользователя. Тип подключения в веб-клиенте игнорируется; фактически:
- Google Chrome, Mozilla Firefox — изолированно;
- Internet Explorer, Safari — не изолированно.

Каждая компонента устанавливается как расширение браузера в профиль пользователя (Chrome, Firefox) или в системный каталог (IE: `%ALLUSERSPROFILE%\Application Data\1C\1CEWebExt` и т.п.). Удаление — через механизмы расширений/дополнений браузера или панель управления (IE).

Подключение внешних компонент **из файла на диске** в веб-клиенте не поддерживается (безопасность). COM доступен только в Internet Explorer на Windows; COMSafeArray и вызовы методов с этим типом не поддерживаются.

Обновление: удалить компоненту и установить заново через `УстановитьВнешнююКомпоненту()` либо изменить имя объекта в коде и манифесте и установить заново.

Рекомендации: установка и расширений платформы должна быть интерактивной; пользователь должен явно согласиться. Желательно встраивать установку в сценарий (например, «Отправить отчёт» → при отсутствии компоненты — диалог с установкой и продолжением). В приложении должны быть средства установки компонент и расширений в любой момент.

---

## 32.4. Сервер

Допустимы только компоненты на **Native API** (файлы или zip). Метод `УстановитьВнешнююКомпоненту()` на сервере недоступен; используется только `ПодключитьВнешнююКомпоненту()`.

Подключать компоненту непосредственно перед использованием. Компонента должна поддерживать все нужные архитектуры и ОС: серверный код может выполняться на разных рабочих процессах и компьютерах. Файл компоненты кешируется рабочим процессом до его перезапуска. При изолированном подключении авария компоненты завершает серверный вызов; на клиенте срабатывает обработчик ошибки.

## 32.6. Асинхронная работа

### 32.6.1. Обратный вызов

Подключение — асинхронный метод `НачатьПодключениеВнешнейКомпоненты()`. Для свойств и методов используются асинхронные методы `НачатьУстановку<ИмяСвойства>()`, `НачатьПолучение<ИмяСвойства>()`, `НачатьВызов<ИмяМетода>()`. У первого параметра `НачатьВызов…()` — описание оповещения; остальные параметры совпадают с параметрами метода компоненты. Компоненту перерабатывать не требуется, но нужна перекомпиляция для поддержки асинхронной работы.

В Chrome и Firefox допустимо только асинхронное подключение и вызовы; для остальных браузеров режим задаётся свойством конфигурации «Режим использования синхронных вызовов расширений платформы и внешних компонент».

### 32.6.2. Обещания

- Установка — `УстановитьВнешнююКомпонентуАсинх()`.
- Подключение — `ПодключитьВнешнююКомпонентуАсинх()`.
- Создание объекта — `СоздатьОбъектВнешнейКомпонентыАсинх()`; имя типа формируется так же, как для `Новый()`.
- Свойства — `Получить<ИмяСвойства>Асинх()`, `Установить<ИмяСвойства>Асинх()`.
- Методы — `<ИмяМетода>Асинх()`.

Интерфейс дополняется платформой автоматически; перекомпиляция компоненты для асинхронной работы необходима. Результат ожидания обещания от `<ИмяМетода>Асинх()` имеет тип `РезультатАсинхВызоваВнешнейКомпоненты`: свойство `Значение` — результат, `Параметры` — параметры вызова (в т.ч. изменённые). Ограничения по браузерам — как в 32.6.1.

## 32.7. Примеры

### 32.7.1. Native API

- **Установка на клиента** — вызов `УстановитьВнешнююКомпоненту()` перед использованием в тонком/веб-клиенте.
- **Из файла на диске** — файл в PATH; `ПодключитьВнешнююКомпоненту()` с путём; создание объекта через `Новый("Addin.<Имя>.ИмяОбъекта")`.
- **Из макета** — общий макет (например, «ДрайверСканераШтрихКодов», тип ДвоичныеДанные) с zip-архивом для всех ОС/браузеров/архитектур; получение ссылки на макет и вызов `ПодключитьВнешнююКомпоненту(Ссылка, "MyName", ТипВнешнейКомпоненты.Native)`; объект — `Новый("AddIn.MyName.ComponentExtension")`.
- **Из ИБ** — справочник с реквизитами типа ХранилищеЗначения; выбор варианта по `СистемнаяИнформация.ТипПлатформы` (Windows_x86, Windows_x86_64, Linux_x86, Linux_x86_64), получение навигационной ссылки на нужное хранилище, затем `ПодключитьВнешнююКомпоненту(Ссылка, "MyName", ТипВнешнейКомпоненты.Native)` и `Новый("AddIn.MyName.ComponentExtension")`.
- **Из zip-архива** — общий макет с zip особой структуры; подключение по ссылке на макет.
- **Изолированно** — макет с компонентой, поддерживающей изолированное подключение; вызов `ПодключитьВнешнююКомпоненту(…, ТипПодключенияВнешнейКомпоненты.Изолированно)`; при неподдержке изоляции — исключение.

На сервере `ПодключитьВнешнююКомпоненту()` нужно вызывать каждый раз перед созданием экземпляра (рабочий процесс и платформа могут быть разными).

### 32.7.2. COM

Только Windows. `ЗагрузитьВнешнююКомпоненту()` — для совместимости с предыдущими версиями. Подключение — `ПодключитьВнешнююКомпоненту()`; создание объекта — `Новый("AddIn.ComponentExtension")` (имя по правилу Addin + имя из подключения + имя объекта). В примерах Native API можно подключать и COM-компоненты при работе под Windows.
