# Design Challenge — universal-xml-exchange2

## Адверсариальная установка

Challenge выполнен независимо: прочитаны `proposal.md` (`## Why`, `## What Changes`), `design.md`,
все четыре `specs/**/spec.md`, `tasks.md`. Решение рассмотрено как чужое. Как источники истины
использованы только постановка change, спеки, выгрузка кода в `src/ЗУП/cfe/рг_УниверсальныйОбменXML/`
и ADR-0002/0003. Отчёты `reports/architecture-*.md` текущей сессии **не** использованы как

## Three-Question Challenge

### Q1 — Problem-Solution Fit

- **Why говорит:** v1 — pull-модель без контроля жизненного цикла сеанса: правила по имени макета на
  источнике, нет подтверждения доставки, «зависшие» выгрузки не отслеживались. Нужны: статусы профиля,
  выгрузка только из «Новое», `ConfirmDataReceipt`, правила в хранилище профиля, журналирование переходов.
- **Design адресует:**
  - Правила по имени макета → реквизит-хранилище `ПравилаОбменаДанными` + проверка при записи (D6, S1.6,
    spec exchange-settings «Правила обмена в хранилище профиля»). **Покрыто.**
  - Нет статусов → перечисление `рг_СтатусыОбмена`, реквизит `Статус`, gate `GetData` по «Новое» с
    атомарным переводом в «Выполняется» (D1, D2, S2.8). **Покрыто.**
  - Нет подтверждения доставки → `ConfirmDataReceipt` (D3, S2.12, S2.16). **Покрыто.**
  - Нет журналирования → запись переходов и ошибок в журнал регистрации (§ Журнал, S3.5). **Покрыто.**
  - «Зависшие выгрузки не отслеживались» → статус «Выполняется» + журнал + **ручной** сброс;
    авто-детект «зависших» вынесен в FU1 (Non-Goal, Open Question Q2). **Покрыто частично.**
- **Покрытие:** полное по четырём из пяти пунктов Why; по пятому («отслеживание зависших») —
  частичное: design даёт **видимость** (статус «Выполняется» виден на форме) и фиксацию в журнале, но
  не даёт активного обнаружения/сброса. Это осознанный Non-Goal, не дефект, но трассировка Why→solution
  по этому пункту неполная: видимость ≠ отслеживание. Достаточно для scope v2.

Дополнительный разрыв надёжности (Q1), не закрытый ни design, ни Open Questions: **потеря ответа
`ConfirmDataReceipt`.** В двухфазной модели после успешной загрузки приёмник вызывает confirm; источник
переводит «Выполняется» → «Выполнено». Если ответ confirm потерян (сеть/таймаут), источник уже
«Выполнено», а приёмник считает цикл неуспешным. При повторе confirm попадает на статус ≠ «Выполняется»
и по S3.4 возвращает ошибку — приёмник трактует доставленные и подтверждённые данные как сбой.
Confirm не идемпотентен. Для change, чей Why — про надёжность сеанса, это значимый незакрытый край.

### Q2 — Optimality

- **Выбранный путь:** статус-атрибут на профиле-источнике + двухфазный WS (`GetData`/`ConfirmDataReceipt`)
  по префиксам + движок V8Exch82.epf из макета обработки + правила в `ХранилищеЗначения` профиля.
- **Альтернативы (включая не упомянутые в `## Implementation Options`):**
  1. **Регистр сведений `рг_СеансыОбмена`** (не упомянут в design) — измерения: пара префиксов / профиль;
     ресурсы: статус, моменты начала/завершения, текст ошибки. Плюсы: естественная история сеансов,
     детект «зависших» одним запросом по `Начало < ТекущаяДата - N`, отсутствие мутации
     справочника-настройки при каждом сеансе. Минусы: +1 объект метаданных, статус «размазан» между
     профилем и регистром. **Отклонение защитимо:** модель v2 — один активный сеанс на профиль, без
     истории и без авто-детекта (Non-Goal); статус-атрибут — минимально достаточный носитель. IR — это
     то, что понадобится при реализации FU1, но не сейчас. По текущему scope атрибут проще и достаточен.
  2. **Движок на базе штатной `УниверсальныйОбменДаннымиXML` (ADR-0003)** — упомянут в design Option B и
     в ADR-0003, где V8Exch82.epf уже был **рассмотрен и отклонён** (риск несовместимости платформы,
     дублирование, «Verify 2026-06-22»). v2 переоткрывает именно отклонённое ранее решение. Преимущества
     ADR-0003 объективны: нет гейта запуска внешней обработки (S2.4: профиль безопасности кластера /
     безопасный режим), нет привязки к версии платформы epf, переиспользование штатного движка КД 2.1.
     **Однако** решение закрыто постановкой заказчика, а verified code fact, делающего epf неработоспособным,
     нет (сам файл epf в выгрузке не проверялся — это и есть гейт S2.1). `reopen-blocked: V8Exch82-engine`.
  3. **UUID профиля вместо префиксов** — упомянут в design (Option B, отклонён). Префиксы несут бизнес-смысл
     идентификации баз и соответствуют ТЗ; UUID раскрывал бы внутренние ссылки. Отклонение защитимо.
- **Вердикт по Q2:** по закрытым осям (движок, идентификация) выбор защитим и переоткрытию не подлежит
  без кода epf. По незакрытой оси (носитель статуса) атрибут — оптимальный минимум для scope; IR избыточен
  сейчас. Существенного превосходства неучтённой альтернативы, требующего REJECT, нет.

### Q3 — Fresh-Eye Approval

- **Согласовал бы:** да, с оговорками (CHALLENGE).
- **Причины «за»:**
  - Двухфазный контракт корректен по 1С: фаза 1 — короткая транзакция под управляемой блокировкой
    профиля с коммитом до выгрузки, фаза 2 (epf) — вне транзакции (D2, S2.8). Это снимает риск долгой
    блокировки и совпадает с инвариантами Behavior Contract.
  - Precedent-regression обработан честно: `## Blast Radius` явно фиксирует supersede ADR-0002 (сигнатура)
    и ADR-0003 (движок), новый ADR запланирован при archive; защищаемый инвариант ADR-0003 «ядро не ищет
    ссылки по примитивам» сохранён в Non-Goals.
  - Контракт epf корректно вынесен в гейт S2.1 (Assumption до static-сверки макета), блокирует S2.9/S2.15.
- **Причины «против» (оговорки):**
  - Не специфицирована идемпотентность `ConfirmDataReceipt` (см. Q1).
  - Порядок проверок отказа и работы с регламентным заданием в существующем `ПередЗаписью` не задан
    (см. Gaps).

## Verdict

**CHALLENGE** — design решает проблему Why и имплементируем, но до apply требуется закрыть несколько
gaps, главный из которых (идемпотентность `ConfirmDataReceipt`) прямо вытекает из цели надёжности.

## Gaps for design.md (всегда при CHALLENGE)

1. **Идемпотентность / потеря ответа `ConfirmDataReceipt`.** Добавить в D3 и spec
   exchange-web-service поведение повторного confirm: для профиля уже в «Выполнено» с совпадающими
   префиксами вернуть успех (no-op), а не ошибку. Текущий S3.4 («≠ Выполняется → ошибка») вступает в
   конфликт с этим сценарием. Уточнить, что считается успешным завершением цикла на приёмнике при
   потерянном ответе confirm.

2. **Порядок в существующем `ПередЗаписью` (verified).** As-built `ПередЗаписью`
   (`Catalogs/рг_НастройкиОбменаXML/Ext/ObjectModule.bsl:6`) **безусловно** создаёт/блокирует
   регламентное задание для любого профиля. D6/S1.9 требуют: задание только при `Источник = Ложь`, а
   S1.6–S1.8 добавляют отказные проверки (`Отказ = Истина`). Зафиксировать в design контракт порядка:
   early-return при `ОбменДанными.Загрузка` → отказные проверки роли/правил/уникальности → работа с
   заданием только при `Источник = Ложь`. Иначе профиль-источник или невалидный профиль всё равно
   затронет регламентное задание.

3. **Трассировка «отслеживания зависших» (Why).** Явно записать в design, что v2 обеспечивает
   видимость + журнал + ручной сброс, а активное обнаружение/сброс — FU1; чтобы Why→solution был полным
   и не читался как нереализованное требование.

4. **Область проверки уникальности префиксов (S1.8).** Уточнить: уникальна ли пара
   (`Префикс`, `ПрефиксБазыПриемника`) глобально по ИБ или только среди `Источник = Истина`. Поиск в
   `GetData`/`ConfirmDataReceipt` фильтрует по `Источник = Истина` (D2, S2.7), поэтому проверку
   уникальности логично сузить до той же области, чтобы не блокировать законные комбинации.

## Architectural alternatives (опционально, только если 2+ реальные равноправные пути по коду)

### Носитель статуса сеанса обмена
**A. Статус-атрибут на профиле (выбран):** один реквизит `Статус` на справочнике; один активный сеанс
на профиль, без истории. Trade-off: минимум объектов, мутация справочника-настройки при каждом сеансе.
**B. Регистр сведений `рг_СеансыОбмена`:** история сеансов + детект «зависших» одним запросом. Trade-off:
+1 объект метаданных и распределение статуса; оправдан при реализации FU1, избыточен для scope v2.
*(Развилка незакрытая, но для текущего scope вариант A защитим; переоткрывать не требуется.)*

### Движок выгрузки/загрузки
**A. V8Exch82.epf из макета (выбран, closed):** независимость от cf, явная версия epf. Trade-off: гейт
запуска внешней обработки (профиль безопасности/безопасный режим), привязка к версии платформы epf.
`reopen-blocked: V8Exch82-engine` — отклонение ADR-0003 закрыто постановкой; verified code fact против
epf отсутствует (проверка — гейт S2.1).
**B. Движок на базе штатной cf (ADR-0003):** нет гейта внешней обработки, переиспользование КД 2.1.
Trade-off: противоречит постановке v2; ранее отклонён за дублирование макетов cf.
