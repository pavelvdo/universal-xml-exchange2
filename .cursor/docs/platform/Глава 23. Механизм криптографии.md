# Глава 23. Механизм криптографии

> **Agent ref:** Криптография 1С — взаимодействие с внешними модулями (не реализация алгоритмов). Ключевые сущности: МенеджерКриптографии, СертификатКриптографии, хранилища сертификатов; ЭЦП (присоединённая/отсоединённая), шифрование (асимметричное + симметричное, CMS/PKCS#7); синхронный режим на сервере, асинхронный на клиенте; форматы подписи CAdES-BES/T/C/X/A, CMS; КриптоПро CSP, CryptoAPI; OID алгоритмов.
> ---

## 23.1. Общая информация

Механизм криптографии нужен для: проверки неизменности документов (например, приложенный файл к объекту «Договор»), передачи подписанной информации, утверждения документов, шифрования данных при передаче по открытым каналам.

В «1С:Предприятие» реализован механизм на базе **асимметричного шифрования** (пара ключей: открытый и закрытый).

**ВНИМАНИЕ.** Платформа не содержит реализации алгоритмов криптографии; предоставляет объекты для работы с **внешними модулями криптографии** сторонних производителей.

- **Открытый ключ** — для передачи по открытым каналам. **Закрытый ключ** — секретный, не для распространения.
- Шифрование: нужны открытые ключи получателей. Расшифровка: закрытый ключ, парный использованному открытому.
- Подпись: генерация — закрытый ключ; проверка — открытый ключ подписанта (часто входит в подпись).

Дополнительно используется **симметричное шифрование** (один ключ для шифрования и расшифровки); особенности описаны ниже.

**Клиент и сервер:** на сервере — **синхронные** методы; на клиенте — **асинхронные**. В веб-клиенте необходимо расширение работы с криптографией; для части операций запрашивается разрешение пользователя (доступ к файловой системе и к закрытому ключу).

---

## 23.2. Основные понятия

- **ЭЦП (электронная цифровая подпись)** — результат криптографического преобразования данных закрытым ключом; подтверждает целостность, неизменность и авторство при наличии открытого ключа и сертификата. **Присоединённая ЭЦП** — подпись и данные в одном файле; **отсоединённая** — данные в одном файле, подпись в другом.
- **Удостоверяющий центр (УЦ)** — субъект, уполномоченный удостоверять принадлежность открытого ключа ЭЦП пользователю; выпускает сертификаты для криптоопераций.
- **Сертификат открытого ключа (сертификат)** — электронный документ с открытым ключом и данными о владельце, подписанный УЦ.
- **Контейнер ключа** — хранилище закрытого (и при необходимости открытого) ключа; доступ защищён паролем или иным средством.
- **Владелец сертификата ключа** — физлицо, на которое выдан сертификат и которое владеет соответствующим закрытым ключом.
- **Модуль криптографии** — библиотека с реализацией криптоалгоритмов или доступом к ним.
- **Метка доверенного времени** — достоверные дата и время подписания, создаются и проверяются доверенной третьей стороной (УЦ или оператор ИС).

## 23.3. Общая схема использования

Два основных сценария: **шифрование/расшифровка** и **подпись/проверка подписи**.

**Рекомендации:** не получать несколько хранилищ сертификатов с одинаковыми характеристиками (модификация одного может по-разному влиять на другое в зависимости от модуля). Алгоритмы (шифрование, хеширование, подпись) предпочтительно задавать **OID**, а не мнемонику: один алгоритм у разных провайдеров может называться по-разному при одном OID. Примеры: SHA-256 — OID 2.16.840.1.101.3.4.2.1; RSA — 1.2.840.113549.1.1.1.

### 23.3.1. Шифрование и расшифровка

1. Создать объект доступа к криптографии (**МенеджерКриптографии**).
2. Выбрать **сертификат получателя** (открытый ключ) — объект **СертификатКриптографии**.
3. Зашифровать файл или двоичные данные методами **Зашифровать()** менеджера криптографии.

Внутри: формируется ключ симметричного шифрования → им шифруются данные → этот ключ шифруется асимметричным ключом. Итоговый пакет (по спецификации **CMS**, PKCS#7): зашифрованные данные + зашифрованный симметричный ключ + список получателей.

4. При получении: создать объект доступа; модуль должен **поддерживать** алгоритмы, использованные при шифровании. Расшифровка — метод **Расшифровать()**.
5. В пакете извлекается зашифрованный симметричный ключ, расшифровывается закрытым ключом получателя; затем симметричным алгоритмом расшифровываются данные. Без закрытого ключа расшифровка невозможна.

### 23.3.2. Подпись и проверка подписи

1. Создать **МенеджерКриптографии**.
2. Выбрать сертификат с **закрытым ключом** подписанта.
3. Задать вариант ЭЦП свойством **ВключениеДанныхВПодпись**: Истина — присоединённая; Ложь — отсоединённая (по умолчанию).
4. Подписать файл или двоичные данные методами **Подписать()**.

Схема: сначала вычисляется **хеш** подписываемых данных, затем подписывается хеш. Рекомендуется явно задавать **АлгоритмХеширования** и **АлгоритмПодписи**; иначе проверка на другой стороне может быть невозможна из-за несовпадения алгоритмов. Объект **ИнформацияМодуляКриптографии** содержит список поддерживаемых алгоритмов; в свойства менеджера криптографии подставляют данные из свойств этого объекта. Информация об алгоритмах входит в подпись, отдельно передавать её не требуется.

5. Передача: один файл (присоединённая) или два (отсоединённая: данные + файл подписи).
6. Проверка: создать объект доступа; модуль должен поддерживать алгоритмы хеширования и подписи, использованные при подписании. **ПроверитьПодпись()** — проверка ЭЦП. При неподдерживаемых алгоритмах проверка выдаст ошибку.

**Форматы подписи** (параметр ТипПодписи метода Подписать(), по умолчанию CAdES-BES):

- CAdES-BES (CAdESBES) — базовый CAdES.
- CAdES-T (CAdEST) — с доверенным временем.
- CAdES-C (CAdESC) — с полным набором проверочных данных.
- CAdES-X Long Type 2 (CAdESXLongType2) — долгосрочная расширенная.
- CAdES-A v3 (CAdESAv3) — для архивного хранения.
- CMS (CMS) — устаревший.

**Архивное хранение:** подписать документ форматом CAdES-A или усовершенствовать существующую подпись (**УсовершенствоватьПодпись()**). Подпись в формате CMS усовершенствовать нельзя — документ переподписывают CAdES-A. После помещения в архив периодически (рекомендуется раз в год) добавлять в подпись архивную метку времени (**ДобавитьАрхивнуюМеткуВремени()**). Сервера меток задаются свойством **АдресаСерверовМетокВремени** (например, 1С: `https://1stdss.1c.ru/TSP/tsp.srf`). Проверка метки времени — **ПроверитьМеткуВремени()**.

## 23.4. Работа с модулями криптографии

В ОС Windows используется **Microsoft CryptoAPI**; в Linux и macOS — интерфейс **КриптоПро CSP**.

Поддерживаемые компоненты: средство КриптоПро — в конструкторе **МенеджерКриптографии** указывать **ТипМодуляКриптографии** = 80.

В Linux/macOS путь к модулю КриптоПРО CSP: если в конструкторе задан **ПутьМодуляКриптографии** — используется он; иначе — каталоги из параметра **CryptoAPILibraryLocation** в conf.cfg; при отсутствии параметра — поиск по стандартным путям. Если параметры модуля неизвестны, можно создать менеджер без указания пути (упрощённый вариант).

## 23.5. Примеры использования

### 23.5.1. Создание объекта доступа

Базовая операция для всех дальнейших операций с криптографией. Свойства менеджера заполняются по данным объекта **ИнформацияМодуляКриптографии** (список поддерживаемых алгоритмов).

### 23.5.2. Получение списка сертификатов

Формируется список сертификатов из выбранных хранилищ для дальнейшей работы.

### 23.5.3. Шифрование файла

Шифрование выбранного файла и запись зашифрованного файла на диск (интерактивно). В примере для шифрования берётся первый сертификат из списка установленных (в реальной задаче выбор по иным критериям).

```bsl
&НаКлиенте
Асинх Процедура ВыполнитьКоманду(Команда)
    ПомещенныйФайл = Ждать ПоместитьФайлНаСерверАсинх(, , , Новый ПараметрыДиалогаПомещенияФайлов);
    Если ПомещенныйФайл = Неопределено Тогда Возврат; КонецЕсли;
    Адрес = ПомещенныйФайл.Адрес;
    ТипыСертификатов = Новый Массив;
    ТипыСертификатов.Добавить(ТипХранилищаСертификатовКриптографии.ПерсональныеСертификаты);
    Список = Ждать ПолучитьСписокСертификатов(80, ТипыСертификатов, Истина);
    Сертификаты = Новый Массив;
    Сертификаты.Добавить(Ждать Список[0].ВыгрузитьАсинх());
    РезультатШифрования = ЗашифроватьНаСервере(Адрес, Сертификаты);
    Если Не РезультатШифрования Тогда Возврат; КонецЕсли;
    ПолучитьФайлССервераАсинх(Адрес, ПомещенныйФайл.СсылкаНаФайл.Файл.ПолноеИмя + ".signed", Новый ПараметрыДиалогаПолученияФайлов);
КонецПроцедуры
```

### 23.5.4. Расшифровка файла

Расшифровка выбранного файла. Требуется реализация **ПолучитьПарольДоступа()** для доступа к закрытому ключу. При использовании **Расшифровать()** исключение возбуждается после неудачных попыток расшифровки по **всем** доступным сертификатам, а не после первой ошибки.

### 23.5.5. Формирование ЭЦП

Подпись выбранного файла. Параметр **ПрисоединеннаяПодпись** метода ПодписатьФайл(): Истина — присоединённая ЭЦП, результат в файле с расширением .sign (то же имя); Ложь — отсоединённая, подпись в signature.sign. Нужны: ПолучитьПарольДоступа(), ПолучитьСписокСертификатов().

```bsl
МенеджерКриптографии.ВключениеДанныхВПодпись = ПрисоединеннаяПодпись;
Если ПрисоединеннаяПодпись Тогда
    ИмяФайла = Ждать МенеджерКриптографии.ПодписатьАсинх(Данные, ФайлДляПодписи.Путь + ФайлДляПодписи.ИмяБезРасширения + ".sign", Сертификат);
Иначе
    ИмяФайла = Ждать МенеджерКриптографии.ПодписатьАсинх(Данные, ФайлДляПодписи.Путь + "signature.sign", Сертификат);
КонецЕсли;
```

### 23.5.6. Проверка ЭЦП

Проверка действительности ЭЦП для файла. Если файл — не подпись с включёнными данными, ожидается отсоединённая подпись в signature.sign в том же каталоге. Для присоединённой подписи из результата проверки можно получить подписанные данные.

### 23.5.7. Усовершенствование ЭЦП

Усовершенствование отсоединённой ЭЦП до типа с отметкой доверенного времени. Подпись — в файле с тем же именем и расширением .sign; результат — в файле с расширением .extsign. В примере выбирают подписанный файл; имена файла подписи и усовершенствованной подписи формируются автоматически.

### 23.5.8. Добавление архивной метки времени

Для документов в архиве периодически добавляют в подпись архивную метку времени. В примере операция выполняется для всех файлов отсоединённой ЭЦП; файл подписи с добавленной меткой сохраняется в тот же файл.

### 23.5.9. Получение информации о подписях

Получение сведений о подписях, которыми подписан файл. Тип подписи определяется автоматически. Для отсоединённой используется схема хранения, как в примерах формирования и проверки ЭЦП.

## 23.6. Асинхронная работа с механизмом криптографии

- Доступна только на **клиенте**; на сервере — только синхронные методы.
- В веб-клиенте необходимо расширение работы с криптографией.

Код на встроенном языке при асинхронной работе практически совпадает с синхронным вариантом; отличие — вызов асинхронных методов (например, получение списка сертификатов из заданного типа хранилища). Пример вывода серийного номера сертификата:

```bsl
Сообщить("Серийный номер сертификата = " + Сертификат.СерийныйНомер);
```
