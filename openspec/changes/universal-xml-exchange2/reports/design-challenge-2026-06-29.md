Challenge независим: прочитаны `proposal.md`, `design.md`, все четыре `specs/**/spec.md` как чужая работа. Отчёт `reports/architecture-extend-coherence-2026-06-29.md` и прочие `reports/architecture-*.md` **не** использовались как источник истины (указание scope). Источник проблемы — `## What Changes` п.1 proposal + design D1/D7 + Behavior Contract UX. Позиция: попытаться отвергнуть переход enum→string + кнопочный UX, а где не удаётся — найти, что должно появиться в артефактах до apply. Закрытые решения (string-хранение, список значений, две кнопки, диалог подтверждения отмены из «Выполняется») не переоткрываются — нет кодовых фактов для reopen.

## Three-Question Challenge

### Q1 — Problem-Solution Fit

- **Why (delta) говорит:** профиль-источник должен управляться кнопками «Подготовить к обмену»/«Отменить обмен», статус — только просмотр, домен статусов — строковые коды (перечисление `рг_СтатусыОбмена` удаляется); прямое редактирование `Статус` пользователем MUST NOT допускаться (Behavior Contract, инвариант).
- **Design адресует:**
  - D1 — тип `Статус` → `Строка(20)`, домен = функции-константы в `рг_УниверсальныйОбменXMLСервер` (SSOT), `СписокСтатусов` (`СписокЗначений`) для подписей, поле read-only.
  - D7 — команды «Подготовить»/«Отменить» с таблицей доступности и кодами; guard `ПередЗаписью` для отклонения недопустимых пользовательских переходов; диалог подтверждения отмены из «Выполняется».
  - S4 — срез с миграцией; Migration Plan (S4).
- **Покрытие: частичное.** Целевой UX и доменная модель покрыты. Два инварианта закрыты не полностью:
  1. **Инвариант «прямое редактирование статуса MUST NOT»** опирается на guard `ПередЗаписью`, который должен «отклонять пользовательские изменения `Статус` вне `{"", "Новое"}`, но WS-запись статусов не блокировать» (D7). В design **не задан механизм различения** WS-инициированной записи (`"Выполняется"`/`"Выполнено"`/`"Ошибка"`) от пользовательской записи с формы. `ОбменДанными.Загрузка` это не покрывает (запись из веб-сервиса — обычная серверная запись, не загрузка обмена). Без явного признака контекста guard либо отклонит легитимную WS-запись, либо станет нерабочим. Это **implementation invariant gap**, а не развилка.
  2. **Spec-носитель противоречив** (см. Q3, gap A).

### Q2 — Optimality

- **Выбранный путь:** строковый реквизит + `СписокЗначений` на форме + две команды + guard `ПередЗаписью`; удаление перечисления.
- **Альтернативы:**
  1. **Сохранить перечисление `рг_СтатусыОбмена` как носитель, UX — поверх enum** (read-only поле, синонимы значений как подписи, установка значения кнопками). Материально отличается от закрытого решения, но **отклонена заказчиком явно** (closed decision: «string storage, not keeping enum»). Отмечается только для полноты оценки оптимальности: этот путь не требует смены типа реквизита и не обнуляет значения существующих профилей (Risk «Очистка статусов при смене типа» + Migration S4 — следствие именно строкового выбора). Reopen не предлагается — нет verified code fact, что enum-путь обязателен. `reopen-blocked: closed-enum-to-string`.
  2. **Строка без `СписокЗначений`** — подпись статуса формируется серверной функцией код→представление при чтении формы. Снимает один реквизит формы и `ЗаполнитьСписокСтатусов()`. Незначимо отличается; заказчик выбрал список значений (closed). Не форк.
- **Вердикт по Q2:** в рамках закрытой оси (строка + список + кнопки) путь близок к минимальному; глобально более дешёвый по миграции enum-путь сознательно закрыт заказчиком. Оптимальность принимается с оговоркой, что миграционная стоимость — следствие закрытого выбора, а не дефект design.

### Q3 — Fresh-Eye Approval

- **Согласовал бы:** нет — до устранения gap A и gap B.
- **Причины «нет»:**
  - **Gap A (носитель спецификации противоречив).** В `specs/exchange-settings/spec.md` требование «Статус жизненного цикла обмена» присутствует **дважды**: как `## ADDED` (тип перечисление `рг_СтатусыОбмена`, значения Новое/Выполняется/Выполнено/Ошибка) и как `## MODIFIED` (тип `Строка(20)`, «перечисление MUST NOT использоваться»). В файле два отдельных заголовка `## MODIFIED Requirements`. Поскольку исходный ADDED-requirement этого change ещё не заархивирован, корректная дельта должна **переписать ADDED на месте**, а не накладывать MODIFIED на одноимённый ADDED. Свежий ревьюер при применении дельты получает два конфликтующих определения одного требования.
  - **Gap B (guard без механизма различения контекста)** — см. Q1.
- **Причины «да» (после устранения A/B):** доменная модель статусов и переходов согласована между Behavior Contract, D1/D7 и сценариями spec; WS-контракт (`GetData`/`ConfirmDataReceipt`, префиксы, идемпотентность) extend-ом не затронут; S4 изолирован и зависит от S2.

## Verdict

**CHALLENGE** — решение реализуемо и не нарушает закрытых решений, но до apply нужно закрыть две конкретные дыры в артефактах (носитель spec и механизм guard), иначе инвариант «статус не редактируется напрямую» и применимость дельты specs не гарантированы.

## Gaps for design.md / specs (обязательно при CHALLENGE)

- **Gap A — устранить двойное определение требования «Статус жизненного цикла обмена»** в `specs/exchange-settings/spec.md`: оставить одно определение (строковый носитель). Поскольку enum-требование принадлежит этой же незаархивированной ЗНИ, переписать исходный `## ADDED`-блок под строковый тип и удалить наложенный второй `## MODIFIED Requirements` + одноимённый MODIFIED-дубль. Цель: дельта не должна содержать конфликтующих определений одного requirement.
- **Gap B — специфицировать в D7 (и в задаче S4) механизм различения записи WS и записи с формы** для guard `ПередЗаписью`: какой признак контекста используется (например, явный флаг/параметр серверного вызова статусного перехода, либо `ДополнительныеСвойства` записи, проставляемый WS-кодом перехода), чтобы переходы `"Выполняется"`/`"Выполнено"`/`"Ошибка"` от веб-сервиса проходили, а пользовательские изменения `Статус` вне `{"", "Новое"}` отклонялись. Это уточнение инварианта на закрытой оси, не архитектурный форк. Выбор конкретного признака — за реализацией (writer); design должен зафиксировать само требование «WS-переходы помечаются и пропускаются guard-ом».
- **Gap C (minor) — Open Question 1 при delta.** Поведение «ошибка на приёмнике после успешного GetData → источник остаётся "Выполняется"» теперь снимается ручным «Отменить обмен» с подтверждением (D7). Стоит явно связать Open Question 1 с этим UX-путём в design, чтобы при archive он не выглядел нерешённым.

## Simplicity Check

- **Бюджет сложности дельты (S4):** смена типа реквизита (1, требует миграции), реквизит формы `СписокСтатусов` (1), `ЗаполнитьСписокСтатусов()` (1), функции-константы статуса (SSOT, уже в D1-базе), две команды формы (2), guard-ветка в `ПередЗаписью` (1).
- **Минимальность на закрытой оси:** в рамках строкового выбора лишних сущностей не выявлено; функции-константы обязательны как SSOT после удаления enum. Альтернатива «без `СписокЗначений`» убрала бы 2 элемента, но противоречит закрытому решению — не предлагается.
- **Замечание:** строковый носитель сам по себе несёт миграционную стоимость (обнуление значений при смене типа), которой нет у enum-пути; это следствие закрытого решения заказчика, зафиксировано в Risks/Migration. Не дефект простоты дизайна.
