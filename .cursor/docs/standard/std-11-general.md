# Домен: Общее (функциональные опции, регламентные задания, обмен, локализация)

> Стандарты и методики (ИТС v8std).

## Использование функциональных опций

- 1.1. В случае если некоторая функциональность конфигурации является необязательной для использования, то для управления доступностью такой функциональности на стадии внедрения следует применять функциональные опции. Для хранения значений функциональных опций в информационной базе необходимо завести в конфигурации соответствующие данные (например, константы).

Допустим, в конфигурации есть функциональность версионирования данных информационной базы, которая является необязательной. Для управления доступностью этой функциональности необходимо:

- создать функциональную опцию ИспользоватьВерсионированиеОбъектов, которая определяет использование прикладного механизма конфигурации для текущей информационной базы
- создать константу ИспользоватьВерсионированиеОбъектов типа Булево для хранения значения этой функциональной опции
- в свойстве Хранение функциональной опции указать константу ИспользоватьВерсионированиеОбъектов. После этого, те или иные объекты конфигурации можно «привязать» к функциональной опции, включив их в ее состав, а в случае необходимости управления доступностью кода – использовать метод ПолучитьФункциональнуюОпцию:

```
ИспользуетсяМеханизмВерсионирования = ПолучитьФункциональнуюОпцию("ИспользоватьВерсионированиеОбъектов");
```

Таким образом, набор функциональных опций описывает функциональность конфигурации, доступность которой на этапе внедрения можно настроить в зависимости от требований конкретного предприятия. При этом платформа автоматически изменяет пользовательский интерфейс в соответствии с установленными значениями функциональных опций.

Функциональные опции могут также влиять на бизнес-логику. Для чего применяются функциональные опции не только булева типа, но и других типов, например, ссылки на справочники или значения перечислений.

1.2. Доступность функциональности может задаваться не только для информационной базы в целом, но и в зависимости от контекста применения этой функциональности. Для параметризованных функциональных опций необходимо создавать параметры, указывать хранение (в т.ч. регистры сведений с измерениями), устанавливать значения параметров в формах (УстановитьПараметрыФункциональныхОпцийФормы) и получать значение опции с параметрами (ПолучитьФункциональнуюОпцию с вторым параметром). Внимание: следует учитывать, что описанный здесь вариант применения функциональных опций не является единственным. Подробнее см. в документации по платформе 1С:Предприятие.

- 1.3. Не следует использовать функциональные опции не по назначению, например:
- создавать функциональные опции ради управления видимостью элементов управления конкретной формы. С помощью функциональных опций следует управлять доступностью той или иной функциональности для всей конфигурации (и, как следствие, доступностью элементов форм и команд во всей конфигурации, а не в одной отдельно взятой форме);
- использовать функциональные опции для оптимизации доступа к тем или иным данным информационной базы (хранения значений на сервере 1С:Предприятия). Для этой цели предназначены модули с повторным использованием возвращаемых значений.

### Установка и получение значений функциональных опций

- 2.1 Платформа 1С:Предприятие не предоставляет каких-либо специальных средств для установки значений функциональных опций: установка значений функциональных опций производится установкой значений соответствующих констант, редактированием элементов справочников или записей регистров сведений. В конфигурации следует предусмотреть соответствующую функциональность.
- 2.2. При использовании функциональных опций с параметрами, следует иметь в виду, что если в справочнике или регистре сведений нет записи, соответствующей параметру, то функциональная опция считается выключенной. Если же параметру соответствует больше, чем одна запись, то значения функциональной опции складываются по «ИЛИ».
- 2.3. Если функциональная опция «привязана» к ресурсу периодического регистра сведений, то система использует срез последних для получения значения опции. Если требуется получать значение опции на какую-либо другую дату, необходимо указать значение для параметра функциональной опции Период типа Дата. Параметр Период не следует создавать в метаданных, так как он предоставляется системой автоматически.
- 2.4. Также необходимо иметь в виду, что установка значения функциональной опции не вызывает автоматического изменения пользовательского командного интерфейса. Для отработки изменения следует вызвать метод ОбновитьИнтерфейс.
- 2.5. С точки зрения производительности системы следует иметь в виду, что значения функциональных опций кешируются на сервере. Однако большой размер кеша может ухудшить производительность. Поэтому не рекомендуется параметризовать функциональные опции такими данными, которые заведомо могут иметь большое число значений. Например, параметризация функциональной опции контрагентом или товаром не допустима. На практике функциональность ставится в зависимость от вида контрагента или иного признака.

### Зависимые функциональные опции

- 3.1. В некоторых случаях та или иная функциональность должна быть доступна при условии использования или отказа от использования другой функциональности. В подобных случаях сложной зависимости значения функциональной опции от значений других функциональных опций необходимо обеспечить непротиворечивость данных, связанных с функциональными опциями. Значения таких зависимых функциональных опций рекомендуется показывать администратору в соответствующей форме настроек; значение зависимой опции должно быть недоступно для редактирования.
- 3.2. Значения взаимоисключающих функциональных опций рекомендуется редактировать при помощи элементов управления «Поле переключателя», «Поле ввода» (со списком выбора) или иного элемента для выбора одного значения из многих.
- 3.3. В том случае, если та или иная незначительная функциональность сложным образом зависит от значений функциональных опций, но при этом не может быть названа так, чтобы ее название было понятно конечному пользователю, рекомендуется воздержаться от создания очередной функциональной опции. Зависимость элементов форм должна обеспечиваться при создании формы на сервере за счет анализа значений функциональных опций из кода на встроенном языке.

### Ограничения на использование параметров функциональных опций

- 4.1. По соображениям производительности не рекомендуется заводить в конфигурации более 10 параметров функциональных опций. Не следует создавать различные параметры функциональных опций одной смысловой нагрузки; рекомендуется объединять их в один параметр (например, ТипОбъектаКонфигурации), связанный с измерениями нескольких регистров.
- 4.2. Для принятия решения по составу функциональных опций и их параметров рекомендуется: определить опциональную функциональность; определить, выключается ли она для всей ИС или нужны несколько «выключателей» (по организации, виду товара и т.п.); выписать список параметризуемых функциональных опций и их параметров без дублирования типов параметров; при избытке параметров составить «рейтинг» и исключить менее востребованные; оставшиеся без параметров опции делать непараметрическими либо удалять.

**См. также:** Влияние изменения значений параметров сеанса и функциональных опций на производительность механизма ограничения доступа к данным.

## Использование параметров сеанса

#std413

- 1.1. Параметры сеанса предназначены для хранения значений определенных типов для каждого клиентского сеанса на время работы этого сеанса. Инициализацию параметров сеанса следует выполнять в модуле сеанса (см. ниже раздел 2.1), а их значения рекомендуется использовать в запросах и условиях ограничения доступа к данным для текущего сеанса. Примеры параметров сеанса: ТекущийПроект (СправочникСсылка.Проекты), ОбменДаннымиВключен (Булево), РабочееМестоКлиента (СправочникСсылка.РабочиеМеста). Параметры сеанса доступны из встроенного языка (ПараметрыСеанса.ТекущийПроект) и в текстах ограничений доступа (ГДЕ Документ.Автор = &ТекущийПользователь).
- 1.2. Не рекомендуется использовать параметры сеанса для хранения значений, используемых исключительно в клиентской логике. В клиент-серверном варианте параметры сеанса хранятся на сервере, любое их считывание или изменение на клиенте потребует дополнительного серверного вызова. В таких случаях следует использовать глобальные переменные модуля управляемого приложения.
- 1.3. Также не рекомендуется использовать параметры сеанса для кеширования вычисленных значений, многократно используемых в серверной бизнес-логике. В таких случаях следует определять функцию в серверном общем модуле с повторным использованием возвращаемых значений.

### Установка параметров сеанса «по требованию»

2.1. Не следует производить инициализацию параметров сеанса при запуске программы, так как не все параметры запрашиваются при запуске и возможно намеренное обнуление значений из кода. Правильным способом является установка значений «по требованию» в обработчике УстановкаПараметровСеанса модуля сеанса: параметры должны инициализироваться только в момент первого обращения к ним как к неустановленным. В обработчике при ИменаПараметровСеанса = Неопределено выполняется установка параметров при начале сеанса; иначе — установка «по требованию» с группировкой параметров, требующих одних и тех же данных, и сохранением имен уже установленных параметров в массиве для избежания повторной инициализации.

**См. также:** Влияние изменения значений параметров сеанса и функциональных опций на производительность механизма ограничения доступа к данным.

## Версия платформы 1С:Предприятие для разработки

- 1. Минимальная версия платформы, на которой обеспечивается работа прикладного решения, устанавливается:
  - не больше номера версии платформы, на которой основан последний сертифицированный релиз платформы – в прикладных решениях для государственных учреждений, оборонных предприятий и корпоративных клиентов, а также в любых библиотеках. См. защищенный программный комплекс «1С:Предприятие 8.3z»;
  - или на усмотрение ответственных за конфигурацию – в остальных прикладных решениях. В качестве минимальной версии выбирается версия платформы, на основе которой собран последний сертифицированный релиз 1С:Предприятие 8.3z. Номер публикуется на страничке 1С:Предприятие 8.3z на сайте online.1c.ru.
- 1.1. При невозможности обойти ошибку на уровне прикладных решений минимальная версия платформы может быть увеличена (для прикладных решений для госучреждений новый релиз платформы должен быть сертифицирован в короткие сроки). #std785
- 1.2. В исправительных релизах библиотек не допускается увеличивать минимальную версию платформы.
- 2. Рекомендуемая версия платформы устанавливается в прикладных конфигурациях на усмотрение ответственного; ориентироваться на последний опубликованный релиз, на котором ведется разработка и тестирование. Не может быть увеличена в исправительных релизах библиотек.
- 3. Запуск на платформе ниже минимальной версии блокируется с помощью БСП. Минимальная и рекомендуемая версии задаются в процедуре ПриОпределенииОбщихПараметровБазовойФункциональности общего модуля ОбщегоНазначенияПереопределяемый. Они также указываются в файле readme.txt дистрибутива.

## Регламентные задания

### Общие требования к регламентным заданиям

Область применения: управляемое приложение, обычное приложение. #std540

1. В общем случае регламентные задания следует использовать, когда необходимо выполнить определенные периодические или однократные действия в соответствии с расписанием.
2. Если регламентные задания не требуется добавлять или удалять в зависимости от действий пользователя или логики конфигурации, следует использовать предопределенные регламентные задания (загрузка курсов валют, извлечение текста для полнотекстового индексирования, обновление агрегатов и т.п.).
3.1. Если выполнение регламентного задания зависит от одной или нескольких функциональных опций (ФО), необходимо программно управлять признаком Использование предопределенного регламентного задания в зависимости от установленных ФО. Иначе задание будет приводить к запуску сеанса и занимать ресурсы сервера. Неправильно: создавать предопределенное регламентное задание, зависящее от ФО, с установленным флажком Использование. Правильно: снять флажок Использование и управлять использованием в зависимости от ФО. При использовании БСП – процедура ПриОпределенииНастроекРегламентныхЗаданий общего модуля РегламентныеЗаданияПереопределяемый (добавить настройку с РегламентноеЗадание и ФункциональнаяОпция). Для конфигураций без БСП – управлять в модуле менеджера при записи константы, соответствующей ФО.
3.2. Дополнительно в начало процедуры обработки регламентного задания вставить проверку ФО и вызов ОбщегоНазначения.ПриНачалеВыполненияРегламентногоЗадания(); при недоступности ФО – ВызватьИсключение. При использовании БСП и настроенных зависимостях от ФО достаточно вызова, как в п.6.
4.1. Если выполнение зависит от данных ИБ (обмен с каждой базой по своему расписанию, запуск дополнительных обработок по отдельному расписанию), флажок Предопределенное следует отключать. Создавать экземпляры и параметризовать их из кода (РегламентныеЗадания.СоздатьРегламентноеЗадание), в свойстве Наименование указывать представление объекта. При использовании БСП – выполнить п.6.
4.2. При использовании БСП необходимо запрещать интерактивное создание и запуск параметризованных регламентных заданий из формы «Регламентные и фоновые задания»: указать такое задание в ПриОпределенииНастроекРегламентныхЗаданий (Настройка.Параметризуется = Истина). Также выполнить п.6.
5. Рекомендуется в копиях ИБ автоматически блокировать все регламентные задания, обращающиеся к внешним ресурсам (рассылка почты, синхронизация и т.п.). При БСП – перечислить такие задания в ПриОпределенииНастроекРегламентныхЗаданий (Настройка.РаботаетСВнешнимиРесурсами = Истина) и выполнить п.6.
6. Если регламентное задание попадает под требования п.3.1, 4.2, 5 и используется подсистема «Регламентные задания» БСП, в начале процедур обработчиков таких заданий необходимо помещать вызов: ОбщегоНазначения.ПриНачалеВыполненияРегламентногоЗадания(Метаданные.РегламентныеЗадания.<ИмяЗадания>); первый параметр заполнять обязательно.

При разработке регламентных заданий необходимо выбирать время и интервал запуска исходя из прикладного назначения и соображений производительности:
- не выполнять чаще, чем нужно с прикладной точки зрения;
- для большинства заданий нормален интервал 1 день;
- ни в каких случаях не задавать периодичность меньше одной минуты;
- периодичность частых заданий (менее дня) сбалансировать со временем выполнения;
- ресурсоемкие операции переносить на время минимальной загрузки сервера; несколько ресурсоемких заданий «разносить» по времени.

**См. также:** Настройка расписания регламентных заданий; Регламентные задания: требования по локализации.

### Запуск регламентных заданий

Область применения: управляемое приложение, обычное приложение. #std539

- 1. Рекомендуется предоставлять пользователям возможность выполнять регламентные задания вручную («по кнопке»), так как работа системы не должна зависеть от автоматического выполнения (задания могут быть выключены на кластере; в файловом режиме в 8.2 и ранее не было автоматического выполнения).
- 1.1. Если регламентное задание изменяет данные, необходимые определенному бизнес-процессу или «рабочему месту», в таких рабочих местах дополнительно размещать команду для выполнения этого действия (например, «Обновить», «Получить почту», «Выполнить»). Рабочие места должны информировать о дате актуальности данных и давать команду обновления. Команда доступна только пользователям с необходимыми правами.
- 1.2. Если результат работы регламентного задания влияет на данные в заранее неизвестном количестве рабочих мест или на ИБ в целом (обновление агрегатов, установка периода рассчитанных итогов), рекомендуется заводить отдельное рабочее место для выполнения таких заданий. При БСП такое рабочее место входит в подсистему «Регламентные задания» (форма «Регламентные и фоновые задания»).
- 1.3. Когда регламентное задание не изменяет данные в системе, а формирует отчеты или рассылки, также рекомендуется предусматривать отдельное рабочее место. При БСП – форма «Регламентные и фоновые задания».
- 2. Для администраторов ИБ: на период обновления ИБ блокировать регламентные задания. Если обновление выполняет неподготовленный пользователь (особенно в файловом режиме), дополнительно: в файловом режиме при неудачной попытке установки монопольного режима предлагать автоматически блокировать регламентные задания (/AllowExecuteScheduledJobs -Off); в начале кода обработчиков регламентных заданий проверять режим работы и прерывать выполнение вызовом исключения, если обновление ИБ еще не завершено. При БСП первое реализовано в подсистеме «Обновление версии ИБ», для второго предусмотрена процедура ПриНачалеВыполненияРегламентногоЗадания общего модуля ОбщегоНазначения.

**См. также:** Настройка расписания регламентных заданий; Предопределенные регламентные задания; Ограничения на регламентные задания при работе в режиме сервиса.

### Ограничения на регламентные задания при работе в режиме сервиса

Область применения: управляемое приложение. #std760

- 1. В прикладных решениях, ориентированных на работу в режиме сервиса по Технологии 1cFresh, не должно быть регламентных заданий, включенных в состав любого из разделителей. Разделенные регламентные задания при большом количестве областей данных могут вызвать перегрузку рабочих процессов.
- 2. Если требуется регулярное выполнение кода в каждой области данных разделенной ИБ, необходимо использовать подсистему БСП «Очередь заданий» или разработать аналогичный механизм очереди заданий. Добавить предопределенное неразделенное регламентное задание и в общем модуле ОчередьЗаданийПереопределяемый зарегистрировать шаблон и псевдоним обработчика (ПриПолученииСпискаШаблонов, ПриОпределенииПсевдонимовОбработчиков); установить в качестве обработчика процедуру соответствующего модуля.
- 3. Исключение: когда регламентное задание обязательно должно выполняться от имени определенного пользователя (например, с учетом RLS). Тогда разделение допускается, но задание должно быть включено в состав всех разделителей.
- 4. В решениях для режима сервиса 1cFresh не должно быть участков, где из кода напрямую выполняется управление регламентными заданиями. Для управления необходимо использовать программный интерфейс БСП (модуль РегламентныеЗаданияСервер), например РегламентныеЗаданияСервер.НайтиЗадания(Отбор), РегламентныеЗаданияСервер.ИзменитьЗадание(УникальныйИдентификатор, Параметры).
- 5. Подсистема «Очередь заданий» не гарантирует выполнение в точном соответствии с расписанием. Рекомендуется в режиме сервиса не предоставлять пользователям возможность настройки расписания регламентных заданий.

## Оформление карты маршрута бизнес-процесса

Область применения: управляемое приложение, обычное приложение. #std480

### 1. Общие рекомендации

- 1.1 Бизнес-процессы рисуются сверху вниз. Другая ориентация (слева направо) допустима, если позволяет изобразить процесс наилучшим образом.
- 1.2 Используется стандартная сетка – шаг 20 точек, нарисованная линиями.
- 1.3 Не используются жирные и разноцветные шрифты.
- 1.4 Количество изгибов соединительных линий должно быть минимальным.
- 1.5 При размещении декораций и точек маршрута соблюдать отступ в одну клетку от левого и верхнего края карты.
- 1.6 Ширину точек маршрута делать равной четному количеству клеток. Размеры точек и декораций приближать к пропорциям спичечного коробка (например, 3х6 клеток). Избегать вытянутых по вертикали или горизонтали точек.
- 1.7 Выравнивание текста в точках маршрута и декорациях – по центру (по умолчанию), за исключением комментариев – выравнивание влево.
- 1.8 Размещать декорации и точки маршрута так, чтобы они не пересекали границы печатного листа (меню Графическая схема – Режим просмотра страниц).

### 2. Рекомендации по оформлению отдельных элементов

- 2.1 Точки Старт и Завершение – размер 2х2 клетки, без наименования. Количество неограниченно. Отступ от точки старта или завершения до другой точки – две клетки.
- 2.5 Точки условного перехода включают короткий текст вопроса, заканчивающийся знаком вопроса. Вопрос формулировать так, чтобы ответ – только Да или Нет (например, «Счет есть?», «Отгрузка разрешена?»). Точки условия делать компактными. Соединительные линии точек условия (входящие и исходящие) – суммарная длина не менее двух клеток, чтобы надписи Да и Нет читались нормально.

## Настройка обмена данными для классификаторов между различными информационными базами

Область применения: управляемое приложение, мобильное приложение. #std637

1. К классификаторам относятся справочники и регистры сведений с условно-постоянной информацией, одинаковой для всех совместно используемых ИБ (классификаторы банков, стран мира, валют, ОКЕИ, адресный классификатор и пр.). При использовании инструмента «Конвертация данных» для создания обменов между различными конфигурациями необходимо придерживаться следующих рекомендаций по синхронизации классификаторов.

2. Данные классификаторов объектного типа должны сопоставляться по уникальным идентификаторам ссылок, а при отсутствии совпадения – по полям поиска (режим автоматического сопоставления). Для этого требуется: в правиле конвертации объекта (ПКО) классификатора установить признаки «Искать объект приемника по внутреннему идентификатору объекта источника» и «Продолжить поиск по полям поиска, если по идентификатору объект приемник не найден»; указать ПКС для полей поиска (например, «БИК» для банков); при необходимости определить код обработчика ПКО «Поля поиска» для сложного сценария. В противном случае данные классификаторов могут продублироваться.

3. Данные классификаторов, представленные наборами записей (например, адресный классификатор), в обмене участвовать не должны. Они должны заполняться и поддерживаться в актуальном состоянии отдельно в каждой ИБ. Это предотвращает избыточную миграцию данных при обмене.

**См. также:** Описание инструмента «Конвертация данных» в разделе «Методическая поддержка 1С:Предприятия 8».

## Разработка планов обмена с отборами

Область применения: управляемое приложение, мобильное приложение. #std701

- 1.1. Для синхронизации данных между различными конфигурациями и для РИБ используется технология планов обмена. Часто возникает задача обмена не всеми данными, а частью (в разрезе организаций, складов, подразделений и пр.). Для определения узлов-получателей следует использовать события ПередЗаписью и ПередУдалением объектов ИБ с логикой регистрации изменений на узлах плана обмена. При использовании подсистемы «Обмен данными» БСП логика регистрации может задаваться декларативно в правилах регистрации объектов (ПРО) в конфигурации «Конвертация данных».
- 1.2. При проектировании логики регистрации учитывать: данные из сообщения обмена загружаются в произвольном порядке; логика выполняется в транзакции записи объекта, признак ОбменДанными.Загрузка игнорируется – ошибки приведут к аварийному завершению загрузки; в РИБ в обмене могут участвовать не все данные (например, движения мигрируют, регистраторы – нет); обращение к полям связанных таблиц «через точку» снижает производительность. При обмене по правилам конвертации (не РИБ) с подсистемой «Обмен данными» БСП первые два пункта не требуются для ссылочных типов – регистрация выполняется отложенно после загрузки всех данных.

- 2.1. Обеспечить самодостаточность данных, участвующих в обмене. Логика регистрации не должна обращаться к полям связанных таблиц, только к данным основной таблицы. Пример: регистрация в разрезе организаций; в обмене документ и регистр накопления с движениями – оба самодостаточны (содержат ссылку на организацию). См. также: Самодостаточность регистров, Разыменование ссылочных полей составного типа в языке запросов.
- 2.2. Исключить из обмена вторичные данные, которые могут быть вычислены независимо в каждой ИБ.
- 2.3. Если в обмене участвуют несколько логически взаимосвязанных таблиц, логика регистрации должна регистрировать к выгрузке данные всех взаимосвязанных таблиц. Логика не должна аварийно прерываться при отсутствии необходимых данных. Пример: справочник «Основные средства» и регистр сведений «Основные средства организаций» – при записи элемента справочника и набора записей регистра регистрировать оба; при загрузке сначала элемента справочника он не регистрируется к выгрузке до загрузки набора записей регистра, затем регистрация выполняется для обоих.

## Интеграция прикладных решений через формат EnterpriseData

Область применения: управляемое приложение. #std771

1. Переходы с конфигурации на конфигурацию следует разрабатывать на основе правил конвертации (ПК). Обмены между конфигурациями, использующими БСП, следует делать на основе формата EnterpriseData; разрабатывать новые обмены на основе ПК запрещается.
2. Актуальные версии EnterpriseData содержатся в последней опубликованной версии конфигурации БСП в виде объектов метаданных Пакет XDTO: EnterpriseData\_X\_Y\_Z (Major, Major, Minor). Список актуальных версий поддерживается в состоянии, обеспечивающем баланс между затратами на поддержку и степенью совместимости.
3. При разработке новых версий прикладных решений в обменах данными необходимо поддерживать все актуальные версии EnterpriseData. Запрещается выпускать версию конфигурации, не поддерживающую какую-либо из версий формата, поддерживаемых в версии БСП, встроенной в конфигурацию.
4. Версия формата EnterpriseData не должна поддерживаться в обмене, если она не соответствует требованиям к функциональности этого обмена (например, для обмена чеками ККМ необходима поддержка не младше версии, в которой есть передача этих данных).
5. В редких случаях обмен невозможен из-за отсутствия совместно поддерживаемых версий (сильно устаревшая версия одного из решений). Рекомендуется информировать пользователей о поддерживаемых в прикладном решении версиях EnterpriseData.

**См. также:** Обмен данными с прикладными решениями на платформе «1С:Предприятие» в формате EnterpriseData в разделе «Разработка и администрирование» ИТС.

## Версионирование и нумерация сборок

Изменение номеров версий регламентируется стандартом «Нумерация редакций и версий». Ниже – правила изменения номера сборки (четвертое число в номере версии).

### 1. Определения

Плановая версия конфигурации – версия, содержащая существенное развитие функционала, срок выпуска которой назначается заранее. Исправительная версия – версия, которая выпускается при необходимости срочной публикации исправлений критичных ошибок. В исключительных случаях исправительная версия может содержать новый функционал (например, доработки, связанные с поддержкой изменения законодательства). Срок выпуска определяется при анализе количества и критичности обнаруженных ошибок плановой версии. Технический проект – задание на доработку конфигурации с четко сформулированной целью и конечным списком изменений. Для организации работ по разработке и сопровождению конфигураций (ведение информации о технических проектах и списка ошибок) рекомендуется использовать Систему проектирования прикладных решений (СППР). #std709

### 2. Разработка исправительных версий

- 2.1. Для выпуска каждой исправительной версии создается новое хранилище на основе конфигурации последней выпущенной версии. Важно – создавать новое хранилище, а не копировать основное.
- 2.2. В исправительной версии не должно быть объемных доработок конфигурации; иначе пересматривать сроки выпуска плановой версии.
- 2.3. Все закладки в хранилище исправительной версии должны содержать комментарий (требования к содержанию аналогичны п. 3.4 для плановой версии).
- 2.4. Все изменения, выполняемые в исправительном релизе, должны синхронно повторяться в основном хранилище. Новые объекты (реквизиты) переносить исключительно с помощью сравнения/объединения конфигураций, чтобы не различались внутренние идентификаторы объектов.
- 2.5. При сборке исправительной версии рекомендуется устанавливать метку с информацией о номере сборки на закладке той версии хранилища, конфигурация которой идет в сборку (обычно последняя на момент сборки закладка).

### 5. Нумерация сборок

- 5.1. Номер сборки следует увеличивать как в основном хранилище, так и в хранилище исправительного релиза в двух случаях: непосредственно перед сборкой релиза (чтобы полный номер собранного релиза гарантированно отличался от предыдущего); при закладке в хранилище обработчика обновления информационной базы (чтобы после обновления из хранилища у всех участников разработки добавленный обработчик обновления запускался автоматически; только для конфигураций на БСП).
- 5.2.1. При добавлении обработчиков обновления ИБ рекомендуется в рамках той же закладки повышать номер сборки. Сценарии: обработчик добавляется в хранилище технического проекта – при переносе в основное хранилище увеличить номер сборки основного хранилища; обработчик добавляется при исправлении ошибки – повышать номер в том хранилище (или обоих), где исправляется ошибка.
- 5.2.2. Обработчик и изменение номера сборки помещать в хранилище в рамках одной закладки. Обработчик обновления «привязывать» к тому номеру сборки, который с ним помещается в хранилище.
- 5.2.3. Если обработчики обновления разбиты по технологическим подсистемам (например, в 1С:ERP – УправлениеПредприятием и УправлениеТорговлей), повышать номер сборки и подсистемы, и конфигурации.
- 5.3. Номер сборки необходимо изменять: 1) в свойствах конфигурации; 2) в процедуре ОбновлениеИнформационнойБазы<ИмяБиблиотеки>.ПриДобавленииПодсистемы (только для конфигураций на БСП).
