## Работа с пользовательскими настройками

Область применения: управляемое приложение, обычное приложение. #std557

- 1.1. Для хранения персональных настроек пользователя следует использовать хранилище общих настроек. Например, чтение и запись значения настройки «Задавать вопрос при выходе из программы» для текущего пользователя реализуется на встроенном языке с помощью объекта ХранилищеОбщихНастроек:

ЗначениеНастройки = ХранилищеОбщихНастроек.Загрузить("НастройкиПрограммы", "ЗадаватьВопросПриВыходе"); ХранилищеОбщихНастроек.Сохранить("НастройкиПрограммы", "ЗадаватьВопросПриВыходе", ЗначениеНастройки);

При этом для хранения настроек пользователя не следует использовать какие-либо другие способы, в частности, другие объекты метаданных (регистры, реквизиты и табличные части справочников и др.), внешние файлы и пр.

- 1.2. Для работы с пользовательскими настройками требуется, чтобы для пользователя было доступно право СохранениеДанныхПользователя. См. также: Стандартные роли
- 1.3. Для обращения к каждой настройке в хранилище общих настроек необходимо использовать уникальный строковый ключ настройки.
- 2.1. В конфигурации должно быть предусмотрено общее место для редактирования всех пользовательских настроек. Как правило, это общая форма персональных настроек пользователя.
- 2.2. Поля с отдельными настройками могут быть размещены непосредственно в тех рабочих местах, к которым эти настройки относятся.
- 2.3. Форма персональных настроек пользователя, другие формы (рабочие места), а также отдельные элементы форм для работы с персональными настройками должны быть доступны только пользователям с правом СохранениеДанныхПользователя.
- 3.1. Настройки не мигрируют между узлами информационной базы, а специфичны для определенного узла.
- 3.2. Все настройки в хранилище общих настроек сохраняются в разрезе пользователей по строковому имени пользователя. При переименовании пользователя прежние настройки теряются. Рекомендуется переносить настройки при переименовании пользователя и очищать при удалении.

## Установка параметров выбора и связей параметров выбора для объектов метаданных

Область применения: управляемое приложение, мобильное приложение, обычное приложение.

1. Как правило, ограничения выбора должны быть одинаковыми для всех форм, в которых редактируется тот или иной объект. Поэтому задавать параметры выбора и связи параметров выбора рекомендуется в свойствах объектов метаданных — в реквизитах справочников, документов и т.п.
2. Однако могут встречаться случаи, когда ограничения выбора зависят от конкретного сценария работы. В таких случаях параметры выбора могут быть уточнены по месту, в конкретной форме.

#std411

Пример: справочник Сотрудники с реквизитами Организация и ВидСотрудника; документ ПриказОПриеме с реквизитом Сотрудник и связью Отбор.Организация с реквизитом Организация. Требуется выбор только основных сотрудников в зависимости от функциональной опции. Для этого:
- в форме документа ПриказОПриеме реализовать дополнительный реквизит формы ВидыСотрудников;
- для поля формы Сотрудник установить связь параметра выбора Отбор.ВидСотрудника с реквизитом формы ВидыСотрудников;
- реквизит формы ВидыСотрудников заполняется на основании анализа функциональной опции.

(Установить для реквизита Сотрудник документа связь Отбор.ВидСотрудника в свойствах метаданных нельзя, т.к. реквизита ВидСотрудника в документе ПриказОПриеме не существует.)

## Использование РеквизитФормыВЗначение и ДанныеФормыВЗначение

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std409

Методическая рекомендация (полезный совет)

- В большинстве случаев в модулях форм следует использовать метод формы РеквизитФормыВЗначение вместо метода ДанныеФормыВЗначение. Рекомендация обусловлена соображениями унификации прикладного кода и тем, что синтаксис метода РеквизитФормыВЗначение проще (меньше вероятность ошибки). В ДанныеФормыВЗначение необходимо дополнительно передавать тип значения:

ТаблицаПодписей = ДанныеФормыВЗначение(ТаблицаПодписей, Тип("ТаблицаЗначений"));

а для РеквизитФормыВЗначение это не обязательно:

ТаблицаПодписей = РеквизитФормыВЗначение("ТаблицаПодписей");

С точки зрения эффективности и результата методы работают одинаково.

## Обработчики событий модуля формы, подключаемые из кода

#std492

Обработчикам событий модуля формы, которые устанавливаются из кода с помощью метода УстановитьДействие, рекомендуется задавать префикс Подключаемый_ (англ. Attachable_). Например:

Процедура Подключаемый_РазрешитьРедактированиеРеквизитовОбъекта()
…
КонецПроцедуры

Процедура Подключаемый_КонтактнаяИнформацияНачалоВыбора()
…
КонецПроцедуры

Если подключение выполняется не в тексте модуля формы (например, в общем модуле), в результатах проверки конфигурации с флажком "Поиск неиспользуемых процедур и функций" окажутся ошибки. Использование префикса позволяет легко идентифицировать такие обработчики и отсеивать как исключения.

## Информирование пользователя

Область применения: управляемое приложение. #std400

1. Случаи, когда необходимо довести информацию до пользователя:
   - Множество сообщений (протокол) о невозможности выполнения действия.
   - Множество сообщений (протокол) о выполнении действия.
   - Информация об ошибке выполнения действия.
   - Информация об успешном выполнении действия при работе в форме.
   - Информация об успешном выполнении действия в условиях отсутствия формы.

1.1. Информация об ошибках, выявленных при проверке заполнения, должна выводиться в панели сообщений формы.
- А) Проверку заполнения реквизитов объекта следует выполнять вне транзакции записи объекта. Средства: свойство Проверка заполнения, ОбработкаПроверкиЗаполнения (для сообщений — объект СообщениеПользователю).
- Б) Проверку целостности объекта и связанных с ним данных — в транзакции записи объекта. Средства: ПриЗаписи, СообщениеПользователю, параметр Отказ = Истина. Следует минимизировать число проверок во время записи; во время записи — только проверки транзакционного характера.

1.2. Информация о протоколе выполненных действий может выводиться в отдельном поле формы или в отдельной форме со списком сообщений.

1.3. Информация об ошибке должна доводиться в отдельном диалоге. Для предупреждений — ПоказатьПредупреждение; при варианте решения — ПоказатьВопрос. В исключительных случаях (ошибка настройки системы, бессмысленность остальных проверок) — ВызватьИсключение.

1.4. Информация об успешном выполнении действия в форме выводится, если факт выполнения не очевиден. Не следует «проглатывать» молча нажатие на кнопку. Рекомендуется Предупреждение или ПоказатьОповещениеПользователя.

1.5. В условиях отсутствия формы — ПоказатьОповещениеПользователя или Состояние при изменении состояния выполняемого действия на клиенте. Для серверных операций не рекомендуется оповещение через Состояние (многократные вызовы сервера). См. Длительные операции на сервере.

## Длительные операции на сервере

Область применения: управляемое приложение. #std642

1. Следует избегать длительных вызовов из клиентского кода в серверный. Все длительные серверные вызовы (более 8 секунд в обычных сценариях) следует выполнять асинхронно, с помощью фонового задания. К таким операциям относятся: формирование отчета, групповая обработка, загрузка/выгрузка данных, заполнение больших табличных частей и т.п.

Иначе: браузер может предложить прекратить сценарий; веб-сервер может вернуть 504; пользователь не может отменить операцию.

2.1. Общий подход к асинхронному выполнению:
- Код длительной обработки — в модуле менеджера или общем модуле; результат — во временное хранилище. Для фоновых заданий использовать процедуру-обертку в общем модуле, вызывающую процедуру через ОбщегоНазначения.ВыполнитьМетодКонфигурации().
- Запуск фонового задания на сервере, ожидание до 0,8 сек; при незавершении — возврат на клиент, подключение обработчика ожидания с периодической проверкой состояния задания (интервал опроса от 1 до 15 сек, коэффициент 1,4).
- На время операции пользователю отображается индикатор (для отчетов — свойство ОтображениеСостояния поля табличного документа).
- По завершении — загрузка результата из временного хранилища и обработка.

2.2. Асинхронное формирование отчета — для отчетов без СКД или с переопределенной процедурой формирования. Поведение должно быть похожим на отчеты на базе СКД: форма не блокируется; пользователь может изменить настройки и переформировать; при закрытии формы формирование прерывается.

3. При использовании БСП — функции и процедуры общих модулей ДлительныеОперации, ДлительныеОперацииКлиент, УстановитьСостояниеПоляТабличногоДокумента (ОбщегоНазначенияКлиентСервер).

3.1. Результат фонового задания помещается во временное хранилище на время жизни формы. Рекомендуется очищать временное хранилище сразу после получения результата. Если результат нужно сохранять между вызовами — передавать фиксированный адрес заранее инициализированного хранилища.

4. В алгоритмах, инициирующих запуск фоновых заданий или запись данных без участия пользователя, следует проверять, что не установлен монопольный режим; при монопольном режиме — блокировать такие действия.

5. Для длительных операций, требующих монопольного режима (обновление ИБ при первом запуске, удаление помеченных, выгрузка в файл и т.п.): сначала установить монопольный режим, затем запускать фоновое задание. На время выполнения блокировать весь интерфейс (форма ожидания с РежимОткрытияОкна = БлокироватьВесьИнтерфейс).

## Установка внешних компонент и расширений платформы

Область применения: управляемое приложение. #std700

- 1.1. Установка внешних компонент и расширений должна быть интерактивной. Пользователь самостоятельно принимает решение. В диалоге должно быть указано, для чего нужна компонента и что не будет работать без нее. Неправильно: безусловно вызывать УстановитьВнешнююКомпоненту. Правильно: задавать пользователю вопрос в явном виде (например: «Для продолжения работы требуется установить внешнюю компоненту… Нажмите "Установить". После установки нажмите "Продолжить".»).
- 1.2. Рекомендуется выводить предложение об установке перед выполнением прикладного действия (пользователь выбрал команду → проверка наличия компоненты → при отсутствии — информация и кнопка установки → после установки — продолжение).
- 1.3. При использовании БСП для работы с файлами использовать процедуры ФайловаяСистемаКлиент: ВыбратьКаталог, ЗагрузитьФайл, ЗагрузитьФайлы, ОткрытьФайл, СохранитьФайл, СохранитьФайлы — вместо соответствующих методов глобального контекста и ДиалогВыбораФайла. В остальных случаях — ПодключитьРасширениеДляРаботыСФайлами.
- 2. В прикладном решении должны быть предоставлены инструменты для установки внешних компонент и расширений в любой момент (в т.ч. из административного режима). При БСП — общая команда УстановитьРасширениеРаботыСФайлами в форме персональных настроек.

## Общие рекомендации (размеры экрана 1280x768, DPI)

Область применения: управляемое приложение.

**Размеры экрана** #std727

1. Типовое разрешение экрана — 1280×768 пикселей. Разработку вести в стандартном разрешении 96 DPI.
2. Условия: основное окно растянуто на весь экран; панель задач видна (уменьшает высоту на 40 px); в браузере высота сокращается на ~60 px.
3. Область, доступная для работы с программой, — 1280×668 пикселей.
4. При типовом разрешении элементы внутри форм должны помещаться без вертикальной и горизонтальной полос прокрутки. Исключение: списки (вертикальная прокрутка допускается, горизонтальная нежелательна).

**Общие рекомендации (8.3)** #std566

1. Типовое разрешение — 1280×768 pt. Разработку вести в 96 DPI.
2. Основное окно растянуто на весь экран, видна панель задач.
3. Не рекомендуется вносить массовые изменения, нарушающие умолчание платформы (для контекстных меню — умолчание, в специфических случаях допускаются изменения).
4. Рекомендуется использовать Чек-лист проверки интерфейса.

## Командный интерфейс

Область применения: управляемое приложение. #std600

Командный интерфейс (КИ) — средство навигации по функциональности конфигурации. КИ создается панелью разделов, панелью навигации и панелью действий. При проектировании учитывать: все три элемента связаны контекстом и предназначены для повышения эффективности работы и быстрого освоения. Критерии: скорость работы, скорость обучения, удовлетворенность пользователя.

**Общие принципы (8.2):** КИ включает панель разделов, панель функций текущего раздела, меню функций, команды навигации и действий. Количество элементов — без прокрутки при стандартном разрешении. Элементы — в порядке убывания важности и частоты. Не располагать рядом команды с совпадающими первыми символами в названиях. Проектировать КИ с учетом мнения пользователей.

## Панель разделов

#std712 #std711

Область применения: управляемое приложение. Панель разделов может выводиться как отдельная панель или в составе меню функций.

1. Состав панели: «лицо» программы; состав и порядок разделов проектировать тщательно. Количество и состав разделов должны соответствовать реальным участкам работ. Раздел «Главное» по умолчанию первый. Списки с условно-постоянной информацией размещать внутри раздела, в специальном разделе (Справочники, Настройки, НСИ) или в ПН формы подчиненного списка. Разделы для настройки и администрирования — в конце панели.

## Названия разделов

- 2.1. Общая длина названия не более 35 знаков с учетом пробелов (при большем — многоточие). Названия примерно одной ширины.
- 2.2. Названия конкретные и запоминающиеся.
- 2.3. По возможности не использовать длинные слова; комбинировать слова (1–2 средних; одно среднее и короткое; одно длинное и короткое; два коротких и длинное).
- 2.4. Только общеупотребительные сокращения и аббревиатуры (НДС, МСФО); расшифровка во всплывающей подсказке.
- 2.5. Не рекомендуется раздел с названием «Сервис» (перекликается с меню и группой «Сервис»).

(8.3): длина до ~50 знаков (~150 pt при 96 DPI); панель отображает до двух строк с переносом без разрыва слов.

## Картинки разделов

- 3.1. Названия разделов рекомендуется выводить в режиме «Картинка и текст».
- 3.2. Картинки разными по начертанию и ведущим цветам, в одном стиле, с одинаковым направлением света, во фронтальной плоскости.

(8.3): картинка 48×48 pt, PNG с альфа-каналом. В режиме «только картинки» панель должна выглядеть ровно и гармонично.

## Навигация внутри раздела

Область применения: управляемое приложение. #std714

Навигация внутри раздела — через «Панель функций текущего раздела» или область навигации и действий в составе «Меню функций». Панель функций рекомендуется в конфигурациях с небольшим количеством команд в разделе; при сложной структуре — скрывать. Область команд показывает все возможности раздела с возможностью группировки.

1. Состав команд: переходы к спискам и журналам документов, спискам первичных данных, рабочим местам, специальным обработкам (типа списков), отчетам. Второстепенные объекты можно не выносить в КИ.
2. Названия команд: до 38 символов (лучше до 30 для важных). Начинаться с разных слов.
3. Группировка: по смыслу для пользователя, не по техническим признакам. В группе не более 5–6 команд. Не делать группы из одной команды. Группам добавлять название; не пересекаться с системными («Создать», «Отчеты», «Сервис», «Важное», «См. также»).

## Формы списков

Область применения: управляемое приложение.

- Состав и порядок колонок — по прогнозируемому сценарию. Отбор и группировка колонок в несколько этажей без горизонтальной прокрутки.
- Фиксировать колонки там, где уместно (например, «№» и «Номенклатура» в списках документов).
- Колонки «с запасом» делать по умолчанию невидимыми (для настройки пользователем); для табличных частей не применять.
- По умолчанию активная колонка — та, по которой поиск наиболее вероятен.
- Не рекомендуется режим выделения «строка — строка».

**Размеры списков** #std618: высота по умолчанию — с учетом типового количества строк. Высота строки при обычном шрифте — 19 pt. Ширина колонки ≈ количество символов / 1,25.

**Быстрые отборы:** использовать при частом использовании списка и типовых «запросах»; при большом количестве объектов; при частотной операции выбора/поиска. Поля отбора — перед списком (сверху/слева/справа); необязательные отборы — кнопка очистки; без кнопки открытия.

**Заголовки списков:** если список с командной панелью — заголовок в виде группы с рамкой «линия»; без командной панели — положение «верх». Количество элементов — когда уместно, в круглых скобках после заголовка; в пустом списке не выводить.

Область применения: управляемое приложение.

- Рекомендуется не создавать формы выбора, а использовать генерируемые платформой по умолчанию.
- Если нужен особый состав команд или колонок: в командной панели — минимально необходимый набор (выбор, создание, поиск/отбор); в часто используемых формах выбора из больших наборов — область быстрого отбора/поиска; состав колонок — для быстрого визуального поиска.

## Формы пошаговых помощников (мастеров)

Область применения: управляемое приложение. #std589 #std612 #std696

Пошаговые помощники используются для последовательного, контекстно-зависимого ввода данных (помощник ввода партнера, приема сотрудника, создания опроса, обновления системы и т.д.).

1. Расположение и оформление кнопок: командная панель в нижней части формы. На каждом этапе одна кнопка — по умолчанию (визуально: желтый фон, жирный шрифт).
2. Состав кнопок: «Далее» (с «>»), кнопка подтверждения ввода, «Закрыть», «Назад» (с «<», слева, не на первом экране), «Отмена», «Справка». «Назад» — крайняя слева; «Отмена» — между кнопкой подтверждения и «Справка»; «Справка» — крайняя справа. Кнопка подтверждения — глагол или «Готово»; на промежуточном экране — с «>», на последнем — без «>». При «Отмена» при вводе данных — вопрос о сохранении.

## Тексты

Область применения: управляемое приложение. #std598 #std645

**Общие рекомендации:** избегать технологических терминов и аббревиатур; единообразие понятий; короткие предложения; главная мысль — первой; небольшие абзацы; проверка правописания и грамматики. Поле ввода и подпись — одинаковый шрифт (начертание, размер, жирность), не цвет. В интерфейсных текстах — только прямые кавычки. Буква «ё» — при возможности неправильного прочтения, в собственных именах; иначе «е».

**Названия и заголовки:** избегать неоднозначности; не более 60 символов. Термин «родитель» — для родственных отношений физлиц, не для иерархии объектов ИБ (использовать: группа товаров, головное подразделение, входит в группу и т.д.). Если выводится количество элементов в заголовках — применять везде единообразно.

**Многоточие:** использовать, если название команды/ссылки отражает действие в форме инфинитива или отглагольного существительного, а при нажатии — переход к форме для выполнения. Многоточие примыкает к последнему слову без пробела. Если название описывает запускаемый процесс — многоточие не нужно.

**Концевые пробелы:** не допускается использование концевых пробельных символов в заголовках элементов управляемых форм и в представлениях объектов (представление, расширенное представление объекта, списка, записи).

**Одинаковые тексты:** рекомендуется не размещать в форме элементы управления с одинаковым текстом (путаница, затруднение справки и объяснения действий).

## Элементы стиля

Область применения: управляемое приложение. #std660 #std665 #std667

Оформление по умолчанию задается платформой; умолчаний придерживаться в большинстве случаев. Для визуального выделения элемента использовать элементы стиля, а не задавать значения непосредственно в элементах — чтобы аналогичные элементы выглядели одинаково во всех формах. Виды: Цвет (RGB), Шрифт (вид, размер, начертание), Рамка (тип и ширина границ).

1. Элементы стиля использовать всегда, когда нужно изменить оформление по умолчанию (Цвет, Шрифт, Рамку). Например, информационные надписи — элемент стиля «ИнформационнаяНадпись», а не RGB.
2. Не использовать элементы стиля для подмены оформления по умолчанию платформы (например, цвет гиперссылок — из платформы).
3. Каждый элемент стиля — для конкретной ситуации. Для другой ситуации — отдельный элемент стиля.
4. Название элемента стиля должно отражать назначение.
5. Для элементов с одинаковым названием, но разным видом — включать вид («Цвет», «Шрифт», «Рамка») в название; вид указывать после названия. Пример: ТекстНевыбраннойКартинкиЦвет и ТекстНевыбраннойКартинкиШрифт. В названии указывать только фактически используемый вид.
