## Phase 0 — Диагностика и бутстрап

Агент **молча** проверяет наличие ключевых каталогов и файлов (Glob/Read) и выводит пользователю **чеклист состояния**:

### Шаг 1 — Сбор состояния

Проверить (Glob):

| Что | Как проверить | Статус |
|---|---|---|
| Выгрузка конфигураций | `src/*/cf/Configuration.xml` — каждый найденный `Configuration.xml` = одна конфигурация | `[V]` / `[X]` |
| Выгрузка расширений | `src/*/cfe/*/Configuration.xml` — каждый найденный = одно расширение | `[V]` / `[X]` |
| Каталог OpenSpec | `openspec/` (наличие каталога) | `[V]` / `[X]` |
| Промпты агентов | `.cursor/agents/*.md` (количество файлов) | `[V]` / `[X]` |
| Правила | `.cursor/rules/*.mdc` (количество файлов) | `[V]` / `[X]` |
| Скиллы | `.cursor/skills/*/SKILL.md` (количество файлов) | `[V]` / `[X]` |
| AGENTS.md | `AGENTS.md` в корне | `[V]` / `[X]` |

**Дополнительно: KB состояние.** Отдельный мини-чеклист по `openspec/knowledge/` (проверять каждый запуск `/init-project`, не только при отсутствии `openspec/`):

| Что | Как проверить | Критерий валидности | Статус |
|---|---|---|---|
| Каталог `openspec/knowledge/` | Glob/наличие | Каталог существует | `[V]` / `[X]` |
| `openspec/knowledge/_index.yaml` | Read + parse YAML | Парсится; есть `version: 1`; есть поле `facts:` (даже если пустое) | `[V]` / `[X]` |
| `openspec/knowledge/_taxonomy.yaml` | Read + parse YAML | Парсится; есть `version`; `domains:` непустой | `[V]` / `[X]` |
| `openspec/knowledge/README.md` | Read | Файл существует и непустой | `[V]` / `[X]` |
| `openspec/knowledge/_archive/` | Glob | Каталог существует | `[V]` / `[X]` |
| Whitelist соблюдён | Glob по `openspec/knowledge/` | Только `<domain>/`, `_archive/`, служебные `_index.yaml` / `_taxonomy.yaml` / `_taxonomy.template.yaml` / `README.md` (см. `.cursor/rules/knowledge-format.mdc` секция РАСПОЛОЖЕНИЕ) | `[V]` / `[X] <список нарушений>` |

### Шаг 2 — Вывести чеклист

```
Состояние проекта:
  [V] src/<Конфиг1>/cf/     — конфигурация <Имя> (<Версия>)
  [V] src/<Конфиг1>/cfe/<Расш1>/  — расширение <Имя>
  [V] src/<Конфиг1>/cfe/<Расш2>/  — расширение <Имя>
  [X] openspec/              — нет каталога OpenSpec
  [V] .cursor/agents/        — N промптов агентов
  [V] .cursor/rules/         — N правил
  [V] .cursor/skills/        — N скиллов
  [V] AGENTS.md              — индекс

KB состояние:
  [V] openspec/knowledge/                — каталог
  [V] openspec/knowledge/_index.yaml     — индекс (valid, N фактов)
  [V] openspec/knowledge/_taxonomy.yaml  — таксономия (M доменов)
  [V] openspec/knowledge/README.md       — описание модели
  [V] openspec/knowledge/_archive/       — каталог архива
  [X] openspec/knowledge/<нарушение>     — посторонний файл/каталог (см. шаг 3)
```

Имена и версии конфигураций/расширений — из `Configuration.xml` (теги `<Name>`, `<Synonym>`, `<Version>`).

### Шаг 3 — Действия по каждому `[X]`

| Чего нет | Действие |
|---|---|
| `src/` пуст или нет `Configuration.xml` | **СТОП.** Подсказать: «Выгрузите конфигурацию через Конфигуратор: Конфигурация → Выгрузить конфигурацию в файлы. Путь: `src/<ИмяКонфигурации>/cf/`. Для расширений: Конфигурация → Расширения → <Имя> → Выгрузить в файлы. Путь: `src/<ИмяКонфигурации>/cfe/<ИмяРасширения>/`.» Ждать, пока пользователь выгрузит |
| `openspec/` | Агент создаёт сам: `openspec/`, `openspec/changes/`, `openspec/changes/_template/`, `openspec/specs/`, `openspec/docs/`, `openspec/adrs/`, `openspec/knowledge/` (+ `_archive/` с `.gitkeep`, стартовый `README.md` из шаблона `.cursor/skills/openspec-knowledge-init/templates/README-template.md`, пустой `_index.yaml` с `version: 1`, `facts: []`, актуальный `updated-at`), пустой `openspec/project.md`, `openspec/config.yaml`. `openspec/knowledge/_taxonomy.yaml` создаётся в Phase 5 (после разведки архитектуры) |
| `.cursor/agents/` | Подсказать: «Скопируйте промпты агентов из шаблонного проекта в `.cursor/agents/`» |
| `AGENTS.md` | Агент создаёт |

**Действия по `[X]` в KB-чеклисте (self-healing).** Выполняются автоматически при каждом запуске `/init-project`, без ручных действий пользователя за пределами AskQuestion:

| Чего нет / что не так | Действие |
|---|---|
| Нет `openspec/knowledge/` | Создать каталог + `_archive/` (с `.gitkeep`), а также служебные файлы из строк ниже |
| Нет `_index.yaml` или невалиден (не парсится / нет `version: 1` / нет `facts:`) | Если файл есть, но невалиден — сделать backup в `temp/backups/_index-<timestamp>.yaml`. Затем создать/перезаписать минимальный: `version: 1`, `facts: []`, актуальный `updated-at` (ISO-now) |
| Нет `README.md` или пустой | Создать из стандартного шаблона `.cursor/skills/openspec-knowledge-init/templates/README-template.md` (Read шаблона → Write в `openspec/knowledge/README.md`) |
| Нет `_archive/` | Создать каталог + `.gitkeep` |
| Нет `_taxonomy.yaml` | Если Phase 5 будет выполнена — оставить (сгенерируется в Phase 5 шаг 6). Если Phase 5 пропускается (по решению пользователя или при повторном запуске с уже существующей `architecture.md` без обновления) — выдать предупреждение в Phase 6 и предложить запустить `/opsx:knowledge-init` отдельно |
| Whitelist нарушен (посторонние файлы / каталоги в корне `knowledge/`) | AskQuestion: «В `openspec/knowledge/` найдены файлы вне формата: <список>. Для каждого выберите: Перенести в `temp/reports/` / Перенести в архив соответствующего ЗНИ (если узнаваемо) / Оставить (только подтверждение, тогда warning сохраняется)». По выбору пользователя выполнить перенос |

**Логика перехода:**
- Все `[V]` (включая KB-чеклист) → Phase 1.
- `[X]` только в openspec/ / AGENTS.md / KB → агент создаёт/чинит и продолжает в Phase 1.
- `[X]` в src/ (нет выгрузки) → СТОП, дать инструкции, ждать.
- `[X]` в .cursor/agents/ — предупреждение (агенты не будут работать), но не блокирует.
- `[X]` в KB whitelist (посторонние файлы) — не блокирует, но требует AskQuestion из таблицы выше.

### Эталонная структура каталогов

```
src/
  <ИмяКонфигурации>/
    cf/                — выгрузка конфигурации (XML)
    cfe/
      <ИмяРасширения>/ — выгрузка расширения (XML)
```

Одна конфигурация → один подкаталог в `src/`. Несколько конфигураций → несколько подкаталогов. Расширения вложены в `cfe/` соответствующей конфигурации.

### Шаг 4 — Документация

Поискать в `docs/` файлы с проектной документацией (функциональные требования, ТЗ, концепции). Если найдены — предложить пользователю использовать их как основу:

> Нашёл проектную документацию: `docs/<файл>`. Хотите использовать её как основу? Я извлеку информацию и предложу вам подтвердить/скорректировать.

Если пользователь согласен — извлечь данные и предзаполнять ответы в Phase 3.

## Phase 1 — Разведка кодовой базы

Перед интервью агент **молча** (без вопросов пользователю) собирает контекст:

- Структура `src/*/` — какие конфигурации и расширения, их `Configuration.xml` (название, синоним, версия, назначение, префикс)
- Бизнес-процессы, HTTP-сервисы, общие модули (по Glob в каждой конфигурации и расширении)
- Имена ключевых объектов метаданных (справочники, документы, регистры) из структуры каталогов или ConfigDumpInfo.xml
- Файлы документации в `docs/` (если есть)
- Паттерны комментариев в `.bsl` расширений: Grep по `*.bsl` в каталогах `cfe` внутри `src/` (фактические пути — как в Phase 0; не ограничиваться только `src/*/cfe/`, если структура глубже). Искать повторяющиеся маркеры в однострочных комментариях `//`: `// +++`, `// ---`, `// НАЧАЛО`, `// КОНЕЦ`, `// Автор:`, шаблоны с датой, kebab-case идентификаторы сразу после `//`. Подсчитать число файлов с каждым заметным паттерном. Результат — для предзаполнения Блока 7.

Результат разведки — **внутренний контекст** агента. Не показывать пользователю целиком; использовать для предзаполнения вариантов ответов и точечных вопросов в Phase 3.

## Phase 2 — Триаж секций (при повторном запуске)

Если `openspec/project.md` уже существует:

1. Прочитать файл.
2. Оценить секции `project.md`. Для **типовых** секций (Продукт, Назначение, Область и т.д.) — по критериям:
   - **Пустая** — нет содержимого.
   - **Stub** — формальная отписка («уточнять позже»), тавтология (проблема = цель), менее половины ожидаемых пунктов (напр. 2 возможности при 4+ расширениях).
   - **Достаточная** — осмысленный контент, согласованный с кодовой базой.
   Для секции **«Форматы и соглашения по комментариям BSL»** — отдельно:
     - **Пустая:** секции нет или нет ни одной таблицы.
     - **Stub:** таблицы есть, но все строки — заглушки (*(пока нет)*, EX-001 с пометкой «пример» без подтверждения команды).
     - **Достаточная:** хотя бы одна реальная запись в whitelist или обязательном контроле, согласованная с пользователем (в т.ч. явное «формата нет» с пустыми таблицами и пометкой).
3. Показать пользователю **карту секций** со статусами и предложить:
   - Дополнить только пустые/stub
   - Пройти по всем с возможностью пропуска
   - Указать конкретные секции для доработки

Если project.md пуст или отсутствует — перейти к Phase 3 без триажа.

## Phase 3 — Цикл по блокам интервью

Для **каждого** блока выполнять три шага: **вопрос с предзаполнением → проработка тезиса → черновик секции и подтверждение**. Не переходить к следующему блоку без подтверждения пользователя.

### Шаг 1 — Вопрос с предзаполнением

Агент формулирует вопрос и **сразу** предлагает свой вариант ответа на основе разведки (Phase 1) и уже заполненных секций. Пример для блока 3:

> Я вижу в коде бизнес-процессы «Согласование» и «Подписание», HTTP-сервис `arsCompareFiles`, обработку «РедакторСкриптов». Предлагаю такие ключевые возможности:
> 1. Согласовывать договоры через БШИ
> 2. Подписывать документы ЭП через Контур.Диадок
> 3. Сравнивать версии файлов через веб-сервис
> 4. Писать и тестировать скрипты обработки документов
>
> Это верно? Что убрать, добавить, переформулировать?

### Шаг 2 — Проработка тезиса

Получив ответ пользователя (в т.ч. краткий), агент:

- Развёртывает тезис в полноценную формулировку
- Дополняет из кодовой базы (конкретные имена модулей, процессов)
- Раскрывает ключевые термины и аббревиатуры (БШИ, ЭП, ДО и т.п.)
- При необходимости задаёт **точечные** уточняющие вопросы **с вариантами ответа** (не открытые)

### Шаг 3 — Черновик секции и подтверждение

Агент показывает готовый текст секции в формате project.md и просит подтверждения. Только после «ок» или явной правки пользователя — фиксировать и переходить к следующему блоку.

---

### Блок 1 — Продукт

- Как называется проект / расширение / подсистема?
- Для какой конфигурации (1С:Документооборот, УТ, ERP и т.д.)?
- Одним предложением: что это делает?

Предзаполнить из разведки: имена конфигураций из Configuration.xml, синонимы, версии.

---

### Блок 2 — Бизнес-назначение

**Вопросы (конкретные, не открытые):**

- Как сейчас работают **без** этой системы? Что именно плохо? (проблема = текущее состояние)
- Что должно стать возможным **с** системой? (цель = желаемое состояние)
- Кто целевая аудитория? (юристы, исполнители, администраторы — можно несколько)

**Гейт:** проблема и цель не должны дублировать друг друга (нет тавтологии). Если «проблема = автоматизация», «цель = автоматизация» — переформулировать и показать варианты.

---

### Блок 3 — Ключевые возможности

- Перечислите 3–7 ключевых возможностей с точки зрения пользователя (что пользователь может делать?). Формулировки от первого лица.

Предзаполнить из разведки: бизнес-процессы, HTTP-сервисы, обработки, подсистемы.

**Гейт:** минимум 3 пункта. Если расширений больше, чем пунктов — предложить недостающие из разведки.

---

### Блок 4 — Принципы работы

- Опишите 3–5 принципов работы системы «с высоты птичьего полёта». Какие ключевые идеи заложены?

Предзаполнить из разведки и уже названных возможностей (интеграция только через БШИ, разделение расширений и т.п.).

**Гейт:** каждый принцип должен быть подкреплён наблюдением из кода или словами пользователя. Не записывать собственные домыслы без подтверждения. При подтверждении «ты правильно понял» — можно зафиксировать, при необходимости пометить источник: «из анализа кода» / «со слов пользователя».

---

### Блок 5 — Внешние зависимости и инфраструктура

- Внешние сервисы (API, облака)?
- Базовая конфигурация и версия?
- Серверное ПО (криптопровайдеры, внешние компоненты)?
- Сетевые требования (интернет, HTTPS, VPN)?

По каждому обнаруженному в разведке сервису/интеграции задать **конкретный** вопрос с вариантами, например: «Для подписания через Диадок нужен криптопровайдер (КриптоПро, VipNet) — какой у вас?»

**Гейт:** по каждому внешнему сервису — хотя бы «что за сервис и зачем». Пропущенные подвопросы — задать повторно с конкретным вариантом ответа, не писать «уточнять позже».

---

### Блок 6 — Структура репозитория

Агент **сам** строит таблицу из реальной структуры `src/*/` (путь / содержимое / редактирование) и принцип расширения (база — эталон, доработки в расширениях, Перед/После/Вместо). Формат:

```
src/
  <ИмяКонфигурации>/
    cf/                — выгрузка конфигурации (XML)
    cfe/
      <ИмяРасширения>/ — выгрузка расширения (XML)
```

Показывает пользователю и просит подтвердить или скорректировать.

Если обнаружено **более одного** расширения в `cfe/` — предложить пользователю в одном–двух предложениях на каждое расширение указать **назначение** (зона ответственности). Это пригодится для Phase 5 и для спецификаций в `openspec/docs/`.

---

### Блок 7 — Соглашения по комментариям в коде

Вопросы (с вариантами):

1. Есть ли принятый формат маркеров при вставке кода? Канон проекта — `domain_label` (1 фраза «что/зачем» на русском) в open-маркере; см. `openspec/project.md` § Канон маркеров и `.cursor/docs/marker-layers-guide.md`. Примеры:
   - cfe: `// +++ <ФИО> <дата> <domain_label>` / `// --- <ФИО>`
   - cf: `// {cfMarkerPrefix} <domain_label> +++`
   - Нет формата / каждый пишет по-своему
   - Другой (опишите)

2. **defaultDeveloper** и **cfMarkerPrefix** — заполнить в секции «Разработчик по умолчанию».

3. Есть ли другие обязательные шаблоны комментариев для whitelist?

**Предзаполнить из Phase 1:** если разведка нашла повторяющиеся паттерны в комментариях `//` (например `// +++`, `// ---`, `// НАЧАЛО`, `// Автор:`) — показать примеры и спросить: «Вижу маркеры вида … в N файлах. Это принятый формат команды?»

**Гейт:**

- Если пользователь указал формат — записать как минимум одну строку в таблицу «Обязательный контроль» (с regex, если формат формализуем) и/или в таблицу «Whitelist».
- Если пользователь ответил «нет формата» — записать пустые таблицы (заглушки) и явно отметить: формат не принят, строгая гигиена комментариев без исключений по whitelist.
- Не записывать пример EX-001, если пользователь его не подтвердил (удалить строку-пример или заменить на согласованное правило).

---

### Ключевые термины

Если в ответах есть аббревиатура или термин, не общеизвестный (БШИ, ЭП, ДО и т.п.) — **раскрыть** в project.md при первом упоминании и при необходимости согласовать определение с пользователем.

## Phase 4 — Запись project.md

После сбора и подтверждения всех блоков сформировать (или обновить) `openspec/project.md` по структуре:

```markdown
# OpenSpec: <название проекта>

<одно предложение описание из блока 1>

## Продукт

<название, целевая конфигурация, расширения, краткое описание>

## Назначение

**Проблема:** <текущее состояние без системы>
**Цель:** <желаемое состояние с системой>
**Аудитория:** <из блока 2>

## Область

### Ключевые возможности
<маркированный список, от первого лица>

### Принципы
<маркированный список: **заголовок** — пояснение>

### Техническая область
<интеграции, подсистемы, объекты метаданных — из контекста ответов>

## Внешние зависимости

<внешние сервисы, базовая конфигурация, серверное ПО, сеть>

## Структура репозитория

| Путь | Содержимое | Редактирование |
| ... | ... | ... |
```

## Phase 5 — Исследование архитектуры

После записи project.md агент **автоматически** запускает исследование кодовой базы и фиксирует текущую архитектуру.

**Шаг 1 — Бриф для исследования.** Сформировать структурированный бриф из заполненного project.md (расширения, назначение, ключевые возможности, принципы) и данных Phase 1 (обнаруженные модули, бизнес-процессы, HTTP-сервисы).

> Перед вызовом Task в шагах 2–3 — **Task Pre-call Checklist** из `.cursor/rules/tool-name-guard.mdc` (subagent_type из списка 1С-агентов; `model` — по `.cursor/rules/model-selection.mdc`).

**Шаг 2 — Делегирование onec-code-explorer.** Task tool, `subagent_type="onec-code-explorer"`. Задача: по каждому расширению и ключевым модулям базы — определить entry points, цепочки вызовов, зависимости между расширениями, ключевые общие модули. Передать бриф и пути к расширениям.

**Шаг 3 — Делегирование onec-code-architect.** Task tool, `subagent_type="onec-code-architect"`. По результатам explorer: карта компонентов (расширения, зоны ответственности, точки интеграции с базой), ключевые потоки данных (согласование, подписание и т.п.), Mermaid-диаграммы (component diagram, sequence для ключевых потоков), зависимости между расширениями.

**Шаг 4 — Показать пользователю ключевые находки.** Краткое резюме: количество модулей, основные потоки, неожиданные находки. Пользователь может скорректировать или дополнить.

**Шаг 5 — Записать `openspec/specs/architecture.md`.** Полный отчёт с диаграммами (по правилу preserve-subagent-reports — без сокращений). Структура:

```markdown
# Архитектура: <название проекта>

## Обзор
<1–2 абзаца из project.md + результаты исследования>

## Компоненты
<карта расширений и зон ответственности, Mermaid component diagram>

## Ключевые потоки данных
<для каждого ключевого сценария — Mermaid sequence diagram + описание>

## Зависимости
<между расширениями, от базовой конфигурации, от внешних сервисов>

## Ключевые модули
<таблица: модуль / расширение / назначение / тип>
```

**Шаг 6 — Генерация таксономии (`openspec/knowledge/_taxonomy.yaml`).** Шаг идемпотентен: безопасно запускать повторно.

1. **Вход:** карта расширений и границ доменов из выхода `onec-code-architect` (и при необходимости `openspec/specs/architecture.md`, `project.md`).
2. **Проверка идемпотентности.** Если `openspec/knowledge/_taxonomy.yaml` уже существует:
   - Сравнить набор расширений в `src/*/cfe/*` с `domains[*].source` в текущем `_taxonomy.yaml`.
   - **Если совпадает** — пропустить с сообщением «taxonomy актуальна, расширения совпадают, sync не требуется».
   - **Если расширения добавлены/удалены** — предложить sync через `/opsx:knowledge-init` (diff + merge), не перезаписывать вслепую.
3. **Если файла нет — автогенерация draft `_taxonomy.yaml`:** по одному `domain` на каждое расширение из `src/*/cfe/*` с корректным `source`; `subdomains` — из подсистем / ключевых общих модулей (эвристика + ручная правка в draft); обязательный блок `cross` из шаблона `openspec/knowledge/_taxonomy.template.yaml`.
4. **Confirm пользователя** перед записью (показ diff или краткого summary).

**При повторном запуске /init-project:** если `openspec/specs/architecture.md` уже существует — Phase 5 **предложить** обновить (если project.md изменился) или пропустить. Не пересоздавать автоматически без согласия. Шаг 6 (taxonomy) выполняется отдельно от пересоздания `architecture.md` — таксономия идемпотентна и проверяется всегда, даже если `architecture.md` не обновлялся.

## Guardrails

- **Не переспрашивай то, что уже заполнено и достаточного качества** в project.md — только дополняй или уточняй по триажу.
- **Не записывай свои домыслы без явного подтверждения** — принципы и формулировки подкреплять словами пользователя или наблюдением из кода; при подтверждении «ты правильно понял» можно фиксировать, при необходимости пометить источник.
- **Не принимай минимальный ответ** — если получено менее половины ожидаемых пунктов (напр. 2 возможности при 4 расширениях), предложить свои варианты из разведки.
- **Тавтология = stub** — если проблема и цель дублируют друг друга, переформулировать и показать пользователю.
- **Ключевые термины раскрывать** — каждая аббревиатура (БШИ, ЭП, ДО и т.п.) расшифрована в project.md при первом упоминании.
- **Не навязывай технологию** — если проект не на 1С, адаптировать вопросы (Phase 0 по-прежнему может проверять `Configuration.xml`; блок 7 — про BSL-комментарии: для других стеков пропустить или заменить эквивалентом).
- **Если есть документация в docs/** — максимально использовать её, минимизировать ручной ввод.
- **Русский язык** в project.md и в интервью.
- **Не пиши код** — только project.md и openspec/specs/architecture.md.
