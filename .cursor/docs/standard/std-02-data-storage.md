## Уточнение сущности объекта метаданных #std541

Область применения: управляемое приложение, мобильное приложение, обычное приложение.

1. При описании структуры прикладного решения следует использовать разные типы объектов метаданных для реализации разных по смыслу сущностей. Например, сущность «организации» описывается справочником Организации, а сущность «подразделение» — отдельным справочником Подразделения.

2. В то же время для уточнения сущности того или иного объекта у объекта могут быть заведены реквизиты, в зависимости от значения которых меняется его смысловая нагрузка и поведение. Например, в справочнике Организации могут одновременно храниться «обычные» организации и обособленные подразделения (т.е. организации, у которых имеется головная организация), а в справочнике Сотрудники — принятые на работу и уволенные сотрудники.

В этих случаях рекомендуется заводить отдельные реквизиты, которые бы однозначно определяли вид или состояние объекта. Это могут быть реквизиты типа Булево или перечисления с видами объекта. Не следует трактовать вид объекта по косвенным признакам, в частности в зависимости от заполнения того или иного реквизита.

**Неправильно:** заводить в справочнике Организации реквизит ГоловнаяОрганизация, в зависимости от заполнения которого прикладная логика трактует вид организации — «обычная» или обособленное подразделение.

**Правильно:** помимо реквизита ГоловнаяОрганизация завести в справочнике Организации булев реквизит ОбособленноеПодразделение, в зависимости от значения которого (Истина или Ложь) однозначно трактуется вид организации и в частности необходимость заполнения реквизита ГоловнаяОрганизация.

3. В то же время, если у справочника Сотрудники имеются реквизиты ДатаПриема и ДатаУвольнения, то введение пары реквизитов булево типа ПринятНаРаботу и Уволен не оправдано. Для подобных сущностей с несколькими состояниями целесообразно завести один реквизит типа перечисление со статусами, например: «работает», «уволен», состав которых при необходимости может быть расширен и другими значениями.

## Имя, синоним, комментарий #std474

- **1.1.** Синоним объекта должен быть определен так, чтобы осмысленно, лаконично описывать объект. Заполняется обязательно. Требование распространяется также на реквизиты объектов метаданных, табличные части, реквизиты табличных частей, измерения регистров, ресурсы и другие объекты конфигурации, у которых имеется синоним.

- **1.2.** Не рекомендуется в синонимах объектов использовать сокращения. Исключением являются только общеупотребительные и соответствующие целевой аудитории сокращения (например, Сумма (регл.)) и аббревиатуры (например, НДС или МСФО).

- **1.3.** В синонимах объектов и текстовых сообщениях пользователю должны использоваться общепринятые термины. Не должно быть сленга, искажения названий продуктов и компаний; англоязычных фраз, записанных русскими буквами; русскоязычных английскими буквами и т.п. В частности, если для англоязычного термина нет общепринятого перевода на русский язык, следует использовать оригинальный англоязычный термин. Например, правильно: «Загрузка данных из Microsoft Excel».

- **1.4.** Для стандартных реквизитов (стандартных табличных частей) также следует указывать синонимы, исходя из прикладного смысла каждого реквизита.

- **1.5.** Для стандартных реквизитов Родитель и Владелец следует всегда указывать синонимы, отличные от синонимов по умолчанию. Например, для справочника Файлы со стандартным реквизитом Владелец типа СправочникСсылка.ПапкиФайлов правильно вложить в синоним прикладной смысл: «Папка» или «Папка с файлом». Для справочника Файлы целесообразнее синоним «Имя файла», для справочника ФизическиеЛица — «ФИО».

- **1.6.** Если есть два (или более) объекта метаданных со схожим назначением, синонимы каждого объекта должны полностью описывать каждый объект. Например, правильно: «Банковские счета организаций» и «Банковские счета контрагентов». Для реквизитов табличной части «Товары» документа «Пересчет товаров» правильно: «Количество (в наличии)» и «Количество (по учету)». Для справочника «Номенклатура» правильно: «Рабочее наименование» и «Наименование для печати».

- **2.1.** Имя объекта рекомендуется строить на основе синонима: пробелы и недопустимые в имени символы удаляются, первые буквы слов — прописные. Допустимы ситуации, когда имя более кратко описывает объект, чем синоним, или наоборот. Имена не должны превышать 80 символов.

- **2.2.** Исключение: объекты с префиксом Удалить. Не переименовывать объекты при пересмотре синонимов, если на них распространяются требования обратной совместимости.

- **2.3.** Для подчиненных объектов (реквизиты, измерения, ресурсы) не использовать имена, совпадающие с именами объектов-владельцев. Не использовать имена, совпадающие с именами таблиц языка запросов (Документ, Справочник, РегистрСведений и т.д.).

- **3.1.** Комментарий задается только при необходимости дать пояснения участнику разработки. Комментарий начинается с прописной буквы, точки — только после сокращений.

- **4.** В именах, синонимах и комментариях не допускается использовать букву «ё».

## Подсказка и проверка заполнения #std478

Область применения (уточнение): управляемое приложение, обычное приложение.

- **1.1.** Свойство «Подсказка» задается для тех объектов (реквизитов, измерений, ресурсов), которые выводятся пользователю и требуют пояснения. Для реквизитов, используемых в ежедневной работе, подсказки не добавлять. Бессмысленные подсказки, автоматически генерируемые конфигуратором (например, «Группа шапка»), нужно очищать.

- **2.1.** Свойство «Проверка заполнения» для всех типизированных объектов метаданных и стандартных реквизитов/табличных частей, обязательных по логике объекта, должно быть установлено в «Выдавать ошибку». Например, у реквизита Клиент и табличной части Товары документа Заказ клиента — «Выдавать ошибку».

- **2.2.** Все ограничения и проверки по возможности должны быть описаны в метаданных. Если хотя бы один сценарий требует обязательного заполнения реквизита, свойство «Проверка заполнения» — «Выдавать ошибку». Условные случаи — в обработчике ОбработкаПроверкиЗаполнения. Не использовать обратную схему: «Не проверять» в метаданных и проверки только в обработчике.

- **2.3.** Если проверка заполнения зависит от условий, рекомендуется управлять автопометкой незаполненного значения с помощью условного оформления форм.

## Использование кодов (номеров) объектов конфигурации #std432

Длины кодов (номеров) задаются в зависимости от прикладного смысла.

**Основания для применения кодов (номеров):**
- 1.1. Пользователям предстоит работать со списками с большим количеством элементов (например, справочник Номенклатура).
- 1.2. Искать по коду (номеру) удобнее, чем по другим свойствам (например, справочник Статьи расходов).
- 1.3. Код имеет прикладной смысл (код Единицы измерения — ОКЕИ, Номера ГТД — номер ГТД и т.д.).

**Автонумерация не применяется:**
- 2.1. Код используется как краткое представление элемента (например, справочник Пользователи — логин).
- 2.2. Код (номер) приходит из внешних источников (классификаторы, Номера ГТД по входящим документам).

**Длина кода (номера):**
- 3.1. Для объектов с автонумерацией — исходя из потенциального количества объектов (в БД, по периоду, по владельцу). Учитывать длину префиксов нумерации. Рекомендуемый ряд длин: 3, 5, 9, 11. При подсистеме «Префиксация объектов» БСП — не менее 11 символов с учетом префикса.
- 3.2. Для кода как краткого представления — достаточная для хранения краткого строкового представления.
- 3.3. Для входящих данных — в зависимости от источника.
- 3.4. Допустимая длина кода (номера) рекомендуется переменная.

**Параллельный ввод (РИБ, другие программы):** при выполнении условий (строковый код/номер, автонумерация, данные могут вводиться параллельно и консолидироваться) должна быть реализована префиксация. БСП: подсистема «Префиксация объектов».

## Использование реквизитов строкового типа #std432, #std473

- **1.1.** Для реквизитов строкового типа использовать переменную длину строки (Допустимая длина = Переменная) и указывать максимально допустимую длину. Фиксированная длина — только когда нужна гарантия определенной длины (дополнение пробелами).

- **1.2.** Когда максимальная длина известна (регламентирована), указывать ее в свойстве Длина (Длина наименования для Наименования). Например, ИНН — 12 символов.

- **1.3.** Если строка — конкатенация других строк, длина может быть суммой длин исходных (например, адрес).

- **1.4.** Если длина не регламентирована, выбирать длину, достаточную в большинстве случаев (например, полное наименование контрагента — 250, полное имя файла — 260, ФИО — 100).

**Строки неограниченной длины допускаются:**
- 2.1. Пользовательский текст значительного объема (многострочные поля: дополнительное описание, комментарий и т.п.).
- 2.2. Техническая информация, генерируемая программами (xml, заголовки сообщений и т.п.) — методическая рекомендация.

**Ограничения в языке запросов при неограниченной длине:**
- 3.1. При сравнении, группировке, получении различных — выражать как строку определенной длины, например: `ВЫРАЗИТЬ КАК СТРОКА(1000)`.
- 3.2. В отчетах СКД задавать параметр «Тип значения поля» (закладка Наборы данных). Частое приведение к длине в запросах/СКД может указывать на неверное проектное решение — пересмотреть тип в пользу ограниченной длины.
- 3.3. В остальных случаях урезать строку в запросах не требуется.

**4.** В печатных формах выводить строковые поля полностью, без обрезания (например, адрес доставки).

Для выявления реквизитов неограниченной длины можно использовать обработку СтрокиНеограниченнойДлины.erf.

## Ограничения на использование реквизитов составного типа #std728

- **1.1.** Реквизиты составного типа, используемые в условиях соединений, отборах и для упорядочивания, должны содержать только ссылочные типы (СправочникСсылка.…, ДокументСсылка.… и т.п.). В состав типов не рекомендуется включать нессылочные типы (Строка, Число, Дата, УникальныйИдентификатор, Булево, ХранилищеЗначения) — производительность запросов снижается. См. «Особенности хранения составных типов данных» (ИТС).

- **1.2.** Альтернатива: вместо состава «ссылка + строка» (например, реквизит Адрес) предусмотреть отдельный справочник (например, ПроизвольныеАдреса), указывать его в реквизите, добавлять элементы автоматически при записи, удаление лишних — регламентным заданием.

- **1.3.** Исключение — таблицы с заведомо малым объемом данных (до 1000 записей).

- **2.1.** Для типизированных объектов метаданных, хранящихся в ИБ, не использовать составные типы ЛюбаяСсылка, СправочникСсылка, ДокументСсылка и аналогичные. Состав типов должен определяться явно. Исключение — универсальные механизмы, рассчитанные на произвольные ссылочные объекты. Избыточное использование сильно составных типов ведет к: неявным соединениям со всеми таблицами состава типа и деградации производительности; избыточной реструктуризации при удалении объектов; усложнению алгоритмов и анализа; неудобному выбору из большого списка типов на формах; избыточным блокировкам при удалении помеченных.

Область применения (уточнение): управляемое приложение, обычное приложение.

- **2.2.** При массовом использовании составного типа в подсистеме или конфигурации рекомендуется использовать определяемые типы. Исключение — ведущие измерения регистров универсальных (библиотечных) механизмов, где состав ЛюбаяСсылка/СправочникСсылка/ДокументСсылка оправдан.

См. также: Разыменование ссылочных полей составного типа в языке запросов.

## Требования к проведению документов #std603

1. Документы предназначены для ввода первичной информации о событиях, воздействующих на учитываемые показатели (хозяйственные операции, производственные операции и т.д.).

2. **2.1.** Регистрация события выполняется проведением документа. Большинство документов должны проводиться (свойство Проведение — Разрешить). Непроведенный документ — «черновик», не отраженный в учете; проведенный — «чистовик», участвующий в учете.

**2.2.** Если жизненный цикл документа состоит из нескольких этапов (статусы: Не согласован, К обеспечению, Закрыт и т.п.), проведение соответствует моменту первичного отражения в учете, статусы уточняют, как событие отражено. До проведения перевод по статусам не контролируется; после проведения при смене статусов могут требоваться дозаполнение и проверки.

**2.3.** Исключения (документы не проводятся): документы только для регистрации событий (входящая корреспонденция, звонки, встречи); документы с технологией, сильно отличающейся от возможностей платформы, но выглядящие для пользователя как проводящиеся (например, Операция, Регламентная операция).

**2.4.** Если регистрация и отражение в учете должны выполняться за одно действие — записывать новый документ в режиме проведения. Недопустимо отключать проведение у документа для этой цели.

3. «Вторичные» данные со сложными привязками к времени и объектам следует помещать в регистры. Движения формировать при проведении — автоматически или вручную (ручные операции для начальных остатков и непредусмотренных операций).

4. В отдельных случаях формирование движений выполняется отдельным документом (схожая обработка разных видов, групповая обработка, разделение функций исполнителей). В цепочке документов движения формируют только определенные документы (например, Платежное поручение не делает движений, движения — документ Списание с расчетного счета).

5. Непроведенные и помеченные на удаление документы не должны иметь активных движений.

6. Даже если документ не формирует движений, он должен проводиться, чтобы логически отличаться от черновика.

7. **7.1.1.** Если в документе не менялись данные, влияющие на проведение (например, только Комментарий), перепроведение не должно менять движения. Исключение — когда движения формируют внешние по отношению к документу алгоритмы (п. 4).

**7.1.2.** Избегать решений, при которых результат формирования движений зависит от состояния учетных регистров (например, остатков), т.к. результат будет зависеть от порядка ввода. Исключение — обоснованные случаи (например, партионный учет).

**7.2.** Документы при формировании движений должны по максимуму опираться на данные самого документа. Данные, не сохраняемые в документе, должны быть защищены от изменения:
- 7.2.1. Внешние данные (реквизиты НСИ), влияющие на движения, либо сохранять в документе, либо блокировать изменение (в БСП — механизм «Запрет редактирования реквизитов»).
- 7.2.2. Минимизировать влияние настроек (функциональных опций) на формирование движений: дата начала действия настройки, значения по умолчанию для отключенных полей, формирование движений без учета настройки с сокрытием в отчетах и т.п.
- 7.2.3. Если зависимость от настроек неизбежна: автоматическая фоновая обработка после изменения настройки с предупреждением; или ручной запуск обработки с уведомлением перед изменением настройки и в интерфейсах по затронутым данным; либо предупреждение о нежелательности изменения после начала учета. Допустима разная реакция на включение и отключение настройки.

**7.3.** При изменении логики формирования движений для выполнения п. 7.1 необходимо предусмотреть обработчики обновления ИБ либо поддержку старых алгоритмов формирования движений для существующих документов.

8. Для большинства событий отражение в учете должно быть обратимым — использовать механизм отмены проведения.

См. также: Имена объектов метаданных в конфигурациях, Порядок записи движений документов, Самодостаточность регистров.

## Использование активности движений #std633

Если в конфигурации применяется переключение активности движений определенных регистров (например, для ручной корректировки):

1. Активность записей таких регистров должна учитываться во всех запросах и отчетах. Для игнорирования неактивных записей в условие выборки добавлять:

```bsl
ГДЕ Активность = ИСТИНА
```

В виртуальных таблицах регистров платформа учитывает только активные записи автоматически.

2. Активность должна учитываться в универсальных отчетах и универсальной бизнес-логике, работающей с произвольными регистрами.

3. При отмене проведения документов с ручной корректировкой движений следует отключать активность движений, а не удалять их (например, документ «Операция (бухгалтерский и налоговый учет)»).

См. также: Требования к проведению документов, Порядок записи движений документов.

## Самодостаточность регистров #std477

Регистр должен быть логически независим от регистраторов. Любая логика и отчеты по регистру не должны обращаться к полям регистратора; данных самого регистра должно быть достаточно.

Обращение к полям регистратора «через точку» приводит к неявному соединению с дополнительными таблицами. В РИБ регистратора может не быть (движения мигрируют между узлами, регистраторы — нет).

См. также: Разыменование ссылочных полей составного типа в языке запросов, Разработка планов обмена с отборами.

## Реквизит «Комментарий» у документов #std531

Методическая рекомендация (полезный совет).

- **1.** Для всех документов рекомендуется создавать реквизит Комментарий (строка неограниченной длины) для служебных заметок (причина пометки на удаление и т.п.). Доступ к реквизиту — как к самому документу (только чтение / запись вместе с документом).

- **2.** Для произвольной текстовой информации по прикладному сценарию предусматривать отдельные реквизиты (например, в Заказе клиента — «Дополнительная информация»), а не использовать служебный Комментарий.

- **3.** В простейшем случае для редактирования комментария — ВвестиСтроку. В БСП — процедура ПоказатьФормуРедактированияКомментария общего модуля ОбщегоНазначенияКлиент.

См. также: Поля «Ответственный» и «Комментарий» (8.2), (8.3).

## Использование констант

Запись константы выполнять вне транзакций, т.к. запись может стать узким местом при конкурентной работе.

**Неправильно** (увеличение счетчика в константе внутри проведения):

```bsl
Процедура ОбработкаПроведения(Отказ, РежимПроведения)
    ТекущееЗначение = Константы.СчетчикПроведенныхДокументов.Получить();
    Константы.СчетчикПроведенныхДокументов.Установить(ТекущееЗначение + 1);
КонецПроцедуры
```

На время записи значения в константу работа других сеансов, записывающих ту же константу, приостанавливается. Подробнее: база знаний «Технологические вопросы крупных внедрений». Недопустимо решать проблему блокировок тотальным кешированием констант в общих модулях с повторным использованием возвращаемых значений.

## Работа с неактуальными (недействительными) объектами #std632, #std638

Методическая рекомендация (полезный совет).

- **1.** Рекомендация действует, когда объект ИБ перестает быть актуальным (навсегда или на время): сотрудник уволился, подразделение расформировано и т.д. Удалять объект недопустимо из-за ссылочной целостности (например, поле Автор типа Файл должно хранить ссылку на уволившегося сотрудника).

- **2.** Для запрета выбора неактуальных объектов в полях ввода (автоподбор, быстрый выбор) — один из двух подходов. Пример: справочник Пользователи, реквизит Недействителен (Булево), по умолчанию Ложь.

**2.1.** Запрет по умолчанию во всех или большинстве полей:
- 2.1.1. В модуле менеджера справочника реализовать ОбработкаПолученияДанныхВыбора и ОбработкаПолученияФормы с установкой отбора (например, Недействителен = Ложь). При необходимости передавать выбранную форму выбора и обновлять параметры.
- 2.1.2. Там, где поведение нужно изменить (выводить всех пользователей или другое ограничение), явно задавать «Параметры выбора» и «Связи параметров выбора»: например, Отбор.Недействителен(Ложь) или оба значения Истина/Ложь при отсутствии ограничения.

**2.2.** Запрет зависит от контекста — не задавать по умолчанию. В объектах с реквизитами типа СправочникСсылка.Пользователи задавать «Параметры выбора» и «Связи параметров выбора» по контексту. Если критерий не описывается параметрами выбора — обработчики формы ОбработкаПолученияДанныхВыбора, ОбработкаВыбора, ОкончаниеВводаТекста и/или отдельная форма выбора с той же логикой.

- **3.** В формах списка и выбора рекомендуется флажок «Показывать недействительных пользователей» для выбора/открытия карточки и перевода объекта снова в действительные.

- **4.** Для отображения неактуальных объектов в списках рекомендуется элемент стиля ТекстЗапрещеннойЯчейкиЦвет (192, 192, 192).
