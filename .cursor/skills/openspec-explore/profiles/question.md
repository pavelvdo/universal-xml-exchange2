# Профиль `explore-question` (вопрос / справка)

Ответ — в чате; `/opsx:new` **не** предлагается по умолчанию. Полный отчёт `Task` сохраняется в `temp/reports/exploration-YYYY-MM-DD-<slug>.md`.

## Подсказки для брифа (слот **Вопрос**; в промптах — `user-goal`)

| Типичная цель заказчика | Критерий «решено» |
|-------------------------|-------------------|
| Получить ответ да/нет | Есть прямой ответ |
| Понять механизм для своего решения | Могу применить к своему случаю |

## Типичные предметы

- Точка входа механизма.
- Внутренняя реализация (модули, регистры).
- Параметры и настройки.
- Связанные сценарии.
- Применимость к кейсу пользователя.

## Маршрут

Обычно 1, реже 2 `Task` → `onec-code-explorer`. Лимит ≤ 3 `Task` (см. SKILL.md).

1. `Task` → `onec-code-explorer` с узкой целью (точка входа / реализация / применимость).
2. Превью 8–15 строк в чате + «Итог / Вердикт / Дальше».
3. (Опционально) Второй `Task` — если первый поднял уточняющий вопрос.

## Финал

В чате — блок «Итог / Вердикт / Дальше», где **Дальше** — обычно «достаточно» или один уточняющий вопрос. Файл-handoff и блок `## Постановка ЗНИ` — **не** обязательны.

Если в ходе ответа пользователь говорит «теперь починим» / «давай заведём ЗНИ» — переключиться на bug-профиль (`profiles/bug.md`): сформировать блок `## Постановка ЗНИ` из накопленного контекста и предложить `/opsx:new <name>`.

## Постановка фичи (не путать)

Маркеры «нужно сделать», «добавить», «изменить поведение» — это **не** `explore-question`. Перенаправить на `/opsx:new <name>`. Если непонятно, что трогать — короткая разведка через `explore-question` (1–2 explorer), затем `/opsx:new`.

## Handoff после ответа

- Свод в чате — основной результат; файл-handoff — только по словесной просьбе.
- Полный отчёт `Task` живёт в `temp/reports/`, доступен для `/opsx:extend <change> --from-report <path>` при необходимости.
