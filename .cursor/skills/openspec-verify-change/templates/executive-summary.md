# Структура файла `reports/verification-YYYY-MM-DD.md`

**Принцип.** Файл читается **сверху вниз** разработчиком 1С, который **не открывал** артефакты OpenSpec и **не знает** про слои verify. Поэтому первый видимый раздел — на языке кода 1С, без жаргона движка. Технический аудит со статусами слоёв уходит вниз, в отдельную секцию, как блок для самого OpenSpec и для будущих verify-прогонов.

## Порядок блоков (строго)

1. **YAML frontmatter** — `verdict`, `layer_status`, `snapshot`, `verify_mode`. Нужен фильтру новизны и аудиту. **Не для чтения человеком.** Формат — `templates/report-header.md`.
2. **`## Резюме для разработчика`** — первый видимый раздел. Тот же текст, что в чате (`templates/chat-summary.md`), плюс 1–2 абзаца контекста: какие модули / расширения / процедуры / директивы задействованы. **Здесь только язык кода 1С.**
3. **`## Что доработать в постановке`** — только при NO-GO. Содержит рекомендации (пунктами) и опционально развёрнутые развилки прозой, если есть равноправные альтернативы. Если вердикт «можно запускать apply» — секция **отсутствует**.
4. **`## Что меняется в постановке`** — что план меняет в коде, какие модули / процедуры / директивы затрагиваются, ссылки на ADR / KB / архив (если есть). Это «карточка изменений» уровня архитектора для разработчика, не цитирование `design.md`.
5. **`### Подправил в постановке`** — список авто-правок гигиены (если были). Простыми словами: «убрал пустые чекбоксы», «закрыл маркер среза», «нормализовал отступы». По одной строке на правку. Если правок не было — секция отсутствует.
6. **`### К сведению`** — мелочи, не влияющие на запуск apply (старый формат приёмки, отсутствие подсистемы и т.п.). По одной строке.
7. **`## Технический аудит (для движка OpenSpec)`** — статусы слоёв, авто-правки в техническом формате, развёрнутые карточки развилок (см. шаблон ниже). **Только здесь** появляются имена `Layer 1..5`, `PASS`/`WARNING`/`FAIL`, `CHALLENGE`/`APPROVE`/`REJECT`, `design-challenge`, `task-readiness`. Это блок для самого OpenSpec, не для первого чтения.
8. **`## Источники`** — пути к `quality-control-*.md`, `design-challenge-*.md`, `architecture-task-readiness-*.md`, технические коды алертов.

## Шаблон секции `## Резюме для разработчика`

```markdown
## Резюме для разработчика

<change-name> — <можно запускать apply | до старта нужен ваш выбор по логике <тема> | apply пока нельзя: не удалось автоматически дописать постановку за 2 итерации>.

**Следующий шаг:** <тот же текст, что в чате — `/opsx:apply`, `/opsx:archive`, «ответьте в чате», или `/opsx:extend`>

<2–4 предложения «архитектор — разработчику»: что именно делает план в коде. Имена общих модулей, процедур, объектов метаданных, директив расширения, реквизитов — в backticks.>

<При вердикте «можно apply» + авто-правки — одна строка «Подправил в постановке: <что>».>

<При наличии нюансов к сведению, не блокирующих apply — одна-две строки.>
```

**Правила:**

- Первый абзац — **зеркало чата**. Один в один тот же текст, что выдан пользователю. Различия запрещены.
- При `decision_round >= 1` — после первого абзаца **тот же** блок «Уже зафиксировано» / «Новый вопрос», что в `templates/chat-summary.md` (прозой, без `D0`–`Dn`, `decision_id`, `closed_decisions`).
- При GO-saturated — та же формулировка «можно apply; остаточный риск … проверяется в S1.accept», **без** «GO-with-assumptions» и без agent-keys.
- Без слов «согласованность», «реализуемость», «слой», «когерентность», «PASS/FAIL», `Layer N`, `verdict`.
- Backtick-цитирование имён 1С обязательно: `ОбщийМодуль`, `ИмяПроцедуры`, `&Вместо`, `СсылкаШаблон`, `Тип("СправочникСсылка.ИмяСправочника")`.

## Шаблон секции `## Что доработать в постановке`

Используется **только** при `verdict: NO-GO`. Содержит список рекомендаций и опционально развилки.

```markdown
## Что доработать в постановке

### Рекомендации

- **<Имя темы>:** <2–4 строки описания проблемы и что нужно дописать в design.md / tasks.md>.
- ...

<Опционально, ТОЛЬКО если архитектор-адверсарий вернул реальные равноправные альтернативы:>

### Развилки

#### 1. <Заголовок развилки на языке кода 1С>

**Цель ЗНИ:** <ссылка на пункт ## Why в proposal.md одной строкой>. <Маппинг: обе ветки закрывают / только A закрывает / только B закрывает.>

**Что в коде сейчас.** <2–4 предложения с конкретными именами `Функция/Процедура/Модуль/Реквизит`. Ссылки на пути файлов в `cf` / `cfe`, если уместно.>

**Что предлагает план.** <2–4 предложения о том, что меняется и где.>

**Почему это развилка.** <1–2 предложения: что не закрыто в design / что нашёл независимый аудит.>

**Варианты решения.**

- **A. <Название варианта>** — <что именно делается в коде>; **компромисс:** <одна фраза «но/зато»>.
- **B. <Название варианта>** — <то же>; **компромисс:** <одна фраза>.

**Влияет на:** <что увидит пользователь>.

**Что изменится после выбора.** <1–2 предложения на языке эффекта: что будет зафиксировано в постановке после ответа пользователя. Internal: user-extend `--from-verify` — не цитировать в чате.>

**Источники** *(техническое):* <технические коды алертов, ссылки на дочерние отчёты>.

#### 2. <Следующая развилка, если есть>

...
```

**Заголовки `#### N. ...`** генерируются с предсказуемым slug-якорем (markdown auto-id) — на них ведёт ссылка «Подробнее: …#…» из чата (`templates/chat-summary.md`).

**Таблицы «Ситуация → Сейчас / После», цепочки вызовов, листинги** — допустимы в этой секции файла (например, для развилки с побочным UX-эффектом). В чате их быть не должно.

## Шаблон секции `## Что меняется в постановке`

Описывает план на языке архитектора для разработчика 1С. Не цитирует design.md, а пересказывает суть.

```markdown
## Что меняется в постановке

**Расширение / конфигурация:** <путь в `src/`>.

**Точки изменения:**

- `<ОбщийМодуль>.<ПроцедураИлиФункция>` — <что меняется одной фразой>.
- `<ОбъектМетаданных>.<ИмяФормы>.<ИмяПроцедуры>` — <что меняется>.
- ...

**Что НЕ меняется** (важно для регресс-сценариев): <1–3 строки про инварианты, которые план обязуется сохранить — типовая запись виз, типовой расчёт, поведение для одного шаблона и т.п.>.

**Связанные ADR / KB / архив** (если есть): <ссылки>.
```

## Шаблон секции `## Технический аудит (для движка OpenSpec)`

Это блок **внутри** файла отчёта, не для чтения человеком при первом открытии. Содержит то, что раньше было в `## Executive Summary` со слоями.

```markdown
## Технический аудит (для движка OpenSpec)

### Слои проверки

- **Layer 1 (Гигиена артефактов):** <PASS / AUTOFIXED + перечень / FAIL>.
- **Layer 2 (Internal Coherence):** <PASS / WARNING / FAIL>; QC отчёт: `<path>`.
- **Layer 3 (Problem-Solution Trace):** <PASS / WARNING / FAIL>; алерты: `<коды>`.
- **Layer 4 (Independent Challenge):** <APPROVE / CHALLENGE / REJECT / SKIPPED-novelty / SKIPPED-override>; отчёт: `<path>`.
- **Layer 5 (Implementation Readiness):** <PASS / WARNING / FAIL>; отчёт: `<path>`.

### Авто-исправлено (Layer 1)

*(Список авто-правок гигиены в техническом формате: файл, строка, что изменено. Либо «не применялось»)*

### Развёрнутые карточки развилок

<Здесь допустимы внутренние коды алертов и ссылки на дочерние отчёты — это техническая секция.>
```

## Правила оформления

### Первый абзац секции «Резюме для разработчика»

- Зеркало чата. Точно та же первая строка вердикта, что в чате.
- Без «PASS», «verdict: GO», `Layer N`, `CHALLENGE` — это YAML и `## Технический аудит`.

### Секция «Что доработать в постановке»

- Только при `verdict: NO-GO` (внутренний код в YAML).
- Рекомендации — списком (1 пункт на блокер, 2–4 строки каждый).
- Развилки (если есть) — отдельным `####`, пронумерованы, заголовок на языке кода 1С. Тело — связная проза, не таблица. Цель: разработчик 1С понимает за 30 секунд, что от него хотят и какие варианты есть.

### Секция «Технический аудит»

- **Всегда** все пять буллетов в указанном порядке. Если слой не выполнялся (Layer 4 пропущен по новизне) — буллет остаётся со статусом `SKIPPED-novelty` и одной фразой обоснования.
- Имена `Layer N`, `design-challenge`, `task-readiness` — **разрешены** в этой секции (только здесь).

## Анти-примеры

**Анти-пример 1: «Executive Summary» как первый раздел**

```
## Executive Summary

**На обсуждение.** План решает... но независимый аудит постановки выявил...

### Слои проверки
...
```

Что не так: разработчик сразу попадает в язык слоёв и движка verify, до того как увидит «что меняется в коде» и «что доработать». Правильно — сначала «Резюме для разработчика» и «Что доработать в постановке», и только потом, ниже по файлу, технический аудит.

**Анти-пример 2: размазывание языка слоёв по всему файлу**

«Внутренняя согласованность плана: WARNING — есть пометки по приёмке.»

Что не так: фраза непонятна разработчику 1С, не открывавшему OpenSpec. Правильно: «Старый формат приёмки (S1.T1–T4 вместо одного S1.accept) — не блокирует apply.» в разделе «К сведению»; имя `Layer 2 / WARNING` — только в `## Технический аудит`.
