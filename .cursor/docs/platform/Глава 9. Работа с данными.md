## 9.1. Механизм транзакций

### 9.1.1. Общая информация

Независимо от варианта работы (файловый или клиент-серверный) система «1С:Предприятие» обеспечивает работу с информацией в базе данных с использованием механизма транзакций.

Транзакция — неделимая, с точки зрения воздействия на базу данных, последовательность операций. Она выполняется по принципу «все или ничего» и переводит базу из одного целостного состояния в другое. При невыполнимости одного из действий или нарушении работы системы выполняется откат транзакции.

Система определяет неявную транзакцию при любых действиях, связанных с модификацией данных (например, обработчики событий в модулях объектов и наборов записей). Чтение объектов (обращение через точку, .ПолучитьОбъект(), открытие формы) выполняется вне транзакции; при необходимости платформа открывает неявные транзакции для целостности чтения.

### 9.1.2. Использование явного вызова транзакций

Используются процедуры глобального контекста: НачатьТранзакцию(), ЗафиксироватьТранзакцию(), ОтменитьТранзакцию().

- НачатьТранзакцию() — открыть транзакцию; последующие изменения могут быть приняты или отвергнуты целиком.
- ЗафиксироватьТранзакцию() — принять все изменения.
- ОтменитьТранзакцию() — отменить все изменения открытой транзакции.

Если вызовов НачатьТранзакцию() больше, чем ЗафиксироватьТранзакцию()/ОтменитьТранзакцию(), система выполняет неявный ОтменитьТранзакцию() при окончании выполнения встроенного языка или при передаче управления с сервера на клиента. Если фиксаций/отмен больше, чем начатий — порождается исключение.

Типичная схема с обработкой ошибок:

```
Попытка
    НачатьТранзакцию();
    // … операции с данными
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
КонецПопытки;
```

Восстановимые ошибки: система не отменяет транзакцию автоматически; разработчик должен обработать исключение (продолжить или отменить). Если ошибка связана с базой данных, дальнейшая фиксация транзакции невозможна — допустима только отмена, после чего можно повторить попытку.

Невосстановимые ошибки: выполнение прекращается; изменения текущей транзакции отменяются системой.

Пример повторных попыток записи при восстановимой ошибке БД:

```
Записано = Ложь;
Пока Не Записано Цикл
    Попытка
        НачатьТранзакцию();
        Данные.Записать();
        ЗафиксироватьТранзакцию();
        Записано = Истина;
    Исключение
        ОтменитьТранзакцию();
    КонецПопытки;
КонецЦикла;
```

### 9.1.3. Вложенный вызов транзакций

В рамках уже открытой транзакции можно вызывать НачатьТранзакцию(), ЗафиксироватьТранзакцию(), ОтменитьТранзакцию(). Фактически действует только транзакция верхнего уровня: вложенные вызовы относятся к той же транзакции. Отмена во «вложенной» откатывает всю транзакцию верхнего уровня; фиксация во вложенной игнорируется.

### 9.1.4. Влияние транзакций на работу программных объектов

Программные объекты в общем случае «прозрачны» для транзакций БД: откат транзакции не отменяет изменения в памяти. Разработчик при необходимости должен сам восстанавливать состояние (повторное чтение или изменение реквизитов).

Исключения (откат влияет на объекты):
- признак проведения документа восстанавливается к значению до начала транзакции;
- если объект создан и записан в транзакции, при откате очищается значение ссылки;
- если объект создан вне транзакции и при записи использовался автосгенерированный код/номер, при откате код/номер очищается.

## 9.2. Блокировки

### 9.2.1. Общая информация

Блокировка — информация о том, что объект БД захвачен для действий некоторым потребителем (пользователем, сеансом). Другие пользователи не могут использовать заблокированный объект до снятия блокировки. Механизм обеспечивает конкурентный доступ и непротиворечивость данных.

Блокировки в 1С: объектные и транзакционные. Объектные — конкурентный доступ в терминах объектов (редактирование в формах и т.д.). Транзакционные — согласованное изменение данных в СУБД.

Для объектных данных (справочники, документы, счета и пр.) предусмотрены пессимистическая и оптимистическая объектные блокировки.

### 9.2.2. Механизм объектных блокировок

#### 9.2.2.1. Общая информация

(См. подразделы ниже.)

#### 9.2.2.2. Пессимистическая блокировка

Пессимистическая блокировка запрещает изменение данных объекта другими сеансами (или этим же сеансом в другом контексте) до снятия блокировки. Платформа использует её в расширениях форм: при начале модификации объекта в форме блокировка устанавливается, при закрытии формы — снимается. При попытке другого пользователя редактировать тот же объект выводится сообщение; в зависимости от варианта (файловый/клиент-сервер) возможны варианты: информация о заблокировавшем, утрата изменений другого сеанса, сообщение через систему взаимодействия.

В коде: ЗаблокироватьДанныеДляРедактирования() — установка, РазблокироватьДанныеДляРедактирования() — снятие.

Два варианта установки:
- С идентификатором формы — блокировка на время жизни формы; снимается при закрытии формы, завершении сеанса, через 1 минуту после сброса признака модифицированности, при установке других блокировок от имени формы, при закрытии формы с незавершёнными фоновыми заданиями или при вызове РазблокироватьДанныеДляРедактирования() с тем же идентификатором.
- Без идентификатора формы — блокировка не привязана к форме; снимается при завершении сеанса, при возврате управления с сервера или при завершении транзакции (если блокировка устанавливалась в транзакции), либо методом РазблокироватьДанныеДляРедактирования() без идентификатора.

Установка блокировки сама по себе не препятствует изменению/удалению в БД; в другом сеансе операция изменения также должна предваряться попыткой блокировки. Попытка заблокировать уже заблокированный объект вызывает исключение (можно обработать Попытка/Исключение). Несовместимы блокировки одного объекта с указанием идентификатора формы и без него.

Для нестандартной формы: методы формы ЗаблокироватьДанныеФормыДляРедактирования() и РазблокироватьДанныеФормыДляРедактирования().

Пример установки блокировки и записи:

```
ТоварСсылка = Справочники.Товары.НайтиПоКоду("000000001");
Попытка
    ЗаблокироватьДанныеДляРедактирования(ТоварСсылка);
    ТоварОбъект = ТоварСсылка.ПолучитьОбъект();
    ТоварОбъект.Наименование = "Новое наименование";
    ТоварОбъект.Записать();
    Возврат Истина;
Исключение
    // объект уже заблокирован
    Возврат Ложь;
КонецПопытки;
```

#### 9.2.2.3. Пессимистическая блокировка и транзакции

Операции объектной блокировки не влияют на течение транзакций и операции над данными. Блокировка уже заблокированного объекта вызывает исключение, которое можно обработать; откат транзакции не обязателен. Объектные блокировки, установленные в транзакции без идентификатора формы, снимаются при окончании транзакции.

#### 9.2.2.4. Оптимистическая блокировка

Оптимистическая блокировка запрещает запись объекта, если после считывания он был изменён в БД. При чтении считывается версия объекта; при записи проверяется соответствие версии в памяти и в БД. При несовпадении выдаётся предупреждение (версия изменилась или объект удалён). Гарантирует, что изменения текущего сеанса не перезапишут изменения других сеансов.

### 9.2.3. Механизм управляемых блокировок

#### 9.2.3.1. Общая информация

Для одновременного выполнения транзакций используются уровни изоляции и блокировки данных. Режимы работы с данными в 1С: в транзакции и вне транзакции. Вне транзакции допускается только чтение (безответственное: возможны устаревшие или незафиксированные данные). В транзакции — чтение и модификация; правила: воспроизводимость чтения, актуальность (никакая другая транзакция не изменит прочитанные данные до завершения текущей), отсутствие незафиксированных изменений в результате чтения.

Два режима в рамках транзакции: автоматические блокировки (управляет платформа за счёт уровня изоляции СУБД) и управляемые блокировки (низкий уровень изоляции в СУБД, блокировки задаёт разработчик). Рекомендуется режим управляемых блокировок для максимальной параллельности; автоматический — для совместимости с прежними версиями.

Уровни изоляции (кратко): при автоматических блокировках — Serializable (файловая БД, PostgreSQL, Oracle) или Repeatable Read/Serializable (MS SQL, IBM Db2); при управляемых — Read Committed (или Read Committed Snapshot для MS SQL 2005+). Безответственное чтение: в автоматическом режиме — «грязное» чтение (кроме PostgreSQL, Oracle); в управляемом — согласованное чтение (MS SQL 2005+, Db2 9.7+, PostgreSQL, Oracle). Объекты с табличными частями и наборы записей — всегда согласованное безответственное чтение.

#### 9.2.3.2. Управляемые блокировки (средства языка)

Объект БлокировкаДанных: создаётся конструктором, описывает пространства блокировок и режимы; метод Заблокировать() устанавливает блокировки. В транзакции (явной или неявной) блокировки снимаются при окончании транзакции. Вне транзакции или в автоматическом режиме вызов Заблокировать() может привести к неожиданному поведению (в документации уточнять).

Пространства блокировок соответствуют структуре объектов конфигурации. Примеры имён: Справочник.<имя>, Документ.<имя>, РегистрСведений.<имя>, РегистрНакопления.<имя>, РегистрБухгалтерии.<имя>, РегистрРасчета.<имя>, ПланОбмена.<имя>, ПланСчетов.<имя>, БизнесПроцесс.<имя>, Задача.<имя>, ПланВидовРасчета.<имя>, ПланВидовХарактеристик.<имя>, Последовательность.<имя>, Константа.<имя> и др. Для каждого пространства заданы поля (Ссылка, измерения, Регистратор и т.д.). Перечень полей, по которым разрешена блокировка, задаётся в свойстве «Поля блокировки данных» объекта метаданных. Запрещено блокировать по реквизитам неограниченной строки, хранилищу значения, составному типу с такими полями; по полям Предопределенный, ИмяПредопределенныхДанных в справочниках и планах.

Условия задаются: УстановитьЗначение(имяПоля, значение) или через ИсточникДанных и ИспользоватьИзИсточникаДанных(); источник — запрос, табличная часть, набор записей, таблица значений. Для диапазона — объект Диапазон. Режимы: Разделяемый (другие транзакции не могут изменить данные до конца текущей) и Исключительный (также не могут прочитать с разделяемой блокировкой). Совместимость: разделяемая с разделяемой — да; исключительная с разделяемой/исключительной — нет. При записи данных платформа устанавливает неявные управляемые исключительные блокировки; при чтении по ссылке блокировка не ставится — при необходимости ставить явно до обращения к объекту.

Эскалация: при более 100 000 блокировок на одно пространство возможна эскалация — блокируется всё пространство (или в рамках разделителей при независимых разделителях). Поглощение блокировок: блокировка с меньшим числом заданных измерений может поглощать блокировки с большим (при совпадении по общим измерениям). Рекомендация: при большом числе блокировок с разными значениями одного измерения — одна блокировка без этого измерения.

#### 9.2.3.3. Установка режима блокировок в конфигурации

Свойство конфигурации «Режим управления блокировкой данных»: Автоматический, Управляемый (по умолчанию для новой конфигурации), Автоматический и управляемый. Для объекта метаданных — своё свойство; при значении «Автоматический и управляемый» режим берётся из объекта. Режим объекта применяется к транзакциям, инициируемым платформой при работе с данным объектом. Для явной транзакции (НачатьТранзакцию()) режим задаётся параметром РежимБлокировок метода НачатьТранзакцию(); по умолчанию — Автоматический, для управляемых блокировок указывать РежимУправленияБлокировкойДанных.Управляемый.

(Содержание объединено с 9.2.3.2.)

#### 9.2.3.5. Особенности работы в режиме «Автоматический и управляемый»

Режим позволяет перевести часть конфигурации в управляемые блокировки (например, отдельные документы), остальное оставить в автоматическом. Особенности: при любом режиме транзакции система устанавливает соответствующие управляемые блокировки (транзакции в управляемом режиме могут конфликтовать с транзакциями в автоматическом); фактический режим определяется транзакцией верхнего уровня — вложенная транзакция выполняется в режиме уже открытой (при несовпадении желаемого и текущего режима при начале управляемой при открытой автоматической возможна исключительная ситуация). Для записи наборов записей при проведении документа управляемый режим нужно задать и для регистра, и для документа.

#### 9.2.3.6. Особенности работы с большим количеством управляемых блокировок

(Эскалация и поглощение — см. 9.2.3.2.)

#### 9.2.3.7. Модификация конфигураций при переходе к режиму управляемых блокировок

Участки кода, где нужны управляемые блокировки: считываются данные с последующим изменением; считывается согласованная совокупность данных из нескольких объектов и нужно сохранить согласованность; требуется неизменность считанных данных до конца транзакции. Способы: объект БлокировкаДанных или свойство БлокироватьДляИзменения у наборов записей регистров накопления/бухгалтерии. Исключительный режим — для данных, которые будут изменяться в этой транзакции; разделяемый — для данных только на чтение, но с защитой от изменений. При удалении/изменении объекта блокировка по полям выполняется и по «старым», и по «новым» значениям записываемого объекта.

## 9.3. Монопольный режим

Монопольный режим — доступ к базе только одним сеансом. Применяется для существенных согласованных изменений без обязательной упаковки в одну транзакцию, при необходимости исключить влияние других сеансов. Требуется для части методов (например, УдалитьОбъекты()) и для работы Конфигуратора.

Установка/снятие: УстановитьМонопольныйРежим(); проверка: МонопольныйРежим(). Нельзя изменить монопольный режим при активной транзакции. В монопольном режиме запрещено создание новых сеансов (кроме одного фонового задания; при его запуске оно «забирает» монопольный режим у родительского сеанса до своего завершения). Управляемые блокировки в монопольном режиме не устанавливаются (попытки игнорируются).

ПараметрыМонопольногоРежима: свойство Сообщение — текст (или ФорматированнаяСтрока) для отображения пользователю при попытке подключения (причина блокировки). РазрешитьЗавершатьПриНачалеСеанса + право ЗавершениеМонопольногоРежимаПриНачалеСеанса — возможность завершить сеанс с монопольным режимом при начале нового сеанса (кнопка в диалоге). ПолучитьПараметрыМонопольногоРежима() — текущие параметры; Неопределено — монопольный режим не устанавливался.

## 9.4. Индексы объектов метаданных

### 9.4.1. Общая информация

Индекс — объект БД для ускорения поиска; строится по одному или нескольким столбцам (ключевые поля). Типы: кластерный (физический порядок строк, один на таблицу) и некластерный (указатели на записи, несколько на таблицу). Могут быть уникальные и неуникальные. Платформа создаёт набор индексов для объектов конфигурации (описано на ИТС). Разработчик может указывать необходимость индексирования реквизитов.

### 9.4.2. Возможности дополнительных индексов

Дополнительные индексы (справочники, документы, регистры, бизнес-процессы, задачи, журналы, последовательности, планы и т.д.) — некластерные, неуникальные; обычные или составные. Доступны для лицензии КОРП. Управление созданием: УстановитьИспользованиеДополнительныхИндексов() / ПолучитьИспользованиеДополнительныхИндексов() или диалог параметров ИБ. Физическое создание/удаление — при обновлении конфигурации БД или при тестировании и исправлении (с включённой функциональностью КОРП). Учитывается параметр ИБ «Использовать дополнительные индексы». Большое число индексов увеличивает время записи и объём БД; баланс между скоростью выборки, записью и размером.

Дополнительный индекс: имя, выбранные (ключевые) поля, дополнительные (неключевые) поля. В ключ не входят ХранилищеЗначения и строка неограниченной длины. Составной индекс — не более 16 физических полей таблицы (ограничение Конфигуратора). Поля могут браться из основной таблицы, табличных частей, виртуальных таблиц (для регистров); в один индекс — поля только одного вида таблиц. Дублирующие индексы в СУБД не создаются.

### 9.4.3. Дополнительные индексы во встроенном языке

Только чтение: свойство ДополнительныеИндексы объекта метаданных; коллекция с объектами ДополнительныйИндекс (Имя, ИндексируемыеПоля, ДополнительныеПоля); ПолеИндекса — Таблица, Поле. Настройка состава — только в Конфигураторе (диалог «Дополнительные индексы»).

### 9.4.4. Редактирование дополнительных индексов

(См. 9.4.3.)

## 9.5. Динамические выборки данных

Динамические выборки (методы Выбрать(), ВыбратьИерархически(), ВыбратьПоСсылкам()) не считывают данные целиком, а получают их блоками при обходе Следующий(), что позволяет обходить большие объёмы без переполнения памяти.

- Выбрать() / ВыбратьИерархически(): отбор по реквизиту, упорядочивание; при обходе используется индекс. Параметры задают отбор и сортировку.
- ВыбратьПоСсылкам(): состав задаётся набором ссылок (параметр метода); порядок чтения — порядок ссылок в наборе.

Блок по 25 записей (по выбранному упорядочиванию). Считываются все данные объекта, включая табличные части. Следующий блок читается после полного обхода предыдущего — разработчик не контролирует момент физического чтения и актуальность при обходе в транзакции. Выборка.ПолучитьОбъект() возвращает объект из данных выборки (уже считанных); Выборка.Ссылка.ПолучитьОбъект() — объект из БД (повторное чтение).

ВыбратьИерархически(): обход по уровням (элемент → подчинённый → следующий на том же уровне); каждое считывание подчинённых — новый блок. При изменении данных во время обхода возможны: получение удалённого объекта, двойное попадание объекта в выборку, пропуск записей. При обходе и удалении иерархии в обычной выборке возможны ошибки из-за удаления родителя до обхода подчинённых в блоке; при иерархической выборке в блоке только элементы одного родителя — проблемы нет.

Рекомендации: применять для задач без ответственного чтения или регламентных задач (в т.ч. с монопольным режимом); учитывать влияние изменений объектов во время обхода на состав и порядок выборки.
