# Глава 25. История данных

> **Agent ref:** История данных (версионирование) — хранение данных объектов по шкале времени; версия = данные объекта + метаданные на момент изменения; настройка по объекту и по реквизиту (в т.ч. ТЧ); дополнительные данные и комментарий к версии; постобработка после записи версий; очередь версий и очередь постобработки; управление из конфигуратора и из встроенного языка.
> **Ключевые разделы:** 25.1 Общая информация, 25.2 Устройство механизма (очередь версий, ОбновитьИсторию, постобработка), 25.3 Использование (настройки, запись версии, обычная/ускоренная обработка, дополнительные данные, обработка после записи, ВыбратьВерсии, получение данных версии, комментарий, удаление, ПолучитьРазличияВерсий, расширения).
> ---

## 25.1. Общая информация

История данных — механизм хранения в базе данных данных объектов конфигурации, упорядоченных по шкале времени (версионирование данных). В БД хранится версия: данные объекта на момент изменения и состояние метаданных на момент изменения. Настройка хранения возможна для объекта целиком и для каждого реквизита (включая табличные части). К каждой версии можно добавлять произвольные дополнительные данные (состав задаётся разработчиком) и выполнять постобработку после фактической записи версии в таблицы БД.

Включение и выключение истории — из конфигуратора или из встроенного языка (включение из встроенного языка не требует снятия конфигурации с поддержки).

Операции механизма:
- запись версии для поддерживаемого объекта;
- запись дополнительных данных, отсутствующих в фактических данных объекта;
- получение данных выбранной версии (включая дополнительные);
- удаление данных версии;
- получение разницы между двумя версиями.

Поддерживаемые объекты: общие реквизиты, константы, планы обмена, справочники, документы, планы видов характеристик, планы счетов, планы видов расчета, бизнес-процессы, задачи, регистры сведений.

Работа с историей данных регламентируется правами доступа и протоколируется в журнале регистрации.

---

## 25.2. Устройство механизма

История хранится в специальных таблицах той же ИБ. Хранятся данные версий и метаданные прежних версий версионируемых объектов. Версия метаданных создаётся при изменении метаданных объекта и не привязана к изменению данных. В истории не хранятся типы реквизитов, только фактические значения. С точки зрения системы эквивалентны: включение версионирования реквизита и создание нового реквизита; отключение версионирования реквизита и удаление реквизита.

Для каждой версии можно указать комментарий и дополнительные данные.

**Формирование версии — шаги:**

1. **Фиксация изменения** — автоматически или через API. Информация попадает в очередь версий. В этот момент можно добавить произвольные данные: `ЗаписьИсторииДанных.ДобавитьДополнительныеДанные()`.

2. **Перенос из очереди в таблицы БД** — метод `ИсторияДанных.ОбновитьИсторию()`. Можно выполнять асинхронно (например, регламентным заданием). Версия считается записанной только после этого переноса. Каждое изменение объекта фиксируется отдельно. Версии с `ЗаписьИсторииДанных.ВыполнятьОбработкуПослеЗаписи` = Истина переносятся в очередь постобработки.

3. **Обработка очереди постобработки** — в рамках `ОбновитьИсторию()` или методом `ИсторияДанных.ВыполнитьОбработкуПослеЗаписиВерсий()`. Обработка — в обработчиках `ОбработкаПослеЗаписиВерсийИсторииДанных()` модуля менеджера версионируемого объекта.

4. **Удаление версии из очереди постобработки** — обязательно после обработки: `ИсторияДанных.УдалитьИзОбработкиПослеЗаписиВерсий()`. Иначе версия будет снова передаваться в обработчики.

5. **Принудительная очистка очереди постобработки** — `ИсторияДанных.УдалитьИзОбработкиПослеЗаписиВерсий()`.

Шаги 1–2 обязательны; 3–5 — при использовании очереди постобработки.

Два способа формирования истории: **обычный** (очередь версий + периодическое `ОбновитьИсторию()`) и **ускоренный** — принудительная обработка одной версии одного объекта (сразу после записи или вызовом `ОбновитьИсторию(Объект)`).

## 25.3. Использование механизма

### 25.3.1. Управление использованием истории данных

**Конфигуратор:** свойство объекта **История данных** = **Использование**; затем регламентное задание на обновление истории по очереди. Аналогичное свойство у реквизитов (стандартных и обычных), табличных частей и их реквизитов, ресурсов регистров сведений.

**Значения по умолчанию:**
- Справочники, документы, ПВХ, планы счетов, бизнес-процессы, задачи, регистры сведений — **Не использовать**.
- Стандартные реквизиты, реквизиты объектов, реквизиты ТЧ, ресурсы регистров сведений — **Использовать**.
- Измерения регистров сведений — всегда **Использовать** (без возможности отключения).

Для типообразующих объектов ведение истории включают принудительно; реквизиты по умолчанию попадают в историю — при необходимости исключают принудительно.

**Программный доступ:** глобальный контекст `ИсторияДанных` (менеджер истории). Методы: `ИсторияДанных.ИмяМетода()`.

**Настройки в конфигураторе** — через свойство `ИсторияДанных` объекта метаданных (например, признак включения истории для справочника). **Настройки в режиме 1С:Предприятие** — метод `ПолучитьНастройки(ОбъектКонфигурации)`. Возвращает объект `НастройкиИсторииДанных` или Неопределено, если изменений нет. Метод `ПолучитьМетаданные()` — список реквизитов, сохраняемых в истории (конфигуратор + режим Предприятия). Установка настроек — `УстановитьНастройки(НастройкиИсторииДанных)`. В объекте задаются: `Использование` (для объекта в целом), `ИспользованиеПолей` (соответствие: ключ — имя реквизита или `ТабличнаяЧасть.Реквизит`, значение — признак участия в истории). Для реквизитов, не перечисленных в `ИспользованиеПолей`, участие приводится в соответствие с конфигуратором. Вызов `УстановитьНастройки(Настройки, Неопределено)` сбрасывает настройки к конфигуратору.

### 25.3.2. Запись версии

**Два способа:**

1. **Автоматически** — при записи версионируемого объекта система фиксирует необходимость создания версии. На этом этапе доступно: отказаться от версии (установить признак отказа в Истина); добавить дополнительные данные — `ЗаписьИсторииДанных.ДобавитьДополнительныеДанные()`, представление — `ЗаписьИсторииДанных.ДобавитьПредставлениеПоляДополнительныхДанных()`; включить версию в очередь постобработки — свойство `ЗаписьИсторииДанных.ВыполнятьОбработкуПослеЗаписиВерсии`; ускоренная обработка — свойство `ЗаписьИсторииДанных.ОбновлятьИсториюСразуПослеЗаписи`. Фактическая запись в таблицы истории — только после вызова `ОбновитьИсторию()` (рекомендуется в регламентном задании). После модификации таблиц формируется очередь постобработки; обработка — методом `ОбновитьИсторию()` (если параметр выполнить постобработку Истина) или `ВыполнитьОбработкуПослеЗаписиВерсий()`.

2. **Встроенный язык** — метод `ЗаписатьВерсию()`: запись сразу в таблицы истории, без очереди (например, при переносе из другого хранилища).

**Ускоренная обработка:** установить в конфигураторе флажок «Обновлять историю данных сразу после записи» (для всех записей объекта, не рекомендуется при большом количестве и интенсивных изменениях) или при записи установить `ЗаписьИсторииДанных.ОбновлятьИсториюСразуПослеЗаписи` (для выборочных случаев). Либо вызвать `ОбновитьИсторию(ТребуемыйОбъект)` — обрабатывается одна версия одного объекта.

---

### 25.3.3. Дополнительные данные в версии

Метод `ЗаписьИсторииДанных.ДобавитьДополнительныеДанные(Имя, Значение, Представление)`. Параметры: **Имя** — имя «реквизита» дополнительных данных; **Значение** — значение (если передана `ТаблицаЗначений`, в дополнительные данные добавляется табличная часть, структура ТЗ задаёт структуру ТЧ); **Представление** — строка (для всех языков) или соответствие (код языка — представление). Представление можно задать отдельно: `ЗаписьИсторииДанных.ДобавитьПредставлениеДополнительныхДанных()`.

---

### 25.3.4. Обработка после записи истории

Включение версии в очередь постобработки: свойство объекта конфигурации «Выполнять обработку после записи версии истории данных» (все версии объекта) или свойство `ЗаписьИсторииДанных.ВыполнятьОбработкуПослеЗаписиВерсии` при записи (по условию).

При вызове `ОбновитьИсторию()` с параметром выполнить постобработку Истина версии с включённой постобработкой помещаются в очередь, затем вызывается `ВыполнитьОбработкуПослеЗаписиВерсий()`. В процессе: формируется коллекция версий одного типа (не более 1000); вызывается обработчик `ОбработкаПослеЗаписиВерсийИсторииДанных` в модуле менеджера. В обработчике каждую обработанную версию нужно явно удалить из очереди постобработки — методом `УдалитьИзОбработкиПослеЗаписиВерсий()`; иначе версия будет снова передаваться в обработчики. Очистка очереди постобработки — по времени: по умолчанию удаляются версии старше 20 дней; параметр **Дата** метода `УдалитьИзОбработкиПослеЗаписиВерсий()` задаёт интервал.

### 25.3.5. Получение списка версий

Метод `ВыбратьВерсии()` — параметры: критерии отбора, какие данные возвращать, упорядочивание, максимальное количество записей.

**Отбор (параметр Отбор):** версия удовлетворяет отбору, если выполняются все критерии (И). Если параметр допускает массив значений — версия подходит при совпадении хотя бы одного (ИЛИ). Исключения: отборы `ЗначенияПолей` и `ИзменениеЗначенийПолей` (описаны ниже). Строка в отборе обычно сравнивается по вхождению, не на полное равенство.

Обязательно указать **Метаданные** или **Данные** (и только один из них; иначе исключение; исключение не генерируется, если метаданные параметра Данные совпадают с параметром Метаданные). Для списка версий по конкретному объекту рекомендуется предварительно вызвать `ОбновитьИсторию(ЭлементСсылка)` для актуализации. Отбор по пользователю — по идентификатору пользователя ИБ (не объект); идентификатор — `ПользовательИнформационнойБазы.УникальныйИдентификатор()`. Допускается отбор по имени и полному имени пользователя. **ИзменениеЗначенийПолей** — отбор только тех версий, где изменялись указанные поля (факт изменения, без указания значений).

**ЗначенияПолей** — массив структур: **Поле** — имя реквизита (для ТЧ: `ТабличнаяЧасть.Реквизит`); **ЗначениеДоИзменения**; **ЗначениеПослеИзменения**. В структуре обязателен ключ Поле и хотя бы один из ключей значений. Одинаковые Поле в массиве объединяются по ИЛИ, разные — по И. Поддерживается только сравнение на равенство.

### 25.3.6. Получение данных версии

По ссылке или ключу записи и номеру версии: данные версии в объект нужного типа — метод получения по версии (в синтакс-помощнике); сформированный объект можно записать в БД. Версия удаления возвращает объект с данными на момент удаления. Вмешательство в формирование объекта — обработчик `ОбработкаФормированияПоВерсииИсторииДанных()` в модуле объекта и набора записей регистра сведений (вызывается после метода формирования по версии и при интерактивном переходе на версию). Параметр формы **НомерВерсииПереходаНаВерсиюИсторииДанных** — если не Неопределено, форма открыта с данными указанной версии. Для подчинённого регистра сведений метод по версии формирует набор записей по ключу; при записи обновляются все строки набора — при параллельном изменении из другого сеанса возможна потеря данных; при необходимости применять блокировки.

Метод `ПолучитьДанныеВерсии()` — возвращает структуру (ключ — имя реквизита, значение — значение реквизита). Список версионируемых реквизитов (и представлений) для версии — для сравнения с текущим набором; метод `ПолучитьМетаданные()` возвращает структуру со списком реквизитов (не объект метаданных; сравнение свойств реквизитов недоступно).

### 25.3.7. Комментарий к версии

Изменение комментария — метод `ЗаписатьКомментарий()`. Максимум 1024 символа. При записи объекта — свойство `ЗаписьИсторииДанных.КомментарийВерсии`; комментарий относится к версии, создаваемой при этой записи.

---

### 25.3.8. Удаление версии

Методы: `УдалитьВерсии()` и глобальный контекст `УдалитьДанныеИнформационнойБазы()`.

`УдалитьВерсии()`:
1. Все исторические данные по объекту конфигурации: `УдалитьВерсии(Метаданные.Справочники.Товары)`.
2. Данные объекта ранее указанной даты: `УдалитьВерсии(Метаданные.Справочники.Товары, НачалоГода(ТекущаяДата()))`.
3. Набор версий объекта данных: `УдалитьВерсии(СсылкаНаОбъект, НачалоИнтервала, ОкончаниеИнтервала)`.
4. Все версии объекта ранее указанной даты: `УдалитьВерсии(СсылкаНаОбъект, НачалоГода(ТекущаяДата()))`.

При включённом разделении данных действуют особые правила удаления истории.

### 25.3.9. Различия между версиями

Метод `ПолучитьРазличияВерсий()` — возвращает значения версионируемых реквизитов для двух версий (без вычислений и анализа промежуточных версий). Интерпретация результата — на стороне разработчика. Приведение типов не выполняется; смена типа реквизита даёт различие. Параметры: объект, версия после изменения (правее по времени), версия до изменения (левее); если версия «до» не указана — сравнение с предыдущей версией.

Результат строится по структуре метаданных версии после изменения (второй параметр). Если в версии «до» был реквизит, а в версии «после» его нет — по удалённому реквизиту в разнице информации нет. Если реквизит появился в версии «после» — в качестве значения «до» используется значение по умолчанию типа.

Результат — структура: ключ — имя реквизита или табличной части; значение — структура (реквизит) или массив структур (ТЧ). Для реквизита: **ЗначениеПослеИзменения** (и при наличии **ЗначениеДоИзменения**; если в более ранней версии реквизита не было или версионирование было отключено, элемента ЗначениеДоИзменения нет). Для ссылочного реквизита значение передаётся структурой: **Представление**, **Ссылка**. Для табличной части каждый элемент массива: **НомерСтрокиВВерсииПослеИзменения**, **НомерСтрокиВВерсииДоИзменения**, **ВидИзмененияСтроки**, **Поля** (структура изменений). Вид изменения строки: создание (для каждого реквизита ТЧ — элемент с именем реквизита и значением); изменение (для каждого реквизита — структура как для реквизита объекта); удаление (значение реквизита до удаления); перемещение строки в Поля не отражается.

---

### 25.3.10. История данных и расширение

**Собственный объект расширения:** история ведётся как для собственного объекта; данные истории удаляются при полном удалении расширения из ИБ.

**Заимствованный объект расширения:** настраивать использование истории можно только для реквизитов, добавленных в расширении; ведение истории определяется настройками заимствованного объекта. Удаление расширения не очищает историю (аналогично удалению версионируемого реквизита без расширения).
