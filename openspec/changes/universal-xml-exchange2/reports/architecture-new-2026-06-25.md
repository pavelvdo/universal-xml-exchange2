## Детальный разбор

### F1 — BLOCKER: контракт движка epf отсутствует

D4 описывает движок одной фразой: «получает обработку через `ПолучитьОбработку()` / аналог и вызывает экспортные методы epf». Проблемы:

- `ПолучитьОбработку()` — не платформенный метод. Реальный путь: `ПолучитьМакет("ДвижокОбмена")` → `BinaryData` → запись во временный файл → `ВнешниеОбработки.Создать(Путь)` (или подключение). Это надо зафиксировать.
- **Не определены** имена и сигнатуры экспортных методов `V8Exch82.epf`: как передаются правила (XML из `ПравилаОбменаДанными`), как инициализируются параметры из ТЧ, как извлекается XML данных, как передаётся имя файла/каталога. Без этого задачи S2 (выгрузка) и S3 (загрузка) — неисполнимы.
- **Регресс относительно архива:** архивный design имел отдельную секцию `## Engine Contract` + риск-митигацию static-сверки. v2 убрал движок cf, но **не** ввёл аналогичный контракт для epf. Это шаг назад по управляемости риска.

**Требование:** добавить в design секцию `## Engine Contract (V8Exch82.epf)` или явный Open Question-блокер + acceptance-шаг «определить публичный API epf до BSL S2»: способ загрузки epf из макета, перечень экспортных методов выгрузки/загрузки, способ передачи правил и параметров, формат результата (XML/файл).

### F2 — BLOCKER: транзакционная модель статуса внутренне противоречива

Три места говорят разное:

- Behavior Contract (инвариант): «Перевод «Новое» → «Выполняется» MUST быть в **одной транзакции с началом выгрузки**» и таблица: «Выполняется (в той же транзакции, до выгрузки)».
- exchange-web-service spec: «атомарно перевести статус в «Выполняется» **до формирования ZIP**».
- exchange-export spec / D2: при ошибке выгрузки — `Статус → Ошибка`.

Противоречие: если перевод статуса находится в одной транзакции с выгрузкой epf, то при ошибке транзакция откатывается и статус возвращается в «Новое» — **что прямо противоречит** требуемому переходу в «Ошибка». Кроме того, держать транзакцию открытой на всё время выгрузки через внешний epf — антипаттерн (длинная транзакция, блокировки, риск таймаута WS).

Отдельно: **не выбран механизм защиты от конкурентного `GetData`**. Инвариант «GetData MUST NOT выгружать, если статус ≠ Новое» требует атомарного check-and-set. Два одновременных вызова оба прочитают «Новое». Нужен явный механизм: управляемая блокировка профиля (`БлокировкаДанных` на `Справочник.рг_НастройкиОбменаXML`) либо оптимистичная проверка версии. В design этого нет — это сердце корректности двухфазной модели.

**Требование:** в D2 разделить два действия:
1. короткая транзакция под управляемой блокировкой: проверка `Статус = Новое` → перевод в «Выполняется» → **коммит**;
2. вне транзакции — выгрузка epf; при ошибке отдельной записью `Статус → Ошибка` + журнал.
Убрать из Behavior Contract формулировку «в одной транзакции с выгрузкой»; заменить на «атомарный check-and-set до выгрузки, отдельной короткой транзакцией под блокировкой».

### F3 — High: as-built модуль объекта создаёт рег. задание для каждого профиля

В `Catalogs/рг_НастройкиОбменаXML/Ext/ObjectModule.bsl` (as-built) `ПередЗаписью`/`ПриЗаписи` **безусловно** создают и синхронизируют регламентное задание `рг_ВыполнениеОбменаXML` для любого профиля (нет ветвления по роли). В v2:

- `Источник = Истина` (сервер) — обмен **инициируется извне** (pull через GetData), задание-инициатор для него бессмысленно.
- Новая «проверка наличия правил при записи источника» должна встроиться **в существующий** `ПередЗаписью(Отказ)` (который сейчас уходит в early-return при `ОбменДанными.Загрузка` и занимается заданиями), а не появиться как параллельный обработчик.

Design (D6, S1) перечисляет «модуль объекта» в файлах, но **не описывает** интеграцию новой логики со существующими обработчиками и не отвечает: создаётся ли рег. задание для профиля-источника. Это приведёт writer к догадкам.

**Требование:** в D6 или отдельном решении зафиксировать: (а) рег. задание создаётся только при `Источник = Ложь` (или сохранить текущее поведение с обоснованием); (б) проверка правил-в-хранилище встраивается в существующий `ПередЗаписью`, с учётом early-return по `ОбменДанными.Загрузка`.

### F4 — High: ADR-0002 (load-bearing) отменяется, но не назван в Blast Radius

ADR-0002 фиксирует контракт `GetData(ИмяМакетаПравил: String, Параметры: Structure)`. v2 меняет вход на `GetData(Префикс, ПрефиксБазыПриемника)` — это **отмена** контракта load-bearing ADR. В Blast Radius строка про GetData ссылается только на «exchange-web-service main spec», ADR-0002 по имени **не упомянут**. По precedent-regression-gate отмена контракта load-bearing ADR должна быть явной.

Также из запроса: «ADR-0002 GetData HTTP Basic — проверить совместимость». Аутентификация HTTP Basic сохраняется (совместимо), но **сигнатура** меняется — это и есть отменяемая часть ADR-0002.

**Требование:** в Blast Radius назвать ADR-0002 в колонке «Архивный источник» для строки GetData; пометить, что HTTP Basic сохраняется, а сигнатура заменяется (новый ADR при archive — суперсед ADR-0002 в части контракта операции).

### F5 — Medium: расхождение spec по префиксам приёмника

- `exchange-import` (MODIFIED «Чтение настроек профиля»): приёмник читает «префиксы для идентификации сеанса на источнике».
- `exchange-settings` (Scenario «Профиль приёмника»): приёмник «отображает и сохраняет адрес сервиса и логин» — **префиксы не упомянуты**, а Requirement «Префиксы идентификации сеанса» озаглавлен «Профиль-источник».

Получается, приёмник обязан хранить префиксы (чтобы вызвать GetData), но settings-spec этого не требует. Реальный конфликт между capability.

**Требование:** в exchange-settings явно указать, что профиль-приёмник тоже хранит пару префиксов (цель — идентификация сеанса на источнике), либо вынести префиксы в общий блок реквизитов профиля независимо от роли.

### F6 — Medium: не описан путь «не найдено / найдено несколько»

D2/D3 и Behavior Contract описывают только happy-path поиска профиля по префиксам. Не определено поведение GetData/ConfirmDataReceipt, когда по паре префиксов профиль **не найден** или найдено **несколько** (несмотря на проверку уникальности при записи — гонки/ручные правки возможны). Какой ответ клиенту, есть ли запись в журнал.

**Требование:** добавить в Behavior Contract строки для «профиль не найден» и «найдено >1» (ошибка клиенту + журнал, без смены статуса).

### F7 — Medium: видимость реквизитов формы по роли не спроектирована

Сценарии exchange-settings требуют, чтобы форма «отображала реквизиты источника» / «адрес и логин приёмника» в зависимости от `Источник`. Manual Configuration перечисляет добавление реквизитов, но не упоминает логику видимости/доступности по роли (`ИсточникПриИзменении` → перенастройка видимости). Для S1 acceptance это поведенческий пробел.

**Требование:** в Manual Configuration/D6 добавить, что форма элемента переключает видимость групп реквизитов по `Источник` (программно в модуле формы или через условное оформление).

### F8 — Medium: Open Questions не покрывают настоящие блокеры

Текущие OQ (статус при ошибке приёмника; джоб сброса зависших) — второстепенны. Отсутствуют OQ/решения по: контракту epf (F1), механизму блокировки статуса (F2), безопасному режиму запуска epf (F9). Самые рискованные неизвестные не зафиксированы.

**Требование:** поднять F1/F2/F9 в Open Questions (или закрыть решениями в design до apply).

### F9 — Medium: безопасный режим / профили безопасности для epf на сервере WS

Запуск внешней обработки (`ВнешниеОбработки.Создать`) в контексте опубликованного веб-сервиса может блокироваться профилями безопасности кластера / безопасным режимом. Risk-таблица упоминает «epf несовместим с версией платформы», но **не** упоминает разрешение на запуск внешних обработок на сервере. Это эксплуатационный блокер.

**Требование:** добавить риск + митигацию (требование к настройке профиля безопасности на источнике; проверка в Primary acceptance S2).

### F10 — Low: неточности Blast Radius

Строка «Автозавершение без confirm | archive design **D2**» — архивный D2 это контракт GetData, а не отсутствие подтверждения. Прецедент здесь — однофазная pull-модель архива (подтверждения не было вовсе). Мис-цитата вводит в заблуждение.

**Требование:** заменить ссылку на корректный прецедент (однофазная модель архивного change), либо убрать привязку к D2.

## Конкретные правки design.md (перед `/opsx:apply`)

1. **Новая секция `## Engine Contract (V8Exch82.epf)`** — способ загрузки epf из макета, экспортные методы выгрузки/загрузки, передача правил/параметров, формат результата (F1). При невозможности зафиксировать сейчас — Open Question-блокер + acceptance-шаг «discover epf API до BSL S2».
2. **D2 (Контракт GetData)** — разделить на короткую транзакцию под управляемой блокировкой (check-and-set `Новое`→`Выполняется`, коммит) и последующую выгрузку вне транзакции; ошибка → отдельный перевод в `Ошибка` (F2).
3. **Behavior Contract → Инварианты** — заменить «в одной транзакции с выгрузкой» на «атомарный check-and-set до выгрузки, короткой транзакцией под блокировкой»; добавить строки negative-path «профиль не найден» и «найдено несколько» (F2, F6).
4. **D6 / новое решение по as-built** — рег. задание только для `Источник = Ложь` (или обоснование сохранения); встраивание проверки правил-в-хранилище в существующий `ПередЗаписью` (F3).
5. **Blast Radius** — добавить строку/ссылку на ADR-0002 (отмена сигнатуры GetData, HTTP Basic сохраняется); исправить мис-цитату «archive design D2» (F4, F10).
6. **exchange-settings spec** — указать, что профиль-приёмник тоже хранит пару префиксов (F5).
7. **Manual Configuration / форма** — логика видимости реквизитов по `Источник` (F7).
8. **Risks** — запуск внешней обработки на сервере WS под профилем безопасности (F9).
9. **Open Questions** — поднять F1/F2/F9 (F8).
