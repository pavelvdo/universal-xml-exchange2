# Explorer — шаблоны промптов

Шаблоны для делегирования `Task(subagent_type="onec-code-explorer")`. Общие правила, выбор модели, обработка ошибок — `SKILL.md` (навигатор).

---

## Explorer (исследование кода)

```
Task(
  description="Исследовать код [область]",
  prompt="Проанализируй существующий код для [область].
         Найди паттерны, архитектуру, похожие доработки.
         Верни: entry points, потоки выполнения, зависимости, список ключевых файлов.
         
         Design maturity context: для каждой функции, рекомендуемой
         к использованию — задокументировать входной контракт (допустимые
         типы, внутренняя нормализация, гарантии). Указать альтернативные
         точки реализации, если они есть. Указать существующие паттерны
         решения аналогичных задач в проекте.
         Для каждого домена задачи найти существующие механизмы базы
         (регистры, документы, справочники, БП, подписки, точки расширения,
         экспортные функции): что есть, какой контракт/API дают, можно ли
         использовать как есть или расширить без создания параллельного
         механизма.

         ## Existing Knowledge
         [Вставить KB из Discovery по формату KB CONTEXT или: Discovery выполнен, совпадений нет]
         Инструкция: применить блок EXISTING_KNOWLEDGE.

         Контекст задачи: [из proposal/design или описания пользователя]",
  subagent_type="onec-code-explorer"
)
```

## Explorer (верификация гипотез trace-analyst)

Used when trace-analyst report contains `## Verification queries for explorer`.

```
Task(
  description="Верифицировать гипотезы трассы",
  prompt="Отчёт trace-analyst: [путь к trace-analysis-*.md]

         ## Existing Knowledge
         [Вставить KB из Discovery по формату KB CONTEXT или: Discovery выполнен, совпадений нет]
         Инструкция: применить блок EXISTING_KNOWLEDGE.

         Trace-analyst проанализировал трассу выполнения и сформулировал
         гипотезы, которые не удалось верифицировать по коду в рамках
         бюджета (5 модулей). Твоя задача — проверить каждую гипотезу
         по коду и дать однозначный ответ.

         Verification queries:
         [вставить список VQ-1..VQ-N из отчёта trace-analyst]

         Для каждого VQ:
         1. Найди указанные процедуры/модули (modules_hint из запроса)
         2. Восстанови путь вызова / проверь наличие-отсутствие вызова
         3. Ответь: confirms / refutes / inconclusive
         4. Приведи цитату из кода (file:line) как доказательство

         Формат ответа (для каждого VQ):
         ## VQ-N: [вопрос из запроса]
         **Результат:** confirms | refutes | inconclusive
         **Доказательство:** [file:line — что найдено в коде]
         **Контекст:** [краткое пояснение, почему этот код подтверждает/опровергает]

         ОГРАНИЧЕНИЯ:
         - Не исследовать за рамками запросов
         - Не строить альтернативные гипотезы
         - Не додумывать — если код не найден, ответ inconclusive
         - Для каждого confirms/refutes обязательна цитата из кода",
  subagent_type="onec-code-explorer"
)
```

## Explorer — contract resolution (deep)

Используется оркестратором в investigation loop (`review/SKILL.md` шаг 3.5) по таблице Investigation Request из отчёта ревьювера. Вход: вся таблица из Investigation Request (батч). Выход: полный блок `## Resolved Contracts` в формате из `1c-agent-delegation.mdc`.

```
Task(
  description="Глубокий резолв контрактов (батч)",
  prompt="Определи контракты возврата для списка функций. Это нужно ревьюверу для квалификации defensive checks и Попытка-блоков.

         ## Existing Knowledge
         [Вставить KB из Discovery по формату KB CONTEXT или: Discovery выполнен, совпадений нет]
         Инструкция: применить блок EXISTING_KNOWLEDGE.

         ## Точки для резолва (из Investigation Request ревьювера)

         [Вставить таблицу из Investigation Request целиком]

         ## Инструкции

         Для КАЖДОЙ точки из таблицы выше выполни глубокий анализ:

         ### Шаг 1. Найди определение
         - Grep `Функция МетодИмя|Процедура МетодИмя` в src/ (CommonModules, объектные модули, модули форм).
         - **Цепочки вызовов:** если метод вызывается на объекте (например `Ядро.МЧД_СписокДоверенностей`), сначала определи тип объекта Ядро (Grep по коду вызывающего модуля: как создаётся/получается Ядро), затем найди метод в модуле этого типа. Резолвить нужно **возвращаемое значение конечного метода**, а не тип промежуточного объекта.

         ### Шаг 2. Проанализируй тело функции
         - Прочитай тело функции (все Возврат).
         - Тип возврата: Структура | Соответствие | Массив | ТаблицаЗначений | примитив | Произвольный.
         - Для Структура: перечисли ВСЕ ключи (поля).
         - Для Соответствие/Массив: определи тип элементов (если элементы тоже структуры — перечисли их ключи).
         - **Вложенность:** если значение по ключу — тоже Структура/Соответствие, раскрой на один уровень (например: Доверитель(Структура: ИНН, Наименование, КПП)).
         - Если метод вызывает другую функцию и возвращает её результат — проследи цепочку до фактического конструирования значения.

         ### Шаг 3. Классифицируй контракт
         - **fixed**: значение конструируется в коде (Новый Структура, прямой запрос с известными полями, заполнение Соответствие из данных БД). Caller может доверять набору полей/ключей.
         - **dynamic**: значение приходит извне (десериализация JSON/XML, HTTP-ответ, чтение бинарных данных, Выполнить/Вычислить). Набор полей непредсказуем.
         - **Ключевое различие:** если функция сама собирает Структуру через ВставитьЗначение/Вставить из фиксированных данных — это fixed, даже если данные далее пройдут через сериализацию. Если функция десериализует чужой ответ (ПрочитатьJSON, HTTPОтвет) — это dynamic.

         ### Шаг 4. Evidence
         - Укажи файл и номера строк, где видно конструирование возврата.
         - При цепочке вызовов — укажи всю цепочку (файл1:строка → файл2:строка → файл3:строка конструирования).

         ## Формат ответа

         Верни готовый блок для вставки в промпт повторного ревью:

         ## Resolved Contracts

         ### МетодИмя (ИмяМодуляИлиОбъекта)
         - Returns: [тип]
         - Values/Keys: [перечень ключей; при вложенности: Ключ(Тип: подключи)]
         - Contract: fixed | dynamic | unknown
         - Evidence: [файл:строки; при цепочке: файл1:строка → файл2:строка]

         (повторить для каждой точки из таблицы)

         Если контракт не удаётся определить — укажи Contract: unknown с пояснением почему.",
  subagent_type="onec-code-explorer"
)
```
