# 1C Forms — анализ и валидация (без генераторов Form.xml)

В репозитории **нет** конвейера JSON → Form.xml и скриптов создания/редактирования форм: это выровнено с запретом на правку метаданных и `Form.xml` агентом (см. `.cursor/rules/1c-agent-delegation.mdc` § XML WRITE GUARD, `1c-no-metadata-creation.mdc`).

**Как менять формы:** Конфигуратор → выгрузка в `src/`, либо **программное создание элементов** в модуле формы (сервер/клиент BSL: `Элементы.Добавить`, привязки к данным и т. п. по документации платформы).

Платформенный референс: `.cursor/docs/platform/Глава 7. Формы.md` (Grep+Read, не целиком).

---

## Подскиллы

| Подскилл | Назначение |
|----------|------------|
| **`1c-forms/info`** | Сводка по существующему `Form.xml` (дерево элементов, реквизиты, команды). |
| **`1c-forms/validate`** | Структурная проверка `Form.xml` (ID, спутники, DataPath). |
| **`1c-forms/patterns`** | Справочник архетипов и соглашений при **проектировании** UI (без генерации XML). |

Команды см. в `info/SKILL.md` и `validate/SKILL.md` (пути к `form-info.ps1` / `form-validate.ps1` от корня репозитория: `.cursor/skills/1c-forms/...`).

## MCP (опционально)

- **`user-1c-forms-get_xsd_schema`** — XSD для ориентира при разборе/валидации.
- **`user-1c-forms-get_instructions`** — интеграция форм в метаданные (справочно).

Инструменты JSON-схемы для несуществующего в репозитории конвейера компиляции не задействовать как «генератор форм».

---

## OpenSpec / apply

Задачи вида «добавить колонку / поле на форме» в артефактах ЗНИ формулировать как:

1. **Ручное конфигурирование** в Конфигураторе + выгрузка (шаблон в `1c-agent-delegation.mdc` § XML WRITE GUARD), и/или  
2. **Программная доставка в BSL** модуля формы (создание элементов кода, видимость, данные) с ожидаемым поведением приёмки **без** изменения `Form.xml` из скиллов/агентов.
