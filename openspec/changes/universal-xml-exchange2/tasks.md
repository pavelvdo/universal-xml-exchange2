# Срез S1: Модель профиля и статусы

**Сценарий:** администратор настраивает единый профиль `рг_НастройкиОбменаXML` с ролью `Источник`: на источнике задаёт правила в хранилище, параметры, префиксы и статус «Новое»; на приёмнике — адрес, логин, пароль и пару префиксов; форма переключает видимость реквизитов по роли.
**Primary acceptance:** создать профиль-источник (`Источник = Истина`) с загруженными правилами, параметрами и префиксами → установить статус «Новое» → записать без ошибок; создать профиль-приёмник (`Источник = Ложь`) с адресом, логином и теми же префиксами → записать без ошибок.
**Приёмка:** ручной тест в ИБ с развёрнутым расширением.
**Связь со spec:** exchange-settings — Requirement «Ролевая модель профиля источник или приёмник» (Scenario «Профиль источника», «Профиль приёмника»), Requirement «Префиксы идентификации сеанса» (Scenario «Настройка префиксов»), Requirement «Статус жизненного цикла обмена» (Scenario «Ручная инициация обмена»), Requirement «Правила обмена в хранилище профиля» (Scenario «Загрузка правил пользователем», «Проверка при записи источника»), Requirement «Профиль обмена хранит параметры подключения» (Scenario «Создание профиля приёмника», «Создание профиля источника»), Requirement «Табличная часть произвольных параметров» (Scenario «Настройка параметров на источнике»).
**Зависимости:** нет.

# Срез S2: Полный цикл обмена (GetData → загрузка → Confirm)

**Сценарий:** оператор на приёмнике запускает обмен по профилю-приёмнику; приёмник вызывает `GetData` с префиксами; источник под блокировкой переводит статус «Новое» → «Выполняется», выгружает встроенным движком `рг_УниверсальныйОбменДаннымиXML` по правилам из хранилища, возвращает ZIP (правила + данные); приёмник распаковывает, загружает данные тем же движком и вызывает `ConfirmDataReceipt`; источник переходит в «Выполнено».
**Primary acceptance:** на источнике опубликован `rgExchangeService` → на приёмнике открыть обработку `рг_УниверсальныйОбменДаннымиXML` → выбрать профиль-приёмник → запустить обмен → данные из источника появились в базе-приёмнике, статус профиля-источника стал «Выполнено».
**Приёмка:** ручной end-to-end тест на двух ИБ.
**Связь со spec:** exchange-web-service — Requirement «Операция GetData» (Scenario «Успешный вызов GetData из статуса Новое»), Requirement «Входные параметры GetData» (Scenario «Использование параметров профиля»), Requirement «Идентификация профиля по префиксам» (Scenario «Поиск профиля по префиксам»), Requirement «Операция ConfirmDataReceipt» (Scenario «Успешное подтверждение»); exchange-export — Requirement «Загрузка правил из хранилища профиля» (Scenario «Выгрузка с правилами из профиля»), Requirement «Выгрузка через обработку рг_УниверсальныйОбменДаннымиXML» (Scenario «Запуск встроенного движка»), Requirement «Инициализация параметров правил» (Scenario «Подстановка параметров перед выгрузкой»), Requirement «Упаковка результата в ZIP» (Scenario «Состав архива»); exchange-import — Requirement «Чтение настроек профиля» (Scenario «Подготовка сеанса по профилю приёмника»), Requirement «Вызов веб-сервиса и передача параметров» (Scenario «Успешный запрос данных»), Requirement «Распаковка и загрузка XML» (Scenario «Загрузка после получения архива»), Requirement «Подтверждение доставки ConfirmDataReceipt» (Scenario «Успешный цикл с подтверждением»); exchange-settings — Requirement «Статус жизненного цикла обмена» (Scenario «Завершение обмена»).
**Зависимости:** S1.

# Срез S3: Ошибки и журнал регистрации

**Сценарий:** при недопустимом статусе, ошибке выгрузки, ненайденном/неоднозначном профиле и ошибке загрузки система отказывает корректно, не повреждает статусы happy-path и фиксирует переходы и ошибки в журнале регистрации.
**Primary acceptance:** вызвать `GetData` для профиля в статусе ≠ «Новое» → сервис отклоняет вызов без выгрузки; смоделировать ошибку выгрузки после перевода в «Выполняется» → статус профиля становится «Ошибка» и в журнал регистрации записано описание ошибки; на форме обработки приёмника при отказе WS — краткий текст ошибки без стека вызовов.
**Приёмка:** ручной тест отказных путей на двух ИБ с просмотром журнала регистрации.
**Связь со spec:** exchange-web-service — Requirement «Операция GetData» (Scenario «Отклонение GetData при другом статусе»), Requirement «Операция ConfirmDataReceipt» (Scenario «Подтверждение при неверном статусе»); exchange-export — Requirement «Перевод статуса при выгрузке» (Scenario «Ошибка выгрузки»); exchange-import — Requirement «Подтверждение доставки ConfirmDataReceipt» (Scenario «Ошибка загрузки без подтверждения»).
**Зависимости:** S2.

## 1. Отказные пути и журнал (BSL)

- [x] S3.1 В серверной выгрузке (`рг_УниверсальныйОбменXMLСервер.Ext/Module.bsl` / `rgExchangeService`): при статусе профиля ≠ «Новое» `GetData` MUST NOT выполнять выгрузку и возвращать ошибку клиенту (design Behavior Contract инварианты; spec exchange-web-service «Отклонение GetData при другом статусе»)
- [x] S3.2 В серверной выгрузке (`рг_УниверсальныйОбменXMLСервер.Ext/Module.bsl`): при неустранимой ошибке фазы 2 после перевода в «Выполняется» — отдельной записью установить «Ошибка» и записать описание в журнал регистрации; исключение клиенту (design D2, Behavior Contract; spec exchange-export «Ошибка выгрузки»)
- [x] S3.3 В серверном модуле (`рг_УниверсальныйОбменXMLСервер.Ext/Module.bsl`): для `GetData` и `ConfirmDataReceipt` при «профиль не найден» или «найдено более одного» по префиксам — ошибка клиенту и запись в журнал без смены статусов существующих профилей (design Behavior Contract инварианты; F6)
- [x] S3.4 В модуле веб-сервиса `rgExchangeService` (`Ext/Module.bsl`): `ConfirmDataReceipt` при статусе «Выполняется» → «Выполнено»; при статусе «Выполнено» с теми же префиксами — успех no-op; при ином статусе — ошибка без смены на «Выполнено» (design D3; spec exchange-web-service «Подтверждение при неверном статусе», «Повторное подтверждение»)
- [x] S3.5 В серверном модуле (`рг_УниверсальныйОбменXMLСервер.Ext/Module.bsl`): при каждом изменении `Статус` и при ошибках обмена записывать событие подсистемы обмена в журнал регистрации с префиксом профиля и текстом ошибки (design § Журнал регистрации; spec exchange-export «Ошибка выгрузки»)
- [x] S3.6 В приёмнике (`рг_УниверсальныйОбменXMLСервер.Ext/Module.bsl` / форма обработки): при ошибке загрузки на приёмнике NOT вызывать `ConfirmDataReceipt`, информировать пользователя или записать ошибку в журнал (spec exchange-import «Ошибка загрузки без подтверждения»)
- [x] S3.7 В `рг_УниверсальныйОбменXMLСервер` (`Ext/Module.bsl`): при ошибках вызова WS (`ВызватьGetDataНаИсточнике`, `ВызватьConfirmDataReceiptНаИсточнике`) и в `ВыполнитьСеансОбменаПоПрофилюПриемника` формировать **краткое сообщение для пользователя** — извлечь бизнес-текст из исключения SOAP/WS (внутреннее «по причине: …» / текст сервера), **без** стека вызовов и служебных обёрток «Ошибка при вызове метода контекста»; полный текст — только в журнал регистрации (design § UX ошибок WS; spec exchange-import «Понятное сообщение об ошибке веб-сервиса»; verified: as-built показывает `ПодробноеПредставлениеОшибки` на форме обработки)

  - Scenario «Отклонение GetData при другом статусе» (включено в Primary): выгрузка не выполняется, возвращена ошибка
  - Scenario «Ошибка выгрузки» (включено в Primary): статус «Ошибка», в журнал записано описание ошибки

  - Scenario «Ошибка загрузки без подтверждения» (опционально): при ошибке загрузки на приёмнике подтверждение не вызывается, ошибка показана/записана
