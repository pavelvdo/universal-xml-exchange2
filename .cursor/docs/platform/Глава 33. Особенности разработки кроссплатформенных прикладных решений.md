# Глава 33. Особенности разработки кроссплатформенных прикладных решений

## 33.1. Общая информация

Рекомендации по разработке прикладных решений, работающих под Windows, Linux и (где применимо) macOS: учёт различий файловых систем, путей, прав доступа, шрифтов и недоступности части технологий на не-Windows ОС.

## 33.2. Особенности файловой системы

**Регистр имён.** В Linux имена файлов и каталогов регистрочувствительны; `file.txt` и `File.txt` — разные файлы и могут находиться в одном каталоге.

**Диски и монтирование.** В Linux нет букв дисков; есть корневая ФС `/`, в произвольные точки которой монтируются другие ФС (диски, разделы, флешки, ISO). Типичные точки монтирования — `/mnt`, `/media` (например `/media/<Метка раздела>` или `/media/<Идентификатор устройства>`). Автоматическое монтирование не гарантируется.

**Типичные проблемы:**
- Жёстко заданные пути, рассчитанные только на Windows.
- Проверки введённых путей с требованием буквы диска.
- Путь от корня: в Linux `/temp/file.txt` однозначно задаёт файл в каталоге `temp` корневой ФС; в Windows путь `\temp\file.txt` — каталог `temp` в корне *текущего* диска, который может меняться между запусками.

**Разделители пути.** Windows — `\`, Linux — `/`. Методы встроенного языка принимают любой разделитель. При передаче пути во внешнее приложение нужно использовать разделитель целевой ОС. Функции: `ПолучитьРазделительПути()`, `ПолучитьРазделительПутиКлиента()`, `ПолучитьРазделительПутиСервера()`.

**Файловые маски.** Маска `*.*` в Windows означает «все файлы», в Linux — только файлы, в имени которых есть «.». Это влияет на `НайтиФайлы()`, `УдалитьФайлы()` и `ДиалогВыбораФайла`. В синтакс-помощнике к этим функциям описаны особенности масок по ОС. Универсальная маска «все файлы»: `ПолучитьМаскуВсеФайлы()`, `ПолучитьМаскуВсеФайлыКлиента()`, `ПолучитьМаскуВсеФайлыСервера()`.

**Сетевые ресурсы.** В Linux нет доступа к сетевым ресурсам в UNC-нотации (`\\server\folder\file.ext`). Ресурс нужно монтировать в корневую ФС (например через smbclient); такие приложения могут отсутствовать в конкретной установке Linux.

## 33.3. Работа с внешними устройствами

Имена и доступ к устройствам зависят от ОС. Последовательные порты: в Windows — COM1, COM2; в Linux (зависит от дистрибутива и устройства) — `/dev/ttyS0`, `/dev/ttyS1` или `/dev/ttyUSB0`, `/dev/ttyUSB1` и т.д. Учитывать при разработке внешних компонент и диалогов настройки: список портов получать из самой внешней компоненты (специальный метод).

## 33.4. Ограниченность в правах

В Linux не принято работать от имени root. У обычного пользователя запись обычно возможна только в домашний каталог и каталог временных файлов; остальная ФС может быть только для чтения или недоступна.

Доступ к каталогам, возвращаемым методами глобального контекста:
- `КаталогДокументов()` — чтение и запись;
- `КаталогПрограммы()` — только чтение;
- `КаталогВременныхФайлов()` — чтение и запись;
- `РабочийКаталогДанныхПользователя()` — чтение и запись.

Доступ к внешним устройствам может требовать членства пользователя в определённой группе (уточнять в документации ОС).

## 33.5. Шрифты

Рекомендуется использовать стилевые шрифты без явного указания гарнитуры. При необходимости — шрифты из набора Microsoft Core Fonts (кросс-платформенно доступны). Другие гарнитуры могут отсутствовать в части ОС, что искажает внешний вид решения.

## 33.6. Механизмы COM, OLE, ActiveDocument

В Linux и macOS эти механизмы недоступны. Для интеграции использовать альтернативы: обмен файлами (например XML), веб-сервисы. Внешние компоненты на COM рекомендуется перевести на технологию NativeAPI.
