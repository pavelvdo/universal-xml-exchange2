# Вендорские стандарты 1С — конспект для агентов

Стандарты и методики разработки конфигураций для платформы 1С:Предприятие 8 (ИТС v8std). Полный текст по доменам: `.cursor/docs/standard/std-NN-<domain>.md`. Навигация: `.cursor/docs/standard/1c-standards-navigator.md`.

**Когда загружать:** architect — при проектировании (метаданные, хранение, обработчики, транзакции, безопасность); reviewer — при проверке соответствия вендорскому стандарту.

---

## 1. Метаданные и именование

- Объекты верхнего уровня сортировать по имени; подчинённые — по проектной логике. Общие реквизиты-разделители — по порядку установки параметров сеанса.
- Подсистемы: по функциональному признаку; «Включать в командный интерфейс» только у нужных.
- Общие модули: постфиксы ВызовСервера, Клиент, КлиентСервер, ПовтИсп, ПолныеПрава, Переопределяемый; серверные — без «Вызов сервера», с Внешнее соединение и Клиент (обычное приложение).
- Роли: от должности (Бухгалтер) или от действия (ДобавлениеИзменениеНСИ). Стандартные: ПолныеПрава, АдминистраторСистемы, ИнтерактивноеОткрытиеВнешнихОтчетовИОбработок.
- Регламентные задания — единственное число, от существительного. Определяемые типы — единственное число, от назначения; не дублировать имя типа.
- Документы — суть процесса, лаконично. Регистры (сведения/накопления) — множественное число, от состава записей. Бизнес-процессы и задачи — единственное число.
- Неиспользуемых объектов и кода быть не должно. Устаревшие — префикс Удалить, в конце ветки.

**Глубина:** Read `.cursor/docs/standard/std-01-metadata.md`.

---

## 2. Хранение данных

- Выбор типа: условно-постоянная информация — регистр сведений/справочник; события с документооборотом — документ; накопленные показатели — регистр накопления/сведений.
- Имя, синоним, комментарий — по правилам русского языка; подсказка и проверка заполнения для реквизитов.
- Коды/номера — единая система; строковые типы — обоснованная длина. Составной тип — только где необходимо.
- Проведение: движения в ПриЗаписи; отмена проведения — симметрично. Не выполнять в транзакции записи тяжёлые запросы и взаимодействия с пользователем.
- Активность движений: отбор по Активность = ИСТИНА где релевантно. Самодостаточность регистров. Реквизит «Комментарий» у документов — стандартный.
- Константы — для редко меняющихся значений. Неактуальные объекты — помечение на удаление, учёт в запросах.

**Глубина:** Read `.cursor/docs/standard/std-02-data-storage.md`.

---

## 3. Обработчики событий объектов

- ПередЗаписью: проверки и подготовка; не тяжёлые запросы; не менять данные вне записываемого объекта.
- ПриЗаписи: движения, уведомления, побочные эффекты после записи.
- ПередУдалением: проверки возможности удаления. ПриКопировании — копирование. ОбработкаПроверкиЗаполнения — проверки заполнения; ОбработкаЗаполнения — заполнение по умолчанию.
- Проверки, требующие актуальных данных из БД, — в транзакции записи; остальные — по возможности до/вне.
- ОбменДанными.Загрузка: в обработчиках учитывать — при загрузке обмена не выполнять лишние проверки и движения.
- ОбработкаПолученияПредставления / ОбработкаПолученияПолейПредставления — представление объекта.

**Глубина:** Read `.cursor/docs/standard/std-03-event-handlers.md`.

## 4. Запросы

- Оформление: псевдонимы источников (КАК), единый стиль; не конкатенировать фрагменты без необходимости. Проверка пустого результата — через .Пустой() или явная проверка выборки.
- ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ — только при обосновании. ОБЪЕДИНИТЬ / ОБЪЕДИНИТЬ ВСЕ — соответствие полей и типов. Упорядочивание — явное УПОРЯДОЧИТЬ ПО при необходимости.
- ПОДОБНО — осторожно с производительностью. Условия — по возможности по индексам; не разыменовывать составные типы без ВЫРАЗИТЬ.
- Ограничения на соединения с вложенными запросами и виртуальными таблицами; на вложенные запросы в условии соединения. Виртуальные таблицы (Остатки, СрезПоследних) — по правилам платформы.
- Временные таблицы (ПОМЕСТИТЬ) — по назначению. Минимизировать объём выборки и количество запросов; не усложнять запрос без необходимости.

**Глубина:** Read `.cursor/docs/standard/std-04-queries.md`.

## 5. Транзакции и блокировки

- Транзакции: явное начало/конец; не плодить вложенность; откат при ошибке. В процедурах внутри транзакции — не выводить сообщения и вопросы пользователю.
- Управляемая блокировка — при необходимости; не блокировать избыточно. Блокировка данных объекта для редактирования из кода — по правилам.
- Ответственное чтение — при чтении с последующей записью. Чтение отдельных реквизитов — не загружать объект целиком. Остатки перед изменением — при необходимости блокирующее чтение в начале транзакции.
- Режим разделения итогов (бухгалтерия, накопление) — по стандарту. Сдвиг границы последовательности документов — учитывать.

**Глубина:** Read `.cursor/docs/standard/std-05-transactions.md`.

## 6. Код и модули

- Структура модуля: #Область ПрограммныйИнтерфейс → СлужебныйПрограммныйИнтерфейс → СлужебныеПроцедурыИФункции. Имена процедур/функций — глаголы/предикаты; описание в комментариях.
- Параметры — по значению где возможно; структуры как параметры — по правилам. Переменные — осмысленные имена; предварительная инициализация где нужно.
- Обработчики событий: параметр «Отказ» — устанавливать при невозможности выполнить действие. Дублирование — выносить в общие процедуры/функции.
- Директивы &НаКлиенте, &НаСервере — явно; в общих модулях не ветвить препроцессором по контексту — разделять по модулям Клиент/Сервер.
- Перехват исключений — с логированием; не глотать без записи. Журнал регистрации — для важных событий и ошибок.
- Модуль объекта / менеджера / общие: разделение ответственности. Экспортные процедуры — только где нужен вызов извне. Переменные модуля — не вместо параметров и ДополнительныеСвойства.
- Получение метаданных — через Метаданные() объекта/ссылки, не через глобальный Метаданные. Тип значения — ТипЗнч() = Тип(...).

**Глубина:** Read `.cursor/docs/standard/std-06-code-modules.md`.

## 7. Клиент-сервер

- На сервере вместо ТекущаяДата() — ТекущаяДатаСеанса(). На клиенте ТекущаяДата() недопустима — дату/время передавать с сервера или брать из данных (дата документа). Универсальная отметка времени — УниверсальноеВремя().
- Минимизировать количество серверных вызовов; тяжёлую логику — на сервере. Предопределённые значения на клиенте — получать с сервера или из модуля с повторным использованием.
- Модули с повторным использованием возвращаемых значений (ПовтИсп) — для кеширования; не кешировать в параметрах сеанса то, что для ПовтИсп.
- Файловая система: доступ с учётом безопасности; временные каталоги, передача файлов клиент–сервер. Таймауты при работе с внешними ресурсами — задавать.

**Глубина:** Read `.cursor/docs/standard/std-07-client-server.md`.

---

## 8. Безопасность

- Вызов сервера — только из доверенного кода; общие модули с «Вызов сервера» — минимальный набор; без мутабельных параметров в экспорте.
- Пароли — не в коде и не в открытом виде. Выполнить/Вычислить на сервере — не использовать (или обоснованно). Запуск внешних приложений — с проверкой.
- Роли и права: при добавлении объектов/полей — настраивать права. Проверка прав в коде — ПравоДоступа(действие, метаданные), не РольДоступна для прикладных прав. Привилегированный режим — только при необходимости.
- РАЗРЕШЕННЫЕ в запросах — с учётом влияния на производительность и RLS.

**Глубина:** Read `.cursor/docs/standard/std-08-security.md`.

---

## 9. Формы и UI

- Пользовательские настройки — через хранилище настроек. Параметры выбора и связи — задавать для отборов в формах.
- РеквизитФормыВЗначение / ДанныеФормыВЗначение — по назначению. Обработчики, подключаемые из кода, — префикс Подключаемый_.
- Информирование пользователя — не Сообщить(), а БСП (ОбщегоНазначения.СообщитьПользователю и т.п.); сообщения привязать к полю где возможно. В транзакции — не выводить сообщения/вопросы.
- Длительные операции — через механизм длительных операций на сервере.

**Глубина:** Read `.cursor/docs/standard/std-09-forms-ui.md`.

---

## 10. БСП и библиотеки

- Именование в иерархии библиотек — по стандартам. Переопределяемые и поставляемые объекты — различать. Переопределение общих модулей — через иерархию, не дублирование.
- Обратная совместимость библиотек при изменениях API и метаданных. Роли в библиотеках — по общим правилам. Обработчики обновления ИБ — по правилам БСП.

**Глубина:** Read `.cursor/docs/standard/std-10-bsp-libraries.md`.

---

## 11. Локализация

- Тексты для пользователя — через НСтр("ru = '...'"); не хардкодить строки. Имена объектов метаданных в сообщениях — из метаданных (Представление(), ПолноеИмя()), не литералы.
- СтрШаблон + НСтр для подстановки имён объектов в сообщения.

**Глубина:** Read `.cursor/docs/standard/std-11-general.md` (разделы по локализации и интерфейсным текстам).

---

## Использование агентом

- **Architect:** при проектировании прочитать скилл, применить чеклист по затронутым доменам (метаданные, хранение, обработчики, транзакции, безопасность, БСП). При необходимости детали — Read соответствующий доменный файл из `.cursor/docs/standard/` (см. навигатор).
- **Reviewer:** при проверке на соответствие вендорскому стандарту прочитать скилл, проверить все домены, затронутые кодом. При сомнении — Read соответствующий доменный файл (навигатор).
- **Writer:** скилл не загружать; достаточно правил из `.cursor/docs/1c-coding-standards.md` и секции «Ключевые вендорские принципы» там же.
