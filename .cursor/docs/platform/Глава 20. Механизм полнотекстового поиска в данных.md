# Глава 20. Механизм полнотекстового поиска в данных

> **Agent ref:** Полнотекстовый поиск в данных; индекс; поисковые операторы (И, ИЛИ, НЕ, РЯДОМ, *, !); хештеги (#Хэштег); объекты поиска (справочники, документы, регистры, БП, задачи и др.); версия 1 / версия 2; менеджер ПолнотекстовыйПоиск; СписокПолнотекстовогоПоиска; регламентное задание обновления индекса. Синтаксис выражений — Приложение 5.
> **Ключевые разделы:** 20.1 Общие сведения и индекс; 20.2 Использование (API, ранжирование); 20.3 Дополнительные словари (только v1); 20.4 Актуальность индекса; 20.5 Мобильная платформа.

## 20.1. Общие сведения

Механизм полнотекстового поиска позволяет искать информацию в базе с поисковыми операторами (И, ИЛИ, НЕ, РЯДОМ и др.) и хештегами (#Хэштег). Основан на:

- **Индексе полнотекстового поиска** — словарь слов и мест их появления в индексируемых объектах; создаётся и поддерживается платформой.
- **Средствах выполнения** полнотекстового поиска.

**Объекты поиска:** планы обмена, справочники, документы, планы видов характеристик, планы счетов, планы видов расчёта, регистры сведений, накопления, бухгалтерии, расчёта, бизнес-процессы, задачи. Для каждого объекта есть свойство **Полнотекстовый поиск** (включить/исключить из индексирования).

Изменения объектов поиска фиксируются в **журнале регистрации изменений** при записи в БД. В регистрацию попадают только объекты с включённым полнотекстовым поиском. При откате транзакции запись в БД отменяется, запись в журнале изменений остаётся.

Индексирование выполняется в **привилегированном режиме** (сервер), без монопольного захвата БД. Индексируются реквизиты типов: **строка**, **дата**, **число**. В индекс попадают: имя объекта/реквизита, синонимы (все языки конфигурации), представление объекта (все языки). Стандартные и пользовательские реквизиты индексируются на всех языках. При индексировании — транслитерация и замена латиницы на кириллицу (индекс хранит обе формы). Расположение файлов индекса — см. документацию платформы (ИТС).

**Версии поиска:**

- **Версия 1** (с 8.3): основной индекс + дополнительный по изменениям после последнего обновления; поиск по основному быстрый, по дополнительному — медленнее; предусмотрено слияние индексов (может быть длительным). Буква «ё» при индексации заменяется на «е» (ёж = еж). Обновление индекса — фоновое задание, реализуемое разработчиком.
- **Версия 2** (с 8.3.22): один индекс, автоматическое обновление при изменениях в БД; «е» и «ё» различаются. Версия 2 не поддерживается в файловом варианте ИБ (кроме варианта с автономным сервером).

Полнотекстовый поиск инициируется на клиенте; **программный интерфейс доступен только на сервере** (в файловом варианте поиск выполняется на клиенте, в клиент-серверном — в кластере). Поиск учитывает **права пользователя** (в т.ч. RLS и ограничения на уровне полей). Учитываются варианты написания с латиницей вместо кириллицы (например, «системf» вместо «система»).

## 20.2. Использование механизма

Ограничения поисковой строки: длина не более 1 000 символов, не более 32 слов; при превышении — поиск не выполняется, формируется исключение. Синтаксис поисковых выражений — **Приложение 5**.

**Хештеги:** #Заказ — объекты, где встречается хештег #Заказ; #Заказ* — все хештеги, начинающиеся с «Заказ» (#Заказ, #ЗаказВнешний и т.д.). Результаты возвращаются **порциями** (размер задаётся при вызове).

**Ранжирование:** приоритет — усиление «веса» слова в запросе; «вес» объекта метаданных (больше ссылок — выше вес); дата (новее — выше).

**Программный интерфейс:** свойство глобального контекста `ПолнотекстовыйПоиск` — менеджер типа `МенеджерПолнотекстовогоПоиска`. Вызов: `ПолнотекстовыйПоиск.ИмяМетода()`.

**Управление:**

- `ПолучитьРежимПолнотекстовогоПоиска()` — перечисление РежимПолнотекстовогоПоиска (Разрешить / Запретить).
- `ПолучитьВерсиюПолнотекстовогоПоиска()` — текущая версия.
- `ДатаАктуальности()` — дата последнего момента полной актуальности индекса.
- `ИндексАктуален()` — Истина, если индекс соответствует состоянию ИБ (v2: при большом потоке изменений может «мигать» при частых вызовах).
- `ОбновлениеИндексаЗавершено()` — v1: Истина, если слияние не требуется; v2: Ложь, если `ИндексАктуален()` = Ложь, иначе Истина.
- `ПолучитьМаксимальныйРазмерИндексируемыхДанных()` — макс. размер индексируемых данных (байты). До первого вызова — 1 МБ.
- `ПолучитьМаксимальныйРазмерИндексируемыхДанныхРеквизита()` — макс. размер данных одного реквизита, индексируемых поиском.
- `ПолучитьКоличествоЗаданийИндексирования()` / `УстановитьКоличествоЗаданийИндексирования()` — число фоновых заданий индексирования (можно менять по нагрузке).
- `ПолучитьРежимРазделенияСоставныхСлов()` / `УстановитьРежимРазделенияСоставныхСлов()` — только v1; смена режима требует перестроения индекса.

**Индексирование и версия:**

- `УстановитьВерсиюПолнотекстовогоПоиска()` — выбор версии; смена версии требует **монопольного доступа** к ИБ. При переходе 1→2: проверка возможности v2, новые задания и соединения используют v2, используется текущий индекс v2 (при неактуальности поиск ограничен им), индекс v1 не трогается (при необходимости очистить до смены). При переходе 2→1: задания индексации прерываются, обновление v2 прекращается, новые соединения — v1, индекс v2 остаётся (очистка при необходимости до смены).
- `УстановитьРежимПолнотекстовогоПоиска()` — Разрешить/Запретить; при снятии запрета индекс очищается.
- `ОбновитьИндекс()` — обновление индекса; при отсутствии — полная переиндексация. v1: параметры РазрешитьСлияние, Порционное (Истина — порции по 10 000 объектов). v2: запуск обновления и немедленный возврат; обновление в фоне; состояние — `ИндексАктуален()`.
- `ОчиститьИндекс()` — удаление файлов индекса (рекомендуется после массового обновления/загрузки ИБ). v1 — синхронно; v2 — асинхронно, задание индексации при вызове прерывается.

**Выполнение поиска:** `СоздатьСписок(СтрокаПоиска, РазмерПорции)` — возвращает `СписокПолнотекстовогоПоиска`. У объекта можно менять `СтрокаПоиска` и `РазмерПорции`. Методы: `ПерваяЧасть()` — первые результаты; `СледующаяЧасть(ТекущаяНачальнаяПозиция)` и `ПредыдущаяЧасть(ТекущаяНачальнаяПозиция)` — следующие/предыдущие порции (без параметра используется `НачальнаяПозиция()`; параметр предпочтителен для скорости). `Количество()` — число элементов в текущей части (≤ размера порции); `ПолноеКоличество()` — всего найдено. При достижении конца/начала данных методы порции вызывают исключение (обработка Попытка/Исключение). `СлишкомМногоРезультатов()` — Истина, если результаты были усечены по производительности (анализировать при последней порции).

**Ограничения:** свойство `ОграничиватьСтрокуПоиска` (Истина) — проверка: длина ≤ 1 000 символов, слов ≤ 32; иначе исключение. `ПолучатьОписание` — Истина заполняет поле Описание у результатов (контекст найденных слов), Ложь ускоряет поиск.

**Обход результатов:** цикл `Для Каждого … Из … Цикл` по списку. Элемент — `ЭлементСпискаПолнотекстовогоПоиска`: **Значение** (объект или набор записей), **Метаданные**, **Представление**, **Описание** (пары реквизит:значение, где найдено выражение). Формат Описание: v1 — каждая пара с новой строки; v2 — в одну строку, разделитель «;», имя реквизита может опускаться. Отображение результата — метод `ПолучитьОтображение()`.

## 20.3. Дополнительные словари

**Только для полнотекстового поиска версии 1.**

Дополнительные словари морфологии и синонимов расширяют системные (спецтермины, слова конфигурации). Источники: макеты (двоичные и текстовые), константы типа строка и ХранилищеЗначения. Задаются в свойстве **Дополнительные словари** корневого объекта метаданных.

Структура словаря (элемент Dictionary):

- **Words:** lemma — основная форма (им. п.), forms — падежные формы. Нечёткий поиск по умолчанию выключен; оператор «*» (например, руне*) находит рунет, рунета, рунете и т.д.
- **Synonyms:** наборы синонимов в тегах item. Оператор «!»: !ошибка находит ошибка, баг, сбой. Элементы Words и Synonyms могут идти в любом порядке.
- **Roots:** словарь значащих слов для языков, отличных от русского — включение режима разделения составных слов и поиск по частям.

Словари загружаются при первом поиске или индексировании. При ошибках в макете загрузка останавливается. После изменения словаря нужен перезапуск: файловая ИБ (прямое подключение) — все клиенты; файловая через веб-сервер — расширение веб-сервера; клиент-серверный — сервер «1С:Предприятие». Индекс при этом не обновляется автоматически — перестроить вручную.

## 20.4. Поддержка актуальности индекса

Индекс нужно поддерживать в актуальном состоянии: добавлять новые объекты, удалять данные удалённых, обновлять изменённые.

**Версия 1:** регламентное задание с вызовом `ОбновитьИндекс()`; расписание — минимизация задержки (часто — раз в 60 с). Порционное обновление без слияния уменьшает нагрузку при первичном заполнении (по 10 000 объектов за запуск). Слияние основного и дополнительного индекса — отдельно или в том же задании в период минимальной нагрузки.

**Версия 2:** достаточно один раз инициировать первоначальное заполнение (например, через «Управление полнотекстовым поиском»); актуализация выполняется платформой автоматически. При формировании индекса v2 проверяется свободное место на диске каталога индекса: при менее 10% индексация прерывается, индекс становится неактуальным, поиск недоступен; после освобождения места индексация продолжается.

Для гарантированного обновления индекса после синхронизации ИБ (в т.ч. с мобильного) можно явно вызвать `ОбновитьИндекс(Истина, Ложь)` в коде начального заполнения/синхронизации.

## 20.5. Мобильная платформа

Ограничения: нет версии 2; нет поиска с морфологией; нет дополнительных словарей; нет операции извлечения текста (ИзвлечениеТекст).

Особенности: обновление индекса — автоматически после создания/изменения объектов с включённым полнотекстовым поиском; создаётся фоновое задание (если нет серверного вызова и другого фонового задания), порции по 10 000 объектов; при полной проиндексированности ИБ слияние индексов запускается через 20 с бездействия. После синхронизации ИБ с ПК для гарантии индексирования — явный вызов `ОбновитьИндекс(Истина, Ложь)`.
