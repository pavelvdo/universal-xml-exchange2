# Design Challenge — universal-xml-exchange2 (incremental, delta extend 2026-06-25)

## Адверсариальная установка

Challenge ограничен **delta** после extend 2026-06-25 (движок: внешний epf в макете `ДвижокОбмена` → встроенная обработка `рг_УниверсальныйОбменДаннымиXML`, runtime `Обработки.рг_УниверсальныйОбменДаннымиXML.Создать()`) и инвариантами двухфазного обмена. Прочитаны `proposal.md`, `design.md`, `tasks.md`, delta-specs `exchange-export`/`exchange-import`, `debug.md` (§ Extend), as-built `ObjectModule.bsl` обработки. Прошлые `reports/architecture-*.md` как источник истины **не** использованы; публичный API движка сверен непосредственно по коду as-built. Закрытая ось (as-built встроенная обработка, без epf/контейнера/внешних обработок) переоткрывается только при verified code fact.

## Three-Question Challenge

### Q1 — Problem-Solution Fit

- **Why говорит:** двухфазная модель — статусы профиля, выгрузка только из «Новое», подтверждение приёмником `ConfirmDataReceipt`, правила в хранилище профиля, журналирование переходов; «зависшие» выгрузки видимы.
- **Чего касается delta:** способ создания движка выгрузки/загрузки (epf vs встроенная обработка). Это **транспортный механизм** фазы 2, ортогональный жизненному циклу сеанса.
- **Design адресует:** статусы (`рг_СтатусыОбмена`, D1), атомарный check-and-set «Новое»→«Выполняется» под управляемой блокировкой с коммитом до выгрузки (D2, Behavior Contract), `ConfirmDataReceipt` с идемпотентностью (D3), правила в `ПравилаОбменаДанными` (S1, exchange-export ADDED), журнал переходов (§ Журнал, S3.5).
- **Влияние delta на Why:** нейтральное-к-положительному. Встроенная обработка выполняет ту же выгрузку/загрузку, что и epf, и **усиливает** под-цель «надёжность» — снимает требование «разрешить внешние обработки на сервере» (S2.4 cancelled) и риск отсутствия/несовместимости epf (исходный блокер S2.1).
- **Покрытие:** полное. Delta не вводит регресса инвариантов двухфазного обмена; ни один инвариант Behavior Contract не зависит от способа создания движка.

### Q2 — Optimality

- **Выбранный путь:** встроенная обработка-копия `рг_УниверсальныйОбменДаннымиXML` в cfe; экземпляр через менеджер обработки; XML правил/данных через реквизиты `ИмяФайлаПравилОбмена`/`ИмяФайлаОбмена`; ZIP формирует серверный модуль, не движок.
- **Verified code facts (as-built API существует и экспортирован):**
  - `ЗагрузитьПравилаОбмена(Источник="", ТипИсточника="XMLФайл")` — `ObjectModule.bsl:12395` Экспорт
  - `ИнициализироватьПервоначальныеЗначенияПараметров()` — `:6101` Экспорт
  - `УстановитьЗначениеПараметраВТаблице(ИмяПараметра, ЗначениеПараметра)` — `:1500` Экспорт
  - `ВыполнитьВыгрузку()` — `:13300` Экспорт
  - `ОткрытьФайлЗагрузки(ТолькоПрочитатьШапку=Ложь, ДанныеФайлаОбмена="")` — `:11979` Экспорт
  - `ВыполнитьЗагрузку()` — `:13160` Экспорт
  - Реквизиты `ИмяФайлаОбмена`, `ИмяФайлаПравилОбмена`, `ТаблицаНастройкиПараметров` используются объектным модулем (`:11991`, `:12050`, `:1502`).
  Это снимает основной риск delta (фантомный API движка): Engine Contract в design.md соответствует коду.
- **Альтернативы (включая не упомянутые в `## Implementation Options`):**
  1. **Option A — внешний epf в макете `ДвижокОбмена`** (упомянута, отклонена extend). Минусы verified: требует разрешения внешних обработок на сервере; epf отсутствовал в src (блокер S2.1); drift версии epf. Хуже по надёжности и Blast Radius. Не превосходит.
  2. **Option B — встроенная обработка-копия (выбрана).** Плюсы: as-built уже в src, без внешних обработок. Минус, признанный самим design: «[Risk] Drift движка от cf `УниверсальныйОбменДаннымиXML`».
  3. **Не упомянута: прямой вызов базового объекта cf — `Обработки.УниверсальныйОбменДаннымиXML.Создать()`.** Базовый объект существует в `src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML/`, S2.1 подтвердил совпадение API cf ↔ cfe; вызов читает cf в runtime и **не** нарушает ограничение «cf read-only» и закрытую ось «без внешних обработок». Преимущество: устраняет именно тот drift, который design фиксирует как риск (нет копии на ~13k строк, версия всегда трекает cf). Недостаток: связывает расширение с присутствием/конфигурацией базового объекта и теряет возможность автономно править движок в cfe. Это **равноправная по коду** альтернатива, а не workflow-вариант.
  4. **Не упомянута: общий модуль-обёртка как «движок».** Отклоняется — это не движок выгрузки/загрузки, а лишь фасад; не устраняет потребность в самой обработке-конвертере.
- **Вердикт по Q2:** выбранный путь **не превосходит безусловно**. Альтернатива 3 прямо адресует задокументированный риск drift, но не превосходит однозначно (теряется decoupling), поэтому это не REJECT. Однако `## Implementation Options` сравнивает только Option A/B и **умалчивает** наиболее естественную третью опцию, на которую опирается признанный риск, — это gap постановки.
  - **Reopen-blocked:** альтернатива 3 отменяет закрытое решение «оставить as-built встроенную обработку `рг_УниверсальныйОбменДаннымиXML`». Тег: `reopen-blocked: extend-engine-2026-06-25`. Verified new fact, оправдывающий полный fork, **отсутствует** (копия рабочая, API сверен). Поэтому — **implementation_invariant gap** (документировать обоснование copy-vs-reference и стратегию drift), а не архитектурная развилка.

### Q3 — Fresh-Eye Approval

- **Согласовал бы:** с оговорками (да по существу delta, с двумя уточнениями постановки).
- **Причины «да»:**
  - API движка не выдуман — сверен по as-built коду (см. Q2), Engine Contract исполним.
  - Инварианты двухфазной модели целостны и не затронуты сменой движка; атомарность статуса вынесена в короткую транзакцию, выгрузка — вне неё.
- **Причины «с оговоркой»:**
  - «Свежий глаз» сразу спросит: зачем копировать вендорскую обработку на ~13k строк в расширение и вручную синхронизировать её с cf при каждом обновлении, если есть базовый `Обработки.УниверсальныйОбменДаннымиXML`? В Options этого вопроса нет (Gap 1).
  - Контракт загрузки на приёмнике пересекается с поведением самого движка: as-built `ОткрытьФайлЗагрузки` сам распаковывает `.zip` (`ObjectModule.bsl:11994-11996`) и при `БезопаснаяЗагрузка` сам грузит правила из `ИмяФайлаПравилОбмена` (`:12049-12050`). Постановка же (D-import sequence, S2.15) предписывает серверному модулю распаковать ZIP и явно вызвать загрузку правил — возможен двойной разбор ZIP / несогласованная уборка временных файлов (Gap 2).

## Verdict

**CHALLENGE** — delta движка обоснована и подтверждена кодом, инварианты двухфазного обмена целы; но постановка не закрыла два уточнения, влияющих на сопровождаемость и корректность фазы загрузки.

## Gaps for design.md (обязательно при CHALLENGE)

1. **Copy-vs-reference rationale (Q2, implementation_invariant; `reopen-blocked: extend-engine-2026-06-25`).** В `## Implementation Options` / `## Risks` добавить строку об отклонённой опции «прямой вызов базового `Обработки.УниверсальныйОбменДаннымиXML.Создать()`»: почему выбрана копия в cfe (decoupling, автономная правка движка) и как именно купируется задокументированный drift (триггер пере-выгрузки обработки при смене версии cf, кто и на каком шаге проверяет соответствие). Без fork — закрытая ось держится.
2. **Разграничение распаковки/загрузки на приёмнике (Q3, implementation_invariant).** В D-import sequence и постановке S2.15 явно зафиксировать, кто распаковывает ZIP и кто грузит правила: либо серверный модуль распаковывает и передаёт XML данных + XML правил движку явными вызовами (тогда **не** передавать `.zip` в `ИмяФайлаОбмена`, чтобы не сработала встроенная `РаспаковатьZipФайл`), либо положиться на встроенную распаковку движка. Зафиксировать ответственного за удаление временных файлов, чтобы исключить двойную распаковку.
