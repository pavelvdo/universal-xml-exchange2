## 5.1. Свойства конфигурации

### 5.1.1. Категория свойств «Основные»

**Основной режим запуска** — режим по умолчанию (Управляемое приложение или Обычное приложение). Для новой конфигурации — Управляемое приложение. Нельзя изменить, если Режим совместимости = Версия 8.1.

**Вариант встроенного языка** — основной язык программирования (русский или английский); влияет на синтакс-помощник, имена свойств, имена компонентов форм. Оба варианта конструкций доступны независимо от значения.

**Основные роли** — роли при пустом списке пользователей (без авторизации). Пользователь считается с административными правами вне зависимости от права Администрирование у этих ролей. Основной режим запуска должен соответствовать правам на допустимые приложения у всех выбранных основных ролей.

**Модуль управляемого приложения** — по ссылке Открыть: редактор модуля управляемого приложения.

**Модуль сеанса** — по ссылке Открыть: редактор модуля сеанса.

**Модуль внешнего соединения** — по ссылке Открыть: редактор модуля внешнего соединения.

**Использовать управляемые формы в обычном приложении** — использование управляемых форм в обычном толстом клиенте; меняет правила выбора формы и централизованной проверки конфигурации.

**Использовать обычные формы в управляемом приложении** — доступно только при режиме «Управляемое приложение и обычное приложение»; меняет правила выбора формы и проверки конфигурации.

### 5.1.2. Параметры интерфейса и представления

**Дополнительные словари полнотекстового поиска** — общие макеты или константы как дополнительные словари полнотекстового поиска.

**Хранилища (общие настройки, пользовательские настройки отчетов, варианты отчетов, настройки данных форм, настройки динамических списков, внешние данные навигационных ссылок)** — каждое свойство задаёт хранилище для соответствующего типа настроек. Платформа сама не пишет в хранилище общих настроек — это делает разработчик.

**Командный интерфейс, Рабочая область начальной страницы, Командный интерфейс основного раздела, Картинка основного раздела, Интерфейс клиентского приложения** — настройка интерфейса и панелей.

**Режим основного окна клиентского приложения:**
- **Обычный** — режим по настройкам интерфейса.
- **Рабочее место** — скрыты меню Сервис, панели (кроме сообщений и показателей), навигация по формам, ряд сочетаний клавиш (Ctrl+N, Ctrl+P, F1 и др.).
- **Полноэкранное рабочее место** (только Такси) — плюс к Рабочему месту: при ESC в веб-клиенте выход из полноэкранного режима; скрыты кнопки Свернуть/Развернуть/Закрыть.
- **Киоск** (только Такси) — плюс скрыты гиперссылка с именем пользователя и кнопка О программе.
- **Встроенное рабочее место** — только при встраивании веб-клиента в сторонний сайт; для остальных клиентов игнорируется.

Режим можно задать: в свойствах конфигурации; командой MainWindowMode в командной строке; методом УстановитьРежимОсновногоОкна().

**Основной стиль, Основной язык, Краткая информация, Подробная информация** — представление конфигурации.

### 5.1.2.3. Основные формы

ПРИМЕЧАНИЕ. Размер картинки для логотипа 64×64 точки или меньше.

**Заставка** — картинка 600×255 точек, без прозрачности.

**Основная форма констант** — общая форма для ввода/редактирования констант (из Общие формы). Доступно только при режиме «Управляемое приложение и обычное приложение».

**Основная форма отчета, Основная форма настроек отчета, Основная форма варианта отчета** — общие формы по умолчанию для отчётов без своих форм.

**Основная форма настроек динамического списка** — форма редактирования настроек динамического списка; если не задана — используется автоматически генерируемая.

**Основной макет оформления отчета** — макет по умолчанию для отчётов на СКД. Порядок выбора: настройки компоновки отчёта → УстановитьМакетОформленияПользователя() → УстановитьОбщийМакетОформления() → свойство конфигурации → встроенный «Основной».

### 5.1.3. Категория свойств «Разработка»

**Режим управления блокировкой данных** — вариант управления блокировкой в транзакции.

**Режим хранилища двоичных данных** — Использовать / Не использовать. Доступно при Режим совместимости 8.3.10–8.3.21; с 8.3.22 свойство недоступно, хранилище всегда доступно.

**Режим автонумерации объектов:**
- **ОсвобождатьАвтоматически** — неиспользованные автоматические номера/коды возвращаются в пул (аналогично 8.0).
- **НеОсвобождатьАвтоматически** — номера при записи, не при открытии формы; неиспользованные вне транзакции не переиспользуются. Номер, выделенный в транзакции, повторно используется даже при НеОсвобождатьАвтоматически, если объект не записан.

**Режим использования модальности** — Использовать / Не использовать / Использовать с предупреждением. При «Не использовать» модальные методы приводят к ошибке (в т.ч. при синтаксической проверке). С 8.3.5 для тонкого/толстого клиента проверка только с параметром /EnableCheckModal; веб-клиент всегда с проверкой.

**Режим использования синхронных вызовов расширений платформы и внешних компонент** — Использовать / Не использовать / С предупреждениями. При «Не использовать» на клиенте синхронные методы недоступны и дают ошибки при расширенной проверке; в режиме 1С:Предприятие — исключение. Толстый клиент игнорирует свойство. В веб-клиенте (Chrome/Firefox) при «Не использовать» не загружаются расширения с синхронными методами. Проверка — при /EnableCheckExtensionsAndAddInsSyncCalls; из конфигуратора параметр устанавливается принудительно (кроме толстого клиента).

**Режим совместимости интерфейса** — Версия 8.2 / 8.2. Разрешить Такси / Такси. Разрешить 8.2 / Такси. Веб-клиент поддерживает только Такси.

**Режим использования табличных пространств** — доступно при режиме совместимости ≤ 8.3.22; при > 8.3.22 недоступно, табличные пространства всегда используются. При включении — только 8.3.23+.

**Режим совместимости** — Версия 8.3.25 … 8.3.1, 8.2.16, 8.2.13, 8.1, Не использовать. Конфигурация с неизвестным режимом не поддерживается. Загрузка 1cv8.dt из 8.3.1+ в младшие версии запрещена, кроме случая Режим совместимости = 8.2.16. Не рекомендуется проверять режим на НеИспользовать — сравнивать версию платформы.

### 5.1.4. Разработка для мобильных устройств

**Назначения использования** — Приложение для платформы / для мобильной платформы / оба. Доступно при Основной режим запуска = Управляемое приложение.

**Используемая функциональность мобильного приложения** — отмечается вручную; определяет разрешения сборки, текст запроса разрешения и возможность удаления модуля при отключении функциональности.

**Подпись мобильного клиента** — управление цифровой подписью; проверка соответствия, создание ключа, формирование/очистка подписи. Хеш публичного ключа в Base64 — для MobilePublicKey в .v8i.

**Роли ограничения автономного мобильного приложения** — роли объединяются «по И» с результатом для пользователя (роли пользователя — «по ИЛИ», роли свойства — «по ИЛИ»), ограничивая права на мобильном устройстве.

## 5.5. Ветвь конфигурации «Общие»

Общие объекты: подсистемы, общие модули, параметры сеанса, роли, общие реквизиты, планы обмена, критерии отбора, подписки на события, регламентные задания, функциональные опции и др.

### 5.5.1. Подсистемы

Формируют глобальный командный интерфейс.

ПРИМЕЧАНИЕ. Не рекомендуется имя подсистемы Desktop.

Подсистемы могут быть вложенными. **Подсистема с одной командой** — при активизации раздела с одной доступной командой она выполняется без смены раздела; при обновлении конфигурации проверяется, что в подсистеме не более одной команды (иначе предупреждение). **Картинка** — для панели разделов (вертикальный размер 48 точек, расчёт по максимальному размеру). **Включать в содержание справки** — ветвь справки по подсистеме. **Состав** — закладка редактора для включения объектов в подсистему.

**Метаданные.Подсистемы** — иерархия подсистем: свойство Подсистемы, метод Родитель(), свойство Состав (объекты, входящие в подсистему).

### 5.5.2. Общие модули

ВНИМАНИЕ! Общий модуль может содержать только определения процедур и функций.

Экспортные процедуры и функции — часть глобального контекста. Редактирование: свойство Модуль → Открыть. Сохранение только в составе конфигурации.

**Глобальный (Истина/Ложь):** при Истина — методы доступны как методы глобального контекста; при Ложь — в глобальном контексте свойство с именем модуля (только чтение), значение — ОбщийМодуль, вызов: ИмяМодуля.ИмяМетода().

**Контексты компиляции:** Клиент (обычное приложение), Клиент (управляемое приложение), Сервер, Внешнее соединение. Несколько флажков — код компилируется для каждого контекста. При Сервер + клиент — клиентский вызов использует клиентскую копию, серверный — серверную; для разного поведения использовать препроцессор (#Если ТонкийКлиент и т.д.). С клиента на сервер: вызов серверного общего модуля или процедуры с &НаСервере / &НаСервереБезКонтекста. С сервера нельзя вызывать клиентские общие модули и клиентские методы формы.

**Вызов сервера** — разрешает вызов экспортируемых методов серверного общего модуля с клиента. Рекомендуется Ложь для модулей с методами, не предназначенными для вызова с клиента. При одновременной установке Клиент (обычное)+Клиент (управляемое)+Внешнее соединение свойство Вызов сервера сбрасывается; при установке Вызов сервера эти три сбрасываются.

**Привилегированный** — отключение контроля прав при выполнении методов. Привилегированный общий модуль только на сервере.

**Повторное использование возвращаемых значений:** Не использовать / На время вызова / На время сеанса. Для неглобальных модулей. Параметры и результат запоминаются; при повторном вызове с теми же параметрами возвращается кеш без выполнения. Очистка: На время вызова — при возврате с сервера или завершении процедуры верхнего уровня на клиенте; На время сеанса — конец сеанса / закрытие клиента. Дополнительно значения удаляются через 20 мин после вычисления или 6 мин после последнего использования (в веб/тонком с низкой скоростью — только 20 мин), при нехватке памяти, перезапуске процесса, переключении на другой процесс. Ограничения на типы параметров: примитивы, ссылки на объекты БД, структуры с такими значениями. При возврате объекта возвращается ссылка на объект в кеше; изменение объекта влияет на последующие вызовы. При На время сеанса в возвращаемых значениях нельзя использовать МенеджерВременныхТаблиц, Запрос, прикладные объекты (кроме ссылочных типов). Для использования кеша при вызове из того же модуля вызывать по полному имени: ОбщийМодуль.МояФункция(). ОбновитьПовторноИспользуемыеЗначения() — сброс кеша.

### 5.5.3. Параметры сеанса

Для использования в запросах и RLS; снижают время доступа. Настройка в палитре свойств. Два права: Получение и Установка. Инициализация — в модуле сеанса, обработчик УстановкаПараметровСеанса(). До инициализации значение неопределено; при чтении вызывается УстановкаПараметровСеанса(), если осталось неопределено — исключение.

Отличия от глобальных переменных модуля приложения: объекты метаданных, типизация, контроль прав, хранение на сервере (доступ с сервера и клиента). В RLS для получения значения параметра право не требуется.

ПРИМЕЧАНИЕ. Объём данных при установке одного параметра сеанса между клиентом и сервером не более 4 Гбайт (в сериализованном виде).

Пример: ПараметрыСеанса.ТекущийПользователь = …; в запросе &ТекущийПользователь.

### 5.5.4. Роли и права доступа

#### 5.5.4.1. Общая информация

Два типа прав: основные и интерактивные. Права задаются на конфигурацию, объекты, реквизиты, табличные части, реквизиты ТЧ, стандартные реквизиты. Иерархия: снятие «старшего» снимает «младшие»; установка «младшего» устанавливает «старших». При Просмотр без Редактирование — реквизит только для просмотра; без Просмотр — при открытии формы «Нарушение прав доступа!». Новая роль по умолчанию: на корень — ТонкийКлиент, ВебКлиент, СохранениеДанныхПользователя, Вывод.

#### 5.5.4.2. Привилегированный режим

УстановитьПривилегированныйРежим(Истина/Ложь). В клиент-серверном варианте на клиенте не действует. Число включений должно совпадать с числом выключений; при выходе из процедуры незавершённые включения автоматически выключаются. ПривилегированныйРежим() — текущее состояние. Привилегированный сеанс: ключ UsePrivilegedMode или prmod в строке соединения; только для пользователя с правом Администрирование.

#### 5.5.4.3. Безопасный режим

УстановитьБезопасныйРежим(Истина/Ложь). В безопасном режиме: привилегированный режим отменяется и не включается; запрещены COM, загрузка внешних компонент, доступ к файлам (ЗначениеВФайл, КопироватьФайл, Новый Файл, ЧтениеТекста/ЗаписьТекста с путём и т.д.), Интернет (ИнтернетСоединение, HTTPСоединение и т.д.). Запрещённые операции → исключение. Внешние отчёты/обработки (Файл → Открыть) выполняются в безопасном режиме при отсутствии у пользователя прав Администрирование. Вложенность включений/выключений как у привилегированного режима. Для обработчиков событий объектов метаданных (не в расширении) безопасный режим автоматически отключается. ПолучитьОтключениеБезопасногоРежима(), УстановитьОтключениеБезопасногоРежима() — для управления внутри обработчика; УстановитьОтключениеБезопасногоРежима в расширениях не работает. Не более одного уровня вложенности установки безопасного режима и отключения безопасного режима в одном методе.

#### 5.5.4.4. Режимы удаления данных

Непосредственное удаление или контроль ссылочной целостности (пометка на удаление + проверка ссылок). Отключение права непосредственного удаления в целом запрещает непосредственное удаление в рамках конфигурации. Непосредственное удаление возможно и из встроенного языка.

#### 5.5.4.5. Правила сочетания ролей

По каждому праву: разрешение хотя бы в одной роли → доступ открыт; запрет во всех ролях → доступ закрыт.

#### 5.5.4.6–5.5.4.7. Редактор прав, все роли

Флажки: Устанавливать права для новых объектов; Устанавливать права для реквизитов и ТЧ по умолчанию. Независимые права подчиненных объектов — права родителя не учитываются при определении права подчинённого (иначе отсутствие права у родителя снимает право у подчинённого). Для реквизитов/ТЧ отчётов/обработок: при снятом Независимые права и Редактирование на реквизит без Просмотр на отчёт/обработку — Редактирование для реквизита не установлено. Проверка родителя выполняется до сочетания прав по ролям.

#### 5.5.4.8. Ограничение доступа к данным (RLS)

Используются: роли, параметры сеанса, функциональные опции, привилегированные общие модули, РАЗРЕШЕННЫЕ в запросе.

Операции: чтение, добавление, изменение, удаление. Для чтения — несколько ограничений (по полям и «Прочие поля»); для добавления/изменения/удаления — одно условие. Для изменения объект должен удовлетворять ограничению до и после.

ВНИМАНИЕ! При обращении к полям через свойства прикладных объектов выполняется чтение всего объекта, а не одного поля (кроме представления).

Язык ограничения — подмножество языка запросов: только секции ИЗ и ГДЕ; один источник — таблица основного объекта ограничения. В условиях — параметры сеанса, функциональные опции (только не зависящие от параметров), шаблоны. Запись доступна, если условие для неё возвращает непустую таблицу. Вложенные запросы: без В ИЕРАРХИИ и ИТОГИ. Обращение к табличной части — напрямую к ТЧ в ИЗ.

Регистры: в регистрах бухгалтерии в RLS только балансовые измерения; в накопления — только измерения; в регистрах сведений — измерения, ресурсы, реквизиты, Период, Регистратор, общие реквизиты. В оборотных регистрах накопления при использовании в RLS измерений не из итогов виртуальная таблица оборотов не использует хранимые итоги. При объектном доступе к регистру проверяется доступ ко всем полям с ограничениями (И). При записи набора записей действуют только ограничения на изменение.

Способ действия: **Все** — операция должна быть выполнена над всеми подразумеваемыми объектами, иначе ошибка; **Разрешенные** — недоступные записи считаются отсутствующими. Из встроенного языка и при изменениях — всегда «все». В запросах РАЗРЕШЕННЫЕ переключает на «разрешенные». При отсутствии РАЗРЕШЕННЫЕ отборы запроса не должны противоречить ограничениям на чтение. В объектной технике режим РАЗРЕШЕННЫЕ не поддерживается.

Наложение: из ролей выбираются ограничения для прав/таблиц/полей запроса; ограничения из одной роли — И, из разных ролей — ИЛИ; получаются параметры сеанса и функциональные опции; условия добавляются к SQL. При «все» — проверка на использование запрещённых данных; при «разрешенные» — запрещённые записи исключаются из результата.

Препроцессор в тексте RLS: #ЕСЛИ, #ИНАЧЕЕСЛИ, #ИНАЧЕ, #КОНЕЦЕСЛИ. Выражения — булевы, с параметрами сеанса &Имя. С препроцессором ограничение не проверяется синтаксисом при редактировании и не меняется конструктором.

Шаблоны: имя и текст с параметрами #Параметр(n), #ТекущаяТаблица, #ИмяТекущегоПраваДоступа, #ИмяПараметра, ##. Использование: #ИмяШаблона("значение1", "значение2"). Функция СтрСодержит(), оператор +.

Рекомендации: готовить данные в параметрах сеанса в УстановкаПараметровСеанса(); накладывать RLS только где оправдано (влияет на производительность); выносить операции в привилегированные модули или УстановитьПривилегированныйРежим. Проверки при записи (родитель, владелец, уникальность кода/номера) выполняются в привилегированном режиме. Ограничение: в условии ПО при объединении с таблицей с RLS нельзя использовать ТЧ с ограничением. Таблица в запросе без полей — на неё накладываются все ограничения по объекту (по ИЛИ). Вложенная таблица — ограничения и для ТЧ, и для основной таблицы.

### 5.5.5. Общие реквизиты

Реквизит у многих объектов. Два сценария: общий реквизит (один смысл и тип в разных объектах) и разделение данных (свойство Разделение данных = Разделять). Состав и Автоиспользование: при Использовать — автоматически во все подходящие объекты; при Не использовать — выбор вручную в Состав. Для каждого объекта: Автоматически / Использовать / Не использовать. При записи документа общему реквизиту журнала присваивается значение реквизита документа или NULL. Общий реквизит может использоваться в RLS. Может быть составного типа. Не использовать для данных бизнес-логики конкретных объектов.

### 5.5.6. Планы обмена

Узлы обмена, состав данных, связь с механизмом распределённой БД. В решении может быть несколько планов обмена.

### 5.5.7. Критерии отбора

Тип — любой стандартный или объект конфигурации; допускаются составные типы СправочникСсылка, ДокументСсылка и т.д., план видов характеристик. На закладке Состав — реквизиты справочников и документов для отбора. Параметр формы Отбор с элементом Значение. Основная форма — форма по умолчанию.

### 5.5.8. Подписки на события

Источник — прикладные объекты, наборы записей, менеджеры. Событие — из списка общих для выбранных объектов. Обработчик — процедура в общем модуле с свойствами: Глобальный — снят; Клиент (обычное) — установлен; Клиент (управляемое) — снят; Сервер — установлен; Внешнее соединение — установлен. Параметров на 1 больше, чем у обработчика события (добавляется объект-источник). Порядок: обработчик в объекте/наборе (+ расширения) → при Отказ/исключение прерывание → внешние обработчики в произвольном порядке. ДобавитьОбработчик, УдалитьОбработчик. ДополнительныеСвойства у источника — структура для хранения между вызовами.

### 5.5.9. Регламентные задания

Определение регламентных процедур, выполнение по расписанию, асинхронный вызов, ожидание завершения заданий. Фоновые задания — средствами встроенного языка, асинхронно, могут порождать дочерние; выполняются от имени создавшего пользователя. Регламентное задание — часть конфигурации, по расписанию порождает фоновое задание; может выполняться от заданного пользователя, с перезапуском. В клиент-серверном варианте автоматический запуск регламентных заданий может быть запрещён для базы.

### 5.5.10. Функциональные опции и параметры функциональных опций

#### 5.5.10.1. Назначение

Включение/выключение возможностей на этапе внедрения и в работе. Параметры функциональных опций параметризуют опции (например, по организации).

#### 5.5.10.2. На что влияют функциональные опции

- **Интерфейс:** при выключении опции скрываются связанные элементы (глобальный командный интерфейс, реквизиты и команды формы, отчёты СКД). При файловой базе через веб-сервер смена опции меняет интерфейс только после перезапуска веб-сервера.
- **Алгоритмы:** значение опции доступно из встроенного языка.

ВНИМАНИЕ! Функциональные опции не влияют на состав БД — все таблицы и поля есть независимо от состояния опций.

Глобальный интерфейс: скрываются команды объектов выключенных опций. Команда исключается, если отключён реквизит-параметр команды или все типы параметра команды.

**Форма:** основной реквизит типа ВидОбъект отключается, если опцией отключён объект (только опции без параметров). То же для ДинамическийСписок по основной таблице. Журнал документов в списке не учитывает отключение видов. Реквизит ссылочного типа отключается при отключении типа; составной — при отключении всех составляющих. Таблица формы отключается при отключении реквизита. В диалоге выбора типов отсутствуют отключённые типы; кеш типов 20 мин или ОбновитьИнтерфейс(). Значения параметров функциональных опций для интерфейса задаются только для конкретного экземпляра формы.

**Система компоновки данных:** состав выводимых данных и настроек отчёта (поля, отбор, группировка и т.д.).

**Характеристики:** видимость полей значения характеристики — при включении реквизита (хранящего значение характеристики) в состав функциональной опции.

#### 5.5.10.3. Общая схема работы

Два типа объектов: Функциональная опция (источник значения — свойство Хранение: константа, реквизит справочника, ресурс регистра сведений) и Параметры функциональных опций (оси для выбора значения; один параметр может использоваться несколькими опциями). Для опции в реквизите справочника или ресурсе регистра нужен объект «Параметры функциональных опций» с указанием, как выбирать значение.

#### 5.5.10.4. Взаимодействие с другими объектами

Функциональные опции назначаются подсистемам, общим командам, константам, критериям отбора, справочникам, документам, журналам, планам счетов/видов характеристик/видов расчета, БП, задачам, планам обмена, отчётам, обработкам, регистрам, командам, реквизитам, ТЧ, признакам учёта, измерениям/ресурсам регистров и т.д.

#### 5.5.10.5. Создание

**Функциональная опция:** объект в Общие → Функциональные опции. Свойство Хранение — константа, реквизит справочника или ресурс регистра сведений. Для управления интерфейсом пригодны только опции с типом Булево у хранилища. **Привилегированный режим при получении** — при установке значение получается в привилегированном режиме и кешируется для всех сеансов; при сбросе — обычный режим, кеш для текущего сеанса (в т.ч. признак невозможности получения). Кеш сбрасывается при изменении параметров сеанса.

**Параметр функциональных опций:** объект в Общие → Параметры функциональных опций. Свойство Использование — справочники и/или измерения регистров сведений.

ВНИМАНИЕ! Один объект метаданных нельзя использовать в нескольких параметрах функциональных опций.

#### 5.5.10.6. Использование

Назначение объектам — свойство Функциональные опции у объекта метаданных. Только опции с Хранение типа Булево. Если ни одной опции не назначено — объект виден всегда; иначе виден, если включена хотя бы одна назначенная опция (ИЛИ). Реквизиты и команды формы — свойство Функциональные опции в редакторе формы; логика та же.

В RLS допустимы только опции без параметров (привязанные к константам). Контроль уникальности имён между параметрами сеанса и функциональными опциями.

Значение опции: по объекту Хранение (константа, реквизит справочника, ресурс регистра). Для справочника/регистра нужны значения параметров (элемент справочника или измерения). Пометка удаления элемента справочника игнорируется. Для периодического регистра сведений — срез последних; для другой даты — параметр Период (Period), тип Дата (предоставляется платформой). Не все параметры заданы для опции типа Булево — ИЛИ по всем значениям с незаданными параметрами; для небулевого типа — исключение. В транзакции значение опции не меняется до завершения транзакции. Особенности параметризованных опций: в формах списков колонка показывается, если в БД есть хотя бы одно включённое значение; для отключения по умолчанию задать параметры, отсутствующие в БД; при параметре — ссылка на группу и реквизит только для элемента — ПолучитьФункциональнуюОпцию и др. возвращают NULL (интерфейс считает опцию выключенной); для командообразующих объектов команды видны, если есть хотя бы одна комбинация с Истина, либо заданы УстановитьПараметрыФункциональныхОпцийИнтерфейса().

#### 5.5.10.7. Встроенный язык

ПолучитьФункциональнуюОпцию(ИмяОпции, Параметры), ПолучитьФункциональнуюОпциюИнтерфейса(ИмяОпции), ПолучитьФункциональнуюОпциюФормы(ИмяОпции) — для параметров интерфейса и формы. УстановитьПараметрыФункциональныхОпцийИнтерфейса(), УстановитьПараметрыФункциональныхОпцийФормы(). При изменении значения в БД интерфейс и формы не обновляются автоматически — вызывать ОбновитьИнтерфейс() после записи. ОбновитьИнтерфейс() закрывает вспомогательные формы, обновляет состав основной формы и отображает её с учётом новых опций. ПолучитьПараметрыФункциональныхОпцийИнтерфейса(), ПолучитьПараметрыФункциональныхОпцийФормы() — структура с параметрами. При открытии форма использует параметры, установленные для неё.


## 5.5.12. Хранилища настроек

### 5.5.12.1. Общая информация

Для сохранения информации о настройках пользователя, которые должны сохраняться между сеансами работы, в платформе реализованы хранилища настроек.

Существует два вида хранилищ настроек:
- **Стандартное хранилище** — хранилище, используемое системой по умолчанию и хранящее данные в системных таблицах информационной базы.
- **Хранилища настроек** — специальные объекты метаданных, которые описывают хранение данных в некотором объекте информационной базы (например, в справочнике).

Платформа использует следующие хранилища:
- **Системное хранилище** — система сохраняет в нём все настройки для работы платформы (размеры форм, настройки печати табличного документа и т.п.). В качестве системного хранилища всегда используется стандартное хранилище; данные хранятся в системной таблице информационной базы. В системном хранилище также хранятся настройки по умолчанию.
- **Хранилище общих настроек** — для хранения настроек прикладного решения; платформа в него не пишет. Разработчик использует его из встроенного языка для сохранения/восстановления прикладных настроек пользователя.
- **Хранилище вариантов отчетов** — варианты отчетов.
- **Хранилище настроек данных форм** — данные форм (в т.ч. реквизиты обработок). Можно задать индивидуальное хранилище для каждого отчета и обработки.
- **Хранилище пользовательских настроек динамического списка** — пользовательские настройки динамического списка.
- **Хранилище внешних данных навигационных ссылок** — параметры, передаваемые в навигационных ссылках информационной базы.

При разработке конфигурации можно определить собственные хранилища настроек для всех хранилищ, кроме системного: создать объект хранилища настроек в ветке метаданных и указать его в нужном свойстве конфигурации. Свойства объекта Конфигурация имеют те же имена, что и перечисленные хранилища.

Данные хранилищ могут храниться в системной таблице или в объекте информационной базы (справочник, регистр сведений и т.п.). Собственное хранилище имеет смысл при особой структуре хранения, специальных механизмах управления настройками, обмене настройками в распределённой базе.

**Примечание.** Для хранения настроек рекомендуется выбирать объекты с поддерживаемым способом идентификации, при котором идентифицирующий реквизит можно преобразовать в строку и обратно без потери данных (например, справочник с уникальным кодом).

Объект метаданных **ХранилищеНастроек** обеспечивает хранение прикладных настроек. За счёт обработчиков событий и форм меняется место хранения (объекты конфигурации вместо системных таблиц) и визуальные механизмы работы с настройками. В конфигурации может быть произвольное количество хранилищ настроек.

Хранилище может использоваться только программно или программно и интерактивно. Для только программной работы в модуле объекта ХранилищеНастроек обязательно реализуют:
- **ОбработкаСохранения** — метод `Сохранить()`: сохранение настройки в объект (например, в элемент справочника).
- **ОбработкаЗагрузки** — метод `Загрузить()`: получение настроек из объекта.

**ВНИМАНИЕ.** Если обработчик не реализован, соответствующее действие недоступно (например, без ОбработкаСохранения недоступно сохранение настроек).

Если нужна интерактивность (формы сохранения и восстановления), необходимо реализовать формы и заполнить свойства объекта ХранилищеНастроек (Основная форма сохранения, Основная форма загрузки). **ВНИМАНИЕ.** Формы обязательны для интерактивных операций; программное сохранение/восстановление возможно и без них.

При сохранении/загрузке настроек система получает форму объекта хранилища и отображает её. Форме передаются параметры (см. описание объекта `ОписаниеНастроек` в синтакс-помощнике). В формах нужно использовать переданные параметры и фильтровать список настроек (например, только для `КлючОбъекта`). Результат работы формы — значение типа `ВыборНастроек`: в `КлючНастроек` — ключ выбранной настройки, в `ДополнительныеСвойства` — дополнительная информация.

При работе со стандартным хранилищем значение параметра `КлючОбъекта` не должно превышать 256 символов. Длина `КлючНастроек` и `Пользователь` не ограничена.

### 5.5.12.3. Создание объекта метаданных

Хранилище настроек создаётся в ветке **Общие → Хранилища настроек**. В качестве ключа настройки можно использовать, например, поле Код или Ссылку элемента справочника, чтобы однозначно идентифицировать объект хранилища.

### 5.5.12.4. Стандартное хранилище настроек

Стандартное хранилище представляется объектом `СтандартноеХранилищеНастроекМенеджер`. Оно сохраняет настройки в системных таблицах. В качестве ключа объекта настройки и ключа настройки принимается строка; в качестве настроек — любое значение, помещаемое в хранилище значений.

Методы:
- **ПолучитьСписок()** — список настроек для выбранного объекта настройки (выборка может быть ограничена отбором). Для прикладного хранилища методы получения списка нужно реализовывать самостоятельно.
- **Удалить()** — удаление заданной настройки выбранного объекта настройки.

**Примечание.** При использовании методов `Сохранить()`, `Удалить()` и `УстановитьОписание()` объекта `СтандартноеХранилищеНастроекМенеджер`: если объект (например, форма) в текущем сеансе уже использовался, изменения учтутся в следующем сеансе. При существенном программном изменении настроек рекомендуется предложить пользователю перезапуск клиентского приложения.

### 5.5.12.5. Сохранение настроек форм

Управление сохранением данных форм — через свойства формы:
- **Сохранение данных формы в настройках** — включение сохранения данных формы с выбором настройки; форма получает команды сохранения/загрузки настроек.
- **Автоматическое сохранение данных в настройках** — автосохранение при закрытии и восстановление при открытии формы.

Для формы с сохранением данных указывают, какие реквизиты сохраняются (колонка «Сохранение» на закладке Реквизиты). В качестве ключа объекта используется полное имя формы. Сохраняется объект типа `Соответствие`: ключи — пути к реквизитам, значения — значения реквизитов.

### 5.5.12.6. Сохранение настроек отчетов

У объектов **Отчет** и **Внешний отчет** есть свойства **Хранилище вариантов** и **Хранилище настроек**. Если не указаны — используются хранилища из свойств конфигурации; если и там не указаны — системное хранилище. Форма отчета предоставляет команды сохранения и загрузки вариантов и настроек. Дополнительная информация в настройках компоновки данных может храниться в свойстве **ДополнительныеСвойства** объектов `НастройкиКомпоновкиДанных` и `ПользовательскиеНастройкиКомпоновкиДанных` (тип — `Структура`).

### 5.5.12.7. Сохранение настроек динамических списков

Пользовательские настройки динамических списков сохраняются в хранилище, указанном в свойствах конфигурации. Автосохранение управляется свойством **Автоматическое сохранение пользовательских настроек** реквизита формы типа ДинамическийСписок. Форма динамического списка предоставляет команды сохранения, загрузки и установки стандартных настроек (из `Список.КомпоновщикНастроек.Настройки`). Событие **ПриОбновленииСоставаПользовательскихНастроекНаСервере** вызывается при открытии формы динамического списка и после завершения редактирования пользовательских настроек при изменении состава настроек.

### 5.5.12.8. Внешние данные навигационных ссылок

Внешние данные навигационных ссылок хранятся в хранилище из свойства конфигурации **ХранилищеВнешнихДанныхНавигационныхСсылок**; при отсутствии — системное хранилище. В стандартном хранилище все пользователи имеют доступ на чтение ко всем настройкам.

Если свойство не указано: в режиме совместимости 8.3.18 и ранее стандартное хранилище не создаётся (при обращении из встроенного языка — ошибка; формы отчета и динамического списка не дают ссылки с полными настройками); в режиме 8.3.19 и выше используется стандартное хранилище.

Ключ объекта — объект, для которого формируется ссылка; ключ настройки — строковое представление сформированного уникального идентификатора (тот же идентификатор передаётся как параметр `extdata` в навигационной ссылке).

Платформа не содержит инструментов автоматической очистки этого хранилища; реализация — на уровне прикладного решения (например, удаление настроек старше заданной даты). Для проверки «возраста» записи используется свойство `СтандартноеХранилищеНастроекВыборка.ДатаИзменения`.

### 5.5.12.9. Порядок разработки хранилища настроек

1. Определить перечень объектов метаданных и хранимой информации (структура, типы).
2. Создать объект (и структуру) для хранения настроек (например, справочник с реквизитом типа ХранилищеЗначения).
3. Создать объект типа ХранилищеНастроек, реализовать формы сохранения и загрузки.
4. Реализовать обработчики ОбработкаСохранения и ОбработкаЗагрузки в модуле объекта ХранилищеНастроек.
5. В объектах из п. 1 или в свойствах конфигурации заполнить соответствующие свойства (указать созданное хранилище).
6. При необходимости реализовать обработчики в прикладных объектах.

### 5.5.12.10. Работа с хранилищем настроек из встроенного языка

Для доступа к стандартному хранилищу после создания пользовательского используются методы глобального контекста:
- `ПолучитьСтандартноеХранилищеВариантовОтчетов()`
- `ПолучитьСтандартноеХранилищеВнешнихДанныхНавигационныхСсылок()`
- `ПолучитьСтандартноеХранилищеНастроекДанныхФорм()`
- `ПолучитьСтандартноеХранилищеОбщихНастроек()`
- `ПолучитьСтандартноеХранилищеПользовательскихНастроекДинамическихСписков()`
- `ПолучитьСтандартноеХранилищеПользовательскихНастроекОтчетов()`

Каждый метод возвращает менеджер стандартного хранилища независимо от того, задано пользовательское хранилище или нет (перенос настроек в обе стороны возможен в любой момент).

При переборе стандартного хранилища десериализация настроек выполняется при обращении к свойству `СтандартноеХранилищеНастроекВыборка.Настройки`. Некорректные настройки приведут к исключению в момент доступа к конкретным настройкам; исключение можно перехватить и удалить некорректные настройки.

Выборка упорядочивается по возрастанию: хэш от строкового представления Пользователь; КлючОбъекта (алфавитно-цифровой порядок); хэш от строкового представления КлючНастроек; Версия.

### 5.5.12.11. Настройки по умолчанию

Настройки по умолчанию — системные неименованные настройки, применяемые, если для текущего пользователя нет такой же сохранённой настройки. Не поддерживаются для именованных настроек (пользовательские настройки отчетов и динамических списков, данные формы, внешние данные навигационных ссылок). Хранятся в системном хранилище; переопределить его нельзя.

Алгоритм чтения настроек для отображаемого объекта: сначала попытка считывания для текущего пользователя; при отсутствии — считывание настроек по умолчанию; при нахождении на любом шаге — применение и прекращение поиска. Из встроенного языка читается ровно та настройка, которую запросили.

Команда «Установить стандартные настройки» устанавливает настройки по умолчанию, если они заданы; отменяет текущую настройку пользователя и удаляет её из хранилища.

Для части настроек (форма, интерфейс, начальная страница, командный интерфейс) сохраняются отличия от исходного состояния. Если пользовательская настройка совпадает с настройкой по умолчанию — она не записывается (или удаляется из хранилища); при отличии — записывается пользовательская.

Для записи настроек по умолчанию пользователь должен обладать правом **АдминистрированиеДанных**. После сохранения настроек по умолчанию в текущем сеансе они начинают действовать в сеансе другого пользователя не позже чем через 20 секунд (если у того нет собственных настроек этой формы).

Программный интерфейс (менеджер стандартного хранилища — свойство `ХранилищеСистемныхНастроек`):
- `СохранитьНастройкуПоУмолчанию()` — сохранение;
- `ЗагрузитьНастройкуПоУмолчанию()` — чтение;
- `ВыбратьНастройкиПоУмолчанию()` — перебор;
- `УдалитьНастройкуПоУмолчанию()` — удаление.

Идентификация настройки — по ключу объекта (или элементу отбора).

## 5.5.13. Общие формы

Размещение команды открытия общей формы в командном интерфейсе — через свойство **Использовать стандартные команды**; команда попадает в интерфейс подсистем, которым принадлежит форма. ## 5.5.17. Общие картинки

### 5.5.17.1. Общая информация

В конфигурацию включаются графические изображения для элементов управления, форм, макетов и доступа из встроенного языка. Поддерживаются форматы BMP, GIF, JPEG, PNG, TIFF, ICO, SVG, метафайлы (WMF, EMF), наборы картинок в zip-архиве. Для пиктограмм рекомендуется PNG/GIF (сжатие без потерь).

Для устройств с разным dpi можно задать набор картинок по интервалам dpi; платформа подбирает нужную картинку. Варианты разрешения: bldpi (85%), ldpi (100%, <140 dpi), aldpi (125%), mdpi (150%), amdpi (175%), hdpi (200%), xdpi (300%), udpi (400%). Рекомендуемые размеры при 100%: пиктограммы — не более 16×16; табличное поле — 14×14; кнопка выбора — 9×9; представление подсистемы — до 48×48.

Набор картинок — zip-архив с картинками и манифестом `manifest.xml`. Обязательна базовая картинка для ldpi. Шаблонные картинки — информативная часть в альфа-канале (RGBA), остальные каналы — белый (R=G=B=255); рекомендуется PNG 32 бит/пиксель.

### 5.5.17.2. Управление библиотекой картинок конфигурации

Окно **Библиотека картинок**: ветвь **Общие → Общие картинки**, контекстное меню **Все картинки**. Кнопки: Добавить, Изменить, Удалить; предпросмотр; изменение порядка; Очистить, Сохранить в файл. Для новой картинки — диалог свойств с значениями по умолчанию. Кнопки **Выбрать из файла**, **Редактировать**, **Очистить**, **Сохранить в файл**. Для SVG с анимацией может предлагаться конвертация.

### 5.5.17.3. Свойства общей картинки

**Имя** — идентификатор; используется в `БиблиотекаКартинок`. **Доступность для выбора** — выбор картинки в ячейке/рисунке/колонтитуле табличного документа в режиме 1С:Предприятие. **Доступность для оформления** — использование при условном оформлении отчетов. Прозрачный фон: для BMP, JPEG, TIFF — кнопки «Установить прозрачный фон» / «Убрать прозрачность». **Включить набор вариантов** / **Выключить набор вариантов** — работа с архивом вариантов по масштабу; базовая картинка — ldpi.

В диалоге набора картинок для каждого варианта: **Плотность экрана**, **Шаблонная**, **Направление** (LTR/RTL, возможность переворота), **Интерфейс** (taxi, taxiCompact, taxiMobile, version8_2 и др.), **Веб-браузер** (для веб-клиента), **Картинка**, **Ширина/высота ячейки** (для коллекций). Приоритет выбора варианта в LTR и RTL задаётся платформой. Для коллекций: параметры ячейки в редакторе варианта и в диалоге картинки должны совпадать; разбиение — по заданным ширине/высоте ячейки.

Для SVG игнорируются: JavaScript и интерактивность (в т.ч. :hover); отдельные свойства стилей; маркеры пути нулевой длины; feTile; встроенные шрифты (font, altGlyphDef и др.); enable-background; внешние ресурсы; для element image — относительный путь к файлу.

### 5.5.17.4. Манифест картинки

Набор вариантов — zip с файлами картинок и `manifest.xml`. Схема манифеста: элемент `PictureVariant` с атрибутами name, screenDensity (bldpi, ldpi, aldpi, mdpi, amdpi, hdpi, xdpi, udpi), isTemplate, interfaceVariant (taxi, taxiCompact, version8_2, version8_2_OrdinaryApp, version8_0, taxiMobile), glyphWidth, glyphHeight, direction (rtlCanFlipHorizontally, ltrCanFlipHorizontally, ltr, rtl), webBrowser. direction: rtlCanFlipHorizontally — «Справа налево, можно переворачивать»; ltrCanFlipHorizontally — «Слева направо, можно переворачивать»; ltr — «Слева направо»; rtl — «Справа налево».

## 5.5.18. XDTO-пакеты

XDTO — универсальный способ представления данных для взаимодействия с внешними источниками и системами.

### 5.5.18.1. Импорт схемы XML в глобальную фабрику XDTO

Импорт из файла .xsd. При совпадении пространств имён с существующими пакетами предлагается указать пакеты для обновления (по умолчанию не обновляются). В директивах import/include должен быть атрибут schemaLocation для разыменования транзитивного замыкания схем. После импорта новые пакеты добавляются, отмеченные — обновляются.

### 5.5.18.2–5.5.18.3. Экспорт схемы XML

**Экспорт XML-схемы данных конфигурации**: ветка XDTO → контекстное меню «Экспорт XML-схемы данных конфигурации…» — типы данных конфигурации без учёта созданных в дереве пакетов XDTO. **Экспорт XML-схемы пакета**: выделить пакет XDTO → «Экспорт XML-схемы…» — проверка пакета, при успехе — выбор каталога и имени .xsd.

### 5.5.18.4. Проверка пакета XDTO

Выделить пакет → контекстное меню **Проверить пакет**. Выполняется проверка модели пакета; ошибки выводятся в окно сообщений.

### 5.5.18.5. Окно редактирования пакета XDTO

Открывается автоматически при добавлении пакета; для существующего — выделить пакет → **Открыть пакет**. Иерархия: директивы импорта (зависимости), типы значений, типы объектов, свойства (глобальные объявления). Тип значения может содержать образец, перечисление; тип объекта — набор свойств.

**Свойства пакета** (в палитре): Имя, Синоним, Комментарий, URI пространства имен; ссылка Пакет. Для корневого элемента в окне редактирования — только URI пространства имен.

**Директива импорта** — редактирование в палитре свойств.

**Тип значения XDTO**: Имя, Базовый тип, Вариант (атомарный/список/объединение), Тип элемента, Типы объединения; фасеты: минимальная/максимальная длина, пробельные символы, минимум/максимум (включая/не включая границу), общее количество цифр, количество цифр дробной части.

**Тип объекта XDTO**: Имя, Базовый тип, Открытый, Абстрактный, Смешанный, Упорядоченный, Последовательный (последовательность XDTO для явного порядка элементов в XML).

**Свойство типа объекта**: Имя, Тип, Минимальное/Максимальное количество, Возможно пустое, Фиксированное, По умолчанию, Форма (Текст/Элемент/Атрибут), Локальное имя. Только одно свойство типа может иметь форму Текст; при Текст Минимальное количество = 1.

**Глобальное свойство**: Имя, Ссылка, Тип, Минимальное/Максимальное количество, Возможно пустое, Фиксированное, По умолчанию, Форма, Локальное имя.

## 5.5.19. Web-сервисы

Web-сервисы позволяют использовать 1С:Предприятие как набор сервисов в распределённых и гетерогенных системах и интегрировать с другими системами (SOA).

**Добавление**: ветка **Общие → Web-сервисы** → **Добавить**. В окне редактирования на закладке **Прочее**: **URI пространства имен**, **Пакеты XDTO** (типы возвращаемых значений и параметров операций), **Имя файла публикаций**. Адрес доступа: `<хост>/<виртуальный каталог>/ws/<Имя Web-сервиса>` или `/ws/<Адрес Web-сервиса>`. Кнопка **Модуль** — редактирование модуля Web-сервиса.

Web-сервис содержит набор операций; каждой операции соответствует экспортируемая процедура в модуле Web-сервиса.

### 5.5.19.3–5.5.19.4. Операции и параметры

На закладке **Операции** — добавление операций; свойства в палитре. **Тип возвращаемого значения** — тип XDTO; **Возможно пустое значение**; **В транзакции**; **Имя метода** — процедура модуля; **Режим управления блокировкой данных**. Параметры операции: **Тип значения** (XDTO), **Возможно пустое значение**, **Направление передачи** — Входной, Выходной или Входной‑Выходной. Имена параметров должны соответствовать именам параметров процедуры.

### 5.5.19.5. Типы, определяемые системой 1С

Для использования типов платформы в параметрах и возвращаемых значениях в конфигурации определяют пакеты XDTO и в директивах импорта указывают пакеты платформы. URI пространства имен для типа — в синтакс-помощнике по объекту типа.

### 5.5.19.6. Публикация Web-сервисов

Описывается в документации платформы по публикации Web-сервисов.

## 5.5.21. WS-ссылки

## 5.5.22. Стили и элементы стиля

Стиль — совокупность настроек оформления интерфейса. Основной стиль задаётся свойством конфигурации **Основной стиль** или программно свойством `ГлавныйСтиль` (запись только в обработчике `УстановкаПараметровСеанса`).

Системные элементы стиля (настраиваемые): ЦветАктивности, ЦветАкцента, ЦветНавигации, ЦветДополнительнойНавигации, ЦветВажного — с указанием областей применения (рамка текущего элемента, фон меню, строки таблицы, заголовок формы, кнопки и т.д.) для клиентского приложения ПК и мобильной версии. Другие системные элементы Такси изменить нельзя; можно использовать их для оформления элементов формы.

Элементы стиля (пользовательские): Цвет, Шрифт, Рамка. Используются для единообразного оформления элементов формы. Доступ к значению — свойство **Значение**. Системные названия элементов стиля недоступны в качестве значений пользовательских элементов. В диалоге выбора шрифта: шрифт `<Шрифт текста>` — шрифт интерфейса 1С; размер 0 — размер шрифта интерфейса.

## 5.5.23. Языки

Объекты **Языки** — интерфейс на разных языках. Для каждого языка резервируется строка для реквизитов с многоязычным представлением. Смена языка в конфигураторе: **Конфигурация → Язык редактирования конфигурации** или кнопка выбора языка в панели состояния. Текст надписей задаётся в **Заголовок** (Текст/Синоним); многоязычные строки — окно **Строки на разных языках**. **Код языка** — например, EN. При двух и более языках у свойств Синоним и Заголовок появляется кнопка редактирования текста на разных языках. Изменение кода языка приводит к «потере» введённых текстов (они привязаны к прежнему коду). Для мобильных устройств рекомендуется ISO 639. В коде языка запрещены: #, *, |, \, :, ", <, >, ?, /.

## 5.6. Общие свойства объектов конфигурации

### 5.6.1. Основные свойства

Практически все объекты имеют (категория **Основные**):
- **Имя** — одно слово, с буквы, без специальных символов кроме «_»; доступ из встроенного языка; не совпадать с зарезервированными словами языка запросов; длина до 80 символов (в 8.3.7+).
- **Синоним** — для списков выбора, заголовков, интерфейсов; при отсутствии используется имя.
- **Комментарий** — произвольная строка.

Представление объекта формируется: заголовок на языке конфигурации пользователя → синоним → для стандартных реквизитов представление локализации платформы, для прикладных — имя из конфигурации. Дополнительные свойства представления (расширенное представление объекта/списка, пояснение, картинка подсистемы и т.д.) заполняют при необходимости уточнения отображаемой по умолчанию информации.

### 5.6.3. Произвольное представление данных

Для объектов (планы обмена, справочники, документы, планы видов характеристик/счетов/расчётов, бизнес-процессы, задачи, таблицы внешних источников) можно задать реквизиты и алгоритм формирования представления.

Два этапа:
1. **ОбработкаПолученияПолейПредставления** (менеджер) — заполнить массив `Поля` именами реквизитов; при `СтандартнаяОбработка = Истина` массив перезаполняется стандартными полями. Не рекомендуется включать поля неограниченной длины и ссылочные поля с получением данных «через точку».
2. **ОбработкаПолученияПредставления** (менеджер) — по переданным `Данные` (структура: имя реквизита → значение) сформировать строку в параметр `Представление`. При `СтандартнаяОбработка = Истина` строится стандартное представление. Для нового (не сохранённого) объекта в `Данные.Ссылка` передаётся Неопределено. Ошибка в обработчике может привести к аварийному завершению.

### 5.6.4. Стандартные реквизиты

На уровне конфигурации можно переопределять интерфейсные свойства стандартных реквизитов и стандартных табличных частей (синоним, проверка заполнения и т.д.). Команды списков стандартных реквизитов/табличных частей доступны для объектов, у которых они есть. Нельзя изменить: имя, тип, индексирование стандартного реквизита; имя стандартной табличной части. Наличие описания не меняет имени реквизита во встроенном языке и в запросах.

### 5.6.5. Предопределенные данные

Предопределённые данные — элементы прикладных объектов, создаваемые в конфигураторе и доступные по имени (например, `Справочники.Товары.Услуга`). Поддерживаются для справочников, планов счетов, планов видов характеристик, планов видов расчёта. Связь с элементом данных — свойство **ИмяПредопределенныхДанных**. Обновление (создание/обновление элемента при изменении предопределённых данных) задаётся: свойством объекта в ИБ, свойством конфигурации **ОбновлениеПредопределенныхДанных**, настройкой ИБ, видом узла РИБ (корневой/не распределённая — обновлять автоматически; подчинённый — не обновлять). Значения: Не обновлять автоматически, Обновлять автоматически, Авто. Решение принимается по первому отличному от Авто значению в перечисленном порядке.

Связывание: присвоить `ИмяПредопределенныхДанных` имени предопределённого элемента и записать объект. «Отключение»: присвоить пустую строку и записать. Смена элемента в два этапа: отключить текущий, связать новый. Создание элементов: при реструктуризации (если разрешено и предопределённые ранее создавались); при первом обращении к таблице объекта (если не запрещено); из встроенного языка при создании элемента с заданным `ИмяПредопределенныхДанных`. Метод **ИнициализироватьПредопределенныеДанные()** — принудительное создание при выполнении условий. **ПолучитьИменаПредопределенных()** — список имён предопределённых. При удалении предопределённого в конфигураторе: связанный элемент данных помечается на удаление; в плане счетов/видов расчёта удаляются соответствующие записи в субконто/видах расчёта. При удалении предопределённого элемента свойство `ИмяПредопределенныхДанных` заменяется на `#<UUID>`.

В РИБ предопределённые передаются с конфигурацией; элементы данных — по плану обмена. Автосоздание в периферийной базе не выполняется. Если данные получены раньше конфигурации с предопределёнными — автосвязывания не будет; нужна повторная выгрузка. Метод **ИнициализацияПредопределенныхДанных()** можно вызвать до создания начального образа периферии. При отмене главного узла режим обновления может определиться как «Обновлять автоматически»; для фиксации используют `/SetPredefinedDataUpdate` или `УстановитьОбновлениеПредопределенныхДанныхИнформационнойБазы()`. При универсальном обмене: при записи объекта с предопределённым проверяется наличие имени в конфигурации базы-приёмника; при загрузке (`ОбменДанными.Загрузка = Истина`) свойство сохраняется; иначе возможно исключение. Не допускать двух объектов с одним предопределённым в одной базе. Запись объекта с именем вида `#<UUID>` (удалённый предопределённый) возможна; два объекта с одним таким именем — только при Загрузка.

### 5.6.6. Нумерация объектов

Нумерация (авто или вручную) для справочников, документов, планов видов характеристик, бизнес-процессов, задач, планов обмена — по номеру или по коду. Тип кода/номера — число или строка. Строка: любые символы; допустимая длина (фиксированная/переменная); префиксы. При ручном вводе короткого номера (например, «01») максимум для автонумерации ограничивается (99 при длине 2); рекомендуется не вводить вручную или использовать ведущие нули. **Признак автонумерации** — автоматическое назначение кода/номера при записи; уникальность; следующий на 1 больше максимума; при удалении/уменьшении максимума продолжение с оставшегося (не гарантируется сброс в минимум). **Контроль уникальности** — при записи (при Загрузка не производится). **Нумератор** — для документов общее пространство номеров для нескольких видов. **Серии кодов** (справочники, планы видов характеристик): во всём справочнике/плане; в пределах подчинения; в пределах подчинения владельцу. **Периодичность** (документы, БП): непериодический, в пределах года/квартала/месяца/дня; период по дате документа/БП. **Префиксы** — только для строкового типа; длина кода увеличивается на длину префикса и разделителя; для автосмены префикса по реквизитам — обработчики ПриУстановкеНовогоКода/ПриУстановкеНовогоНомера или методы УстановитьНовыйКод/УстановитьНовыйНомер с префиксом. При неполном префиксе выбирается максимальный существующий префикс (алфавитный порядок).

### 5.6.7. Ввод по строке

В свойстве **Ввод по строке** указывают реквизиты, по которым выполняется поиск (справочники, документы, БП, задачи и т.д.). Ограничения: строка до 1 000 символов, не более 32 слов (разделители — пробел, неразрывный пробел, табуляция). Реквизиты поиска: Код, Наименование (справочники, планы и т.д.); Номер (документы); Номер, Наименование (БП, задачи); индексируемые реквизиты типа Число/Строка. **Способ поиска строки**: Начало (строка целиком); Любая часть (по словам, запрос с ПОДОБНО). **Полнотекстовый поиск** — по началу слов; используется для первоначального отбора, затем поиск по полям ввода по строке; к результатам добавляются непроиндексированные ссылки; к каждому слову добавляется «*». Особенности: при «Любая часть» и полнотекстовом поиске фрагменты не с начала слов могут не находиться; при ошибке полнотекстового поиска используется обычный поиск. **Режим получения данных выбора**: Фоновый (фоновое задание при задержке >200 мс) или Непосредственно. Рекомендации: Фоновый для больших объёмов; полнотекстовый — при большом количестве данных и поиске в любом месте. Алгоритм формирования списка выбора: поиск в первом поле по началу и подстроке; ограничение до 10 записей (5+5 или дополнение из следующего поля). Представление в выпадающем списке: обязательная часть — поле с поисковой строкой; дополнительная — из основного/альтернативного представления (Наименование, Код для справочников и т.д.; Номер, Дата для документов и т.д.).

**Программное формирование списка**: в форме — обработчики НачалоВыбора, АвтоПодбор, ОкончаниеВводаТекста; в менеджере объекта — **ОбработкаПолученияДанныхВыбора**. Параметры: СтрокаПоиска, Отбор, ВыборГруппИЭлементов, СпособПоискаСтроки, ПолнотекстовыйПоиск, РежимПолученияДанныхВыбора, ОграничиватьСтрокуПоиска, параметры формы (Связи параметров выбора, Параметры выбора). Возврат — в `ДанныеВыбора`; при `СтандартнаяОбработка = Ложь` список формируется полностью в обработчике. Элемент списка может быть структурой: **Значение** (обязательно; одно или массив для множественного выбора), **ПометкаУдаления**, **Предупреждение**, **ДополнительныеДанные**. При ПометкаУдаления без Предупреждение текст предупреждения формируется автоматически; текст Предупреждение завершается «Продолжить?».

### 5.6.8. Формы

Форма — объект для ввода, просмотра и управления. При одновременной поддержке обычного и управляемого режимов можно задать дополнительные формы; платформа выбирает форму по режиму. Тонкий и веб-клиент — только управляемые формы (при двух управляемых — основная; при одной — она; при отсутствии — автогенерация). Толстый клиент: при отсутствии назначенных форм генерируется обычная или управляемая по режиму; при одной форме — она; при двух (обычная и управляемая) — по режиму; при двух одного типа — основная. Свойства конфигурации **Использовать управляемые формы в обычном приложении** и **Использовать обычные формы в управляемом приложении** влияют на выбор формы в толстом клиенте. При открытии автогенерированной формы при несовпадении режима запуска с **Основной режим запуска** форма может открываться «Только просмотр», без команд Создать/Создать на основании. **Назначения использования** формы задают, где форма доступна (Персональный компьютер, Мобильное приложение); при отсутствии Персональный компьютер недоступны свойства, не поддерживаемые мобильной платформой; при отсутствии Мобильное приложение форма не включается в конфигурацию мобильного приложения.

### 5.6.9. Команды

Команды объекта выполняют операции с объектом метаданных. Нераметризованные команды доступны в командном интерфейсе подсистем, в которые входит объект; параметризованные — в формах с реквизитами типа параметра команды. Требуется процедура выполнения в модуле команды — обработчик **ОбработкаКоманды()**. Модуль команды может открывать форму отчета/списка с параметрами. Для команды в панели навигации повторный выбор не выполняет команду снова. Директивы компиляции — при необходимости.

### 5.6.10. Механизм заполнения реквизитов новых объектов

Заполнение при интерактивном создании (кроме копирования), вводе на основании, вызове ОткрытьФорму/ПолучитьФорму с созданием формы нового объекта или вызове Заполнить(): из отбора при вводе из списка; из свойств реквизитов (значения заполнения); из обработчика **ОбработкаЗаполнения()**. Параметр **ДанныеЗаполнения**: при вводе на основании — ссылка на объект-основание; при вводе из списка с отбором — структура с элементами отбора (Равно/В списке с одним значением); при создании без отбора — Неопределено; при ОткрытьФорму/ПолучитьФорму — параметр формы ЗначенияЗаполнения или Неопределено; при Заполнить() — переданный параметр. Копирование обрабатывается в **ПриКопировании**; реквизиты ХранилищеЗначения не переносятся автоматически. Свойство реквизита **Заполнять из данных заполнения** (Истина — заполнение из данных заполнения). После ОбработкаЗаполнения при СтандартнаяОбработка = Истина система заполняет реквизиты с Заполнять из данных заполнения или Значение заполнения, не заполненные в обработчике. Значение заполнения — свойство реквизита метаданных (значение по умолчанию при интерактивном создании); заполнение происходит после ОбработкаЗаполнения. Данные заполнения передаются в форму параметром ЗначенияЗаполнения. При программном создании объекта обработка заполнения не вызывается автоматически; для вызова — метод Заполнить().

### 5.6.11. Проверка заполнения реквизитов

Управление — свойство **Проверка заполнения** (Не проверять / Выдавать ошибку). Автоматическая проверка для констант, реквизитов и ТЧ справочников, документов, отчетов, обработок, планов, наборов записей регистров, реквизитов форм. Проверка коллекций — наличие хотя бы одной строки. Вызов: метод **ПроверитьЗаполнение()** (объект/форма) или автоматически перед записью (для документов — перед проведением при Разрешить проведение или перед записью при Запретить; для БП — перед стартом; для отчетов — при Сформировать; для обработок — при OK, Да, Повторить, Пропустить). Для формы необходимо свойство **Проверять заполнение автоматически**. Список проверяемых реквизитов формируется платформой; обработчик формы **ОбработкаПроверкиЗаполненияНаСервере** может изменить список (добавлять только реквизиты поддерживаемых типов и основной реквизит). В модуле объекта/набора записей/формы — **ОбработкаПроверкиЗаполнения**: параметр ПроверяемыеРеквизиты (массив); можно удалять/добавлять реквизиты; при собственной проверке — СообщениеПользователю и Отказ = Истина. Формат имён в ПроверяемыеРеквизиты: ИмяРеквизита; ИмяКоллекции; ИмяКоллекции.ИмяРеквизита. Функциональные опции влияют на состав проверяемых реквизитов; для опций с параметрами реквизиты всегда в списке — исключение в обработчике при необходимости. **Отметка незаполненного**: свойства Проверка заполнения (метаданные/форма), Автоотметка незаполненного (Авто/Да/Нет), ОтметкаНезаполненного (программно). При Авто отметка при пустом значении и Выдавать ошибку. При начале редактирования отметка снимается; восстанавливается после завершения или установки значения. Сообщения в окне сообщений также подсвечивают поле; после ОчиститьСообщения() снимается у элементов с ОтметкаНезаполненного = Ложь.

### 5.6.12. Индексирование реквизитов объектов

Для ускорения отбора и сортировки по реквизиту — свойство **Индексировать** (или **Индексировать с доп. упорядочиванием**). Индексирование с доп. упорядочиванием эффективно, когда в списке задана сортировка: сначала по реквизиту, затем по представлению; иначе эквивалентно обычному индексированию при большем размере индекса. Доступно для справочников, документов, планов видов характеристик/счетов/расчётов; для регистров — обычное индексирование. Дополнительные индексы создаются отдельно.

### 5.6.13. Права

Права доступа редактируются в редакторе прав ролей и в окне редактирования объекта (настройка прав объекта во всех ролях). Задаются права для объектов метаданных, табличных частей, реквизитов.

### 5.6.14. Быстрый выбор

**Способ выбора** и **Быстрый выбор** (при Способ выбора = Обоими способами) задают режим выбора по умолчанию. Трехуровневая схема: 1) свойство Быстрый выбор элемента формы; 2) при Авто — реквизит объекта; 3) при Авто — объект конфигурации по типу реквизита. Переопределение на уровне элемента формы или реквизита объекта возможно при Способ выбора = Обоими способами.

### 5.6.15. История выбора при вводе

Свойство **ИсторияВыбораПриВводе** (объект, реквизит, поле ввода). Трехуровневая схема (поле → реквизит → объект). Значения: Авто, Не использовать. При Авто на всех уровнях история отображается и сохраняется, если: поле ссылочного типа; не режим выбора из списка; не быстрый выбор. История — до 7 значений: не более 5 наиболее часто используемых + последние выбранные до общего лимита 7.

### 5.6.16. Создание нового элемента при вводе

Свойство **СозданиеПриВводе** / **КнопкаСоздания** (объект, реквизит, поле ввода). Трехуровневая схема. При создании из поля ввода можно обработать событие **Создание**. При СтандартнаяОбработка = Истина открывается основная форма с параметрами: РежимВыбора, ТекстЗаполнения, ЗначенияЗаполнения, ПараметрыВыбора, ДополнительныеПараметры. Перед ПослеЗаписиНаСервере проверяется соответствие созданного объекта ПараметрыВыбора → ВыборДоступен. При закрытии вызывается ВыборЗначения; при СтандартнаяОбработка = Истина и ВыборДоступен = Истина выполняется выбор созданного элемента; иначе сообщение и отказ. Владелец формы — поле ввода.

### 5.6.17. Связи параметров выбора

Список реквизитов, поставляющих значения для отбора при выборе (форма выбора, быстрый выбор, ввод по строке). Соответствие имени параметра отбора и реквизита формы. Значения передаются в форму в структуре Параметры (например, Отбор.Владелец → значение реквизита). При совпадении имени с **Параметры выбора** приоритет у Связи параметров выбора, если значение реквизита связи заполнено. **Режим изменения связанного значения**: Очищать — при изменении связи поле очищается до ПриИзменении; Не изменять — не очищается. Для стандартного реквизита Родитель подчиненного справочника система может автоматически добавлять/удалять Отбор.Владелец ↔ Владелец при смене подчиненности.

### 5.6.18. Параметры выбора

Фиксированные значения параметров при выборе (форма выбора, быстрый выбор, ввод по строке). Передаются в форму в Параметры (колонка Имя — ключ, Значение — значение). При совпадении с Связи параметров выбора приоритет у Связи при заполненном значении реквизита связи. Параметры выбора используются также динамическим списком и системой компоновки данных. Для субконто в наборе записей регистра бухгалтерии автоматически устанавливается параметр выбора для Владелец (значение — вид субконто), если вид субконто имеет тип справочника.

### 5.6.19. Прочее

**Использовать стандартные команды** — включение стандартных команд объекта в командный интерфейс; при Ложь объект в интерфейс только через созданную команду. **Форма выбора** — форма для выбора значения реквизита (для типов с формами; например, для реквизита Родитель подчиненного справочника — особая форма). **Формат** — для Число, Булево, Дата отображение при выводе. **Формат редактирования** — для тех же типов при редактировании. **Связь по типу** — для реквизитов составного типа, связанных с планом видов характеристик (ограничение типа вводимых значений по выбранному значению другого реквизита); для субконто — связь с реквизитом типа Ссылка на план счетов с указанием номера вида субконто.

## 5.7. Константы

### 5.7.1. Общая информация

В системе «1С:Предприятие» константы предназначены для хранения постоянной или условно-постоянной информации. Основная причина использования: один раз заносится информация, которая затем используется при формировании документов, в расчётах, при построении отчётных форм. Значение константы может редактироваться.

**ВНИМАНИЕ!** При изменении значения константы прежнее значение теряется. Для получения прежних значений вместо константы нужно использовать регистр сведений без измерений.

Для работы с константами предназначена ветвь **Константы** дерева конфигурации.

### 5.7.2. Свойства константы

- **Тип** — тип константы может быть любым из стандартных типов. В зависимости от выбранного типа палитра свойств может содержать дополнительные свойства (Формат, Формат редактирования и т.д.).
- **Формат** — для типов Число, Булево, Дата задаёт формат отображения при выводе.
- **Формат редактирования** — формат редактирования данных типов Число, Булево, Дата.
- **Связь по типу** — для реквизитов с составным типом, логически связанных с реквизитом типа ПланВидовХарактерик.Ссылка; ограничивает тип вводимых значений.
- **Использовать стандартные команды** — при установке команда открытия редактора константы отображается в подсистемах, к которым отнесена константа. Форма задаётся свойством **Основная форма**.

Формы для ввода констант создаются в ветви **Общие формы**.

## 5.8. Справочники

### 5.8.1. Общая информация

Справочники используются для исключения неоднозначного ввода информации; представляют список однородных объектов (элементов). В конфигураторе описывается структура справочника; содержимое вводится пользователем. Как правило, есть стандартные реквизиты **код** и **наименование** (код — Число или Строка).

Справочник может быть **иерархическим**:
- **Иерархия групп и элементов** — данные делятся на группы и элементы; группы только объединяют.
- **Иерархия элементов** — все элементы равнозначны (например, подразделения, статьи затрат).

Помимо кода и наименования можно создать реквизиты и табличные части. Для работы со справочниками — ветвь **Справочники** дерева конфигурации.

### 5.8.2. Свойства справочника

- **Иерархический справочник** — при установке доступны **Вид иерархии** и **Ограничение количества уровней иерархии**.
- **Вид иерархии** — Иерархия групп и элементов / Иерархия элементов.
- **Размещать группы сверху** — при установке группы отображаются выше элементов.
- **Количество уровней иерархии** — задаётся при ограничении уровней.
- **Владельцы** — справочник может быть подчинён одному или нескольким справочникам (владельцам). Для стандартного реквизита **Владелец** свойство **Режим сокращения типа**: Запрещать / Преобразовывать значения / Удалять данные.
- **Длина кода** — максимальная длина кода (макс. 50). Длина 0 — код не используется.
- **Длина наименования** — макс. 150; 0 — без наименования.
- **Серии кодов** — диапазон проверки уникальности и автонумерации: Во всем справочнике / В пределах подчинения (для иерархических и подчинённых).
- **Тип кода** — Число или Строка. При строковом типе автонумерация возможна (001, 002… или префикс + число, например АА001).

### 5.8.3. Свойства реквизитов справочника

- **Реквизиты** — дополнительные сведения об элементе. **Табличные части** — для составных данных (например, состав семьи). На элементы справочника можно ссылаться; на строки ТЧ — нет.
- **Автонумерация** — код присваивается при записи.
- **Контроль уникальности** — проверка уникальности кода при вводе.
- **Основное представление** — задаётся представление элементов (код или наименование); для форм списков — колонка по умолчанию.
- **Ввод на основании** — указание объектов-оснований и объектов, для которых данный тип является основанием.

### 5.8.4. Предопределённые элементы справочника

Предопределённые элементы и группы создаются в конфигураторе; обращение по имени (например, Справочники.Товары.Услуга). **ВНИМАНИЕ!** Для справочника с владельцем нельзя создать предопределённые элементы; справочнику с предопределёнными нельзя назначить владельца.

## 5.9. Документы

### 5.9.1. Общая информация

Документ — одно из основных понятий; при помощи документов вводится информация о хозяйственных операциях. Характеризуются номером, датой и временем. В конфигурации описывается структура; экземпляры вводятся пользователем. Можно создавать реквизиты и табличные части, формы списков и журналов. Документы могут проводиться (изменять состояние регистров). Ветвь **Документы** дерева конфигурации; там же — нумераторы и последовательности.

### 5.9.2. Свойства документа

**Закладка Основное, Данные:** реквизиты, табличные части, индексирование.

**Закладка Нумерация:**
- **Автонумерация** — номер присваивается автоматически.
- **Нумератор** — назначение нумератора для сквозной нумерации нескольких видов документов.
- **Длина номера**, **Тип номера** (Число/Строка), **Допустимая длина номера** (фиксированная/переменная).
- **Контроль уникальности** — проверка номера в пределах **Периодичность**.

**Закладка Движения:**
- **Проведение** — Разрешить/Запретить; при разрешении вызывается обработчик **ОбработкаПроведения**.
- **Оперативное проведение** — при текущей дате проведение в оперативном режиме; при другой дате — неоперативное (или запрос).
- **Удаление движений** — Удалять автоматически / Не удалять автоматически / Удалять автоматически при отмене проведения.
- **Запись движений при проведении** — Записывать выбранные (по свойству Записывать набора) / Записывать модифицированные.

**Закладка Журналы** — назначение журналов документов. **Ввод на основании** — объекты-основания и объекты, для которых документ является основанием.

**Закладка Права:** Привилегированный режим при проведении / при отмене проведения.

### 5.9.3. Механизм проведения документов

Информация хранится в регистрах; документы изменяют состояние регистров (проведение). Проведение может быть **оперативным** (реальное время, контроль остатков и т.д.) или **неоперативным** (задним числом, фиксация факта). Режим определяется по дате документа; для расширения формы — свойство **Использовать режим проведения** (Неоперативный / Оперативный / Автоматически).

**Момент времени** — дата, время и ссылка на объект БД; реализован объектом **МоментВремени**. **Оперативная отметка времени** — значение типа Дата; создаётся при оперативном проведении; получается методом **ПолучитьОперативнуюОтметкуВремени()**. Учитываются **часовой пояс информационной базы** и **часовой пояс сеанса** (УстановитьЧасовойПоясСеанса()).

### 5.9.4. Нумераторы

Нумератор — объект конфигурации, описывающий правила нумерации: тип и длину номера, периодичность, контроль уникальности. Назначение — сквозная нумерация документов разных видов. Ветвь **Нумераторы** внутри **Документы**. Свойства: Тип номера, Длина номера, Периодичность, Контроль уникальности.

### 5.9.5. Последовательности документов

Последовательности обеспечивают проведение документов в строгой хронологической последовательности (для документов, использующих итоги при проведении). В конфигураторе задаются входящие документы и **Движения, влияющие на последовательность**. Поддерживается граница последовательности; при проведении задним числом или отмене граница отодвигается назад; доступно восстановление последовательности (перепроведение от границы до указанного момента). Измерения последовательности позволяют сузить перепроведение по критериям (например, по номенклатуре).

### 5.9.6. Ввод документов на основании

Режим ввода на основании позволяет заполнять реквизиты путём копирования из другого объекта. Необходим обработчик **ОбработкаЗаполнения** в модуле документа; на закладке **Ввод на основании** задаются объекты-основания и объекты, для которых документ является основанием.

## 5.10. Журналы документов

Журналы позволяют работать с документами разных видов. Создаются формы с табличными полями (вид, дата, номер, дополнительные графы). Ветвь **Журналы документов**. Список документов и граф формируется на закладке **Данные**; в графе указывается реквизит документов. Доступность данных журнала определяется правами на журнал и на документы (Чтение, Просмотр). При нескольких типах документов в журнале — кнопка **Создать** с подменю видов (или диалог выбора типа при старом режиме совместимости).

---

## 5.11. Перечисления

Перечисление — служебный тип данных, список возможных значений реквизита. Используется в реквизитах документов, справочников, констант для однозначного ввода. Список значений задаётся в конфигураторе; пополнение в процессе работы невозможно. Ветвь **Перечисления**. Редактирование — окно объекта Перечисление; закладка Данные — значения (имя, синоним); закладки Формы, Макеты.

## 5.12. Отчёты и обработки

### 5.12.1. Общая информация

**Отчёты** — объекты ветви **Отчеты**; формирование отчёта на встроенном языке или по схеме компоновки данных; формы для ввода параметров, макеты печати. **Обработки** — ветвь **Обработки**; выполнение действий над информацией (импорт, удаление устаревших данных и т.д.). Отличие отчёта от обработки — возможность использования схемы компоновки данных у отчёта. Общие формы отчётов могут быть указаны в свойствах конфигурации для всех отчётов или групп отчётов.

### 5.12.2. Внешние обработки и отчёты

Внешняя обработка/отчёт хранится в отдельном файле (.epf/.erf или XML). Назначение — реализация и поставка возможностей отдельно от конфигурации. Можно сохранять объект конфигурации во внешний файл и наоборот; сравнивать и объединять с конфигурацией или другими внешними файлами. Подключение в 1С:Предприятие — метод **Подключить()** (только на сервере); параметры **БезопасныйРежим**, **КодЯзыка**, **ЗащитаОтОпасныхДействий**. Для интерактивного открытия может запрашиваться разрешение (безопасность).

### 5.12.3. Управление отображением отчёта

Механизм преобразования табличного документа в **компактный вид** (без горизонтальной прокрутки). Свойства расширения формы: **Результат отчёта**, **Режим отображения результата отчёта** (Обычный/Компактный/Авто), **Применение режима отображения при установке результата отчёта**. Методы: УстановитьРежимОтображенияРезультата(), ПолучитьРежимОтображенияРезультата(), ПолучитьТекущийРежимОтображенияРезультата(). Преобразование — методы ТабличныйДокумент: ПреобразоватьВКомпактныйДокумент(), ОтменитьПреобразованиеДокумента(), ПолучитьКомпактныйДокумент(), УстановитьРастягиваниеПоГоризонтали().

## 5.13. Планы видов характеристик

### 5.13.1. Общее описание

План видов характеристик (ПВХ) решает задачу хранения свойств объектов, состав и тип которых заранее не определены (характеристики номенклатуры, контрагентов и т.д.). ПВХ хранит набор типов для значений характеристик (свойство **Тип значения характеристик**) и при необходимости **Дополнительные значения характеристик** (справочник для значений, не описанных в конфигурации). Для объектов метаданных в настройках дополнительных характеристик задаются правила отбора видов характеристик и правила получения значений. Тип реквизита для значения характеристики — **Характеристика.<ИмяПВХ>**.

### 5.13.2. Характеристики объектов конфигурации

По устройству ПВХ похож на справочник (иерархия, предопределённые элементы, реквизиты, ТЧ). Ключевое отличие — свойство **Тип значения характеристик**. Вид характеристики (элемент ПВХ) может иметь ограниченный набор типов в рамках типов ПВХ. Редактирование значений характеристик — по объекту данных, виду характеристики и значению; система настраивает связи по типу и параметры выбора. ПВХ используется и для субконто бухгалтерского учёта (виды субконто задаются предопределёнными элементами ПВХ).

Примеры реализации: (1) свой ПВХ и регистр сведений на объект; (2) справочник дополнительных значений + ПВХ; (3) один ПВХ и один регистр для нескольких объектов с отбором по реквизиту типа объекта; (4) наборы характеристик — справочник с предопределёнными элементами и ТЧ состава набора; (5) хранение характеристик в ТЧ объекта (для RLS и доступа по объекту).

### 5.13.3. Субконто аналитического учёта

ПВХ может содержать виды субконто; в плане счетов для счёта указываются виды субконто из этого ПВХ.

### 5.13.4. Формы и использование в отчётах/списках

При корректной настройке форм для просмотра и редактирования характеристик создавать не обязательно — генерируются автоматически. В отчётах и динамических списках характеристики доступны через поля схемы компоновки (отбор, сортировка, группировка, условное оформление, колонки). Для отображения в колонке списка нужно поле ссылочного типа объекта и добавление вида характеристики через настройку формы.

## 5.14. Регистры

### 5.14.1. Общая информация

Регистры предназначены для хранения и обработки информации, отражающей хозяйственную или организационную деятельность. В регистрах хранится информация об изменении состояний объектов (курсы валют, приход/расход товаров и т.д.). Единица хранения — **запись**; у записей регистров нет ссылок. В главе описаны регистры сведений и регистры накопления.

### 5.14.2. Регистры сведений

#### 5.14.2.1. Общая информация

Регистр сведений хранит информацию, развёрнутую по комбинации измерений и при необходимости во времени. **Измерения** — аргументы; **ресурсы** — результат; возможны **реквизиты**. **Периодический** регистр имеет поле **Период**; поддерживаются срезы последних/первых. Регистр без измерений — набор периодических данных (например, подписи должностных лиц по датам). В одной записи — единственный набор измерений и периода (уникальность). Итоги (срез последних/первых) используются при включённом разрешении и обращении без указания периода. **Режим записи**: независимый (ручной/язык) или подчинение регистратору. **Ведущее** измерение — при ссылке на объект БД записи удаляются при удалении объекта.

#### 5.14.2.2. Записи регистра сведений

Запись идентифицируется **ключом записи** (значения измерений + период для периодического). Записи вносятся вручную или документами (регистратор).

#### 5.14.2.3. Чтение и запись из встроенного языка

**Ключ записи** — объект РегистрСведенийКлючЗаписи (СоздатьКлючЗаписи(Измерения)). **Менеджер записи** (РегистрСведенийМенеджерЗаписи): Прочитать(), Выбран(), установка измерений/ресурсов/реквизитов, Записать(Истина — замещение, Ложь — добавление). **Набор записей** (РегистрСведенийНаборЗаписей): отбор, Прочитать(), Добавить()/Вставить(), Записать() с режимом РежимЗамещения (Замещение, Добавление, Обновление, Удаление, Слияние — только для независимого). Свойство **РасширенныеРежимыЗамещения** разрешает Обновление/Удаление и редактирование НомерСтроки. Для подчинённого регистра — особенности по регистратору и НомерСтроки. Чтение выборкой — метод **Выбрать()** менеджера регистра.

#### 5.14.2.4. Редактирование регистра сведений

Ветвь **Регистры сведений**. Свойства: **Периодичность**, **Режим записи**, **Основной отбор по периоду** (для независимого периодического). На закладке Данные — измерения, ресурсы, реквизиты; для измерения **Режим сокращения типа** (Преобразовывать/Удалять/Запрещать). Закладка Регистраторы — при подчинении регистратору.

#### 5.14.2.5. Разработка структуры. Свойства измерения (ресурса, реквизита)

**Тип** — тип хранимого значения. **Ведущее** — для ссылочного измерения (удаление записей при удалении объекта). **Запрет пустых значений**. **Индексировать** — отдельный индекс для ускорения. Порядок измерений влияет на производительность и позиционный доступ.

### 5.14.3. Регистры накопления

#### 5.14.3.1. Общая информация

Регистр накопления хранит движения и итоги; используется для наличия и движения величин (товары, деньги). Структура: измерения (оси) и ресурсы (значения в узлах). Методы встроенного языка дают остатки на момент времени, фильтрацию по измерениям. Могут быть реквизиты движений.

#### 5.14.3.2. Движения регистра накопления

Движения формируются при проведении документа в обработчике **ОбработкаПроведения** через коллекцию **Движения**. В набор записей регистра добавляются записи с **ВидДвижения** (Приход/Расход), Период, измерения, ресурсы, реквизиты. Регистратор и НомерСтроки заполняются платформой. Запись — метод Записать() набора или автоматически после выхода из обработчика (свойство документа **Запись движений при проведении**). Вне проведения — чтение набора, модификация, Записать() с режимом Замещение/Добавление/Обновление/Удаление (РасширенныеРежимыЗамещения).

#### 5.14.3.3. Итоги регистра накопления

Итоги — сводная информация, получаемая суммированием движений. Таблица итогов: колонки — измерения + ресурсы; строки — уникальные комбинации измерений. Просмотр итогов — через отчёты и запросы.

#### 5.14.3.4. Регистры остатков и регистры оборотов

**Регистр остатков** — получение остатков на момент времени по измерениям. **Регистр оборотов** — для данных без понятия остатка (обороты продаж, объём закупок).

#### 5.14.3.5. Основные свойства регистра накопления

**Вид регистра** — Остатки / Обороты. **Основная форма списка**. **Разрешить разделение итогов** — повышение параллельности записи (разделитель итогов). **Использование в итогах** (для измерения) — исключение из хранимых итогов. Тип измерения не может быть УникальныйИдентификатор, ДвоичныеДанные, строка неограниченной длины.

#### 5.14.3.6. Агрегаты оборотных регистров накопления

Агрегаты — физические таблицы сводных оборотов по ресурсам для ускорения запросов. Параметры: размер агрегата, эффект, период, периодичность. **Режим агрегатов/итогов** переключается (УстановитьРежимАгрегатов()); **Использование агрегатов** можно отключать (например, при массовой загрузке). Схема работы: создание списка агрегатов (вручную или ОпределитьОптимальныеАгрегаты()), включение режима агрегатов, перестроение (ПерестроитьИспользованиеАгрегатов()), обновление (ОбновитьАгрегаты()), накопление статистики. В файловом варианте список агрегатов создаётся в конфигураторе обязательно; расчёт оптимальных и обновление списка — отдельные шаги. Редактирование агрегатов — конструктор агрегатов (только для оборотного регистра). Вариант использования агрегата: Авто / Всегда.

#### 5.14.3.7. Разработка структуры регистра накопления

Закладка **Данные** — измерения, ресурсы, реквизиты. **Ресурс** — только тип Число. **Запрет незаполненных значений** (для измерения). **Индексировать** — только для измерений. **Использование в итогах** — для измерений оборотного регистра. Порядок измерений влияет на оптимизацию доступа к итогам.
