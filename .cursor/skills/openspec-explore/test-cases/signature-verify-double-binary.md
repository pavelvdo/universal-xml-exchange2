## Эталонный entry-бриф B3

```markdown
Бриф: проверка подписи показывает «недействительна»

**Симптом** — подпись Kontur валидна, но проверка в ДО показывает «недействительна»; в отладчике два варианта в списке дополнительных проверок.

**Ожидание** — при валидной подписи проверка должна проходить без «недействительна».

**Маршрут** — разберу трассу проверки, сравню с цепочкой подписания; при связи с архивным ЗНИ по Kontur сверюсь с ним. В чате — ответ «баг / штатно» и при необходимости блок `## Постановка ЗНИ`.

**Вопрос** — что расходится между подписанием и проверкой; чинить или это штатное поведение?

**Контекст**
- `temp/Проверка подписи_TRACE_FULL.txt`
- архив `kontur-signing-contract-regression`

Подтвердить? — «да» или уточните (например, только сценарий «из задачи»).
```

## Эталонный фрагмент `## Для заказчика` в отчёте trace-analyst

В полном отчёте `temp/reports/trace-analysis-YYYY-MM-DD-signature-verify-double-binary.md`, после технического раздела:

1. **Что наблюдаешь:** подпись валидна, проверка в ДО — недействительна.
2. **Почему так устроено:** типовая ДО строит два канона (организация заполнена/пустая); в отладчике два варианта.
3. **Связано ли:** `не объясняет, симптом из другой ветки` (двойное получение ≠ invalid).
4. **Что чинить:** `[hypothesis: byte-diff sign vs verify вариант [0]]` — рассогласование `ДанныеПодписанта` / `Стороны.*` при подписании из задачи (`H-sign-task`, exploration-verify-org-variants).

## Блок `## Постановка ЗНИ` в чате (финал) — доменный пример, не эталон формата

Каркас и правила сборки — `../templates/handoff-block.md` (SSOT). Пример ниже показывает доменное наполнение:

```markdown
## Постановка ЗНИ

**Симптом:**
Подпись Kontur валидна, но проверка в ДО показывает «недействительна». В отладчике видно два варианта проверки в `ВариантыДополнительныхПроверок`; сбой воспроизводится при подписании из задачи.

**Корневая причина:** `cfe/ргОблачныеПодписиKonturCryptoAPI_ЭДОПАО:???` — [hypothesis: byte-diff sign vs verify для варианта [0]]
Канон для варианта [0] («организация заполнена») при подписании из задачи собирается с другим составом `ДанныеПодписанта` / `Стороны.*`, чем при проверке. Подпись считается по одним байтам, проверяется по другим (`H-sign-task` из архивного исследования).

**Что менять:**
Выровнять контракт канона на sign и verify: вариант [0] должен собираться идентично в обеих точках. Сначала верифицировать гипотезу byte-diff (explorer на verify-org-variants vs sign-task), затем фикс в расширении.

**Файлы:** `cfe/ргОблачныеПодписиKonturCryptoAPI_ЭДОПАО/` + типовой модуль построения вариантов проверки
**Приёмка:** для тестового документа из инцидента проверка возвращает «действительна» обоими вариантами; регресс по «организация пустая» не воспроизводится.
**Связь с архивом:** extends `kontur-signing-contract-regression` (закрывает `H-sign-task`)
**Architect / verify:** required (UX-значимый фикс отображения статуса подписи)
```
