# Домен: Метаданные и объекты

> Стандарты и методики (ИТС v8std). Источник — вендорская документация 1С.

---

## Общие требования к конфигурации #std467

Область применения: управляемое приложение, обычное приложение.

- 1.1. Конфигурация должна использовать только штатные и документированные возможности платформы 1С:Предприятие. Взаимодействие с пользователем должно быть организовано асинхронно. Дополнительные материалы: список поддерживаемых ОС и СУБД; особенности работы с различными СУБД (приложение 8 документации платформы); особенности режима низкой скорости соединения и веб-клиента (приложение 7); особенности разработки конфигураций для ОС Linux.
- 1.3. Конфигурация не должна содержать ошибок, обнаруживаемых при проверке конфигурации (Конфигурация → Проверка конфигурации), кроме обоснованных случаев: обработчики событий модуля формы, подключаемые из кода; ограничение на модальные окна и синхронные вызовы; ограничение на признак «Вызов сервера»; несущественные предупреждения проверки.
- 1.4. Для обратной совместимости конфигурация должна поддерживать запуск в режимах обычного приложения (толстый клиент) и внешнего соединения для администраторов. Рекомендуется: свойство «Использовать управляемые формы в обычном приложении» = Истина, «Использовать обычные формы в управляемом режиме» = Ложь; придерживаться общей схемы признаков общих модулей; вести разработку в Конфигураторе в режиме редактирования для обоих режимов (Сервис → Параметры → Общие). Отказ от поддержки допустим только в обоснованных случаях.
- 1.5. Не рекомендуется отходить от умолчаний платформы при проектировании решений, разработке интерфейса и отчётов. Альтернативы допустимы только в обоснованных случаях.
- 2.1. Имена, синонимы, комментарии объектов метаданных, общих модулей и любая текстовая информация (пользователю или разработчику) должны соответствовать правилам русского языка и не содержать грамматических ошибок.
- 2.2. В конфигурации не должно быть неиспользуемых объектов метаданных и неиспользуемого программного кода (ни в конфигурации, ни для интеграции).
- 2.3. Объекты метаданных верхнего уровня (Справочники, Документы, Общие модули и т.д.) рекомендуется сортировать в дереве метаданных по имени. Подчинённые объекты располагаются по проектной логике. Исключения: общие реквизиты-разделители (порядок по порядку установки параметров сеанса); объекты с префиксом «Удалить» допустимо размещать в конце ветки.

## Имена объектов метаданных в конфигурациях

Область применения: управляемое приложение, обычное приложение. См. также: общие правила наименования метаданных.

Правила наименования по типам объектов:

| Объекты метаданных | Правила наименования |
|-------------------|----------------------|
| Подсистемы | По общим правилам. Примеры: Финансы, Маркетинг, НастройкаИАдминистрирование. См. Использование подсистем. |
| Общие модули | См. Правила создания общих модулей. |
| Параметры сеанса | По общим правилам. Примеры: ТекущийПользователь, ОбменДаннымиВключен, РаботаСВнешнимиРесурсамиЗаблокирована. |
| Роли | «Прикладные» роли — от должности (Бухгалтер, Кассир, Администратор). Роли для «тонкой» настройки — от разрешаемого действия (ДобавлениеИзменениеНСИ, ЧтениеДополнительныхСведений). См. стандартные роли. |
| Общие реквизиты | По общим правилам (управляемое и обычное приложение). |
| Планы обмена | По принципам наименования планов обмена. |
| Параметры функциональных опций | От описания параметра. Имя не обязательно совпадает с наименованием реквизитов. См. Использование функциональных опций. |
| Определяемые типы | В единственном числе, от назначения. Не использовать слова «Тип», «Объект», «Ссылка» там, где от удаления смысл не меняется. Пример: Строка25, АртикулНоменклатуры. См. Использование определяемых типов. |
| Хранилища настроек | По общим правилам. Пример: ХранилищеВариантовОтчетов. См. определяемые типы. |
| Общие формы | От существительных. Избегать слов «Форма», «Окно», «Диалог». Примеры: НастройкаСистемы, МоиНастройки, ПараметрыПроксиСервера, ВыборОбъектовМетаданных. |
| Общие команды | Для панели навигации/раздела — имя обозначает список объектов (ДополнительныеОтчетыИОбработки, ЗадачиПоБизнесПроцессу). В остальных случаях — от неопределённой формы глагола (ВыполнитьСопоставление, УстановитьРасширениеРаботыСФайлами). Примеры: ПараметрыОбменаДанными, Печать. |
| Группы команд | От существительных. |
| Интерфейсы | По общим правилам. См. общие интерфейсы. |
| Общие макеты | От существительных, без слова «Макет». Примеры: ДополнительнаяОбработка, КомпонентаTWAIN, ОписаниеИзмененийСистемы. |
| Общие картинки | По общим правилам. Допускается спецификатор размера: Папка (16×16), УправлениеПоиском32 (32×32). Пример: Найти, ЗакрепитьВариантОтчета. |
| XDTO-пакеты | На русском, от существительных. Избегать «Пакет», «ХDTO». Пример: Файлы. |
| Web-сервисы | На английском, от существительных. Не рекомендуется кириллица. Избегать «Service», «WebService». Примеры: Files, Accounts, FlightStatus. Параметры и операции — на английском (GetCurrencyRate). |
| Документы | Суть процесса, лаконично. Избегать «Накладная» и т.п. Исключения обосновывать. |
| Журналы документов | Множественное число, от описания списка объектов. Примеры: СкладскиеДокументы, КорректировкиНДС. Избегать «Документы…». |
| Перечисления | Множественное число. Примеры: ДействияСДокументамиПоДвойномуЩелчку, ВариантыВажностиЗадачи, ВидыSMTPАутентификации. Неправильно: ДействиеСДокументамиПоДвойномуЩелчку, ВажностьЗадачи. |
| Планы видов характеристик | Множественное число, от списка объектов. Примеры: ВидыДоступа, ВопросыДляАнкетирования, ДополнительныеРеквизитыИСведения. |
| Обработки | Избегать «Обработка…». Если форма открывается из панели навигации/раздела — имя обработки совпадает с именем команды (РегламентныеИФоновыеЗадания). |
| Планы счетов | Единственное число, от назначения. В синониме — полное наименование. Избегать «ПланСчетов…». Примеры: ЕПСБУ, БухгалтерскийУчет, УправленческийУчет, МеждународныйУчет, Бюджетирование, НалоговыйУчет. Пояснение — для подсказки к команде открытия. |
| Планы видов расчета | Множественное число, от списка видов. Примеры: ОсновныеНачисления, УправленческиеНачисления, Удержания. |
| Регистры сведений, накопления | Множественное число, от описания записей. Примеры: ДокументыФизическихЛиц, ФайлыВРабочемКаталоге, ДвиженияДенежныхСредств. |
| Регистры бухгалтерии, расчета | От описания списка записей. Примеры: Хозрасчетный, Начисления. |
| Бизнес-процессы | В единственном числе, как документы. Примеры: Задание, Согласование, Утверждение, Поручение. |
| Задачи | В единственном числе. Пример: ЗадачаИсполнителя. |
| Внешние источники данных | От описания импортируемых данных. Избегать «Данные», «ИсточникДанных». |
| Регламентные задания | Единственное число, от существительного. Правильно: УстановкаПериодаРасчитанныхИтогов, УведомлениеИсполнителейОНовыхЗадачах. Неправильно: УстановитьПериодРасчитанныхИтогов, УведомитьИсполнителейОНовыхЗадачах, СсылкиНаКонтактыВзаимодействий. |

См. также: Ограничения на переименование объектов метаданных.

## Правила создания общих модулей #std469

Область применения: управляемое приложение, обычное приложение.

- 1.1. Общие модули создаются для процедур и функций, объединённых по признаку: одна подсистема (продажи, закупки) или сходное назначение (работа со строками, общего назначения).
- 1.2. При разработке следует выбирать один из четырёх контекстов выполнения.

Типы общих модулей:

| Тип | Пример наименования | Вызов сервера | Сервер | Внешнее соединение | Клиент (обычное) | Клиент (управляемое) |
|-----|---------------------|---------------|--------|---------------------|------------------|----------------------|
| Серверный | ОбщегоНазначения, ОбщегоНазначенияСервер | — | + | + | + | — |
| Серверный для вызова с клиента | ОбщегоНазначенияВызовСервера | + | + | — | — | — |
| Клиентский | ОбщегоНазначенияКлиент, ОбщегоНазначенияГлобальный | — | — | — | + | + |
| Клиент-серверный | ОбщегоНазначенияКлиентСервер | — | + | + | + | + |

- 2.1. Серверные общие модули: процедуры и функции не доступны из клиентского кода. Для корректной работы в режимах внешнего соединения, управляемого и обычного приложения устанавливать признаки: Сервер (Вызов сервера снят), Клиент (обычное приложение), Внешнее соединение. Имена — по общим правилам; допустим постфикс «Сервер» (например РегламентныеЗаданияСервер, ОбменДаннымиСервер). См. ограничение на признак «Вызов сервера».
- 2.2. Серверные модули для вызова с клиента: признак «Вызов сервера» установлен. Постфикс «ВызовСервера» (англ. ServerCall). Примеры: РаботаСФайламиВызовСервера. Экортные процедуры и функции не должны содержать параметров мутабельных типов (СправочникОбъект, ДокументОбъект и т.п.). См. ограничение на установку признака «Вызов сервера».
- 2.3. Клиентские общие модули: признаки «Клиент (управляемое приложение)», «Клиент (обычное приложение)». Постфикс «Клиент» (Client). Примеры: РаботаСФайламиКлиент, ОбщегоНазначенияКлиент. См. минимизация кода на клиенте.
- 2.4. Клиент-серверные: одинаковый код на сервере и клиенте. Признаки: Клиент (управляемое и обычное), Сервер (Вызов сервера сброшен), Внешнее соединение. Постфикс «КлиентСервер» (ClientServer). При необходимости ветвления по контексту не использовать препроцессор — выносить в модули соответствующего типа (2.1, 2.3). См. использование директив компиляции и препроцессора.
- 3.1. Имена — по общим правилам. Название должно совпадать с подсистемой или механизмом. Избегать слов «Процедуры», «Функции», «Обработчики», «Модуль», «Функциональность» кроме исключительных случаев. Для различения контекстов — постфиксы из п. 2.
- 3.2. Дополнительные постфиксы: «Глобальный» (Global) — для глобальных, без «Клиент»; «ПолныеПрава» (FullAccess) — привилегированный режим; «ПовтИсп» (Cached) / «КлиентПовтИсп» (ClientCached) — повторное использование возвращаемых значений; «Переопределяемый» (Overridable) / «КлиентПереопределяемый» (ClientOverridable) — библиотечные модули с изменяемой реализацией; «Локализация» (Localization) / «КлиентЛокализация» (ClientLocalization) — национальная специфика.

См. также: Структура модуля, переопределяемые и поставляемые объекты библиотеки.

## Использование подсистем

Область применения: управляемое приложение, обычное приложение. #std543

- 1.1. Подсистемы решают две задачи: формирование глобального командного интерфейса основного окна; группировка объектов метаданных по функциональному признаку для удобства разработки. Структура может совпадать. У подсистем, видимых в интерфейсе, должен быть установлен флажок «Включать в командный интерфейс».
- 1.2. Для логического объединения объектов без отображения в одном разделе интерфейса рекомендуется отдельная иерархия подсистем с снятым флажком «Включать в командный интерфейс». Примеры: справочник Номенклатура в «функциональной» подсистеме «Нормативно-справочная информация» и в разделах «НСИ» и «Маркетинг»; в «Настройка и администрирование» помещаются команды списков объектов из функциональных подсистем, связанных с настройкой.
- 1.3. Общие модули, регламентные задания, константы, подписки на события и прочие объекты без визуального представления рекомендуется включать только в состав «функциональных» подсистем.

## Использование общих реквизитов

Область применения: управляемое приложение, обычное приложение. #std677

- 1. Общие реквизиты добавляют реквизиты сразу нескольким объектам метаданных для: разделения данных (свойство «Разделение данных» = Разделять); расширения состава реквизитов (Разделение данных = Не использовать).
- 2. Общие реквизиты без разделения данных — для функционала, не являющегося частью бизнес-логики прикладных объектов, но требующего хранения данных в самих объектах. Не предназначены для удобства добавления одинаковых реквизитов. Неправильно переносить в общие реквизиты «обычные» реквизиты документов (Ответственный, Комментарий, Организация и т.п.). Права доступа к общим реквизитам настраиваются отдельно от объектов.
- 3. Порядок общих реквизитов-разделителей в дереве метаданных подбирать исходя из порядка установки связанных параметров сеанса.

См. также: Использование общих реквизитов без разделения данных (статья на ИТС).

## Использование определяемых типов

Область применения: управляемое приложение, обычное приложение. #std704

- 1. Определяемые типы задают типы данных для часто используемых сущностей или склонных к изменению при внедрении. Позволяют многократно использовать тип в реквизитах, свойствах, формах и т.п. См. статью на ИТС: «Объекты конфигурации – Определяемые типы».
- 2. Рекомендуется использовать в случаях:
  - 2.1. Простой тип с квалификаторами, имеющий прикладной смысл, в рамках подсистемы или всего решения (одинаковая длина, точность, упрощение доработки). Примеры: НомерСчетаФактуры (Строка, 30) в документах ПоступлениеТоваровИУслуг, ЗаписьКнигиПокупок и др.; АдресДоставки (Строка, 500) в документах ЗаказПоставщику, в обработке ПомощникПродаж и др.
  - 2.2. Составной тип, массово используемый в подсистеме или решении (одинаковый состав типов, упрощение внедрения подсистем). Пример: подсистема Взаимодействия, определяемый тип КонтактВзаимодействий (справочники ФизическиеЛица, Партнеры, КонтактныеЛицаПартнеров) в реквизитах документов Встреча, СообщениеSMS, в формах АдреснаяКнига, ВыборКонтакта и т.д.
  - 2.3. При разработке внедряемой подсистемы — для переопределения прикладного типа при внедрении (например тип Поставщики → Контрагенты).
- 3. Некорректно использовать определяемые типы: как «синоним» существующему типу; для «подмены» сущностей; при локальном (не массовом) использовании без необходимости внедрения; только из соображений удобства доработки. Пример ошибки: справочник Контрагенты не встраиваемая подсистема и не расширяется другими типами — некорректно вводить определяемый тип из одного типа Контрагенты «на всякий случай».

## Использование предопределенных элементов

Область применения (уточнение): управляемое приложение, обычное приложение. Действует для платформы 8.3.3 и выше без режима совместимости с 8.2.

- 1.1. В справочниках, планах счетов, планах видов характеристик и планах видов расчета предопределённые элементы можно создавать автоматически или программно.
- 1.2. В большинстве случаев предопределённые элементы рекомендуется создавать автоматически. Для этого: свойство «ОбновлениеПредопределенныхДанных» = Авто (по умолчанию), не допускать программных вызовов УстановитьОбновлениеПредопределенныхДанных для переключения режима; запретить удаление предопределённых элементов пользователями — во всех ролях выключены права: ИнтерактивноеУдалениеПредопределенныхДанных, ИнтерактивнаяПометкаУдаленияПредопределенныхДанных, ИнтерактивноеСнятиеПометкиУдаленияПредопределенныхДанных, ИнтерактивноеУдалениеПомеченныхПредопределенныхДанных.
- 1.3. Исключение — дочерние узлы РИБ: предопределённые элементы не создаются и не обновляются автоматически, передаются из главного узла. При этом: (а) конфигурация должна обеспечивать загрузку сообщения обмена в подчинённый узел до другого прикладного кода, обращающегося к предопределённым элементам; (б) в прикладной логике загрузки (ПриПолученииДанныхОтГлавного, правила регистрации) избегать обращений к предопределённым элементам до их загрузки; (в) код обновления ИБ по предопределённым элементам не выполнять в подчинённых узлах (проверка ПланыОбмена.ГлавныйУзел() = Неопределено). При использовании подсистемы «Обмен данными» БСП 2.1.4+ требования (а) и (б) снимаются.
- 1.4. Для таблиц с предопределёнными элементами, не входящих в план обмена РИБ, свойство «ОбновлениеПредопределенныхДанных» = ОбновлятьАвтоматически (значение Авто для подчинённого узла означает НеОбновлятьАвтоматически). При режиме совместимости с 8.3.3 при первом запуске подчинённого узла после обновления конфигурации вызывать Справочники.<ИмяСправочника>.УстановитьОбновлениеПредопределенныхДанных(ОбновлениеПредопределенныхДанных.ОбновлятьАвтоматически).
- 2. Если наличие предопределённого элемента зависит от условия (функциональная опция, режим работы и т.п.): свойство «ОбновлениеПредопределенныхДанных» = «Не обновлять автоматически»; предусмотреть код создания (и пометки недействительным) в зависимости от бизнес-логики; в прикладном коде учитывать отсутствие элемента в ИБ (иначе обращение к несуществующему предопределённому элементу вызовет исключение). При использовании БСП 2.1.4+ рекомендуется функция ПредопределенныйЭлемент (ОбщегоНазначения / ОбщегоНазначенияКлиент), возвращающая Неопределено для несуществующих элементов. #std697

Пример для условного создания:

```
Если ПолучитьФункциональнуюОпцию("ИспользоватьУчетВремениСотрудниковВЧасах") Тогда
    НачислениеОбъект = ПланыВидовРасчета.Начисления.СоздатьВидРасчета();
    НачислениеОбъект.ИмяПредопределенныхДанных = "ОкладПоЧасам";
    // ...
    НачислениеОбъект.Записать();
КонецЕсли;
```

## Удаление устаревших объектов метаданных из конфигурации

#std534

- 1. При планировании удаления объекта метаданных, связанного с записями в ИБ, необходимо решить: удаление или перенос данных. При переносе:
  - 1.1. Не удалять устаревшие объекты безвозвратно, а помечать префиксом «Удалить» (Obsolete) к имени. В синоним добавлять «(не используется)» ((not used)). Для стандартного реквизита — префикс в синониме тоже.
  - 1.2. После изменения структуры обеспечить перенос данных из устаревших реквизитов в новую структуру.
  - 1.3. Если удаляемый объект — документ-регистратор, а регистры остаются: запретить генерацию движений при проведении; запретить снятие пометки удаления для документов этого вида; в существующих движениях сменить регистратор на замещающий (например «Перенос данных», «Операция»); пометить все документы этого вида на удаление. Удалить лишние права (кроме ПолныеПрава, АдминистраторСистемы), подписки на события, код, формы, макеты, команды.
  - 1.5. Сортировка устаревших объектов — по общим требованиям к конфигурации.
  - 1.6. Рекомендуется очистка устаревших данных. При сложных алгоритмах переноса очистку можно проводить через один или несколько релизов.
- 2. При пересмотре структуры измерений регистров: создать новый регистр с правильной структурой, старый пометить устаревшим и перенести записи. Новый регистр не требуется, если добавляется измерение или расширяется состав типов у измерения составного типа.
- 3. Безвозвратно удалять объекты и реквизиты с префиксом «Удалить» допустимо при выпуске новой версии, если: (1) переход со старой версии всегда выполняется последовательно через версию с переносом данных, либо (2) вероятность использования старой версии нулевая или пренебрежимо мала. Иначе возможна ошибка реструктуризации регистров при ссылках на устаревшие данные.

## Ограничения на переименование объектов метаданных

#std706

Область применения: управляемое приложение, мобильное приложение, обычное приложение.

- 1. Запрещается переименовывать общий модуль и создавать новый с тем же именем. Следует создавать новый общий модуль с новым именем и переносить код из исходного. Иначе при сравнении и объединении с конфигурацией поставщика подписки на события по имени свяжутся с новым модулем, в котором нет требуемых процедур.
- 2. При необходимости удалить объект метаданных и создать новый с тем же именем (например при «сужении» измерений регистра) следует переименовать устаревший объект, добавив префикс «Удалить». Иначе при обновлении конфигурации на поддержке возможна ошибка: если в устаревший объект вносились изменения с сохранением поддержки, он не будет удалён при объединении, и при обновлении БД окажется два объекта с одним именем. См. Удаление устаревших объектов метаданных из конфигурации.
- 3. Хранение ссылок на объекты метаданных:
  - 3.1. Без БСП — строковые реквизиты (Строка, 255) для полных имён. При переименовании/удалении объектов предусматривать замену (удаление) имён в БД при обновлении ИБ.
  - 3.2. С БСП — использовать справочник ИдентификаторыОбъектовМетаданных (централизованное хранение, отслеживание переименований/удалений). Исключение: роли и подсистемы требуют явного описания переименований в БСП. Иначе рассогласуются ссылки (варианты отчётов, дополнительные отчёты и обработки, роли в профилях групп доступа).
  - 3.3. Справочник ИдентификаторыОбъектовМетаданных не предназначен для ссылок на объекты других конфигураций (интеграция) — использовать строковые реквизиты и свои механизмы актуализации.
  - 3.4. При параллельных ветках разработки (например 2.0 и 3.0): в текущей и младших версиях запрещены переименование с созданием нового объекта с тем же полным именем и двойное переименование. Такие переименования выполнять только в самой новой версии (3.0). При использовании ИдентификаторыОбъектовМетаданных запрет для веток касается только ролей и подсистем. См. документацию БСП.

Методическая рекомендация (полезный совет).
