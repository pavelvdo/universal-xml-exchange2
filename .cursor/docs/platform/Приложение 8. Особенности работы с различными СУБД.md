# Приложение 8. Особенности работы с различными СУБД

> **Agent ref:** Особенности работы 1С:Предприятие с различными СУБД.  
> **Ключевые разделы:** файловая БД, IBM Db2, Microsoft SQL Server, Oracle Database, PostgreSQL.

---

## 8.1. Общие особенности

Платформа поддерживает файловый и клиент-серверный варианты с разными СУБД. Ниже — технические ограничения и рекомендации по каждой СУБД.

## 8.2. Файловая база данных

- Файл базы данных (.1CD) организован как набор внутренних файлов. На каждую таблицу — до четырёх: описание таблицы; записи данных (без полей неограниченной длины); индексы (если есть); значения неограниченной длины.
- Размер каждого внутреннего файла: для формата 8.2.14 — не более 4 Гбайт; для 8.3.8 при странице 4 096 байт — 4 Гбайт; при странице 8 192–65 536 байт — 6 Гбайт.
- Длина ключа индекса — не более 1 920 байт.
- Уникальность индексов в файловом варианте может нарушаться. При выгрузке в .dt и загрузке в клиент-серверную ИБ возможны фатальные ошибки. При ошибках уникальности нужно устранить дубли в файловой ИБ и повторить выгрузку/загрузку.

## 8.3. IBM Db2

- В индекс — не более 16 полей (в файловом варианте — до 256).
- Сравнение строк в языке запросов не учитывает концевые пробелы; во встроенном языке — учитывает (например, «bb» и «bb » — Ложь в коде, Истина в запросе).
- ПОДОБНО (LIKE): правый операнд — только литерал или выражение над литералами. Шаблоны: «_» (любой символ), «%» (последовательность символов).
- Нетипизированный NULL: результат «ВЫБРАТЬ NULL» даёт составной тип; такая колонка не может быть операндом арифметики (например, ЕСТЬNULL(f1+1,1) по полю из подзапроса с NULL даёт ошибку).
- Максимальный размер данных неограниченной длины — 1 Гбайт.
- Не более 1012 колонок в списке выборки (с учётом раскрытия составных типов в отдельные колонки СУБД).
- Свои правила точности результата арифметических операций.
- Подзапрос в разделе ПО может сильно замедлять запрос или приводить к невыполнению при многократном использовании в соединениях.
- Конкатенация 10 и более строк в сравнении может давать SQL0401N (несовместимые типы).
- Не допускается соединение двух таблиц по условию, включающему сравнение полей табличной части.
- COLLATION_SEQUENCE: если не равен UCA500R1_LROOT_AN_CX_EX_FX_HX_NX_S2 (ИБ создана 8.1.12 и младше), то: нет фонового обновления конфигурации; в DISTINCT, GROUP BY, UNION строки различаются по регистру; в УПОРЯДОЧИТЬ ПО — то же при наличии UNION/DISTINCT, ВЫБОР с вложенными запросами или RLS без РАЗРЕШЕННЫЕ. ИБ 8.1.13+ при создании получает нужный COLLATION.
- При медленных запросах с HASH JOIN и большим числом операций с диском можно попробовать db2set DB2_OVERRIDE_BPF=5000 и перезапуск СУБД (не требуется для Db2 9.7 FixPack 5 и выше).

## 8.4. Microsoft SQL Server

- Не более 256 таблиц в запросе (SQL Server 2000 и 2005).
- Размер одной записи в таблице — не более 8 060 байт.
- Размер одной записи индекса — не более 900 байт (ошибка проявляется при вставке, а не при обновлении конфигурации).
- SQL Server 2000: при группировке по выражению и использовании в ИМЕЮЩИЕ агрегатной функции, не входящей в ВЫБРАТЬ, возможна ошибка СУБД — включать выражения из ИМЕЮЩИЕ в ВЫБРАТЬ.
- SQL Server 2000: не рекомендуется использовать функции дат в Сгруппировать По (риск неверных результатов); предпочтительна группировка по полям.
- Для разделителей: при участии таблиц объектов общего реквизита-разделителя в запросе подзапросы в условиях соединения не допускаются.
- Казахстан в региональных настройках ИБ: казахская сортировка не используется, применяется Cyrillic_Generic_CI_AS (аналогично России).

## 8.5. Oracle Database

- Размер записи индекса — не более ~80% размера блока: 4К — 3 318 байт, 8К — 6 398 байт, 16К — 12 958 байт.
- В автоматическом режиме — табличные блокировки (одна запись блокирует таблицу). Рекомендуется режим управляемых блокировок; автоматический — только для совместимости со старыми конфигурациями.
- Ограничение на подзапрос в операторе В: при определённых условиях (ПЕРВЫЕ 1 в подзапросе, константы слева и в списке выборки) вложенный запрос во В может быть недоступен или вести себя иначе.
- При сортировке по возрастанию NULL — последними.
- Подзапрос в разделе ПО может замедлять выполнение или приводить к невыполнению запроса.
- Оптимизация записи: сначала обрабатывается первый по порядку реквизит типа ХранилищеЗначений, при отсутствии — первый реквизит типа Строка неограниченной длины.
- Статистика СУБД влияет на производительность. Рекомендуется актуальная статистика (dbms_stats.gather_schema_stats), в т.ч. после загрузки из .dt и после перезапуска сервера.
- Регистры с более чем тремя измерениями типа Строка: при обновлении конфигурации и загрузке ИБ может потребоваться табличное пространство для индексов с большим ключом. Платформа может использовать пространство V81C_INDEX_BIG. Создание — через CREATE TABLESPACE (BLOCKSIZE); для блока 16384 байт может потребоваться DB_16K_CACHE_SIZE (ALTER SYSTEM).
- ПОДОБНО: квадратные скобки в шаблоне — специальные символы только в текстовом литерале, не в выражении (в переменной/колонке скобки не несут специального смысла).

## 8.6. PostgreSQL

- В автоматическом режиме — табличные блокировки. Рекомендуется режим управляемых блокировок.
- При сортировке по возрастанию NULL — последними; по убыванию — первыми.
- При интенсивном удалении/добавлении записей (например, перепроведение) возможна деградация производительности. Рекомендуется регулярно выполнять REINDEX или VACUUM для активно изменяемых таблиц.
- По умолчанию включён fsync: при COMMIT данные сбрасываются на диск — консистентность при сбое, но ниже производительность записи. Повышение производительности: RAID с кеширующим контроллером с энергонезависимой кеш-памятью, UPS; при необходимости — отключение fsync (ответственность за консистентность — на оборудовании).
- ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ реализовано неэффективно; по возможности переписать запрос без него. Не допускается одновременно ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ и обращение к табличным частям в списке полей выборки.
- ПОДОБНО: квадратные скобки в шаблоне — специальные символы только в текстовом литерале, не в выражении.
