## Three-Question Challenge

### Q1 — Problem-Solution Fit

- **Why говорит:** «нет единого клиент-серверного pull-механизма; правила выборки в макетах на источнике; профили с безопасным паролем; стандартная загрузка на приёмнике».
- **Design адресует:** pull через `rgExchangeService.GetData` (D2/D5); правила КД 2.1 в макетах обработки `ргУниверсальныйОбменXML`; профили `ргНастройкиОбменаXML` + БСП-хранилище (D1); загрузка единым движком `Обработки.ргУниверсальныйОбменXML.Создать()` (D6).
- **Покрытие:** полное. Дельта D6 закрыла прежнюю претензию Q1 — контракт движка больше не висит на непроверенном внешнем `V8Exch82.epf`; происхождение движка зафиксировано как adopt штатной cf-обработки. Боль «стандартная загрузка» теперь имеет конкретный, совместимый с платформой адресат.

### Q2 — Optimality

- **Выбранный путь:** единая обработка расширения `ргУниверсальныйОбменXML` на базе штатной `УниверсальныйОбменДаннымиXML` (adopt); транспорт — SOAP-веб-сервис, XDTO `Structure` → `base64Binary` ZIP; серверная логика в одном общем модуле.
- **Альтернативы:**
  1. **Внешний `V8Exch82.epf` в макете движка** — `reopen-blocked: engine-origin`. Отклонена пользователем и зафиксирована в D6 (альт. A). Не переоткрываю: верифицированного факта кода против решения нет, штатная обработка в cf присутствует и совместима по определению.
  2. **Прямой runtime-вызов `Обработки.УниверсальныйОбменДаннымиXML` cf** — `reopen-blocked: engine-origin`. Зафиксирована в D6 (альт. B, отклонена частично: приёмник может не содержать типовую обработку, нет точки доработок). Не переоткрываю.
  3. **HTTP-сервис (REST), бинарный ZIP телом ответа** (axis `transport`, **не** закрыт) — нативная передача бинарника без +~33 % раздувания base64; бьёт ровно в задокументированный риск «Большие выгрузки — таймаут SOAP». Минус: нет готового типизированного `Structure` и WS-прокси с HTTP Basic «из коробки» — разбор параметров и аутентификация вручную.
- **Вердикт по Q2:** по оси движка решение оптимально и закрыто. По оси транспорта оптимальность по-прежнему **не доказана в постановке**: D5 обосновывает контракт SOAP (типизированный XDTO `Structure`, HTTP Basic на WS-прокси), но не сопоставляет его с HTTP-сервисом относительно собственного риска объёма. Рационал D5 валиден как осознанный trade-off (типобезопасность параметров v1 с примитивами), поэтому это документационный пробел, а не равноправная развилка, требующая пересмотра кода.

### Q3 — Fresh-Eye Approval

- **Согласовал бы:** с оговорками.
- **Причины «да»:**
  - D6 + § Engine Contract сделали центральный контракт движка трассируемым: способ создания `Обработки.ргУниверсальныйОбменXML.Создать()` зафиксирован, верификация публичного API штатной обработки вынесена в гейт-задачу S2.5, блокирующую BSL S2.6–S2.12. Это корректная последовательность (постановка не выдумывает имена методов).
  - § Manual Configuration даёт ревьюеру и конфигуратору единую таблицу объектов/реквизитов с префиксами — снимает прежнюю неопределённость по составу метаданных.
  - Риск «обновление cf меняет эталон adopt» уже отражён (фиксация базовой ревизии + регресс S2.accept).
- **Причины «с оговорками»:**
  - Транспортный trade-off и митигация объёма остались на уровне «документировать лимит» (gap 2/3 прошлого прогона не закрыты дельтой).
  - Adopt-from-cf переносит вспомогательную инфраструктуру движка (макеты `ОбщиеПроцедурыФункции`, `ПараметрыОбработчиков`), не перечисленную в § Manual Configuration / tasks — риск нерабочего движка при копировании только «макетов правил» (см. Gaps §3).

## Verdict

**CHALLENGE** — дельта D6 + Engine Contract + Manual Configuration закрывает прежнюю претензию по происхождению движка (axis `engine-origin` held, не переоткрывается); решение имплементируемо, но до apply постановку нужно дополнить тремя пунктами уровня уточнения (транспортный trade-off, митигация объёма, состав adopt-инфраструктуры). Архитектурная развилка не требуется.

## Gaps for design.md

1. **Транспорт — задокументировать как осознанный trade-off, не умолчание.** В `Implementation Options` (или новым решением D7) явно сопоставить SOAP-веб-сервис vs HTTP-сервис относительно риска «Большие выгрузки — таймаут»: назвать причину выбора SOAP (типизированный XDTO `Structure`, WS-прокси с HTTP Basic из коробки для v1 с примитивными параметрами) и принять +~33 % base64 как сознательную цену. Axis `transport` открыт, но D5 даёт валидный рационал — закрывается документированием, без смены кода. `implementation_invariant`.

2. **Митигация объёма.** Риск «таймаут SOAP» сопроводить проектным поведением (лимит размера выгрузки / поведение при превышении / увеличение таймаута публикации как именованная настройка), а не только строкой «документировать лимит». `implementation_invariant`.

3. **Состав adopt-инфраструктуры движка.** § Manual Configuration и S2.1/S2.2 перечисляют для обработки `ргУниверсальныйОбменXML` только «макеты правил (XML, КД 2.1)». Штатная `УниверсальныйОбменДаннымиXML` использует собственные вспомогательные макеты (`Templates/ОбщиеПроцедурыФункции`, `Templates/ПараметрыОбработчиков`) и экспортный контур ObjectModule (`ЗагрузитьПравилаОбмена`, `ВыполнитьЗагрузку`, `ОткрытьФайлЗагрузки`, `ПроизвестиЧтениеДанных` и выгрузочные методы). При adopt в расширение эти артефакты должны быть скопированы/адаптированы вместе с движком. Добавить в § Manual Configuration строку «вспомогательные макеты движка» и расширить охват S2.5: верифицировать не только имена методов, но и перечень обязательных макетов/зависимостей штатной обработки. `implementation_invariant`.
