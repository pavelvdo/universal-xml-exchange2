## ADDED Requirements

### Requirement: Ролевая модель профиля источник или приёмник

Система SHALL хранить в профиле обмена реквизит `Источник` (булево). Профиль с `Источник = Истина` SHALL использоваться на базе-источнике (сервер); профиль с `Источник = Ложь` — на базе-приёмнике (клиент).

#### Scenario: Профиль источника

- **WHEN** пользователь создаёт профиль с `Источник = Истина`
- **THEN** система отображает и сохраняет реквизиты источника: префиксы, правила в хранилище, параметры, статус; реквизиты подключения к удалённому сервису не обязательны

#### Scenario: Профиль приёмника

- **WHEN** пользователь создаёт профиль с `Источник = Ложь`
- **THEN** система отображает и сохраняет адрес сервиса, логин и пару префиксов для идентификации сеанса на источнике; правила и статус сеанса на этом профиле не используются для выгрузки

### Requirement: Префиксы идентификации сеанса

Профиль SHALL содержать `Префикс` (строка, 9) и `ПрефиксБазыПриемника` (строка, 9) независимо от роли. На профиле-источнике пара идентифицирует сеанс для веб-сервиса; на профиле-приёмнике — те же значения передаются в `GetData`/`ConfirmDataReceipt` для поиска профиля на источнике.

#### Scenario: Настройка префиксов

- **WHEN** пользователь заполняет префикс и префикс базы-приёмника на профиле-источнике
- **THEN** система сохраняет значения для использования в операциях веб-сервиса

### Requirement: Статус жизненного цикла обмена

Профиль-источник SHALL содержать реквизит `Статус` типа перечисление `рг_СтатусыОбмена` со значениями: Новое, Выполняется, Выполнено, Ошибка.

#### Scenario: Ручная инициация обмена

- **WHEN** пользователь на профиле-источнике загрузил правила, заполнил параметры и установил статус «Новое», затем записал элемент
- **THEN** профиль готов к вызову `GetData`

#### Scenario: Завершение обмена

- **WHEN** источник получил успешный `ConfirmDataReceipt` для профиля в статусе «Выполняется»
- **THEN** статус профиля становится «Выполнено»

### Requirement: Правила обмена в хранилище профиля

Профиль-источник SHALL хранить файл правил обмена (XML) в реквизите `ПравилаОбменаДанными` типа «Хранилище значения». Система MUST NOT требовать имя макета конфигурации как единственный способ задания правил.

#### Scenario: Загрузка правил пользователем

- **WHEN** пользователь загружает XML-файл правил в профиль-источник
- **THEN** содержимое сохраняется в `ПравилаОбменаДанными`

#### Scenario: Проверка при записи источника

- **WHEN** пользователь записывает профиль с `Источник = Истина` без загруженных правил
- **THEN** система отклоняет запись с сообщением о необходимости правил обмена

## MODIFIED Requirements

### Requirement: Профиль обмена хранит параметры подключения

Система SHALL хранить профили обмена в справочнике `рг_НастройкиОбменаXML`. Для профиля-приёмника MUST сохраняться: адрес публикации веб-сервиса (URL), имя пользователя для подключения к базе-источнику. Для профиля-источника MUST сохраняться префиксы идентификации сеанса; имя макета правил на конфигурации не является обязательным реквизитом.

#### Scenario: Создание профиля приёмника

- **WHEN** пользователь создаёт профиль с `Источник = Ложь` и заполняет URL и логин
- **THEN** система сохраняет профиль для инициации обмена с удалённым источником

#### Scenario: Создание профиля источника

- **WHEN** пользователь создаёт профиль с `Источник = Истина`, префиксами и правилами в хранилище
- **THEN** система сохраняет профиль для публикации веб-сервиса и выгрузки по запросу

### Requirement: Табличная часть произвольных параметров

Справочник SHALL содержать табличную часть параметров обмена в формате «Имя» — «Значение» для профиля-источника. Допустимые типы значений: строка, число, дата, булево. Параметры SHALL подставляться в правила обмена перед выгрузкой на источнике.

#### Scenario: Настройка параметров на источнике

- **WHEN** пользователь добавляет строки в табличную часть параметров профиля-источника
- **THEN** система сохраняет параметры и использует их при выгрузке по `GetData`
