## Критерий 3. Исполнимость задач — GAP

Сильные стороны: задачи атомарны, помечены «ручное конфигурирование» vs BSL, привязаны к конкретным файлам (`Ext/ObjectModule.bsl`, `Ext/Module.bsl`, `Ext/Form/Module.bsl`) и процедурам. Приёмка по срезам с Primary/optional — корректна.

Пробелы:

- **G2 (исполнимость S2 завязана на неподтверждённый внешний контракт).** Весь срез S2 (выгрузка S2.6/S2.7, загрузка S2.12) строится на публичном API движка V8Exch82.epf — именах параметров обмена и методах выгрузки/загрузки. Они **не верифицированы** (OQ1) и отложены на S2.9 при apply. Это не дефект планирования, но честный риск: если эталонный epf недоступен или его API отличается от ожидаемого, S2.6/S2.12 нельзя реализовать без возврата к постановке. Плейсхолдер из S2.2 закрывает только метаданные макета, не логику движка. → writer войдёт в S2.6 без гарантированного контракта.
- **S2.6 — имя экспортной функции выгрузки не задано.** S2.8 («тонкая обёртка над функцией выгрузки общего модуля») ссылается на неё, но сигнатура/имя не зафиксированы. Внутри одного writer-контекста разрешимо, но при раздельной реализации S2.6/S2.8 — риск рассинхрона.

## Критерий 4. Порядок зависимостей — GAP

- **G3 (инверсия порядка S2.9 ↔ S2.6).** S2.9 верифицирует контракт движка V8Exch82, но в нумерации стоит **после** S2.6/S2.7/S2.8, которые этот контракт уже используют. Верификация контракта должна предшествовать коду, который на него опирается. Нужно: вынести S2.9 перед S2.6 (или явно пометить S2.9 как gating-задачу, блокирующую S2.6–S2.12).
- **S1.5 → S1.8 (мягкая, управляемая).** S1.5 (ручное создание регламентного задания со ссылкой на метод `ВыполнитьОбменПоПрофилю`) стоит перед S1.8 (создание самого метода в BSL). В Конфигураторе имя метода задаётся строкой и не требует существования метода на момент создания задания, поэтому жёсткой блокировки нет. Достаточно явной пометки порядка (уже отмечена «зависит от S1.8»). Не блокер, но при ручной конфигурации стоит предупредить пользователя, что метод появится позже.
- S2.14 (дополнить `ВыполнитьОбменПоПрофилю` полным сеансом) корректно зависит от S1.8 и S2.10–S2.13.

## Критерий 5. Ясность контрактов — PASS с оговорками

Ясно:
- `GetData(ИмяМакетаПравил: String, Параметры: Structure) → base64Binary` (D2, D5; XDTO-пакет `http://v8.1c.ru/8.1/data/core`).
- Аутентификация — HTTP Basic на уровне WS-прокси, не в теле GetData (D5).
- Пароль — БСП `ЗаписатьДанныеВБезопасноеХранилище`/`ПрочитатьДанныеИзБезопасногоХранилища`, владелец — ссылка элемента, ключ `"Пароль"`; передача через `ДополнительныеСвойства` (S1.7, S1.10).
- Регламент — API `РегламентныеЗаданияСервер`, идентификатор в реквизит `обменXML_ИдентификаторРегламентногоЗадания` (S1.9).

Оговорки:
- **Имя реквизита формы для ввода пароля** не зафиксировано (S1.3 «нехранимое поле ввода пароля», S1.10 «при вводе нового значения») — нужна координация имени между ручной формой и BSL.
- Контракт V8Exch82 — см. G2 (OQ1).

## Критерий 6. User Task Contract / ручная конфигурация — GAP

Объём ручной конфигурации значителен (весь состав метаданных создаётся в Конфигураторе: справочник + реквизиты + ТЧ, две формы, общий модуль, регламентное задание, обработка + макеты, веб-сервис). Детали имён/типов в tasks.md есть (S1.1–S1.5, S2.1–S2.4) и в основном достаточны. Пробелы:

- **G1 (права роли не покрыты ни одной задачей).** `обменXML_ОсновнаяРоль` — пустая и является `DefaultRoles` расширения (подтверждено в `src/.../Roles/обменXML_ОсновнаяРоль.xml` и `Configuration.xml`). Ни одна задача S1.*/S2.* не назначает права на новые объекты (справочник, обработка, веб-сервис, регламентное задание). Без этого объекты будут недоступны пользователю/сеансу — гарантированный возврат на этапе ручной приёмки. Нужна отдельная задача ручной конфигурации: «выдать права роли `обменXML_ОсновнаяРоль` на новые объекты».
- **G4 (нет консолидированной таблицы Manual Configuration в design).** Детали ручной конфигурации рассыпаны по tasks; в design § D4/Behavior Contract единого перечня «реквизиты/ТЧ/элементы формы/параметры веб-сервиса» нет. Это совпадает с замечанием verify Layer 5.1 (manual-config-incomplete). Дополнительно недоопределены:
  - длина строкового квалификатора составного типа `Значение` ТЧ `обменXML_Параметры` (S1.2);
  - параметры самого веб-сервиса `rgExchangeService` помимо операции: пространство имён сервиса (namespace URI), имя файла/набор XDTO-пакетов публикации (S2.4 задаёт только операцию и типы параметров);
  - механизм редактирования расписания в форме (S1.3 «команда редактирования расписания») — предполагается стандартный диалог расписания БСП, но это не зафиксировано.

Эти пункты по отдельности мелкие, но в сумме — источник нескольких уточняющих возвратов при ручной сборке. Консолидированная таблица «Manual Configuration» в design закрыла бы и G1, и G4.
