# Task Readiness (Layer 5) — universal-xml-exchange

## Вердикт

**GAP** — apply запускать нельзя. Артефакты (`tasks.md`, `design.md`) описывают объекты и реквизиты с именами и типами, которых **нет** в фактической выгрузке расширения. BSL-задачи в таком виде заставят writer обращаться к несуществующим символам метаданных, а задача S1.11 предписывает действие в Конфигураторе, которое платформа не поддерживает.

Три блокера:

1. **S1.11 — невозможное действие в Конфигураторе** (права роли на регламентное задание).
2. **Сквозной дрейф имён метаданных** между артефактами и выгрузкой (объекты, реквизиты, ТЧ, роль).
3. **Роль не даёт прав на создание/редактирование профиля** — противоречит Primary acceptance S1.

## Критерий 5.1 — Manual Configuration Sufficiency

Таблица `design.md § Manual Configuration` и метаданные в `tasks.md` (S1.1–S1.5, S1.11, S2.1–S2.5) **не соответствуют** фактической выгрузке `src/ЗУП/cfe/рг_УниверсальныйОбменXML/`. Ручное конфигурирование, судя по выгрузке, уже частично выполнено, но под **другими** именами и с другим набором реквизитов. Артефакты не догоняют выгрузку.

D4 («префикс реквизитов `обменXML_`») фактом выгрузки **опровергнут**: реквизиты идут без префикса, объекты — с префиксом `рг_`.

### Дополнительные расхождения контракта (не только имена)

- **`ВыполнитьОбменПоПрофилю` без параметра.** В выгрузке процедура объявлена как `ВыполнитьОбменПоПрофилю() Экспорт` (модуль `рг_УниверсальныйОбменXMLСервер`), а регламентное задание вызывает её тем же методом без параметров. Задачи S1.8 и S2.14 предписывают сигнатуру `ВыполнитьОбменПоПрофилю(Профиль)`. Регламентное задание не передаёт `Профиль` — нужен либо параметр задания, либо итерация по профилям внутри метода. Контракт точки входа в артефактах и в коде расходится.
- **Модель хранения расписания.** В выгрузке у справочника есть реквизит `Расписание` (ХранилищеЗначения) — расписание хранится в самом профиле. Design D3 описывает расписание только на регламентном задании (диалог БСП). Либо в коде дублируются два источника расписания, либо design устарел. Это надо зафиксировать до BSL S1.9/S1.10.
- **Идентификатор задания — UUID, а не Строка 36.** S1.9 предписывает «сохранять идентификатор в реквизит `…ИдентификаторРегламентногоЗадания` (Строка 36)». В выгрузке реквизит имеет тип `UUID`. Код должен писать `УникальныйИдентификатор`, а не строку.

## Критический разбор: задача S1.11 (права роли на регламентное задание)

**Подтверждено пользователем и кодом.** S1.11 требует «выдать роли права на … регламентное задание `ргВыполнениеОбменаXML`». В платформе 1С **у объектов «Регламентное задание» нет прав в редакторе ролей** — этот тип метаданных вообще не присутствует в дереве прав роли. Действие в Конфигураторе невыполнимо.

Доказательства:
- Grep `ScheduledJob\.` по `src/ЗУП/cf/Roles/**` — **0 совпадений**: ни одна типовая роль ЗУП не содержит прав на регламентные задания (их там и не может быть).
- `Roles/рг_ИспользованиеОбменаXML/Ext/Rights.xml` содержит узлы только для `Catalog.рг_НастройкиОбменаXML` и `Configuration.рг_УниверсальныйОбменXML`. Узла для `ScheduledJob.*` нет — Конфигуратор его не создаёт.

**Корень.** Регламентное задание исполняет свой метод под сеансом регл. заданий независимо от ролей пользователя; доступ к данным внутри метода регулируется не правами на задание, а правами на используемые объекты и/или привилегированным режимом. Эталонный паттерн в типовой — `РегламентныеЗаданияСервер` + `УстановитьПривилегированныйРежим(Истина)` в коде, исполняемом заданием (как в обработчиках рассылок отчётов).

**Что менять (рекомендация, не предписание реализации):**
- Из S1.11 убрать часть «права на регламентное задание». Оставить только права на справочник `рг_НастройкиОбменаXML`.
- Зафиксировать в `design.md`: регламентному заданию права роли не нужны; метод `ВыполнитьОбменПоПрофилю` обращается к профилю, безопасному хранилищу и сеансу обмена в привилегированном режиме (паттерн `РегламентныеЗаданияСервер` + `УстановитьПривилегированныйРежим`).
- Соответственно скорректировать S1.9 (управление заданием через `РегламентныеЗаданияСервер`) и серверный метод.

## Критерий 3 — права роли недостаточны для Primary acceptance S1

В `Rights.xml` для `Catalog.рг_НастройкиОбменаXML` все интерактивные права записи выставлены в `false`: `Edit=false`, `InteractiveInsert=false`, `Delete=false`, `InteractiveDelete=false` и т.д. (Read остаётся по умолчанию). Роль даёт **только чтение** профилей.

Primary acceptance S1 — «администратор **создаёт/редактирует** профиль». С текущими правами носитель роли `рг_ИспользованиеОбменаXML` не сможет создать или изменить профиль → приёмка S1 невыполнима под этой ролью. Это самостоятельный блокер: задачи на роль (S1.11, S2.5) должны выдать `Edit`/`InteractiveInsert` (и при необходимости `Delete`) на справочник.
