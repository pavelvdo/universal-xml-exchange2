# Design Challenge — universal-xml-exchange

## Адверсариальная установка

Прочитаны как первичные источники: `proposal.md` (`## Why`, `## What Changes`), `design.md` (целиком), `specs/exchange-{settings,web-service,export,import}/spec.md`, `tasks.md`, `project.md`. `design.md` рассмотрен как чужая работа, без опоры на прошлые `reports/architecture-*.md`. Для проверки оптимальности дополнительно прочитан состав базовой конфигурации `src/ЗУП/cf/DataProcessors/` (верифицированный факт кода — см. Q2). KB-совпадений нет, секция «Решения verify» в design отсутствует, `closed_decisions` пуст — реоткрытие axis не ограничено.

## Three-Question Challenge

### Q1 — Problem-Solution Fit

`## Why` формулирует пять болей. Сопоставление с design:

- **Why: «нет единого клиент-серверного pull-механизма (приёмник инициирует у источника)»** → design адресует через `rgExchangeService.GetData`, приёмник инициирует сеанс (D2, Behavior Contract «Сеанс обмена»). Покрыто.
- **Why: «правила выборки/фильтрации не стандартизированы → макеты правил на источнике»** → макеты правил КД 2.1 в обработке `ргУниверсальныйОбменXML`, активация всех ПВД (spec exchange-export). Покрыто.
- **Why: «настройки и параметры не собраны в профиле»** → справочник `ргНастройкиОбменаXML` + ТЧ параметров (spec exchange-settings). Покрыто.
- **Why: «пароли нельзя хранить открытым текстом»** → БСП безопасное хранилище, владелец — ссылка элемента (D1, S1.7). Покрыто.
- **Why: «стандартная загрузка на приёмнике»** → загрузка движком из макета `ДвижокОбмена` (spec exchange-import). Покрыто по namespace, но способ реализации движка спорен — см. Q2.

**Покрытие:** полное на уровне «что решается». Все пять болей Why имеют адресата в design. Проблема Q1 не в пропуске боли, а в том, что один из контрактов (движок загрузки) держится на непроверенном допущении (OQ1) и на самовнесённом риске (Q2, альтернатива B).

### Q2 — Optimality

- **Выбранный путь:** SOAP-веб-сервис с XDTO `Structure` → `base64Binary` ZIP; движок — внешний `V8Exch82.epf`, упакованный в макет `ДвижокОбмена` обработки; вся серверная логика в одном общем модуле.
- **`## Implementation Options` сравнивает только** монолит vs разделение модулей (организация кода). Транспорт и происхождение движка приняты как безальтернативные — это и есть слабое место оптимальности.

**Альтернативы, не упомянутые в design:**

1. **HTTP-сервис (REST) вместо SOAP-веб-сервиса.**
   Платформа 1С даёт HTTP-сервисы, где бинарный ответ (ZIP) отдаётся телом запроса напрямую, без обёртки в `base64Binary`. SOAP+base64 раздувает трафик примерно на +33 % — и это бьёт ровно в задокументированный риск «Большие выгрузки — таймаут SOAP». Параметры в HTTP-сервисе передаются query/JSON-телом (D2 отклонил JSON, но только в контексте уже выбранного SOAP).
   **Плюсы:** меньше трафик, нативная передача бинарника, проще стриминг крупных пакетов, легче публикация.
   **Минусы:** нет готового типизированного контракта `Structure` и WS-прокси с HTTP Basic «из коробки» — парсинг параметров и аутентификацию надо описать вручную.
   **Почему важно:** design выбрал SOAP не показав, что он лучше HTTP-сервиса именно по заявленному риску объёма. Это не отклонённая, а нерассмотренная развилка.

2. **Вызов штатной обработки `Обработки.УниверсальныйОбменДаннымиXML` базы вместо упаковки `V8Exch82.epf` в макет.**
   **Верифицированный факт кода:** `src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML.xml` — ЗУП КОРП уже содержит штатную обработку универсального обмена КД 2.1 (внутриконфигурационный аналог `V8Exch82.epf`). Её можно вызвать через `Обработки.УниверсальныйОбменДаннымиXML.Создать()` без какого-либо файла и без бинарника в макете.
   **Это напрямую снимает риск №1 design** («V8Exch82.epf несовместим с версией платформы»): обработка из состава cf по определению совместима с текущей платформой и версией ЗУП. Обоснование макета в Design Rationale §4 — «чтобы приёмник не зависел от файла на диске» — внутриконфигурационной обработкой выполняется даже лучше (нет ни файла, ни бинарника в макете, ни плейсхолдера-заглушки из S2.2).
   Для стратегической цели «перенос на любую конфигурацию на БСП» вызов штатной `УниверсальныйОбменДаннымиXML` (стандартный объект БСП-конфигураций) переносимее, чем привязка к конкретной версии `V8Exch82.epf`.
   **Минусы / почему не строго превосходит:** публичный API штатной обработки (имена методов выгрузки/загрузки, инициализация правил) может отличаться от `V8Exch82.epf`; команды иногда осознанно фиксируют точную версию движка, чтобы не зависеть от обновлений типовой. Но этот trade-off в design не назван — там бинарный epf принят как единственный путь.

3. **(Кратко) Регистрация движка через БСП «Дополнительные отчёты и обработки»** вместо макета — обновляемость движка без переустановки расширения. Слабее альтернативы 2 (лишний админ-шаг, отдельное хранилище) и здесь не развивается.

**Вердикт по Q2:** не доказана оптимальность. Две реальные альтернативы (транспорт; происхождение движка) не рассмотрены, при этом альтернатива B нейтрализует самый крупный заявленный риск. Ни одна не доказана строго превосходящей (нужна сверка API штатной обработки и поведения при крупных пакетах), поэтому это CHALLENGE, не REJECT.

### Q3 — Fresh-Eye Approval

- **Согласовал бы:** с оговорками.
- **Причины «да»:** связный, стандартный для 1С контур интеграции; БСП-механизмы (пароль, регламент) переиспользованы корректно; декомпозиция на 2 среза и матрица приёмки прослеживаются до spec; контракт GetData и аутентификация (D5, HTTP Basic на публикации) разумны.
- **Причины «с оговорками / нет»:**
  - Ядро export/import опирается на `V8Exch82.epf`, публичный контракт которого ещё не верифицирован (OQ1, S2.9 откладывает на apply), и сам epf может физически отсутствовать (S2.2 предусматривает плейсхолдер-заглушку). При наличии в cf штатной `УниверсальныйОбменДаннымиXML` (Q2-B) держать центральный риск на внешнем бинарнике — неоправданно на этапе постановки.
  - Транспорт (SOAP+base64) выбран без сопоставления с риском объёма, для которого сам же design завёл строку риска без проектной митигации (только «документировать лимит»).

## Verdict

**CHALLENGE** — design полностью покрывает проблему из `## Why` (Q1), но не доказывает оптимальность транспорта и происхождения движка, причём штатная обработка `УниверсальныйОбменДаннымиXML` из cf снимает заявленный риск №1; решение имплементируемо, но до apply постановку нужно дополнить обоснованиями.

## Gaps for design.md (обязательно при CHALLENGE)

1. **Происхождение движка.** Зафиксировать в `Decisions` (новое решение или дополнение к §4 Design Rationale): почему движок упаковывается как `V8Exch82.epf` в макет, когда в `src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML.xml` есть штатная обработка универсального обмена. Либо обосновать привязку к фиксированной версии epf (контроль версии/совместимости правил), либо переключиться на вызов штатной обработки и убрать риск №1. Сверку публичного API штатной обработки добавить в S2.9 / OQ1.
2. **Транспорт.** Добавить в `Implementation Options` развилку «SOAP-веб-сервис vs HTTP-сервис» с явным обоснованием выбора относительно риска «Большие выгрузки — таймаут» (раздувание base64 ~+33 %, передача бинарника). Если SOAP оставлен — назвать причину (типизированный `Structure`, WS-прокси с HTTP Basic) как осознанный trade-off, а не умолчание.
3. **Митигация объёма.** Риск «таймаут SOAP» сопроводить проектным решением (лимит размера выгрузки / поведение при превышении), а не только строкой «документировать лимит».

## Architectural alternatives (равноправные пути по коду/поведению)

### Происхождение движка универсального обмена

**A. Внешний `V8Exch82.epf` в макете `ДвижокОбмена` (текущий выбор):** движок извлекается из макета во временный файл, `ВнешниеОбработки.Создать`. Trade-off: фиксированная версия движка, но самовнесённый риск платформенной несовместимости и зависимость от наличия файла (плейсхолдер в S2.2).
**B. Штатная `Обработки.УниверсальныйОбменДаннымиXML.Создать()` из состава cf:** без файла и бинарника, гарантированная совместимость с платформой/ЗУП, переносимее на БСП-конфигурации. Trade-off: зависимость от API типовой обработки и её возможных изменений при обновлении конфигурации; требуется сверка публичного контракта.

### Транспорт сетевого контракта

**A. SOAP-веб-сервис + `base64Binary` ZIP (текущий выбор):** типизированный XDTO `Structure`, WS-прокси с HTTP Basic. Trade-off: +~33 % трафика, обострение риска таймаута на крупных выгрузках.
**B. HTTP-сервис (REST), бинарный ZIP телом ответа:** меньше трафик, нативная передача бинарника, проще крупные пакеты. Trade-off: ручной разбор параметров и аутентификации, нет готового контракта `Structure`.
