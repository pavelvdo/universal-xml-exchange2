# Стиль пользовательских выводов opsx-команд

Единый источник правил оформления сообщений, которые skill-ы семейства `/opsx:*` показывают пользователю в чате: брифов делегирования, карточек приёмки, отчётов verify/status, сообщений об архивации/синхронизации.

Документ применяется **ко всему, что выводится пользователю** (чат, AskQuestion-блоки, Markdown в `reports/`). На **содержимое артефактов** в `openspec/changes/<id>/` (proposal.md, design.md, tasks.md, spec deltas) действуют только разделы §1 «Три слоя» и §3 «Запрет внутренних ID в пользовательских полях» — остальное там задаётся schema OpenSpec.

## 1. Три слоя (главное правило)

В любой пользовательский текст входят три принципиально разных слоя:

| Слой | Примеры | Формат |
|------|---------|--------|
| **UX** — что видит пользователь | «Повтор согласования», колонка «На новом цикле», кнопка «Отправить» | Обычный текст, UI-надписи в «ёлочках» |
| **Код/метаданные** | `pav_СохранятьУчастниковПриПовторе`, `ФормаИзменениеПараметров`, `ОбработатьВизыПриНачалеИтерации` | Backticks |
| **Процесс OpenSpec** | S4.T1, D7, Фаза 8, R2, I3, tasks.md §12.9 | Backticks для ID, только в полях «План» / «Ссылки» |

**Правило:** в одном поле/абзаце — **один** слой. Если нужны все три — три разных строки/блока, а не одно предложение.

Плохо (три слоя в одной фразе):

```
Симптом: при pav_СохранятьУчастниковПриПовторе = Истина (D7, S3.T1) пользователь видит только отказавшихся в «Повторе согласования».
```

Хорошо (разделено):

```
Симптом: пользователь видит в «Повторе согласования» только отказавшихся; ожидается — всех участников.
Технический контекст: активен режим `pav_СохранятьУчастниковПриПовторе = Истина`.
Ссылки: инвариант D7 (см. design.md §«RCA…»); охват — `Срез S3: «Название из tasks.md»` (формат §10).
```

## 2. Типографика и языки

- UI-надписи (то, что видно на форме): **«ёлочки»**, обычный шрифт. Пример: «На новом цикле», «Направлять», «Отправить на повторное согласование».
- Идентификаторы кода/метаданных: `backticks`. Пример: `pav_РезультатСохранен`, `ФормаИзменениеПараметров.Ext.Form.Module.bsl`.
- Команды opsx и slug-и агентов: `backticks`, как есть. Пример: `/opsx:apply`, `onec-code-explorer`.
- Внутренние ID процесса: `backticks`. Пример: `S4.T1`, `D7`, `R2`.
- Не ставить подряд 3+ разнотипных термина без хотя бы одного связного русского слова-разделителя. Плохо: `` `pav_РежимСохраненияФормы` `ФормаИзменениеПараметров` `&Вместо` `ТекстЗапросаТаблицаИсполнителей` `` — цепочка без контекста. Хорошо: «перехват `&Вместо("ТекстЗапросаТаблицаИсполнителей")` в модуле `ФормаИзменениеПараметров` срабатывает после установки флага `pav_РежимСохраненияФормы`».
- Язык: русский по умолчанию (совпадает с user rule). Английские slug-и команд/агентов — как есть, без русификации.

## 2.5. Человеческий слой: каноны разговорного вывода

Все финальные сообщения оркестратора в чат (особенно `verify NO-GO` и `apply handoff`) подчиняются правилу **человеческого языка**. Разработчик 1С должен принять решение, прочитав только чат.

1. **Чат самодостаточен.** Если нужно идти в `reports/*.md` — то **один раз**, в конце, за деталями (полные карточки, источники агентов, технический аудит). Ссылки «Подробнее» на каждом пункте — анти-паттерн.
2. **Мини-абзацы, не телеграф.** Каждый блокер — это **2–3 предложения**: что не так → почему это блокирует работу → что нужно дописать. Запрещены списки вида «1–4 пункта одной строкой каждый».
3. **Язык бизнеса и архитектуры.** Имена типа `Implementation Options`, `Behavior Contract п.7`, `Spec ↔ Contract` — это словарь движка, не разработчика. В чате: «в требованиях», «в плане задач», «в постановке нет вариантов реализации».
4. **Имена кода 1С — только когда критичны.** `&Вместо("ПолучитьШаблоны")`, пути `cfe/ДО21_Павлик/Forms/...` — живут внутри файла отчёта. В чате: «подменили типовую функцию подбора», «расширение Павлика». Если процедура **критична для понимания развилки** — оставить в backticks с расшифровкой в скобках (`pav_ПолучитьШаблоны` — наша подмена в модуле объекта).
5. **Один вопрос за раз и не висим в воздухе.** Если от заказчика нужен ответ — задаём **один** конкретный вопрос. Если ответов не нужно, но нужны правки постановки — оркестратор **сам** делает repair внутри verify (см. §2.6 Chat Surface Contract), **без** user-facing extend и без «Подтвердить?».

**Эталон вывода (verify NO-GO):**

```markdown
do2-pavlik-vizy-shablony-soglasovaniya-podbor — до старта разработки нужно решить несколько вопросов по логике подбора шаблонов.

Первая часть (S1) готова: мы успешно подменяем типовой список шаблонов, добавляя туда шаблоны согласования. Форму пока не трогали. Но для второй части (S2) — подбора шаблонов при смене документа — в постановке не хватает деталей. Если начать писать код сейчас, придётся угадывать.

Что нужно доработать в постановке:

**1. Как работает автоподстановка первого согласования.**
В требованиях есть противоречие: в одном месте сказано «подставить первое», в другом — «оставить пустым». Нужно выбрать одно чёткое правило поведения.

**2. Что показывать при предпросмотре не-визовых шаблонов.**
Мы не меняем саму форму. Но для шаблонов исполнения/ознакомления виз нет — нужно зафиксировать, какие колонки заполняем и что скрываем.

**3. Расчёт исполнителей для предпросмотра.**
В плане задач пропущен важный шаг: перед вызовом `ЗаполнитьПoШаблону` нужно передать предметы процесса. Без этого предпросмотр может выдать пустой список.

**Что решить: где собирать список шаблонов при смене документа**

Сейчас вся логика подбора в модуле объекта, а нам нужно добавить заполнение списка выбора при смене документа на форме — и не зафиксировано, где это делать.

- **A. В модуле объекта.** Делаем новую экспортную функцию. Зато: логика отбора в одном месте. Но: новый публичный метод.
- **B. В модуле формы.** Перехватываем смену документа на форме. Зато: не трогаем модуль объекта. Но: правила отбора дублируются.

Оба варианта решают задачу пользователя. Напишите, какой вариант ближе (например: «A, автоподстановку — только если есть согласование»), — зафиксирую в постановке, и можно начинать.

Полный отчёт: openspec/changes/.../verification-2026-05-28.md
```

Эталон занимает ~25 строк — это разрешённый объём **информативного NO-GO** по `chat-output-budget.mdc` §1 («до ~25–30 строк»); при конфликте побеждает лимит (иерархия «лимит > шаблон > пример»).

## 2.6. Chat Surface Contract (сквозной слой для всех `/opsx:*`)

**SSOT для чата.** Все skill-ы семейства `/opsx:*` обязаны следовать этому контракту. Детали verify — в `openspec-verify-change/SKILL.md`; handoff-шаблоны — §5 ниже.

### Принцип

Чат — **единственная** поверхность, по которой человек принимает решение. Файлы (`reports/**`, `debug.md`) — для агента, аудита и углублённых уточнений. Если для решения нужно открыть файл — сообщение в чате **провалено**.

### Правило 1 — Самодостаточность

Человек может ответить или действовать, **не открывая файлов**. Ссылка на отчёт — один якорь в конце, не замена содержания.

### Правило 2 — Команды: internal vs user-action

| Класс | Примеры | В чате |
|-------|---------|--------|
| **Internal-оркестрация** | `/opsx:extend --from-verify <path>`, «запускаю скилл…», «Entry Protocol», пути к дочерним отчётам как next step | **НИКОГДА** |
| **User-action** | `/opsx:apply <name>`, `/opsx:archive <name>`, `/opsx:verify <name>` | **МОЖНО** — одной строкой, без флагов и путей, только когда это **реальный** следующий шаг человека |

### Правило 3 — Один сигнал ответственности

В сообщении ровно **один** из:

| Сигнал | Когда | Форма в чате |
|--------|-------|--------------|
| **user-action next step** | GO verify, new, extend, apply final, status | `**Следующий шаг:** \`/opsx:<command> <name>\`` — одна команда, без флагов и путей |
| **ответ в чате** | verify decision (A/B, supersedes), terminal fail repair | «ответьте в чате …» — без internal-команд |
| **подтвердите** | explore / user-extend с реальной развилкой scope | «Подтвердить?» текстом |

**Смешение запрещено** (например «ничего не требуется» + «Подтвердить?»). **Anti-pattern:** GO verify без слота `**Следующий шаг:**` с явной user-action командой.

### Правило 4 — Типографика скан-читаемости

- Первая строка — суть/вердикт, отдельным абзацем.
- Bold/backticks — **точечно**: максимум 1 имя процедуры/модуля как анкор «где смотреть».
- Предложения ≤25 слов; без цепочек тире; пути файлов не инлайнить в прозу.
- Списки — нумерованные/буллеты; не более 5–7 пунктов.
- **Без таблиц в чате** (таблицы — в файл).

### Правило 5 — Handoff на языке эффекта

Завершение команды описывает **что изменилось для дела** («постановка дополнена, можно проверять»), а не перечень файлов. Список артефактов — в `debug.md` / отчёт.

### Правило 6 — Бриф по адресу

Бриф (Sync Card, §5.1) — точка синхронизации с пользователем; глубина **адаптивная** (B0–B3):

- **B3** (карточка explore, слоты из `templates/brief-card.md`) — только `/opsx:explore`.
- **extend** user-path: **B1** по умолчанию; **B2** при drift/decision/неоднозначности; **никогда B3**.
- **new**: **B1** при свободном тексте; **B0** при свежем `## Постановка ЗНИ` в чате/handoff.
- **repair-from-verify** и детерминированный авторемонт — **B0** (0 строк в чат).

### Self-check (HARD, перед каждым финальным сообщением)

1. Нет internal-команд/путей-как-next-step?
2. Один сигнал ответственности?
3. Можно ответить, не открывая файл?
4. Типографика: первая строка — суть; нет перегруза bold; нет инлайн-путей?
5. Handoff на языке эффекта, не лог файлов?
6. GO verify содержит `**Следующий шаг:**` + корректную команду (`/opsx:apply` при `pre-apply`, `/opsx:archive` при `post-apply`)?

Зеркало HALT — [chat-output-budget.mdc](../rules/chat-output-budget.mdc) §7 и §1 (verify repair loop — 0 промежуточных сообщений).

### Terminal outcomes (сессия завершена для человека)

| Команда | Terminal в чате |
|---------|-----------------|
| verify | «можно apply» + **Следующий шаг** (`/opsx:apply` \| `/opsx:archive` по `verify_mode`) \| decision + ответ в чате \| terminal fail + чат или `/opsx:extend` |
| extend (user) | «постановка дополнена» + опц. hint `/opsx:verify <name>` |
| new | «ЗНИ создано» + **один** next step |
| apply | handoff среза / «срез готов» |
| explore | блок `## Постановка ЗНИ` или один вопрос |
| archive | одна строка «заархивировано» |

### Карта repair vs decision (для verify; детерминированная)

Источник — `layer_status` и коды алертов из отчёта verify:

- **Repair (чинит verify сам, без чата):** Layer 1 AUTOFIXED; Layer 2/3/5 WARNING без развилки; алерты `*-orphan-*`, `*-drift`, `metadata-stale-*`, `task-missing-*`, отсутствие пометки «Конфигуратор», контракт-допущение не записан в design; **`implementation_invariant`** — после Post-challenge classifier; **slice acceptance repair:** `primary-acceptance-missing`, `accept-bullets-missing-scenario`, `acceptance-simplicity-overload`, `slice-not-vertical`, `slice-foundation-with-gate` (merge via architect in extend §6c); **`user-task-contract-violation`** (extend §6d).
- **Decision (STOP, вопрос человеку):** Layer 4 CHALLENGE/REJECT с равноправными альтернативами **после classifier** (не reopen без verified fact); `scope-violation`; FAIL, требующий продуктового выбора A/B; противоречие Why ↔ plan; **`supersedes`** — одна прозаичная эскалация.
- **Assumption defer (`GO-saturated`):** load-bearing допущение, duplicate challenge по closed axis при `decision_round >= max` — GO в чат с defer в S1.accept; **не** новая A/B.
- **Смешанный отчёт:** decision имеет **приоритет** → сначала вопрос человеку; repair после ответа (user-extend `--from-verify` после decision).

**Agent-only keys (не в чат):** `closed_decisions`, `decision_id`, `decision_round`, `verify_depth`, `reopen-blocked`, `supersedes`, `CHALLENGE-saturated`. Пользователь отвечает свободным текстом; для **verify decision** — «A»/«B» по текущей развилке; для **entry-брифов** B2/B3 с **Варианты** — номер варианта (см. `templates/brief-card.md`).

### Acceptance: isolated chat (регрессия UX)

Проверка **обязательно** в **новом чате** без истории repair/extend/verify. Единственное сообщение — команда under test.

| ID | Сценарий | Ожидание |
|----|----------|----------|
| A | verify, repair-only | 1 сообщение; GO; `**Следующий шаг:**` + `/opsx:apply`; нет «Подтвердить?», internal-команд, списка файлов |
| B | verify, decision | 1 блок вопроса; `**Следующий шаг:**` + «ответьте»; END TURN |
| C | verify, silent_ok | ≤6 строк; статус прежний; `**Следующий шаг:**` + команда по `verify_mode` |
| D | new | T-CONFIRM, одна команда next: verify |
| E | apply handoff | язык эффекта; user-action next step |
| F | explore | бриф B3 (§5.1); финал — блок или один вопрос |
| F2 | extend | бриф B1: слот **Изменение** + **Подтвердить?**; нет «Как буду искать»; END TURN |
| F3 | extend | бриф B2: **Варианты** нумерованно; `Drift-check` только в `debug.md` |

Anti-patterns (fail): «как в прошлой сессии»; internal-команды с путями; перечень файлов как handoff; explore-бриф на авторемонте; смешение «ничего не требуется» + «Подтвердить?»; GO verify без `**Следующий шаг:**` + user-action команды; решение требует открыть файл.

Спецификация сценариев: [ux-acceptance-isolated-chat.md](ux-acceptance-isolated-chat.md).

## 3. Запрет внутренних ID и технических категорий в пользовательских полях

**Пользовательские поля:** «Контекст», «Сценарий», «Симптом», «Требования», «Что проверить», «Что реализовано», «Результат», «Что меняется для пользователя», заголовки карточек решений, описания вариантов, Executive Summary.

**Внутренние ID:** `D<N>` (Decisions), `S<N>.T<M>` (Slices), `I<N>` (матрица перехватов), `R<N>` (регрессии), `SC<N>` (static checks), номера задач из `tasks.md` вида `12.9`, номера секций `design.md §5.3`.

Запрещено помещать ID из второго списка в поля из первого. Им место — в полях «План», «Ссылки на артефакты», «Источник», либо в отдельной секции внизу блока.

### 3.1. Технические категории (жаргон движка) → пользовательский язык

**SSOT запрещённых в чате кодов движка — `.cursor/docs/chat-lexicon.md` (Слой 1).** Таблица ниже — расширенная карта артефактного жаргона (имена секций `design.md`, классы дефектов), которой нет в словаре. При расхождении приоритет у `chat-lexicon.md`. Помнить про две аудитории: workflow-чат разработчику допускает профессиональные слова (срез, постановка, область задачи), полная расшифровка — только в документах заказчику.

Категории и имена секций движка (правила, шаблоны artefact-секций, режимы скиллов) в пользовательский вывод **не попадают**. Они допустимы только в строке «Источники: …» в самом конце карточки или в полях, которые читает не человек, а другой скилл (front-matter отчёта).

| Технический термин | Пользовательский язык |
|---|---|
| `Implementation Options` | «варианты реализации» (в чате — прозой: «в постановке нет вариантов, как…») |
| `Behavior Contract` | «правила поведения / что должно происходить»; пункты Contract не цитировать как `п.7` |
| `Spec` / `Specification` | не цитировать; вместо «расхождение Spec ↔ Contract» — «в требованиях есть противоречие» |
| `Open Questions` | «открытые вопросы»; конкретные ID не цитировать |
| `Decisions` (как имя секции `design.md`) | не цитировать в чате; «нужно зафиксировать решение по …» |
| `Blast Radius` | «Что меняется для пользователя» / «Эффект для конечного пользователя» |
| `precedent-regression` / `9b` | «Отмена ранее принятого решения» |
| `invariant-drift` | «Отклонение от ранее зафиксированного правила» |
| `load-bearing-adr-bypass` | «Обход опорного архитектурного решения» |
| `Layer 1`…`Layer 5` / `Repair Loop` / legacy `Phase A`/`Phase B` / `Implementation Impact Gate` / `Card consolidation` | Не показывать вообще; пользователь видит только результат: «Авто-исправлено» / «Решения» / «Уточнения текста» |
| `decision`-класс / `artifact-hygiene` / `INFO` | Не показывать; разбиение через заголовки секций |
| `<!-- slice-gate -->` | Внутренний маркер; в чате не цитируется |
| `accept S<N>` / «срез принят» | «Функция готова: <название среза>» |
| `Promotion Test` / `Determinism Test` | Не показывать; работают внутри verify |
| `CRITICAL` / `WARNING` / `SUGGESTION` | «Блокер» / «Замечание» / «Предложение» (в счётчиках допустимы; в заголовках карточек — нет) |
| `verdict: PASS/FAIL`, `verify_mode: …` | «Готово к реализации / Не готово», «Проверка до реализации / после реализации» |
| `Tier`, `Standard`, `Lite`, `Full` (как метки объёма проверки) | «Объём: Краткая / Полная / Расширенная» |
| `когерентность`, `когерентный` (срезов / плана) | «согласованность», «согласованный» |
| `low-confidence`, `low confidence` (метка готовности) | «низкая уверенность» |
| `capability` (как термин движка) | «область поведения», «возможность» |
| `checkpoint` (как термин паузы apply) | «проверочная пауза», «пауза» |
| `step-by-step`, `степ-бай-степ` | «пошагово», «пошаговый режим» |
| Имена агентов (`onec-code-writer`, `onec-code-reviewer`, `onec-code-architect`, `onec-code-explorer`, `onec-trace-analyst`, `onec-code-simplifier`, `openspec-quality-controller`) | «агент», «агент + ревью», «архитектор», «ревьюер», «исследователь», «оркестратор» (без префикса `onec-` / `openspec-`). **В entry-брифе — полный запрет имён агентов.** |
| Имена гейтов (`Architect Gate`, `Precedent Regression Gate`, `Code-Truth Gate`, `Implementation Impact Gate`, `Slice Gate`, `Symptom Relevance Check`, `Cross-Archive Context`, `KB Discovery`) | Описание результата: «архитектурное ревью закрыто», «сверка с прежними требованиями выполнена», «проверка фактов в коде пройдена», «приёмка среза». **В entry-брифе — полный запрет упоминания служебных проверок.** |
| Метаданные маршрута (`target:`, `prev:`, `Шаг 1 — детали для агента`) | **Полный запрет в чате.** Эти данные живут только в `brief.md`. |

**Правило применения:** категории ID кодов движка (`precedent-regression`, `dependency-cycle`, `task-opaque-acceptance`, `acceptance-without-scenario` и пр.) допустимы **только** в строке «Источники: …» в самом конце карточки. В заголовках карточек, полях «Что не так», «Что меняется для пользователя», описаниях вариантов и Executive Summary — заменить на пользовательский язык по таблице выше или на конкретное описание дефекта.

**Принцип:** технические имена остаются в движке (правила, шаблоны секций design.md, ADR Supersedes, методология `vertical-slices.mdc`). Перевод применяется **только в слое вывода пользователю** — в карточках Блок 2 verify, Executive Summary, follow-up сообщениях.

### 3.1b. HALT для чата (жёстче §3.1)

Сообщения оркестратора **в чат** (не файлы `reports/`, не `debug.md`, не тело артефактов ЗНИ) проходят **обязательный** self-check по [`.cursor/rules/chat-output-budget.mdc`](../rules/chat-output-budget.mdc) §7. Бюджет строк, запрет жаргона, разделение «тонкий чат / полный файл». Нарушение → **переписать** ответ до соответствия, не отправлять «как есть».

«Срез» (vertical slice) — корректный методологический термин, не переименовывается. Это перевод устоявшегося vertical slice из Agile/SDD: функциональная единица, проходящая через все слои (UI + код + тест) и дающая одну готовую к приёмке возможность. Жёсткое правило подачи термина — §10 «Срез всегда с названием» и §11 «Глоссарий пользовательских терминов».

## 4. Правила оформления

- **Многошаговое → нумерованный список.** Если в поле ≥2 действий/требований/пунктов приёмки — нумерованный список. Точки с запятой для перечисления шагов — запрещены.
- **Один объект — одно имя на слот.** Если объект имеет имя в UI ДО2, в UI ДО3 и в коде — оформить короткой таблицей соответствия; не перемешивать три варианта в одном предложении.

Пример таблицы соответствия имён:

| UX ДО2 (сейчас) | UX ДО3 (цель) | Код |
|-----------------|---------------|-----|
| «Направить на новый цикл» (булево) | «На новом цикле» (список «Направлять» / «Не направлять») | `НаправитьНаНовыйЦикл` (колонка формы) |

- **Симптом ≠ ожидание.** В «Симптоме» — только наблюдаемые факты; «должно быть» — в «Сценарии» / «Требованиях».
- **Контекст vs Сценарий.** «Контекст» — зачем существует change (1 предложение из proposal). «Сценарий» — что конкретно должно произойти в отлаживаемой/реализуемой задаче. Не дублировать.
- **Длина полей.** Любое поле брифа/отчёта в чате — до 3 строк текста **или** до 7 пунктов списка. Длиннее — либо разбить на несколько полей, либо вынести в ссылку на отчёт в `reports/`.

## 5. Общие шаблоны (T-*)

T-коды (T-BRIEF, T-HANDOFF, T-REPORT, T-STATUS, T-CONFIRM) — **внутренние имена** макетов для авторов скиллов и навигации внутри `opsx-output-style.md`. В **сообщениях пользователю** (чат, отчёты в `reports/`, AskQuestion-блоки, заголовки секций) эти коды **не цитируются**. См. self-check 10 (§7) и таблицу §9 «Что НЕ цитируется в чате».

Допустимо в инструктивном тексте скиллов: «формат сообщения по шаблону `T-BRIEF`» (для автора скилла). Запрещено в шаблонах сообщений пользователю: «Output style: T-BRIEF» в начале брифа, «использую T-REPORT» в чате, «по T-CONFIRM» в финальной строке.

Содержимое слотов — ниже.

### 5.1. Бриф (Sync Card) — точка синхронизации

**Бриф** — структурированная суммаризация **текущего шага** для сверки и подтверждения. Не отчёт о плане работы агента.

После утверждения бриф **не сохраняется в файл** (Ultra-Lite). Для explore якорь **Вопрос** / `user-goal` живёт в чате и пересогласуется текстом; в промптах агентов и Symptom Lock — имя `user-goal`. Полные отчёты `Task` — в `temp/reports/<тип>-YYYY-MM-DD-<slug>.md`; финал bug — блок `## Постановка ЗНИ` **в чате**.

**Единый заголовок:** `Бриф: <тема>` — без «для исследования», без `/opsx:extend` в заголовке.

#### Уровни глубины (B0–B3)

| Уровень | Когда | Бюджет содержания | Сигнал |
|---------|-------|-------------------|--------|
| **B0** | repair-from-verify; new при свежем `## Постановка ЗНИ` в чате/handoff | 0–1 строка | без полного брифа; new: «Создаю ЗНИ `<slug>` по постановке из чата.» — информирующая строка, **без** согласования имени |
| **B1** | new: конкретная постановка свободным текстом (без блока/handoff — drift-check не применяется); extend: конкретное требование, нет открытой развилки (`Drift-check` — internal, в чат не выводится) | карточка: **Изменение** (+ опц. **Затронутое**) | «Подтвердить?» текстом |
| **B2** | extend: `drift-warning` / `scope-violation`; ≥1 неоднозначность; `--from-verify` после decision (только delta extend); `--code-sync`; `--from-review` с >3 findings | карточка + **Варианты** нумерованно | номер варианта + «Подтвердить?» |
| **B3** | `/opsx:explore`; сырой вход после Readiness Check | карточка explore (см. `brief-card.md`) | **Вопрос** + «Подтвердить?» |

**Правило минимальной глубины:** сомнение B1 vs B2 → **B2**; extend/new **никогда B3**. B3 зарезервирован за explore.

#### Классификатор (3 правила)

1. **B0** — repair-from-verify; new при свежем `## Постановка ЗНИ` в чате/handoff.
2. **B3** — только explore (сырой вход после Readiness Check; одиночное уточнение допускается до брифа).
3. **extend/new по свободному тексту:** конкретно и без открытых развилок → **B1**; drift/неоднозначность/`--from-verify` после decision/`--code-sync`/`--from-review` >3 findings → **B2**. Сомнение → B2. `Drift-check` — internal, в чат не выводится.

#### Форма брифа: карточка слотов

**SSOT:** [`templates/brief-card.md`](templates/brief-card.md) — слоты, эталоны, разбор ответов, запреты.

Бриф — **карточка с подписанными слотами** из мира пользователя (не сплошной абзац, не мета-язык агента). Бюджет: ≤6 заполненных слотов, 1–3 предложения на слот, скан ~30 секунд.

- **B1:** **Изменение** (+ опц. **Затронутое**) + `Подтвердить? — «да» или уточните.`
- **B2:** **Изменение** + **Почему выбор** + **Варианты** (нумерованный список `1.` `2.` …) + `Подтвердить? — укажите номер варианта (1 или 2) или уточните.` Голый «да» на B2 **не принимается**.
- **B3:** **Симптом** (+ опц. **Ожидание**, **Версия для проверки**) + **Маршрут** *или* **Варианты** (взаимоисключающие) + **Вопрос** (якорь `user-goal`) + опц. **Контекст**, **Уточнить до старта** + `Подтвердить?`. Слот **План** не используется.

Запрещённые подписи: `Что понял`, `Моё понимание`, `Как буду искать`, `Бриф для исследования` — см. `brief-card.md`.

**Внутреннее (никогда в чат-бриф):** `Drift-check`, «Соответствие исходному scope», KB-discovery, нумерованный план правки proposal/design/tasks, Scope Coherence Audit — только `debug.md` / `reports/`.

**HALT-строки для greppable self-check (extend/new B1/B2):** `proposal.md / design.md`, `1) уточнить`, `обновить артефакты`, `Drift-check:`.

Подтверждение — **всегда текстом**, не `AskQuestion` (исключения: Metadata Gate команды new — согласование **описания** маркера (`AskQuestion` с preview ФИО+дата; ФИО из project.md); один `AskQuestion` именной сверки в explore до брифа). UX-секции — без внутренних ID (§3). `temp/briefs/*.md` не создаются.

Полные эталоны B1/B2/B3 — в `templates/brief-card.md` и `openspec-explore/test-cases/`.

#### Составной бриф: `/opsx:intake` → `/opsx:<next>` в одном ответе

**Назначение:** при однозначном маршруте из intake скилл intake выполняет auto-handoff (тот же ответ ассистента) без повторного полного `T-BRIEF` целевой команды — **одно сообщение**, два логических блока: полный Intake Brief и **дополнение к брифу** (далее в инструктивном тексте скиллов допустимо «дельта»).

**Когда применяется:** только если Readiness intake пройден и маршрут **однозначен** (`/opsx:explore` или `/opsx:debug`). При неоднозначности маршрута — **только** Intake Brief (и при необходимости `AskQuestion`); дополнение выводится **в следующем** сообщении после ответа пользователя. Для `/opsx:ff` и `/opsx:new` отдельной «дельты» в этом же ходе **нет** (handoff — отдельная команда).

**Структура сообщения в чате (обязательный порядок):**

1. **`## Бриф: /opsx:intake | <тема>`** — полный Intake Brief (**Открытые вопросы** обязательны). **Без** строки **Подтвердить?** в конце intake-блока; допускается одна нейтральная строка: «Intake зафиксирован выше.»
2. **Связка** одной строкой (UX): например «**Перехожу к /opsx:<next> (дополнение к брифу выше).**»
3. **`### Дополнение: /opsx:<next>`** — только недостающие к целевому скиллу секции. **Запрещён** второй заголовок верхнего уровня `## Бриф: /opsx:<next> | …` в том же сообщении.

**Минимум в блоке «Дополнение»:**

| Целевая команда | Обязательные секции в дополнении | Не повторять из intake-блока |
|-----------------|-----------------------------------|-------------------------------|
| `/opsx:explore` | **KB в scope**; **План** (шаг 1 → агент); **Подтвердить?** (запуск исследования) | Контекст, Что я понял, Сценарий, Факты, Гипотезы, Артефакты, Границы, Открытые вопросы |
| `/opsx:debug` | **KB в scope**; **Контекст change** (1 строка: имя ЗНИ и суть из `proposal`); **План**; **Подтвердить?** | Симптом и Артефакты, если уже выведены во intake / trace-brief |

**Self-check (совокупное сообщение):** для режима «continuation-from-intake» обязательные поля §5.1 для **целевой** команды считаются выполненными, если **Контекст** и **Что я понял** присутствуют в intake-блоке; **KB в scope**, **План** и **Подтвердить?** — в дополнении.

### 5.1a. Explore — вывод в чате (Ultra-Lite: превью + Итог / Вердикт / Дальше)

**Роли каналов**

| Канал | Роль |
|-------|------|
| **Чат** | Превью отчёта `Task` (8–15 строк) + **Итог / Вердикт / Дальше** — решение **без** открытия файлов. Финал bug — блок `## Постановка ЗНИ` в чате. |
| **Файлы** | Полные отчёты `Task` — в `temp/reports/<тип>-YYYY-MM-DD-<slug>.md` (детали, цитаты). Handoff-файл `temp/explore-handoff-*.md` — **только** по словесной просьбе пользователя. Каталог `openspec/sessions/` в новых исследованиях не создаётся. |

**HALT:** сообщение только с путём к `temp/reports/...` **без** блока «Итог / Вердикт / Дальше» — запрещено. Финал bug без блока `## Постановка ЗНИ` — сессия не завершена для заказчика.

**Превью отчёта `Task`** (после каждого вызова, **8–15 строк**):

Оркестратор **обязан** прочитать из отчёта агента секции `## Для заказчика`, `## Recommended next steps`, `## Anomalies / concerns` (если есть) перед формированием превью.

```markdown
**Превью** — что нашли (3–7 буллетов простым языком; якоря `модуль:строка`, `#событие` — в полном отчёте, в чат — максимум 1 имя как анкор «где смотреть»).
**Источник:** `temp/reports/<тип>-YYYY-MM-DD-<slug>.md`
```

**Symptom Relevance Check** (только для bug-профиля, **до** «Вердикта»):

- Внутренняя классификация: `объясняет` | `частично` | `не объясняет`. В чат — только UX-фраза: «найденное объясняет симптом: да / частично / нет».
- Если `не объясняет` / `частично` — в «Вердикте» **запрещено** «штатно» / «норма» как итог; назвать **другую** причину или явный «причина не найдена, нужен шаг X».

**Итог / Вердикт / Дальше** (после превью):

```markdown
**Итог** — 2–4 буллета: что доказано, связь с **Вопросом** из брифа.
**Вердикт** — одна фраза: баг | штатно (с оговоркой UX) | нужна правка | не хватает данных.
**Дальше** — ровно одно: действие в Конфигураторе | `/opsx:new` / `/opsx:extend` | один вопрос | «достаточно».
```

**Запрещено:** бинарные меню («хватит / копаем код» без конкретного «Дальше»); `AskQuestion` для продолжения после шага; описания шагов оркестратора («запустить explorer») в «Дальше».

**Финал bug — блок `## Постановка ЗНИ` в чате.** Каркас, правила сборки и композиция финального сообщения (блок заменяет «Итог / Вердикт / Дальше», один «Следующий шаг», поведение при блокирующей развилке) — **только** в `.cursor/skills/openspec-explore/templates/handoff-block.md` (SSOT, здесь не дублируется). Поля Симптом / Корневая причина / Что менять — мини-абзацы 2–3 предложения; «Симптом» — наблюдаемое поведение, **не** дословный слот «Вопрос».

`/opsx:new` и `/opsx:extend` подхватывают блок прямо из истории чата (приоритет 1) или из `temp/explore-handoff-*.md` (приоритет 2, если пользователь просил сохранить).

После каждого шага с отчётом `Task` — **управляющая контрольная точка**: превью + Symptom Relevance Check (для bug) + «Итог / Вердикт / Дальше»; приглашение ответить **текстом**, без бинарного «достаточно? да/нет».

### 5.2. T-HANDOFF — handoff / карточка приёмки

Используется в: `/opsx:apply` для всех трёх вариантов: `acceptance` (срез готов к ручной приёмке), `pause` (прервано с issue), `final` (все задачи закрыты). Те же follow-up строки могут использовать `/review`, `/opsx:explore` для handoff на `/opsx:extend`.

#### Тонкий чат (по умолчанию) — что видит пользователь

Соблюдать [`.cursor/rules/chat-output-budget.mdc`](../rules/chat-output-budget.mdc). В **чат** выводится **не более 4–8 строк**:

| Вариант | Чат (минимум) |
|---------|----------------|
| `acceptance` | Заголовок среза с названием (§10); 1–2 строки «что сделано»; блок **«Карта правок (перед тестом)»** — до 5 пунктов (проза + citation, вне лимита 4–8 основного handoff — см. `chat-output-budget.mdc`); **одна** фраза «что проверить в 1С» или ссылка «сценарий в файле»; после проверки — ответить обычной фразой («принято», «срез принят») **или** снова `/opsx:apply <name>`; **без** таблицы «Следующие задачи». |
| `pause` | Суть проблемы (UX); **варианты** короткими буллетами (без машинных кодов ответа); приглашение ответить **обычным текстом**; без таблиц прогресса. |
| `final` | Одна строка: работы завершены; **следующий шаг** `/opsx:archive <name>` (без цепочки «сначала verify»). |

#### Полная версия — только в файл

Сразу после формирования тонкого чата оркестратор **записывает** полный handoff в:

`openspec/changes/<name>/reports/handoff-<variant>-S<N>-YYYY-MM-DD.md` (для `acceptance` / `pause` с указанием среза; для `final` — `handoff-final-YYYY-MM-DD.md`). Для `acceptance` имя файла: `handoff-acceptance-S<N>-YYYY-MM-DD.md`. Накопительная карта правок — `reports/code-map.md` (отдельный артефакт; см. слот «Карта правок»).

В файле — **полная** структура слотов ниже (все секции 1–9 по варианту). В чат пользователю — **только** ссылка на этот путь (одна строка), если нужно не дублировать сценарий.

#### Язык карты (Code Map)

Два слоя, один источник (journal apply / spot-check), **разный язык**:

| Слой | Аудитория | Формат |
|------|-----------|--------|
| **Чат** (acceptance-handoff) | заказчик перед тестом | одно предложение прозой + один citation [`start:end:path`](...); до **5** пунктов; без internal-терминов |
| **Файл** `code-map.md` | заказчик, ревьювер, архив | **сначала** русское предложение «что изменилось и зачем», **потом** markdown-link + `:строки`; ID задачи `S<N>.<M>` обязателен |

**Citation-конvention:** в чате — `` [`start:end:path`](...) `` (Cursor code-citation); в файле — markdown-link `` [`path`](path):start-end ``. Суть берётся из формулировки задачи в `tasks.md`, не из ответа writer. ID задачи в чате опционален.

#### Слоты полной версии (для файла `reports/handoff-*.md`; эталон содержания)

1. **Заголовок.** `## <Вариант>: <change-name> | срез S<N> (если применимо)`. Варианты: «Срез S<N> — передача на приёмку», «Сессия приостановлена», «Реализация завершена».
2. **Прогресс.** `**Прогресс:** M/N задач [x] (срез S<N>.T<M>: <статус>)`.
3. **Что реализовано.** Нумерованный список: `[x] <id задачи> — <одно предложение>; файл: `<path>`; при расхождении spot-check/линтера/ревью — одна строка «Замечание: …» (успешный путь без строки «OK» — см. §3a `chat-output-budget.mdc`).
3a. **Карта правок** (только вариант `acceptance`). Секция среза из `reports/code-map.md` (полный список пунктов с ID задачи, модулем, процедурой, ссылкой на файл:строки). В чат — компактный блок до 5 пунктов (см. «Язык карты»).
4. **Что проверить СЕЙЧАС** (UX-слой, императив). Нумерованный список шагов из критериев приёмки (`S<N>.T<M>` или аналог). Без внутренних ссылок на R<N>/SC<N> в пользовательском тексте — только в скобках «(регрессия R2)» после пункта.
5. **Следующие задачи** (только варианты `acceptance` и `pause`). Таблица: `Задача | Действие | Тип | Исполнитель | Зависит от | Статус`.
   - **Задача** — ID `S<N>.<M>` / `S<N>.T<M>` в backticks.
   - **Действие** — короткое описание задачи из `tasks.md` (одна фраза, без имён процедур и кодов категорий).
   - **Тип** — русское обозначение из набора: `Код модуля` (BSL общий/объектный модуль), `Код формы` (BSL `Forms/.../Ext/Form/Module.bsl`, программное создание элементов), `Ручной тест`, `Ручная регрессия`, `Конфигуратор` (создание/правка метаданных), `Проверка` (Grep/Read оркестратора).
   - **Исполнитель** — `агент`, `агент + ревью`, `пользователь`, `оркестратор`. Имена `onec-code-*` / `openspec-*` в чат не выводятся (см. §3.1).
   - **Зависит от** — ID задач в backticks; если зависимость `[ ]` — пометить «невыполнима до `<id>`».
   - **Статус** — `[x]` / `[ ]`.
6. **Issue / Options** (только вариант `pause`). 1 абзац: суть проблемы (без длинных логов в чате, логи — в файл). После него — нумерованный или маркированный список **альтернатив простым языком** (можно указать последствия каждого пути одной строкой); **не** использовать коды вида `1a` / `2b`.
7. **Как вернуться** (варианты `acceptance`, `pause`). Одна строка с командой: `/opsx:apply <change-name>`. Если требуется пересмотр scope/design/tasks — добавить строку: `Обновить scope: /opsx:extend <change-name> [--from-review <path>|--from-verify <path>|--from-debug <path>]`.
8. **Short-cut** (только `acceptance`). Одна строка: «Если уже проверено и принято — напишите обычной фразой (например, «принято» или «срез S<N> принят»), отмечу без повторной простыни в чате».
9. **Как продолжить** (только вариант `pause`). Одна-две строки: напишите в чат, какой путь ближе (или задайте уточняющий вопрос); машинный формат ответа **не требовать**.

Имена секций в **файле** handoff совпадают между вариантами. Параллельно — запись в `debug.md` `## Slice Gate Decisions` по контракту skill-а apply.

### 5.3. T-REPORT — отчёт

Используется в: `/opsx:verify`, `/opsx:status --reports`.

#### Тонкий чат (по умолчанию), полный отчёт в файле

- Полное тело отчёта (Executive Summary, Scorecard, таблицы, группы замечаний, Action items) — в файле `reports/*.md` по контракту соответствующего скилла.
- Для **`/opsx:verify`** в чат — **строго по** `openspec-verify-change/templates/chat-summary.md`: **тихий** или **информативный** вариант; обязателен слот `**Следующий шаг:**` (зеркало Action items в файле); без блоков «Решение N» / «Как ответить» / кодов вида `<N>b`. Краткая разговорная развилка в секции «**Что решить**» при необходимости; полное тело блоков решения — только в файле.
- Флага **`--verbose` для `/opsx:verify` нет** — чат всегда «тонкий»; файл отчёта при необходимости полный после шага 18 verify.

#### Слоты полного отчёта (файл; эталон содержания)

*(Примечание: для `/opsx:verify` конкретные Markdown-шаблоны карточек и шапки вынесены в `openspec-verify-change/templates/`. Здесь описана общая структура слотов.)*

1. **Summary.** Два слоя, **не в одном предложении** (для `/opsx:verify` порядок и запреты дублируют шаг 16 `openspec-verify-change/SKILL.md`):
   - **Суть (UX + результат):** 1–2 предложения без имён файлов артефактов (`design.md`, `tasks.md`, `debug.md`), без имён секций движка (`## …`), без слов **PASS** / **FAIL** / **verdict**. Только: что за доработка/сценарий проверены и итог для заказчика («готово к реализации», «не готово — есть блокеры», «готово с оговорками»).
   - **Процесс / документы (отдельно от «Сути»):** при необходимости — отдельный короткий абзац или строка. Пути к файлам постановки, повторный прогон, ссылки на архив — **только здесь** (без запрещённых в чате формулировок из §3.1b и `chat-output-budget.mdc`).
2. **Группы по уровню.** Для verify: блоки по важности (в пользовательском языке: «Блокеры» → «Нужны решения» → «К сведению»).
3. **Внутри группы.** Нумерованный список. Каждая запись:
   - пользовательская формулировка проблемы/факта (UX-слой) — 1 предложение;
   - где смотреть (код/артефакт) — `backticks` с путём и диапазоном строк;
   - технический код замечания — в скобках в конце пункта или в строке «Источники:».
4. **Развилки пользователя** (опционально, для verify после Issue Classification карточек в **файле**; иные команды — по своим скиллам). В **чат verify** блок «Решения» с кодами **`Na`** **не используется** — только проза в «Что решить».
5. **Action items.** Нумерованный список из 3–7 следующих шагов: что сделать пользователю (или что сделает следующая команда). Каждый пункт — одно действие. Для verify GO **первый пункт** = та же команда, что в чате (`**Следующий шаг:**`).
6. **Ссылки.** Пути к полным отчётам субагентов в `reports/` (по одной на строку).
7. **Как ответить** (опционально: для отчётов с пронумерованными вариантами; **`/opsx:verify`** — не использует этот блок, ответ пользователя принимается **свободным текстом**). Если блок есть: три строки — как ответить, свободный текст, поведение при пустом ответе.

Запрещено:

- Перемешивать ID категорий и пользовательские формулировки в одном предложении (ID — только в скобках).
- Выводить список замечаний как сплошной текст.
- Дублировать в чат полное содержимое файла отчёта без запроса пользователя «подробнее».

### 5.4. T-STATUS — снимок состояния

Используется в: `/opsx:status`.

Слоты:

1. **Шапка.** `**Change:** <name> | **Schema:** <schema> | **Фаза:** <pre-apply | post-apply> | **Структура:** <срезы | legacy>`.
2. **Маркеры.** `developer`, `comment_suffix` (domain_label), `marker_style`, `process-only: да|нет`, `marker_scope`, preview `open`/`close` (см. `openspec-status/SKILL.md` §2b). Если Metadata нет — одна строка «не заполнена». Ссылка: `marker-layers-guide.md`.
3. **Прогресс срезов.** Таблица: `Срез | Название | Прогресс (N/M) | Приёмка (S<N>.accept) | Статус (принят / awaiting-acceptance / в работе / не начат)`.
4. **Открытые решения.** Нумерованный короткий список вариантов открытых `awaiting-acceptance` (вариант 1, 2) без развёрнутых абзацев, отсылая за деталями в файл `debug.md` (если есть).
5. **Последние отчёты.** Таблица (только при флаге `--reports`): `Тип | Дата | Путь | First-line summary`.
6. **Рекомендуемая следующая команда.** Одна строка: `/opsx:<команда> <name> [флаги]` + в скобках 1 предложение почему.

Режим `--short`: только слоты 1, 3 (одной строкой), 6.

### 5.5. T-CONFIRM — короткое действие

Используется в: `/opsx:archive`, `/opsx:bulk-archive`, `/opsx:sync`, `/opsx:extend` (после обновления артефактов или остановки на неоднозначности).

**Skip-on-empty и non-events:** см. **§3a** [`.cursor/rules/chat-output-budget.mdc`](../rules/chat-output-budget.mdc). Не перечислять в чате «чего не делали», «что уже было синхронизировано», пустые статусы Knowledge/ADR — только материальный результат и при необходимости одна команда «Дальше».

Слоты (гибко, **опускать** пустые):

1. **Действие.** 1–2 предложения на **языке эффекта** (§2.6 Правило 5): что изменилось для дела, не перечень файлов.
2. **Следующий шаг** (если применим). Одна строка user-action команды (`/opsx:verify <name>` и т.п.) **без** флагов и путей.

**По командам:**

- **`/opsx:extend` (user-extend):** «Постановка дополнена. Следующий шаг: `/opsx:verify <change-name>`.» — hint, не auto-chain.
- **`/opsx:extend` (repair-from-verify, internal):** **нет** сообщения в чат — вызывающий verify продолжает Repair Loop.
- **Прочие (archive, sync, migrate):** слот 1 + опц. слот 2.

Запрещено: «Обновлено: `tasks.md`, `design.md`, … Дальше: `/opsx:extend --from-verify …`»; перечень файлов как handoff.

Запрещено длинное описание «почему сделано» — это место для объяснения выбирает пользователь через отдельный вопрос.

## 6. Пары bad → good

**P1. Смешение слоёв.**

- bad: «После реализации S5.2 при `ПриИзменении` колонки «Направлять/Не направлять» в `ФормаИзменениеПараметров` вызов `pav_УчастникиПовтора` записывает в `ДопСв` по ключу `(НомерСтроки, Исполнитель, РольИсполнителя)` согласно D8».
- good:
  > UX: пользователь переключает значение в колонке «На новом цикле» — состав повтора пересчитывается без сохранения БП.
  > Код: `ФормаИзменениеПараметров.Ext.Form.Module.bsl`, обработчик `ПриИзменении` колонки `НаправитьНаНовыйЦикл`; запись в `ДополнительныеСвойства.pav_УчастникиПовтора`.
  > Процесс: срез `S5`, задача `S5.2`; ключ по контракту `D8`.

**P2. Разнотипные термины без разделителя.**

- bad: `` `pav_РежимСохраненияФормы` `ФормаИзменениеПараметров` `&Вместо` `ТекстЗапросаТаблицаИсполнителей` `onec-code-explorer` ``.
- good: «перехват `&Вместо("ТекстЗапросаТаблицаИсполнителей")` в модуле `ФормаИзменениеПараметров` зависит от флага `pav_РежимСохраненияФормы`; для проверки цепочки вызовов делегировать `onec-code-explorer`».

**P3. Внутренние ID в UX-поле.**

- bad: «Симптом: см. D7 и S3.T1 — на форме отображается только отказавшийся (`12.9` не применено)».
- good: «Симптом: на форме «Повтор согласования» отображается только отказавшийся, ожидаются все участники». *Ссылки (отдельная секция):* инвариант `D7`, срез `S3`, задача `12.9`.

**P4. Три имени одного объекта подряд.**

- bad: «Колонка `НаправитьНаНовыйЦикл` — это "Направить на новый цикл" в ДО2 и "На новом цикле" в ДО3».
- good: таблица соответствия (§4).

**P5a. Explore: только AskQuestion без брифа.**

- bad: первый ответ на `/opsx:explore` — вызов `AskQuestion` («Утверждаем план?») без markdown-тела брифа в сообщении.
- good: бриф B3-карточка: `Бриф:` + слоты **Симптом**, **Маршрут** (или **Варианты**), **Вопрос**, **Подтвердить?** — см. `templates/brief-card.md`.

**P5a-ext. Extend: план агента вместо сверки.**

- bad: бриф B1 с «Как буду искать: 1) обновлю proposal 2) design 3) specs…» или заголовком «Бриф для исследования»; подпись **Что понял**.
- good: бриф B1: слот **Изменение** + `Подтвердить?`; план правки артефактов — в `debug.md` после «да».

**P5. Поток вместо карточки (только B3 explore).**

- bad: сплошной абзац с симптомом, гипотезой и планом в одной фразе; имена агентов; слот **План** вместо **Маршрут** / **Варианты**.
- good:
  > **Симптом** — …
  > **Маршрут** — разберу трассу, сравню с цепочкой подписания.
  > **Вопрос** — …
  > При выборе маршрута:
  > **Варианты**
  > 1. Сначала трасса — …
  > 2. Сначала код — …

**P6. Локальная нумерация vs сквозная.**

- bad:
  > ## Требует вашего решения (1)
  > ### 1. Резолв реализации S1.2
  > a) добавить research-задачу
  > ## Уточнение текста артефактов (1)
  > 1. Покрытие сценариев
  > [a] сузить spec
- good (канонический путь описан простым текстом в «Что решить»; каждая развилка — заголовок + абзац + буллеты-альтернативы, без машинных кодов ответа):
  > ### Что решить
  >
  > **Резолв реализации в срезе S1: «<название>».** … **Логичный путь:** … Альтернативы одной строкой: … Что затронет выбор …
  >
  > **Покрытие сценариев (уточнение текста).** … Логичный путь …

**P6. Телеграф вместо мини-абзаца.**

- bad: «S2.3: явно указать вызовы `Мультипредметность` перед `ЗаполнитьПоШаблону`.»
- good: «**Расчёт исполнителей для предпросмотра.** В плане задач пропущен важный шаг: перед вызовом `ЗаполнитьПoШаблону` нужно передать предметы процесса. Без этого предпросмотр может выдать пустой список.»

**P7. Техцитирование вместо прозы.**

- bad: «В `design.md` нет `Implementation Options` для S2; противоречие `Spec` ↔ `Behavior Contract п.7`.»
- good: «В постановке не хватает деталей: нет вариантов реализации для второй части (S2), а в требованиях есть противоречие по автоподстановке.»

**P8. Висим в воздухе.**

- bad: NO-GO с 3 пунктами «что доработать в постановке» + ссылкой на отчёт без явного следующего хода.
- good: NO-GO с одним вопросом и явной строкой «остальное через extend» — **или** с автозапуском extend без вопроса.

## 7. Self-check-5

Перед выводом любого брифа / отчёта / handoff skill проходит 5 базовых проверок и обязательную 6-ю «канонический переход». Для T-BRIEF добавляется проверка 7 (KB в scope). Если хоть одна обязательная проверка не выполнена — переформатировать вывод.

1. **Слои разделены.** Ни одно пользовательское поле (`Контекст`, `Сценарий`, `Симптом`, `Требования`, `Что проверить`) не содержит внутренних ID (`D<N>`, `S<N>.T<M>`, `I<N>`, `R<N>`, `SC<N>`, `12.9`).
2. **Имена объекта.** Для каждого объекта, у которого есть UX-имя и имя в коде, выбрано ровно одно имя на слот; если нужны оба — таблица соответствия, а не перечисление через «—» / «/».
3. **Списки.** Любое перечисление из ≥2 пунктов оформлено как нумерованный список, а не строка с `;`.
4. **Симптом ≠ ожидание.** В поле «Симптом» нет формулировок «должно быть / ожидается»; ожидание вынесено в «Сценарий» / «Требования».
5. **Длина.** Ни одно поле не длиннее 3 строк текста или 7 пунктов списка; лишнее вынесено в отдельное поле или ссылку на отчёт.
6. **Канонический переход (все шаблоны T-*).** Последний абзац / последняя строка вывода предлагает **канонический переход**: конкретную команду `/opsx:*`, правку существующего файла, ссылку на созданный отчёт или явное END. **Запрещены** формулировки: «могу выписать», «черновик в чате», «список для копирования», «сам запустите команду с этим контекстом», «накидаю формулировки», «в следующем сообщении выпишу...». Промежуточный артефакт, который пользователь должен куда-то «приклеить» сам, — провал self-check.
7. **KB в scope и открытые вопросы (T-BRIEF в чате).** Для `/opsx:explore`, `/opsx:debug`, `/opsx:extend` секция **KB в scope** в чате обязательна: либо факты `KB-NNNN` + заголовок + однострочная выжимка, либо явное «нет совпадений по anchor-paths и домену». Для `/opsx:intake` секция **Открытые вопросы** обязательна (см. `.cursor/rules/conversational-discipline.mdc` §B). Для B1/B2/B3 explore/extend **полный** KB-discovery в промпты агентов — после «да», не вместо строки «KB в scope» в entry-брифе.
8. **Адресуемость развилок.** Для **`/opsx:verify`** — только секция «**Что решить**» без кодов ответа. Для **паузы `/opsx:apply`** (§5.2 вариант `pause`) и прочих handoff без verify — каждая альтернатива имеет понятную **одну строку** последствий для пользователя; пригласить ответить **обычным текстом** (метки вида `<номер><буква>` не обязательны и для verify **`запрещены`** в чате).
9. **Срез всегда с названием.** Каждое упоминание `S<N>` в пользовательском поле (Executive Summary, таблицы Slice Acceptance, заголовки карточек, описания вариантов, handoff-сообщения) идёт в формате `Срез S<N>: «<название из tasks.md>»`. Голый идентификатор `S1`, `S2` без названия — провал self-check. Допустимо без названия только внутри идентификатора задачи `S<N>.<M>` / `S<N>.T<M>` или в строке «Источники: …» (см. §10).
10. **Жаргон движка.** Ни один термин из §3.1 (`Blast Radius`, `precedent-regression`, `9b`, `Phase A/B`, `Promotion Test`, `verdict:`, `verify_mode:`, `target:`, `prev:`, `Шаг 1 — детали для агента`, `Symptom Relevance Check`, `KB Discovery` и т. д.) не появляется в пользовательских полях, заголовках карточек и Executive Summary **файла отчёта в той части, что дублируется в чат**; в полном теле `reports/*.md` технические коды допустимы там, где это зафиксировано шаблоном verify (включая YAML и строки «Источники»). Для **чата** — дополнительно обязательный HALT-список из [`.cursor/rules/chat-output-budget.mdc`](../rules/chat-output-budget.mdc).
11. **Verify — «Суть» и режим.** В **файле** отчёта `/opsx:verify` тело `## Executive Summary` — только **`templates/executive-summary.md`** («Суть», опционально «Что в работе»). Режим и вердикт — только в YAML frontmatter + **`snapshot`**. Чат verify — см. **`chat-summary.md`**, без технической шапки «Этап / Объём» как пользовательское сообщение.
12. **Бюджет чата.** Entry-бриф: бюджет содержания из `templates/brief-card.md` (≤6 слотов, ~30 сек скан); B0: 0–1 строка. Плюс §4 (длина полей). Прочие opsx-выводы — таблица §1 `chat-output-budget.mdc`.
13. **HALT перед отправкой в чат.** Мысленно прогнать запреты жаргона и длину из п.12 и §3.1b; при сомнении — не отправлять длинный текст, заменить на ссылку на файл.
14. **No Acknowledgement.** Отдельная строка-подтверждение («Понял», «Принял к работе», «Сейчас сделаю») в чат **не** выводится (`chat-output-budget.mdc` §4). «Человечность» даёт сам бриф — одна строка перефраза задачи внутри него. Для нетривиальной/неоднозначной команды — сразу адаптивный entry-бриф (§5.1); для простой read-only — сразу результат.
15. **Adaptive-by-default (T-BRIEF).** Выбран **минимальный достаточный** уровень B0–B3 (классификатор §5.1). Форма — карточка `brief-card.md`; для explore B3 — обязательны **Симптом**, **Маршрут** или **Варианты**, **Вопрос**; для B2 — **Варианты** нумерованно. Опциональные слоты — только при непустом содержимом. `temp/briefs/*.md` не создаются.
16. **Risk Surfacing в handoff.** После успешного результата — 1–3 границы изменений на UX-языке («что меняется / что НЕ меняется»), компактно. Развёрнутый разбор рисков и глоссарий терминов — в файл отчёта или по запросу, не в каждый handoff.
17. **Варианты на виду.** Если требуется решение пользователя, альтернативы должны быть **явно описаны простым языком** в последних блоках (для **`/opsx:verify`** — в секции «**Что решить**» без кодов **`1a`/`2b`**). Запрещено прятать содержание развилки только во вложенных файлах, если от пользователя ожидается ответ в этом сообщении (**verify**: краткая версия в чат обязательна).
18. **Explore — превью + синтез.** После брифа каждый ход `/opsx:explore` после `Task` содержит превью отчёта 8–15 строк с путём к `temp/reports/<тип>-YYYY-MM-DD-<slug>.md` и блок **Итог / Вердикт / Дальше** (§5.1a). Путь к файлу — внутри превью или после него, не вместо синтеза.
19. **Explore — финал bug.** Завершение исследования: блок **`## Постановка ЗНИ`** **в чате** (Симптом, Корневая причина с маркером verified/hypothesis, Что менять, Файлы, Приёмка, Связь с архивом, Architect/verify). Опциональный handoff `temp/explore-handoff-*.md` — **только** по словесной просьбе.
20. **Explore — Symptom Lock.** В bug-профиле запрещён вердикт «штатно» / «норма», пока якорь **Вопрос** / `user-goal` не снят или явно не пересогласован в чате; в этих случаях «Вердикт» обязан назвать другую причину или явный «причина не найдена, нужен шаг X».
21. **Explore — `Task` лимиты.** ≤ 3 `Task` (trace + explorer) на одну тему без явного «копаем глубже» от пользователя; architect — вне лимита 3. Каталог `openspec/sessions/` в новых исследованиях не создаётся.

Историческое имя «self-check-5» сохраняется ради существующих ссылок в скиллах; де-факто проверок больше.

Если хоть один пункт self-check не выполнен — переформатировать. Никаких выводов «с примечанием, что пункт N не прошёл» в чат пользователю.

## 8. Дисциплина правок ЗНИ (decision vs hygiene)

Verify в Repair Loop классифицирует замечания на три класса (`decision` / `artifact-hygiene` / `INFO`). Класс задаёт **дисциплину правки** артефактов ЗНИ — кто меняет файлы и через какой путь.

| Класс | Что меняет | Кто меняет | Путь | После правки |
|---|---|---|---|---|
| **decision** | код, поведение системы, приёмочные шаги, нарезку срезов, `Behavior Contract` | оркестратор через `/opsx:extend` | **Только** `/opsx:extend <name> --from-verify <report>` (или `--from-review`, `--from-architecture`, `--from-debug` по контексту). Прямая правка `proposal.md` / `design.md` / `specs/**/spec.md` / `tasks.md` запрещена. | повторный `/opsx:verify <name>` |
| **artifact-hygiene** | только текст артефакта (формулировка, привязка `Связь со spec`, опечатки, согласованность сценариев) — **не** меняет код / поведение / приёмку | оркестратор вручную через StrReplace в файлах ЗНИ | прямая правка артефакта **разрешена** | повторный `/opsx:verify <name>` обязателен |
| **INFO** | ничего | — | действия не требуется | — |

**Принцип:** прямая правка артефактов ЗНИ оркестратором (без `/opsx:extend`) разрешена **только** для класса `artifact-hygiene`. Любая правка, меняющая код / поведение / приёмочные шаги (даже однострочная), должна пройти через `/opsx:extend` — это сохраняет аудит-след и гарантирует, что Architect Gate / Design Gate переоцениваются при изменении scope.

В **файл** verify (внутренняя классификация Repair Loop) попадают **альтернативы в пользовательском языке** (буллеты, абзацы), без машинных кодов ответа; в **чат** — короткий парафраз в «**Что решить**». Приглашение: ответить **обычным текстом**. Куда вести дисциплину правки (`/opsx:extend` vs ручная правка текста) — указать одной финальной строкой в чате, не через меню кодов.

## 9. Что НЕ цитируется в чате

Вспомогательная сводка для self-check 9 и 10 (правило: не упоминать в пользовательском выводе).

| Категория | Где живёт | В чат не попадает |
|---|---|---|
| Внутренние ID без названия (`S1`, `S2`, голый `S<N>`) | tasks.md, debug.md | ✗ — только `Срез S<N>: «<название>»` |
| Имена T-* шаблонов (T-BRIEF, T-HANDOFF, T-REPORT, T-STATUS, T-CONFIRM) | этот документ §5 | ✗ — это имена для авторов скиллов |
| Имена классов проблем (`decision`, `artifact-hygiene`, `INFO`) | `openspec-verify-change/SKILL.md` | ✗ — пользователь видит «Что решить», «По плану», секции уточнения текста **внутри файла** |
| Внутренние имена этапов verify (`Layer 1`…`Layer 5`, `Repair Loop`, legacy `Phase A`/`Phase B`) | `openspec-verify-change/SKILL.md` | ✗ — в чат можно «мы поправили мелочи автоматически» вместо внутреннего имени |
| Имена режимов verify (`slice-pre`, `slice-post`, `slice-transition`, `verify_mode:`, `verdict:`) | `vertical-slices.mdc`, `verify-user-communication.mdc` | ✗ — пользователь видит «Проверка до реализации среза» / «После реализации» / «Готово к реализации» |
| Имена секций `design.md` (`Blast Radius`, `Behavior Contract`, `Implementation Options`, `Simplicity Check`) | `architect-gate.mdc`, `precedent-regression-gate.mdc`, `openspec-new-change/SKILL.md` | ✗ в карточках; ✓ в строке «Источники: …»; ✓ в самом артефакте `design.md` |
| Имена внутренних тестов verify (`Promotion Test`, `Determinism Test`) | `openspec-verify-change/SKILL.md` | ✗ — внутренняя механика |
| Severity-метки в заголовках карточек (`(CRITICAL)`, `(WARNING)`) | verify | ✗ — severity передаётся через секцию или счётчик в Executive Summary, не в каждой карточке |

## 10. Жёсткое правило: срез всегда с названием

В любом пользовательском выводе (Executive Summary, таблицы Slice Acceptance, карточки решений, handoff-сообщения, диалог в чате, `## Что меняется для пользователя`) **запрещено** цитировать голый идентификатор среза `S<N>`. Только формат:

```
Срез S<N>: «<название из tasks.md>»
```

Название берётся из заголовка `# Срез S<N>: <название>` соответствующего блока в `tasks.md`. Если такого заголовка нет (legacy ЗНИ без срезов) — формулировка переходит в текст без `S<N>` («первый этап реализации», «функция Х»).

**Исключения** (формат `S<N>` без названия допустим):

1. Внутри идентификатора задачи: `S3.2`, `S3.T1` — это полный ID задачи / приёмочного теста, цитируется как есть в backticks.
2. В строке «Источники: …» в конце карточки: `Источники: precedent-regression-9b, S3` — компактный технический контекст для пользователя, который хочет копнуть глубже.
3. В команде в строке «Как вернуться»: `/opsx:apply <change-name>` — без `S<N>` вообще.

**Эталон срабатывает для:** verify (Executive Summary, Slice Acceptance Status, карточки решений Repair Loop), apply (handoff-карточки, paused-сообщения), extend (брифы), debug (брифы делегирования, RCA), status (таблицы прогресса).

Self-check 9 (см. §7) отлавливает нарушение: голый `S<N>` вне исключений — провал.

### 10.1. Задача в первом упоминании — с коротким описанием

Расширение правила §10 на **задачи** среза. В заголовке секции / карточки / handoff-сообщения первое упоминание `S<N>.<M>` или `S<N>.T<M>` идёт **с коротким описанием задачи в «ёлочках»** — берётся из формулировки задачи в `tasks.md`.

Хорошо:

```
## Что проверить у себя — задача `S1.2` («перенос флажка из шаблона в БП»)
**Задача `S1.2` («перенос флажка из шаблона в БП») реализована.**
```

Плохо (голый ID без описания в заголовке):

```
## Что имеет смысл проверить у себя (к S1.2)
## S1.2 закрыта с моей стороны
```

Повторное упоминание `S1.2` в той же секции допускается без описания (как раньше). Срез — по §10 (`Срез S1: «<название>»`).

**Эталон срабатывает для:** apply (заголовки шагов пошаговой паузы, handoff-карточки), debug / extend / verify (карточки решений с привязкой к задаче), `/review` (формулировки замечаний с привязкой к задачам ЗНИ).

## 11. Глоссарий пользовательских терминов (one-shot per session)

В первом сообщении новой сессии команд `/opsx:apply`, `/opsx:verify`, `/opsx:extend` (один раз на сессию, не дублировать при повторных вызовах в той же сессии) — справочная строка / маленький блок с определениями ключевых пользовательских терминов, которые встретятся в выводе.

**Определения для one-shot блока:**

> **Срез** — связка UI, кода и тестов, дающая одну готовую к приёмке функцию.
> **Слой кода / поведения / приёмки** — что именно меняется при выборе варианта решения: исходный код, наблюдаемое поведение системы, или сценарий ручной приёмки.
> **Что меняется для пользователя** — описание эффекта решения в терминах конечного пользователя 1С (без кодовых имён процедур и реквизитов в основной формулировке).

**Когда выводится:**

- Первое сообщение скилла в новой сессии — встроить блок в шапку или сразу после заголовка.
- Повторные запуски той же команды в той же сессии — блок **не** дублируется.
- Если сессия начата с `/opsx:status` (read-only) — блок не нужен; он появится при первом `/opsx:apply` / `/opsx:verify` / `/opsx:extend`.

**Когда НЕ выводится:**

- В отчётах в `reports/` — отчёт читают разработчики, знакомые с терминами; в начало вкладывается стандартный заголовок и метаданные.
- В коротких T-CONFIRM сообщениях (одно действие — одна строка), кроме случая, когда это первое сообщение сессии.
- Если пользователь явно отказался («без глоссария») — сохранить выбор в текущей сессии и не выводить.
