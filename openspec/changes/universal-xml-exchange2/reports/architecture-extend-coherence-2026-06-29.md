# Architecture: статус обмена через список значений (удаление перечисления)

## KB references

Discovery выполнен, совпадений в KB нет — секция неприменима.

## Свод

**Задача.** Заказчик отклонил «оставить перечисление + кнопка» и требует: удалить `Перечисление.рг_СтатусыОбмена`; статус вести через `СписокЗначений`, сформированный на форме элемента `рг_НастройкиОбменаXML`; две кнопки — «Подготовить к обмену» и «Отменить обмен» — устанавливают статус из этого списка.

**Ключевой факт платформы.** `СписокЗначений` — это коллекция формы (UI), а **не** тип реквизита для постоянного хранения. Веб-сервис (`GetData` / `ConfirmDataReceipt`) читает статус на сервере **без формы**, поэтому статус обязан где-то персистентно храниться. Удаляя перечисление, мы обязаны заменить тип реквизита `Статус` на примитив (рекомендуется **Строка(20)** со стабильными кодами), а серверный gate — сравнивать с **централизованными константами**, а не со списком значений (которого на сервере нет).

**Вывод.** Запрос реализуем по UX-части **ровно** как просит заказчик (нет перечисления; список значений на форме; две кнопки пишут статус из списка). Единственная неустранимая техническая добавка — реквизит `Статус` становится текстовым кодом, а 7 серверных сравнений переводятся на функции-константы. Поведение двухфазной модели (атомарный check-and-set, идемпотентный Confirm, журнал) **сохраняется без изменений** — меняется только носитель значения.

## Complexity

**Complex** — затрагивает метаданные (удаление enum + смена типа реквизита), серверный модуль (7 точек), форму (список + 2 команды + read-only поле), миграцию данных ИБ и контракт design.md (D1).

## Chosen Approach

**Approach:** Pragmatic balance — удалить перечисление, перенести домен статусов в строковые коды-константы общего модуля, представление и управление вынести на форму через `СписокЗначений` и две команды.

**Rationale:**
- Сохраняет двухфазные инварианты (gate сравнивает значение, а не тип) → нулевой риск регрессии логики обмена.
- Не плодит новых объектов метаданных: константы/функции добавляются в **существующий** `рг_УниверсальныйОбменXMLСервер` (форма вызывает его в `ПриСозданииНаСервере`).
- Список значений на форме = единственный UI-источник статусов и подписей; кнопки берут значение из него.

## Simplicity Check

- **Viable alternatives:**
  1. **Строка(20) + коды-константы в существующем серверном модуле** (выбрано) — 0 новых объектов метаданных; правка enum→строка в 7 точках; список значений на форме.
  2. **Число + коды 0..3** — компактнее, но коды нечитаемы в журнале регистрации и в БД; всё равно нужна функция представления; миграция и текст ошибок усложняются без выигрыша.
  3. **Новый общий модуль `…КлиентСервер` под константы/список** — лишний объект метаданных (ручное создание в Конфигураторе) при том, что форма всё равно вызывает сервер в `ПриСозданииНаСервере`; не оправдано.
- **Selected simplest viable design:** вариант 1 — хранение Строка с человекочитаемыми кодами, SSOT в уже существующем модуле, список значений только как UI-слой.
- **Why not simpler:** убрать персистентный реквизит и «жить только списком значений» нельзя — WS читает статус на сервере без формы; gate без персистентного значения неработоспособен.
- **Complexity budget:** Files touched: 4 (+ авто-обновление Configuration.xml/ConfigDumpInfo.xml/Rights при выгрузке). Hooks/intercepts: 0 новых перехватов. New procedures/functions: ~4 экспортных (`СтатусНовое()`/коды, `ПредставлениеСтатуса()`, `ЗаполнитьСписокСтатусов()`, опц. `СтатусГотовКВыгрузке()`) + 2 обработчика команд формы. Conditional branches / feature flags: 0 флагов.

## Платформенная модель

### Почему список значений не может быть хранилищем

| Слой | Носитель | Контекст | Кто читает |
|------|----------|----------|------------|
| **Хранение (SSOT)** | реквизит `Статус` справочника | сервер + БД | WS gate, журнал, запросы |
| **Домен значений** | функции-константы общего модуля | сервер | gate, форма (через `ПриСозданииНаСервере`) |
| **Представление + выбор** | `СписокЗначений` на форме | клиент/сервер формы | пользователь, кнопки |

`СписокЗначений` обслуживает только верхний слой. Удаление перечисления требует выбрать примитивный тип для нижнего (хранение) и явный домен для среднего (константы).

### Тип реквизита `Статус` после удаления enum

**Решение: `Строка(20)`, переменная длина, со стабильными кодами, совпадающими с прежними именами значений перечисления:** `"Новое"`, `"Выполняется"`, `"Выполнено"`, `"Ошибка"`; «не готов» = **пустая строка** `""`.

Обоснование выбора Строки (а не Числа):
- читаемость в журнале регистрации и при разборе инцидентов (gate уже подставляет статус в текст ошибки — строка остаётся осмысленной);
- совпадение кодов с прежними синонимами enum минимизирует правку текста журнала/ошибок;
- защита от «магических строк» — единый модуль-источник кодов (см. ниже), прямого ввода кодов в коде нет.

Альтернатива (Latin-коды `"New"/"Running"/"Done"/"Error"`) — более «чистое» разделение код/подпись, но требует функцию представления во всех текстах журнала; для данной задачи выигрыш не оправдывает churn. Зафиксировано как возможная замена, не выбрано.

### Домен статусов — централизация (без нового объекта метаданных)

В существующий **`рг_УниверсальныйОбменXMLСервер`** добавить экспортные функции (контракты, не реализация):

| Функция | Назначение |
|---------|-----------|
| `СтатусНовое()` / `СтатусВыполняется()` / `СтатусВыполнено()` / `СтатусОшибка()` | возвращают строковые коды — SSOT, исключают магические строки |
| `СтатусНеГотов()` | возвращает `""` (состояние «не готов / отменён») |
| `ПредставлениеСтатуса(Код)` | код → UX-подпись («Готов к обмену» и т.д.) для журнала, текста ошибок, списка значений |
| `ЗаполнитьСписокСтатусов(СписокЗначений)` | наполняет переданный `СписокЗначений` парами (код, подпись) — единый источник списка для формы |
| `СтатусГотовКВыгрузке(Код)` (опц.) | предикат gate: `Код = СтатусНовое()` |

Форма вызывает `рг_УниверсальныйОбменXMLСервер.ЗаполнитьСписокСтатусов(...)` в `ПриСозданииНаСервере` (серверный контекст формы имеет доступ к серверному общему модулю).

## Дизайн формы и кнопок

### Состав списка значений на форме

Реквизит формы `СписокСтатусов` (тип `СписокЗначений`), наполняется в `ПриСозданииНаСервере` из `ЗаполнитьСписокСтатусов`:

| Значение (код, хранится) | Представление (подпись пользователю) |
|--------------------------|--------------------------------------|
| `""` | Не готов к обмену |
| `"Новое"` | Готов к обмену |
| `"Выполняется"` | Идёт обмен |
| `"Выполнено"` | Обмен завершён |
| `"Ошибка"` | Ошибка обмена |

Маппинг на gate: `"Новое"` = предусловие `GetData`; `"Выполняется"` = промежуточное (между GetData и Confirm); `"Выполнено"`/`"Ошибка"` = терминальные; `""` = не запускает обмен.

### Поле статуса

- `Объект.Статус` отображается **только для просмотра** (read-only) — текстом подписи из списка (через условное представление / отдельное поле-надпись). Прямое редактирование запрещено: единственные UI-мутаторы — две команды.
- Видимость — по-прежнему только на профиле-источнике (`ГруппаИсточник`).

### Поведение кнопок

| Кнопка | Пишет статус (из списка) | Доступна при | Действие |
|--------|--------------------------|--------------|----------|
| **Подготовить к обмену** | `"Новое"` (значение элемента «Готов к обмену») | `Статус ∈ {"", "Выполнено", "Ошибка"}` и правила/префиксы заполнены | устанавливает `Объект.Статус` из списка → `Записать()` |
| **Отменить обмен** | `""` (значение элемента «Не готов к обмену») | `Статус ∈ {"Новое", "Выполняется"}` | устанавливает `Объект.Статус = ""` → `Записать()` |

Паттерн «из списка значений» (буквально по требованию заказчика): обработчик команды находит нужный элемент в `СписокСтатусов` и берёт его `Значение`, например `СписокСтатусов.НайтиПоЗначению(рг_УниверсальныйОбменXMLСервер.СтатусНовое())` → присвоить `Объект.Статус`.

**Зависший «Выполняется».** «Отменить обмен» из «Выполняется» = санкционированный ручной сброс (соответствует Goals design.md «ручной сброс» и Non-Goals «авто-сброс — out of scope»). Перед сбросом — подтверждающий диалог («Сеанс в статусе "Идёт обмен". Сбросить готовность?»). «Подготовить к обмену» из «Выполняется» — недоступна (не затирать незавершённый сеанс), пользователь сначала «Отменить обмен».

**Защита инвариантов.** Поле read-only закрывает основной канал ручного обхода. Дополнительно (рекомендуется) — мягкий guard в `ПередЗаписью` **после** early-return по `ОбменДанными.Загрузка` (привилегированная запись WS не должна блокироваться): отклонять пользовательские переходы вне набора {`""`, `"Новое"`}. Серверные переходы (`Выполняется`/`Выполнено`/`Ошибка`) идут только через привилегированную запись WS и под early-return не попадают под guard.

## План миграции с enum

Реквизит `Статус` нельзя автоматически сконвертировать из `EnumRef` в `Строка`: при смене типа платформа очищает несовместимые значения. Два пути.

### Путь A — простой (рекомендуется при A1: профилей немного)
1. (Конфигуратор) Удалить enum, сменить тип `Статус` → `Строка(20)`, выгрузить cfe.
2. Существующие значения статусов очищаются в `""` («не готов»). Терминальные/пустые профили — без последствий; пользователь повторно жмёт «Подготовить к обмену».
3. Деплой — в окно без активного обмена (A2). Если на момент деплоя был «Выполняется» — после деплоя профиль окажется «не готов»; повторить сеанс с приёмника.
4. (BSL) Рефакторинг серверного модуля и формы.

### Путь B — с сохранением данных (если требуется сохранить in-flight статусы)
1. (Конфигуратор) Добавить временный реквизит `СтатусКод` (`Строка(20)`), оставив `Статус` (enum).
2. Разовая обработка/обработчик обновления: для каждого профиля `СтатусКод = <маппинг enum.Статус → код>`.
3. (BSL) Рефакторинг на `СтатусКод`.
4. (Конфигуратор) Удалить enum и старый `Статус`; при необходимости переименовать `СтатусКод` → `Статус` (чистое переименование, данные сохраняются).

**Порядок BSL/метаданных (общий):** правка `.bsl` (writer) согласуется с моментом смены типа — компиляция не должна ссылаться одновременно и на удаляемый enum, и на новый код. Практично: сначала добавить функции-константы и перевести сравнения на них (значения совпадают со старыми именами), затем удалить enum.

## Риски

| Риск | Эффект | Митигирование |

| Очистка значений при смене типа (Путь A) | in-flight «Выполняется»/«Выполнено» сбросятся в «не готов» | деплой в окно без обмена; либо Путь B |

| Рассинхрон кода и подписи | журнал/ошибки покажут служебный код | всё представление через `ПредставлениеСтатуса()`; коды только через функции-константы |
| Несинхронная правка enum/BSL | ошибка компиляции (ссылка на удалённый enum) | сначала перевести 7 точек на константы, затем удалить enum |
| Регрессия двухфазной логики | сбой обмена | логика не трогается — меняется лишь сравниваемое значение; покрыто приёмкой S2/S3 |

## Рекомендация

**Реализуемо ровно как просит заказчик по UX** (перечисления нет; список значений на форме; две кнопки пишут статус из списка). Неустранимый технический минимум: хранимый реквизит `Статус` становится текстовым кодом, а веб-сервис сравнивает его с централизованными константами — потому что список значений существует только на форме и не может быть полем БД, которое читает обмен.

**Следующий шаг:** зафиксировать решение в `design.md`/`tasks.md` (через `/opsx:extend universal-xml-exchange2` или в рамках apply) — пересмотр D1, миграция, рефакторинг 7 точек, форма с двумя командами.

## Для заказчика

- **Можно сделать как вы просите:** перечисление удаляем, статус выбирается из списка значений на форме, кнопки «Подготовить к обмену» и «Отменить обмен» проставляют статус из этого списка.
- **Одно техническое уточнение:** поле статуса в базе станет коротким текстовым кодом (список значений живёт только на форме и не может быть полем базы, которое читает обмен) — для вас это незаметно, подписи остаются человеческими («Готов к обмену», «Идёт обмен», …).
- **Важно при внедрении:** при удалении перечисления текущие значения статуса у профилей обнулятся — запланируем переключение в момент, когда не идёт обмен; если нужно сохранить статусы «на лету», есть безопасный вариант с временным полем и переносом данных.
