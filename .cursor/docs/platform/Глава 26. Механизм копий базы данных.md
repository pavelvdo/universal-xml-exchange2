## 26.1. Общая информация

- **Лицензия:** только КОРП (по видам лицензий — документация платформы).
- **Назначение:** создание и обновление копий данных в других физических БД; несколько копий одновременно; настраивается состав объектов конфигурации для каждой копии. Данные копии можно использовать в запросах и в системе компоновки данных (СКД).
- **Термин «таблица»** в контексте механизма — таблица СУБД, а не объект конфигурации (например, регистр накопления — несколько таблиц СУБД; переносятся все таблицы, связанные с объектом).
- **Не резервное копирование:** копия не является полноценной информационной базой, с ней нельзя работать как с обычной БД.
- **Не поддерживается** файловый вариант информационной базы.
- **Применение:** обязательный элемент службы «Дата акселератор»; вынос данных отчётов на отдельный сервер СУБД для разгрузки основной БД.
- **Синхронизация:** встроенный механизм платформы или сторонние средства (настраивается по копии). Средства СУБД для внешней копии допустимы только если: (1) копия используется только через механизм копий БД; (2) в копию помещаются все таблицы основной БД (полная копия). Иначе — только встроенная репликация «1С:Предприятие».
- **Разделение данных:** поддерживается; в состав копии могут входить таблицы объектов разделителей.
- **СУБД копии:** Microsoft SQL Server, Oracle Database, PostgreSQL. СУБД кластера и копии могут различаться (версия, разрядность, производитель).

## 26.2. Устройство механизма

В копии создаются таблицы — копии таблиц объектов информационной базы. Состав задаётся свойством **Состав** объекта копии (тип **МенеджерКопииБазыДанных**). Платформа отслеживает изменения в основной БД и по команде передаёт их в копии. Перенос — метод **КопииБазыДанных.Обновить()**. Обновление копий можно выполнять асинхронно, только после переноса в копию изменений, зафиксированных для переносимых таблиц.

**Этапы обновления (стандартная репликация):**

1. **Начальный перенос** в новую таблицу копии (в т.ч. для разделённых объектов — все области). Одно или несколько системных фоновых заданий. Используется после любого изменения структуры в копии (добавление/удаление реквизита или объекта).
2. **Текущий перенос** в существующую таблицу копии (для разделённых данных — в область по разделителям транзакции в основной БД). Без фоновых заданий. Переносятся все успешно завершённые изменения только для объектов, входящих в состав копии. При большом объёме или длинном интервале обновления вызов **КопииБазыДанных.Обновить()** может быть длительным.

**Внешний тип репликации** (свойство **МенеджерКопииБазыДанных.ТипРепликации**): платформа не создаёт таблицы в копии, не синхронизирует данные (**КопииБазыДанных.Обновить()** не нужен), не отвечает за параметры и наполнение копии; только направляет запросы на копию вместо основной БД. Несоответствие структуры копии и основной БД приводит к ошибке. Платформа считает данные такой копии всегда актуальными.

**Версии стандартной репликации** (перечисление **ВерсияСтандартнойРепликацииКопийБазыДанных**): методы **ПолучитьВерсиюСтандартнойРепликации()** / **УстановитьВерсиюСтандартнойРепликации()** менеджера копий БД.
- **Версия1:** отслеживание изменений средствами кластера серверов «1С:Предприятие».
- **Версия2:** отслеживание изменений средствами СУБД основной БД.

При смене версии сбрасывается состояние всех копий со стандартным типом репликации; при следующем обновлении выполняется по сути первоначальное заполнение.

**Версия2, поддержка по СУБД:**
- **Microsoft SQL Server 2008+:** Change Data Capture (CDC). Включается автоматически, если пользователь кластера — член роли sysadmin; иначе — включение вручную (команды в основной БД от имени sysadmin). Требуется работа SQL Server Agent. При ошибке `sp_cdc_create_change_table` (например, из‑за высокой нагрузки) в копию попадут только таблицы, уже добавленные в CDC; в сообщении перечисляются объекты, не попавшие в копию; повторить обновление после снижения нагрузки. В SQL Server 2019 CU 20, 2022 CU 2 и новее при включённом DELAYED_DURABILITY использование CDC невозможно — синхронизация будет с ошибкой; нужно отключить отложенные транзакции в основной БД (документация Microsoft SQL).
- **PostgreSQL 10+:** логическое декодирование. Нужно: СУБД из дистрибутива «1С» 10 или новее; в `postgresql.conf` задать `wal_level=logical`; перезапуск СУБД; в стандартной функции управления копиями включить Версию2. Для PostgreSQL 9 и старше всегда используется Версия1. Если СУБД не из дистрибутива «1С», при настройке Версии2 фактически может использоваться Версия1. Если выбрана Версия2, а СУБД основной БД её не поддерживает, используется Версия1.

**Использование копий в запросах и СКД.** Свойства объектов **Запрос** и **ЗначенияПараметровВыводаКомпоновкиДанных**:
- **ИспользованиеКопийБазыДанных:** НеИспользоватьКопии | ИспользоватьТолькоКопии | ИспользоватьПреимущественноКопии | Авто (трактуется как ПреимущественноКопии).
- **ИспользуемыеКопииБазыДанных:** список копий, на которых может выполниться запрос/компоновка.

**Алгоритм выбора копии для запроса:**
1. Все таблицы запроса (источники, «через точку», таблицы в условиях RLS) должны быть в одной копии.
2. Актуальность данных копии удовлетворяет настройкам (см. ниже).
3. Если в копии задан отбор по таблице — условие в запросе должно соответствовать отбору; иначе запрос идёт в основную БД или в копию без отбора.
4. Из подходящих копий выбираются с наименьшей загрузкой (по числу запросов текущего рабочего процесса).
5. Среди них выбор случайный.

**Актуальность.** Свойства **ТребуемаяАктуальностьДанных** и **ТребуемоеВремяАктуальностиДанных** (Запрос и параметр вывода СКД):
- **ПреимущественноКопии:** Актуальные — данные только из основной БД; Любые — копия со всеми нужными таблицами и требуемой актуальностью, иначе выполнение в основной БД.
- **ИспользоватьТолькоКопии:** Актуальные — копия пригодна, если в основной БД нет неперенесённых изменений; Любые — копия с требуемой актуальностью, иначе исключение.
- **Авто:** для Запроса — Актуальные, для СКД — Любые.

**ТребуемоеВремяАктуальностиДанных** задаётся: целым (интервал в секундах от момента выполнения запроса); датой/временем (конкретный момент); стандартной датой начала (интервал привязан к дате, меняется автоматически).

**Результат запроса / СКД:** **РезультатЗапроса** — свойства **ВремяАктуальности**, **КопияБазыДанных**. В языке выражений СКД — функции **ДанныеАктуальны()**, **ВремяАктуальностиДанных()**, **КопияБазыДанных()**. Параметры вывода СКД: **ВыводитьАктуальностьДанных**, **ВыводитьКопиюБазыДанных**. При вложенных схемах компоновки каждая схема может выполняться на основной БД или на подходящей копии. При нескольких подходящих копиях выбор произвольный, механизма приоритета нет.

## 26.3. Дата акселератор

- **Суть:** СУБД в оперативной памяти рабочего сервера кластера (in-memory). Работает только с механизмом копий БД и только на 64-разрядной ОС.
- **Лицензии:** несколько экземпляров в кластере — КОРП; один экземпляр — ПРОФ. Управление числом экземпляров — требования назначения функциональности.
- **Сценарий:** аналитические запросы с большим объёмом чтения в СУБД и малым результатом; в иных сценариях время выполнения может быть больше, чем на основной БД.
- **Порядок настройки:** настройка рабочего сервера → создание копии БД с признаком **Встроенный Дата акселератор** (всегда встроенная синхронизация) → в копию рекомендуется включать только таблицы, нужные для критичных отчётов, а не всю БД → начальное обновление копии → после обновления отчёты могут использовать акселератор.
- **Перезапуск рабочего сервера:** данные в ОЗУ теряются; после перезапуска обязательно выполнить начальное обновление копии. Настройка расписания актуализации — на разработчике.
- **Ограничения:** реквизиты более 4 КБ не поддерживаются — при чтении возвращаются пустые значения; плюс ограничения общего механизма копий БД.

## 26.4. Особенности и ограничения

- **Запрос не выполняется на копии**, если в сеансе открыта транзакция или у объекта **Запрос** установлен менеджер временных таблиц.
- **Ошибка при обращении к копии:** копия исключается из использования (все сеансы) до обновления копии или перезапуска кластера. Для отчёта СКД с несколькими наборами данных или иерархическим набором пользователю выдаётся ошибка; в остальных случаях запрос повторно выполняется на основной БД или другой доступной копии (без проблемной).
- **ВерсияДанных** в копии может не совпадать с основной БД; для корректного значения использовать только актуальные данные (ТребуемаяАктуальностьДанных = Актуальные).
- **Состав копии (настройка):** новые таблицы добавляются в копию после записи **МенеджерКопииБазыДанных** и вызова **КопииБазыДанных.Обновить()**; удаление таблиц из состава выполняется сразу после записи **МенеджерКопииБазыДанных**, без ожидания обновления копии.
- **Права:** в базе-источнике — администрирование копий, состав, обновление; для каждого синхронизируемого объекта — Чтение, Добавление, Изменение, Удаление (рекомендуется отдельный пользователь для синхронизации). Пользователь подключения к СУБД копии должен иметь права создавать/изменять/удалять таблицы и читать/создавать/изменять/удалять данные во всех таблицах копии.
- **Расширения с расширением данных:** не поддерживаются; таблицы таких объектов в состав копии не включаются; запросы по ним на копии не выполняются.

## 26.5. Управление составом копии

1. **КопииБазыДанных.Добавить()** — создание копии; **КопииБазыДанных.Найти()** — описание существующей.
2. Настройка состава таблиц и полей через **МенеджерКопииБазыДанных**.
3. **МенеджерКопииБазыДанных.Записать()** — сохранение состава в основной БД.
4. **КопииБазыДанных.Обновить(**копия**)** — обновление данных и структуры таблиц в копии.

Состав таблиц — **МенеджерКопииБазыДанных.Состав**. Управление полями (структурой таблиц копии):
- Справочники/документы и т.п.: реквизиты, табличные части и их реквизиты; стандартные и общие реквизиты разделения всегда в копии.
- Регистры: ресурсы и реквизиты; измерения всегда в копии.

**Способы настройки полей:** (1) свойства по видам реквизитов (рекомендуется); (2) пореквизитная настройка.

**Свойства МенеджерКопииБазыДанных и ЭлементСоставаКопииБазыДанных:** ИспользованиеХранилищЗначений, ИспользованиеСтрокНеограниченнойДлины, ИспользованиеПрочихПолей, ИспользованиеТабличныхЧастей. Реквизиты табличных частей подчиняются тем же трём свойствам по типам. **ПолеЭлементаСоставаКопииБазыДанных.Использование** — пореквизитное включение/выключение.

**Перечисление ИспользованиеПоляЭлементаСоставаКопииБазыДанных:** Использовать | НеИспользовать | Авто. При Авто цепочка определения: ПолеЭлементаСоставаКопииБазыДанных.Использование → ЭлементСоставаКопииБазыДанных (по типу поля: табличная часть, ХранилищеЗначения, строка неограниченной длины, прочие) → МенеджерКопииБазыДанных → умолчания платформы. **Стандартная репликация:** табличные части и ХранилищеЗначений по умолчанию НеИспользовать, остальное — Использовать. **Внешняя репликация:** по умолчанию Использовать для всех. Фактический статус использования реквизита — метод **ПолеЭлементаСоставаКопииБазыДанных.ПолучитьТекущееИспользование()**.

## 26.6. Использование базы данных копии

**Сущности:** база данных (набор данных в СУБД); информационная база (определение платформы); копия базы данных (хранение и доступ к копии + объект **МенеджерКопииБазыДанных**). ИБ и копия хранят данные в своих БД; одна БД может быть подключена к разным копиям (одной или разных ИБ); копия может быть подключена к БД информационной базы — риски разрушения данных. Для снижения рисков — интерфейс привязки БД к копии.

- **УстановитьИспользованиеБазыДанныхКопииЭтойКопией()** (менеджер копии БД) — «закрепить» БД копии за данным экземпляром копии.
- **ПолучитьИнформациюОбИспользованииБазыДанныхКопии()** (менеджер копий БД) — по параметрам БД, имени копии или **МенеджерКопииБазыДанных** возвращает **ИнформацияОбИспользованииБазыДанныхКопии**: свойство **ВариантИспользования** (перечисление **ВариантИспользованияБазыДанныхКопии**), **Описание** (строка подключения). Значения: ИспользованиеНеизвестно (можно размещать данные копии), ИспользуетсяЭтойКопией, ИспользуетсяДругойКопией (нужен вызов УстановитьИспользование… для переключения на текущую ИБ), ИспользуетсяИнформационнойБазой (в БД нельзя размещать копию).

**Обновить()** менеджера копий БД проверяет возможность работы каждой копии. Одна БД может быть связана с несколькими копиями (в одной или разных ИБ); при дублировании ИБ для сохранения работы копий нужно дублировать и все БД, используемые этими копиями.

## 26.7. Управление копиями

Управление списком копий, составом и обновлением — стандартная функция «Управление копиями базы данных» в режиме 1С:Предприятие (описание в справке по этой функции).
