# Исследование: подключение к веб-сервису и поиск профиля по префиксам

**Change:** `universal-xml-exchange2`  
**Дата:** 2026-06-26  
**Тип:** Focused Investigation (explore-bug)

---

## Свод

Ошибка «Профиль-источник не найден по префиксам "ИСТ1" / "ПРИЕМ1"» возникает **на стороне базы, где опубликован веб-сервис `rgExchangeService`**, в функции `НайтиЕдинственныйПрофильИсточникаПоПрефиксам`: в **локальном** справочнике `рг_НастройкиОбменаXML` этой базы нет элемента с `Источник = Истина` и парой префиксов `ИСТ1` / `ПРИЕМ1`.

Команда «Выполнить обмен» запускается **только на приёмнике** (база C): приёмник по профилю с `Источник = Ложь` вызывает удалённый `GetData` с префиксами из своего профиля. Поиск профиля-источника выполняется **не на приёмнике и не на «отдельной» базе B**, а в той ИБ, куда указывает `АдресСервиса` (где обрабатывается SOAP-запрос).

При топологии «база A — только WS, база B — профиль-источник, база C — приёмник» ошибка **ожидаема**: профиль лежит в B, а `GetData` ищет его в A. По design v2 WS и профиль-источник должны быть **в одной** базе-источнике.

Пара `ИСТ1` / `ПРИЕМ1` в тексте ошибки означает, что приёмник передал **корректную** пару (не сценарий «ПРИЕМ1 / ПРИЕМ1» из verify-2026-06-25-12).

---

## Для заказчика

**Что происходит.** Вы нажимаете «Выполнить обмен» на базе-приёмнике. Программа подключается к опубликованному веб-сервису и просит у «источника» пакет данных, передавая два кода: префix базы-источника и префix базы-приёмника. На сервере, где висит веб-сервис, ищется настройка «я — источник» с такой же парой кодов. Если её там нет — вы видите эту ошибку.

**Почему ошибка при трёх базах.** Если веб-сервис опубликован на базе A, а профиль-источник создан только на базе B, сервер на A «не знает» про профиль из B — справочники у файловых баз разные. Код не ходит в B за профилем; он смотрит только свою базу.

**Что проверить (по порядку).**

1. **Где опубликован WS** — откройте базу, URL которой указан в `АдресСервиса` профиля-приёмника (база C). Именно **в этой** базе в справочнике `рг_НастройкиОбменаXML` должен быть элемент: `Источник = Да`, `Префикс = ИСТ1`, `Префикс базы приёмника = ПРИЕМ1`, загружены правила, статус **«Новое»**.
2. **Свести топологию к двум ролям:** либо перенести публикацию WS на базу B (источник), либо завести профиль-источник на базе A (дублирование — не рекомендуется).
3. **Профиль на приёмнике (C):** `Источник = Нет`; `АдресСервиса` → URL базы с WS **и** профилем-источником; в реквизитах **`Префикс = ИСТ1`** и **`ПрефиксБазыПриемника = ПРИЕМ1`** (на форме приёмника видно только одно поле «Префикс» — оно пишет в `ПрефиксБазыПриемника`; `Префикс` источника на форме скрыт — см. раздел «Форма профиля»).
4. **Статус на источнике** перед обменом — «Новое» (иначе будет другая ошибка, не «не найден»).

**Вердикт шага:** уточнение / готово к решению — дефект конфигурации стенда (расслоение WS и профиля-источника), не логики передачи префиксов при корректной паре `ИСТ1`/`ПРИЕМ1`.

## Алгоритм поиска профиля по префиксам

### Где ищется

**Verified:** только в **текущей информационной базе** сеанса, обрабатывающего вызов веб-сервиса. Удалённых запросов к другим ИБ нет.

Функция `НайтиЕдинственныйПрофильИсточникаПоПрефиксам` (`Module.bsl:105–193`):

```bsl
// Условие запроса (строки 123–127):
//   Источник = ИСТИНА
//   Префикс = &Префикс
//   ПрефиксБазыПриемника = &ПрефиксБазыПриемника
```

### Откуда берутся параметры `ИСТ1` / `ПРИЕМ1`

| Контекст | Источник значений |
|----------|-------------------|
| Вызов `GetData` с приёмника | Реквизиты **профиля-приёмника** в базе C: `ПодготовитьДанныеСеансаПриемника` (`:1003–1005`) → аргументы `GetData` (`:1029`) |
| Прямой вызов WS | SOAP-параметры `pref`, `prefReceive` → те же формальные имена в `GetData` WS-модуля |

Семантика (комментарии к функции, `:93–95`, `:273–275`):

- **`Префикс`** — префикс базы-**источника** (в примере `ИСТ1`)
- **`ПрефиксБазыПриемника`** — префикс базы-**приёмника** (в примере `ПРИЕМ1`)

### Что **не** участвует в поиске

**Verified:** в запросе **нет** фильтра по `Статус`, `ПометкаУдаления` (если не включена платформенная RLS), `Наименование`, `АдресСервиса`. Статус проверяется **после** нахождения профиля в `НачатьВыгрузкуGetData` (`:527–545`).

### Исходы поиска

| Результат | Поведение | Строки |
|-----------|-----------|--------|
| 0 записей | Ошибка «Профиль-источник не найден…» | `:141–155` |
| 1 запись | Возврат ссылки | `:163–189` |
| 2+ записей | Ошибка «Найдено более одного…» | `:167–181` |

### Форма профиля и риск настройки приёмника

**Verified:** в `Form.xml` группа `ГруппаПриемник` содержит **одно** поле `Префикс1` с `DataPath = Объект.ПрефиксБазыПриемника` (строки 201–206). Реквизит **`Префикс`** (префикс источника) на форме приёмника **скрыт** (он в `ГруппаИсточник`, видимость только при `Источник = Истина`, `Form/Module.bsl:247–250`).

На приёмнике для `GetData` нужны **оба** реквизита (`exchange-settings/spec.md`, requirement «Префиксы идентификации сеанса»). Если пользователь заполнил только видимое поле и не задал `Префикс = ИСТ1`, возможны другие пары в ошибке. Текущий симптом `ИСТ1`/`ПРИЕМ1` указывает, что **оба** реквизита на приёмнике, скорее всего, заданы верно.

## Сверка с топологией 3 баз

| База | Роль по описанию пользователя | Что требует as-built / design |
|------|------------------------------|-------------------------------|
| **A** | Публикация `rgExchangeService` | WS **и** профиль-источник с `Источник=Да`, правилами, статусом — **в одной ИБ** (`design.md` D6) |
| **B** | Профиль-источник `ИСТ1`/`ПРИЕМ1` | Корректное место для источника **если** здесь же опубликован WS |
| **C** | Профиль-приёмник `Префикс=ПРИЕМ1` | Должен быть `Источник=Нет`, `АдресСервиса` → URL базы **с WS и профилем-источником**, **`Префикс=ИСТ1`**, **`ПрефиксБазыПриемника=ПРИЕМ1`** |

### Ожидаемое распределение (design / proposal)

**Verified по `design.md` D6, `exchange-settings/spec.md`:**

| ИБ | Профиль-источник | Профиль-приёмник | WS |
|----|------------------|------------------|-----|
| База-источник | Да (правила, параметры, статус, префиксы) | Нет | Опубликован |
| База-приёмник | Нет | Да (URL, логин, пароль, **та же пара префиксов**) | Не нужен |

Штатная схема — **2 файловые базы**, не 3. Третья база «только шлюз WS» без профиля-источника **не поддерживается** текущим кодом.

### Диаграмма фактического потока

```
[База C — приёмник]
  «Выполнить обмен» / регламент
       │
       ├─ читает профиль-приёмник (Источник=Ложь)
       ├─ WSПрокси → АдресСервиса (база A?)
       │
       ▼
[База публикации WS — A или B]
  rgExchangeService.GetData(ИСТ1, ПРИЕМ1)
       │
       ├─ ЗАПРОС в Справочник.рг_НастройкиОбменаXML ЭТОЙ базы
       │     WHERE Источник И Префикс И ПрефиксБазыПриемника
       │
       ├─ НЕ найден → ошибка строка 155  ◄── симптом пользователя
       └─ найден → выгрузка ZIP → ответ
       │
       ▼
[База C] загрузка ZIP → ConfirmDataReceipt → снова WS → смена статуса на источнике
```

## Корневая причина ошибки пользователя

### Verified (основная)

**Профиль-источник с парой `ИСТ1`/`ПРИЕМ1` отсутствует в справочнике той информационной базы, где выполняется `GetData`** (база публикации веб-сервиса).

Доказательства:

1. Текст ошибки формируется только при пустой выборке запроса с `Источник = Истина` и переданными префиксами (`Module.bsl:141–155`).
2. Префиксы в ошибке — те, что приёмник передал в `GetData` (`:1029`); для пары `ИСТ1`/`ПРИЕМ1` приёмник, вероятно, настроен согласованно с источником.
3. Поиск локальный — нет обращения к базе B из кода WS (`:109–127`).
4. Топология пользователя (WS на A, профиль на B) противоречит D6 design («источник = сервер + публикация WS»).

**Что опровергнет:** в базе A (URL из `АдресСервиса`) в списке справочника есть элемент с `Источник=Да`, `Префикс=ИСТ1`, `ПрефиксБазыПриемника=ПРИЕМ1`, не помечен на удаление.

### Hypothesis (дополнительные сценарии при той же ошибке)

| # | Гипотеза | Подтвердит | Опровергнет |
|---|----------|------------|-------------|
| H1 | На базе WS профиль есть, но `Источник = Ложь` | В ИБ WS элемент с нужными префиксами и `Источник=Нет` | `Источник=Да` |
| H2 | Профиль помечен на удаление / недоступен по RLS | Запрос в консоли от пользователя WS не видит строку | Запрос находит строку |
| H3 | Опечатка в префиксах на источнике (`ИСТ1 ` с пробелом и т.п.) | Реквизиты источника отличаются от переданных | Точное совпадение строк |
| H4 | `АдресСервиса` на C указывает не на ту базу, где лежит профиль B | URL ведёт на A без профиля | URL ведёт на B с профилем |

## Рекомендации

### Настройка (предпочтительно, без правки кода)

1. **Объединить WS и источник:** опубликовать `rgExchangeService` на **базе B** (где профиль-источник). В профиле-приёмника на C указать URL публикации B.
2. **На B — профиль-источник:** `Источник=Да`, `Префикс=ИСТ1`, `ПрефиксБазыПриemника=ПРИЕМ1`, правила в хранилище, `Статус=Новое`, пользователь WS с правами расширения.
3. **На C — профиль-приёмник:** `Источник=Нет`, `АдресСервиса`, логин/пароль; **`Префикс=ИСТ1`** (через конфигуратор или временно включив «Источник», если поле недоступно на форме); **`ПрефиксБазыПриемника=ПРИЕМ1`** (поле на форме приёмника).
4. **База A:** либо не использовать в контуре обмена, либо перенести на неё **копию** профиля-источника (нежелательно — два источника правды).
5. **Перед повтором:** на источнике статус «Новое»; в журнале регистрации событие `рг.УниверсальныйОбменXML` на базе WS.

### Правка кода (опционально, не для устранения текущего симптома)

| Улучшение | Зачем |
|-----------|--------|
| На форме приёмника показать **оба** префикса с подписями «Префикс источника» / «Префикс приёмника» | Снизить риск H4 и сценария «ПРИЕМ1/ПРИЕМ1»; закрыть gap формы |
| Проверка при записи профиля-приёмника: оба префикса заполнены | Сейчас `ПередЗаписью` требует префиксы только для источника (`ObjectModule.bsl:12–48`) |
| Текст ошибки: «проверьте профиль-источник **в базе публикации веб-сервиса**» | UX при трёх базах |

Текущая ошибка при корректной паре `ИСТ1`/`ПРИЕМ1` **не требует** изменения алгоритма поиска — требуется выравнивание стенда с двухролевой моделью.
