# Verification — три файловые ИБ: профиль не найден

## Корневая причина (verified)

Публикация WS и поиск профиля выполняются в **одной и той же информационной базе** — той, которая указана в каталоге публикации `test_zup`.

У заказчика:
- **ИБ-1** — опубликован `rgExchangeService` (`localhost/test_zup`)
- **ИБ-2** — профиль-источник (`Источник=Да`, `ИСТ1`/`ПРИЕМ1`)
- **ИБ-3** — профиль-приёмник, вызывает WS

`GetData` на сервере 1С читает **`Справочник.рг_НастройкиОбменаXML` только ИБ-1**. Профиль в ИБ-2 для WS **невидим** → ошибка «не найден по префиксам».

## Цепочка в коде

1. **Приёмник (ИБ-3):** `ВызватьGetDataНаИсточнике` → WS-прокси по `АдресСервиса` из профиля → `GetData(Префикс, ПрефиксБазыПриемника)`.
2. **Сервер публикации (ИБ-1):** `rgExchangeService.GetData` → `ПолучитьДанныеОбмена` → `НайтиЕдинственныйПрофильИсточникаПоПрефиксам` — запрос к справочнику **текущей** ИБ.
3. Выгрузка данных — из правил/параметров найденного профиля **в этой же ИБ**.

Межбазового обращения к справочнику другой файловой ИБ **нет**.

## Допустимая топология (2 ИБ)

| ИБ | Роль |
|----|------|
| **Источник** | Профиль `Источник=Да`, правила, статус «Новое»; **здесь же** публикация WS |
| **Приёмник** | Профиль `Источник=Нет`, адрес WS → источник, префиксы |

## Варианты исправления стенда

**A (рекомендуется):** опубликовать `test_zup` из **ИБ-2** (источник). ИБ-1 не использовать или удалить публикацию с неё.

**B:** перенести профиль-источник (и данные для выгрузки) в **ИБ-1** — та, что уже опубликована.

**C:** одна ИБ = источник + публикация; вторая = только приёмник.

## Профиль на приёмнике (ИБ-3)

Помимо адреса WS: `Префикс=ИСТ1`, `ПрефиксБазыПриемника=ПРИЕМ1` (поле формы «Префикс» → второй реквизит).

## Следующий шаг

Перепубликовать WS из ИБ-источника **или** создать профиль-источник в ИБ публикации; в профиле приёмника указать новый URL.
