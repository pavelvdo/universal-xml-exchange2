| Обработка `ргУниверсальныйОбменXML` (макет `ДвижокОбмена`) | S2.2 | частичное (см. ниже) |

**Не отражено в таблице Manual Configuration (есть в tasks как «Ручное конфигурирование», но не в таблице design):**

1. **S1.3 — форма `ФормаЭлемента`:** группы реквизитов источника/приёмника, нехранимый реквизит и команда загрузки XML правил в `ПравилаОбменаДанными`, обработчик `ИсточникПриИзменении`. В design упоминается только в D6 прозой; в таблице Manual Configuration строки нет. Без неё конфигурирующий по таблице пропустит вёрстку формы и команду загрузки правил — а это primary acceptance S1 («форма отображает реквизиты по роли», «загрузка правил пользователем»).
2. **S1.5 / S2.6 — права роли `рг_ИспользованиеОбменаXML`:** чтение перечисления, новые реквизиты, обработка, веб-сервис. В таблице отсутствуют.

Рекомендация: дополнить таблицу `## Manual Configuration` строками: форма `ФормаЭлемента` (группы + команда загрузки правил), роль `рг_ИспользованиеОбменаXML` (S1.5, S2.6), форма запуска обработки, профиль безопасности (S2.4).

Самостоятельных фантомов в BSL (вызовов объектов/реквизитов, не покрытых ручной задачей) **не обнаружено**. Все ссылки BSL ведут либо на существующие as-built объекты, либо на объекты с предшествующей ручной задачей создания.

**GAP-1 — feasibility S2.1 (epf API как статическая agent-задача).**
S2.1 требует «верифицировать по коду/макету публичный API `V8Exch82.epf` … зафиксировать имена методов в design.md (agent-задача, не user-spike)». Однако:
- макет `ДвижокОбмена` и сам файл `V8Exch82.epf` отсутствуют в `src/` (обработка ещё phantom, S2.2);
- `*.epf` — бинарный контейнер; извлечение публичного API экспортных методов не выполняется чтением/grep по репозиторию без открытия в Конфигураторе/распаковки.

design это частично признаёт: § Engine Contract помечает имена методов как **Assumption до apply S2**. Но в текущем виде S2.1 как «чистая static agent-задача» нереализуема инструментами агента — фактически это либо распаковка epf пользователем в исходники, либо короткий конфигураторный спайк. Поскольку S2.9 и S2.15 (фаза 2 выгрузки и загрузка на приёмнике) **блокированы** S2.1, нерешённый контракт epf — главный риск готовности S2.

User Task Contract («No user runtime-spike в S<N>.<M>») формально соблюдён: ни одна задача не содержит пользовательского рантайм-спайка. Но S2.1, заявленная как agent-static, по факту требует артефакта (исходники epf), которого нет. Рекомендация до apply S2: добавить ручную задачу «распаковать/выгрузить `V8Exch82.epf` в исходный вид (внешняя обработка → модуль)» или зафиксировать конкретный способ получения API epf, чтобы S2.1 стала исполнимой агентом.
