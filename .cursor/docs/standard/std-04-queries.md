## Оформление текстов запросов

#std437

1. Все ключевые слова языка запросов пишутся заглавными буквами.

2. Рекомендуется указывать и необязательные конструкции запроса, прежде всего — явно назначать псевдонимы полям, в целях повышения наглядности текста запроса и устойчивости использующего его кода. Например, если в алгоритме используется запрос с полем, объявленным как `Касса. Валюта`, при изменении имени реквизита нужно будет также изменить и код, осуществляющий обращение по имени свойства Валюта к выборке из результата запроса. Если же поле будет объявлено как `Касса. Валюта КАК Валюта`, то изменение имени реквизита приведет только к изменению текста запроса.
   - 2а. Особенно внимательно следует относиться к автоматически присваиваемым псевдонимам для полей-реквизитов других полей, типа «… Касса. Валюта. Наименование …». В приведенном выше примере поле получит автоматический псевдоним ВалютаНаименование, а не Наименование.
   - 2б. Следует обязательно указывать ключевое слово КАК перед псевдонимом поля источника.

3. Текст запроса должен быть структурирован, не следует писать запрос в одну строку, даже короткий. Текст запроса должен быть нагляден, поскольку это существенно улучшает его понимание другими разработчиками.

4. В запросы, сложные для понимания, в которых используются вложенные запросы, объединения или соединения, рекомендуется вставлять комментарии. Комментарии, например, могут объяснять, для получения каких данных используется та или иная таблица в соединении или объединении. При этом необходимо иметь в виду, что при использовании конструктора запросов все комментарии в запросе удаляются автоматически без предупреждения.

5. При создании объекта Запрос рекомендуется указывать комментарии, для получения какой информации или каких иных целей будет использован данный запрос.

6. При программной «сборке» текста запроса рекомендуется комментировать все этапы его сборки. Нужно стараться, чтобы каждая часть формируемого запроса могла быть открыта с помощью конструктора запросов: это позволяет осуществить экспресс-проверку корректности синтаксиса запроса и упрощает разработку и сопровождение кода конфигурации, в том числе сторонними разработчиками.

### Типичные случаи программной модификации текста запроса

**Изменение имени поля выборки или таблицы**

```
Неправильно:
ТекстЗапроса = "ВЫБРАТЬ | Номенклатура." + ИмяПоляКод + " КАК КодАртикул | ИЗ Справочник.Номенклатура КАК Номенклатура";

Правильно:
ТекстЗапроса = "ВЫБРАТЬ | Номенклатура.Наименование КАК Наименование, | &ИмяПоляКод КАК КодАртикул | ИЗ Справочник.Номенклатура КАК Номенклатура";
ТекстЗапроса = СтрЗаменить(ТекстЗапроса, "&ИмяПоляКод", "Номенклатура." + ИмяПоляКод);
```

Или для имени таблицы:
```
ТекстЗапроса = "ВЫБРАТЬ | ТаблицаСправочника.Наименование КАК Наименование, | ТаблицаСправочника.Код КАК Код | ИЗ &ТаблицаСправочника КАК ТаблицаСправочника";
ТекстЗапроса = СтрЗаменить(ТекстЗапроса, "&ТаблицаСправочника", "Справочник." + ИмяСправочника);
```

**Конкатенация нескольких текстов запросов в пакет**

```
Неправильно:
ТекстЗапроса = "";
Если ИспользоватьУпаковки Тогда ТекстЗапроса = "ВЫБРАТЬ ... ИЗ Справочник.Упаковки ...; |////////////////////////////////////////////| "; КонецЕсли;
ТекстЗапроса = ТекстЗапроса + "ВЫБРАТЬ ... ИЗ Справочник.Номенклатура ...";

Правильно:
ТекстЗапроса = "";
Если ИспользоватьУпаковки Тогда ТекстЗапроса = "ВЫБРАТЬ ... ИЗ Справочник.Упаковки ..."; ТекстЗапроса = ТекстЗапроса + " |; |////////////////////////////////////////////| "; КонецЕсли;
ТекстЗапроса = ТекстЗапроса + "ВЫБРАТЬ ... ИЗ Справочник.Номенклатура ...";
```

## Проверка на пустой результат выполнения запроса

#std438

1. Проверку того, что результат выполнения запроса не содержит строк, следует выполнять с помощью метода `Пустой()`. Поскольку на получение выборки из результата запроса (выгрузка его в таблицу значений) будет затрачиваться дополнительное время.

Неправильно:
```
Выборка = Запрос.Выполнить().Выбрать();
Если Выборка.Следующий() Тогда Возврат Истина; Иначе Возврат Ложь; КонецЕсли;
```

Правильно:
```
Возврат НЕ Запрос.Выполнить().Пустой();
```

2. Если требуется выбрать (или выгрузить) результат запроса, предварительный вызов метода `Пустой()` не требуется. Правильно: `Выборка = Запрос.Выполнить().Выбрать(); Пока Выборка.Следующий() Цикл ...`

## Ограничение на использование конструкции «ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ» в запросах

#std435

1.1. При разработке текстов запросов следует иметь в виду, что при работе в клиент-серверном варианте, когда в качестве СУБД используется PostgreSQL, производительность выполнения запросов с конструкцией ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ значительно снижается. В особенности это касается случаев, когда в запросе встречаются две и более таких конструкций. Поэтому в общем случае не рекомендуется использовать конструкцию ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ в запросах. И в тех случаях, где это возможно, рекомендуется переписать текст исходного запроса без использования этой конструкции.

Пример: запрос
```
ВЫБРАТЬ ЕСТЬNULL(ПланПродаж.Номенклатура, ФактическиеПродажи.Номенклатура) КАК Номенклатура, ...
ИЗ ПланПродаж КАК ПланПродаж ПОЛНОЕ СОЕДИНЕНИЕ ФактическиеПродажи КАК ФактическиеПродажи ПО ...
```
может быть реализован без конструкции ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ с помощью ОБЪЕДИНИТЬ ВСЕ и СГРУППИРОВАТЬ ПО.

1.2. Исключение составляют случаи, когда текст исходного запроса не может быть переписан без использования конструкции ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ по объективным причинам. При выполнении данной конструкции на СУБД PostgreSQL она автоматически заменяется платформой 1С:Предприятие на эквивалентную. В таких случаях не следует механически заменять конструкцию только с целью избавиться от неё в тексте запроса.

2. Не допускается одновременно использовать конструкцию ПОЛНОЕ СОЕДИНЕНИЕ и обращение к табличным частям из раздела ВЫБРАТЬ. Данное требование продиктовано особенностью выполнения подобных запросов на СУБД PostgreSQL и необходимостью переносимости прикладных решений на эту СУБД.

## Использование ключевых слов «ОБЪЕДИНИТЬ» и «ОБЪЕДИНИТЬ ВСЕ» в запросах

#std434

В общем случае при объединении в запросе результатов нескольких запросов следует использовать конструкцию ОБЪЕДИНИТЬ ВСЕ, а не ОБЪЕДИНИТЬ. Поскольку во втором варианте при объединении запросов полностью одинаковые строки заменяются одной, на что затрачивается дополнительное время, даже в случаях, когда одинаковых строк в запросах заведомо быть не может. Исключением являются ситуации, когда выполнение замены нескольких одинаковых строк одной является необходимым условием выполнения запроса.

Правильно:
```
ВЫБРАТЬ ПоступлениеТоваровУслуг.Ссылка ИЗ Документ.ПоступлениеТоваровУслуг КАК ПоступлениеТоваровУслуг
ОБЪЕДИНИТЬ ВСЕ
ВЫБРАТЬ РеализацияТоваровУслуг.Ссылка ИЗ Документ.РеализацияТоваровУслуг КАК РеализацияТоваровУслуг
```

Неправильно:
```
... ОБЪЕДИНИТЬ
ВЫБРАТЬ РеализацияТоваровУслуг.Ссылка ...
```

## Упорядочивание результатов запроса

#std412

1.1. Если алгоритм обработки результатов запроса зависит от порядка записей в запросе или если результат обработки запроса в той или иной форме представляется пользователю, то в тексте запроса следует использовать предложение УПОРЯДОЧИТЬ ПО. В отсутствие УПОРЯДОЧИТЬ ПО невозможно сделать никаких предположений о порядке записей в результатах запроса. Типичные примеры проблем: разная последовательность строк табличной части при заполнении по результатам запроса; разный порядок вывода данных в отчетах; разное заполнение движений документа по результатам запроса. Вероятность возникновения разных результатов повышается при переносе ИБ на другую СУБД, при смене версии СУБД, при изменении параметров СУБД.

*Примечание:* упорядочивание результатов запросов, по которым формируются движения, оправдано только если упорядочивание является частью алгоритма формирования движений (например, списание остатков партий по FIFO). В остальных случаях упорядочивать записи не следует.

1.2. При сортировке по полю, которое может содержать NULL, следует учитывать, что в разных СУБД порядок сортировки по этому полю может отличаться. Рекомендуется использовать ЕСТЬNULL() для поля в УПОРЯДОЧИТЬ ПО.

1.3. Если результаты запроса отображаются пользователю: упорядочивать необходимо по полям примитивных типов; упорядочивание по полям ссылочных типов нужно заменять на упорядочивание по строковым представлениям этих полей.

1.4. Отсутствие УПОРЯДОЧИТЬ ПО оправдано только когда: алгоритм не рассчитывает на определенный порядок; результат не показывается пользователю; результат заведомо одна запись. В таких случаях не добавлять УПОРЯДОЧИТЬ ПО, так как это приводит к дополнительным затратам времени.

2. Если в запросе используется конструкция РАЗЛИЧНЫЕ, упорядочивание следует выполнять только по полям, включенным в выборку (в секции ВЫБРАТЬ).

3. Использование конструкции ПЕРВЫЕ совместно с конструкцией АВТОУПОРЯДОЧИВАНИЕ запрещено. Конструкцию АВТОУПОРЯДОЧИВАНИЕ не рекомендуется использовать, так как разработчик не контролирует, какие именно поля будут использованы для упорядочивания. Применение оправдано только когда порядок записей не важен, но он должен быть одинаковым независимо от СУБД. Причины использования АВТОУПОРЯДОЧИВАНИЕ следует указывать в комментарии перед текстом запроса.

См. также: Особенности работы с различными СУБД.

## Округление результатов арифметических операций в запросах

#std535

1. Если в операции деления заранее известны порядки числителя и знаменателя, следует по возможности избегать выполнения деления числа заведомо маленького порядка на число заведомо большого порядка.

2. Платформа 1С:Предприятие поддерживает точность вычислений до 8 разрядов дробной части. Однако из-за особенностей работы различных СУБД в некоторых ситуациях точность результатов может отличаться. Если точность результата запроса, содержащего арифметические операции деления, агрегатные функции СРЕДНЕЕ, арифметические операции умножения с дробными множителями, различается на различных СУБД, то рекомендуется к операндам и/или результатам применять оператор явного приведения разрядности и точности: `ВЫРАЗИТЬ(... КАК Число(m, n))`. Оператор ВЫРАЗИТЬ следует применять к операндам, если на какой-либо СУБД точность результата недостаточна. К результату — если точность на всех СУБД достаточна, но на разных СУБД разная. Указанная разрядность должна быть минимальной, но достаточной. На разных СУБД имеются ограничения на максимальную разрядность (самое жесткое — 31 разряд в целой и дробной частях).

## Особенности использования в запросах оператора ПОДОБНО

Область применения: управляемое приложение, мобильное приложение, обычное приложение.

#std726

1. При использовании в тексте запроса оператора ПОДОБНО допустимо использовать только константные строковые литералы или параметры запроса. Запрещается формировать строку шаблона при помощи вычислений, использовать конкатенацию строк средствами языка запросов.

Допустимо: `Реквизит ПОДОБНО "123%"`  
Недопустимо: `Реквизит ПОДОБНО "123" + "%"` или `Реквизит ПОДОБНО Таблица.Шаблон`

2. Запросы, в которых управляющие символы шаблона оператора ПОДОБНО находятся в полях запроса или в вычисляемых выражениях, по-разному интерпретируются на различных СУБД. Запрос, успешно выполняющийся при работе с файловой базой, может возвращать неверные результаты в режиме клиент-сервера. Подобные выражения необходимо переформулировать: формировать полный шаблон в параметре запроса, а не конкатенировать в тексте запроса.

См. также: Оформление текстов запросов.

## Псевдонимы источников данных в запросах

- Псевдонимы не должны противоречить именам переменных в коде.
- Псевдонимы следует образовывать от терминов предметной области так, чтобы было понятно, как источник данных будет использоваться в запросе.
- Псевдонимы следует образовывать путем удаления пробелов между словами. Каждое слово пишется с прописной буквы (например, ТоварыНаСкладах). Предлоги и местоимения из одной буквы также пишутся прописными.
- Псевдонимы запрещается начинать с подчеркивания.
- Псевдонимы не должны состоять из одного символа.

Неправильно:
```
ВЫБРАТЬ Таблица1.Ссылка КАК Товар, ЕстьNULL(Таблица2.КоличествоОстаток, 0) КАК Остаток
ИЗ Справочник.Номенклатура КАК Таблица1
ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки КАК Таблица2
ПО Таблица1.Ссылка = Таблица2.Номенклатура
```

Правильно:
```
ВЫБРАТЬ ВсяНоменклатура.Ссылка КАК Товар, ЕстьNULL(ОстаткиНаСкладах.КоличествоОстаток, 0) КАК Остаток
ИЗ Справочник.Номенклатура КАК ВсяНоменклатура
ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки КАК ОстаткиНаСкладах
ПО ВсяНоменклатура.Ссылка = ОстаткиНаСкладах.Номенклатура
```

Не рекомендуется использовать имена классов объектов метаданных («Справочник», «Документ» и т.д.) в качестве псевдонимов. В ряде случаев при разработке универсальных механизмов допустимо использование универсальных псевдонимов (например, «Таблица» при подстановке имени таблицы через СтрЗаменить).

## Общие требования по разработке оптимальных запросов

- 1.1. Следует минимизировать объем выборки: выбирать ровно те данные, которые требуются. Не выбирать все поля «на всякий случай» (ВЫБРАТЬ * ИЗ …). По возможности перекладывать свертку, сортировку, вычисления на СУБД, чтобы получить готовый результат.
- 1.2. В большинстве случаев следует минимизировать и общее количество запросов к СУБД. #std758

#std729

См. также: Многократное выполнение однотипных запросов.

- 2. Не следует любой ценой переносить выполнение задачи в СУБД. СУБД обычно оптимизирует простые запросы эффективнее сложных.
  - 2.1. Рассмотреть: подготовку различных (более простых, частных) текстов запроса в зависимости от параметров — вместо одного большого универсального запроса; более эффективную постобработку на сервере 1С средствами встроенного языка.
  - 2.2. Нужно быть уверенным, что запросы используют эффективные планы выполнения. Для сложных запросов СУБД с высокой вероятностью выберет неправильный план (особенно актуально для DB2, PostgreSQL, Oracle). Не неоправданно усложнять запрос: не добавлять вложенные запросы только для читаемости; избегать сложных условий в ГДЕ и соединениях, содержащих подзапросы и ВЫБОР; использовать минимально необходимое число таблиц (уже 5–7 таблиц в одном запросе может давать плохой план). Для анализа плана — консоль запросов, технологический журнал или средства СУБД. Признак проблемы — timeout warning в плане выполнения.

См. также: Запросы в динамических списках.

## Несоответствие индексов и условий запроса

#std652

1.1. Необходимо убедиться, что для всех условий, использованных в запросе, имеются подходящие индексы. Условия используются в секциях: ВЫБРАТЬ … ИЗ … ГДЕ; СОЕДИНЕНИЕ … ПО; виртуальные таблицы (параметры); ИМЕЮЩИЕ.

Подходящий индекс:
- содержит все поля, перечисленные в условии;
- эти поля находятся в самом начале индекса;
- эти поля идут подряд (между ними нет полей, не участвующих в условии).

1.2. При отсутствии подходящего индекса СУБД будет сканировать таблицу или индекс, что увеличит время выполнения и количество блокировок.

2. Платформа автоматически создает индексы: по ссылке для объектных сущностей; по регистратору для таблиц движений; по периоду и измерениям для итогов регистров накопления/бухгалтерии.

3. При недостатке автоматических индексов можно дополнительно индексировать реквизиты (для справочников и документов рекомендуется с дополнительным упорядочиванием). Создавать индексы осознанно, не «на всякий случай». Не индексировать первое измерение регистра дополнительно (для него уже есть основной индекс). Не создавать индексы по низкоселективным полям (например, Булево — только если в запросах всегда отбор по одному значению и записей с ним незначительная часть).

### Примеры

**Запрос 1.** Условие только по Номенклатура при порядке измерений (Склад, Номенклатура, Качество). Нарушено требование 2: отсутствует отбор по первому полю индекса (Склад). Варианты оптимизации: проиндексировать измерение Номенклатура или поставить Номенклатура первым (осторожно — может замедлить другие запросы).

**Запрос 2.** Условие «Качество = &Качество И Склад = &Склад» при порядке Склад, Номенклатура, Качество. Нарушено требование 3: между Склад и Качество в индексе есть Номенклатура. Варианты: добавить условие по Номенклатура, убрать условие по Качество или поменять местами измерения Номенклатура и Качество.

**Запрос 3.** Условие «Номенклатура = &Номенклатура И Склад = &Склад» при том же порядке измерений — требования соблюдены, порядок условий в запросе может не совпадать с порядком полей в индексе.

См. также: Индексы таблиц базы данных; Эффективные условия запросов.

## Разыменование ссылочных полей составного типа в языке запросов

#std654

1.1. В языке запросов возможно обращение к полям таблицы, на которую ссылается поле исходной таблицы (поля ссылочного типа), «через точку». Это приводит к неявному соединению с дополнительными таблицами. Для поля составного типа (ссылка на несколько таблиц) обращение «через точку» приводит к соединению со всеми таблицами, ссылки на которые могут оказаться в данном поле, и с RLS к этим таблицам. Получение данных «через точку» от ссылочных полей составного типа крайне нежелательно (например, Регистратор.Дата для регистра с регистратором любого документа — неявное соединение со всеми таблицами документов). Каждое исключение должно тщательно анализироваться.

2.1. Следует избегать избыточности при создании полей составных ссылочных типов: указывать ровно столько возможных типов, сколько необходимо; не использовать без необходимости «любая ссылка» / «любой документ».

2.2. Чтобы избежать запросов с большим числом исходных таблиц, допустимо дублировать данные в исходную таблицу (например, реквизит ДатаРегистратора в регистре, заполняемый при проведении) и использовать его в запросах вместо разыменования Регистратор.Дата.

2.3. Для конкретного запроса ограничивать количество возможных типов функцией ВЫРАЗИТЬ; при универсальном запросе — формировать запрос динамически, подставляя в ВЫРАЗИТЬ нужный тип. Пример: вместо обращения к Регистратор.Номер, Регистратор.Дата по регистру с регистратором из 56 видов документов (56 левых соединений) — использовать ВЫБОР КОГДА ... ВЫРАЗИТЬ(Продажи.Регистратор КАК Документ.РеализацияТоваровУслуг).Номер ... с условием в ГДЕ по типу регистратора (Регистратор ССЫЛКА Документ.РеализацияТоваровУслуг ИЛИ ...).

См. также: Самодостаточность регистров; Ограничения на использование реквизитов составного типа.

## Ограничения на соединения с вложенными запросами и виртуальными таблицами

#std655

1.1. Не следует использовать соединения с вложенными запросами. Следует соединять друг с другом только объекты метаданных или временные таблицы. Если запрос использует соединения с вложенными запросами, его следует переписать с использованием временных таблиц (кроме случая, когда вложенный запрос сканирует мало записей). Иначе возможны: крайне медленное выполнение; нестабильная работа; большая разница по времени на разных СУБД; повышенная чувствительность к актуальности статистик.

Пример переписывания: вместо
```
... ИЗ Документ.РеализацияТоваровУслуг ЛЕВОЕ СОЕДИНЕНИЕ (ВЫБРАТЬ ... ИЗ РегистрСведений.Лимиты ...) ПО ...
```
— пакетный запрос: первый запрос — ПОМЕСТИТЬ Лимиты ... ИНДЕКСИРОВАТЬ ПО ...; второй — ВЫБРАТЬ ... ИЗ Документ.РеализацияТоваровУслуг ЛЕВОЕ СОЕДИНЕНИЕ Лимиты ПО ...

1.2. Если запрос с соединением с виртуальной таблицей (например, Остатки) работает неудовлетворительно, рекомендуется вынести обращение к виртуальной таблице в отдельный запрос с сохранением результатов во временной таблице.

2. Следует избегать неявных подзапросов при вложенных соединениях. Вместо вложенных соединений использовать последовательные соединения. Если вложенное соединение использовано по ошибке (считали, что аналогично последовательному), — переписать на последовательное. Если вложенное соединение осмысленно, от него лучше отказаться или заменить на временную таблицу.

См. также: Ограничения на вложенные запросы в условии соединения; Использование временных таблиц.

## Ограничения на использование вложенных запросов в условии соединения

#std656

Не следует использовать вложенные запросы в условии соединения. Это может привести к значительному замедлению запроса и (в отдельных случаях) к его полной неработоспособности на некоторых СУБД.

Пример: условие соединения вида «Цены.Период В (ВЫБРАТЬ МАКСИМУМ(...) ИЗ РегистрСведений.Цена ...)». Рекомендуется переписать с использованием временных таблиц: первый запрос — выборка в временную таблицу (например, ДатыПоНоменклатурам — Номенклатура, МАКСИМУМ(Период)); второй — соединение по этой временной таблице и РегистрСведений.Цена по Номенклатура и Период.

См. также: Ограничения на соединения с вложенными запросами и виртуальными таблицами; Использование временных таблиц.

## Обращения к виртуальным таблицам

#std657

1. При использовании виртуальных таблиц следует передавать в параметры таблиц все условия, относящиеся к данной виртуальной таблице. Не рекомендуется обращаться к виртуальным таблицам при помощи условий в секции ГДЕ — СУБД сложнее выбрать оптимальный план, возможны ошибки оптимизатора и замедление. Ограничивать выборку на раннем этапе: передавать условия в параметры виртуальной таблицы (например, Остатки(, Склад = &Склад), а не Остатки() с ГДЕ Склад = &Склад).

2.1. В параметры виртуальной таблицы передавать наиболее простые конструкции (например, «Измерение = Значение»). Не рекомендуется использовать подзапросы и соединения в параметрах виртуальной таблицы (в т.ч. неявные — обращения «через точку» и соединения из RLS).

2.2. При необходимости использовать подзапросы в условиях виртуальной таблицы: в подзапросе только одна таблица, без соединений; для табличной части — без обращения к реквизитам шапки; для объекта с табличными частями — без обращений к табличным частям в условии; для временной таблицы — без раздела ГДЕ; для постоянной таблицы условие ГДЕ допустимо, только если выполняется для 80%+ случаев; в RLS не должно быть подзапросов и соединений (только простые условия). При использовании стандартных шаблонов RLS БСП с Exists и подзапросами следует переписать запрос с временной таблицей или привилегированным режимом.

2.3. Если несколько условий с подзапросами — выбрать одно, удовлетворяющее условиям выше и отсекающее максимум записей; остальные накладывать на внешний запрос. Либо использовать одну временную таблицу и условие (Поле1, Поле2, ...) В (ВЫБРАТЬ ... ИЗ ВременнаяТаблица).

См. также: Разыменование ссылочных полей составного типа; Эффективное обращение к виртуальной таблице «Остатки».

## Эффективные условия запросов

#std658

1. Условия запросов должны быть оптимальны по производительности. Поля основного условия в ГДЕ, ПО и виртуальных таблицах должны быть проиндексированы. Основное условие может быть уточнено дополнительным, объединять их следует только по И. Основное условие — то, что сильнее всего ограничивает выборку, его составляющие объединены по И. В основном условии допустимы только операции, позволяющие поиск по индексу: для всех полей индекса кроме последнего — только = и И; для последнего/единственного — =, >, <, >=, <=, ПОДОБНО, МЕЖДУ, В, ИЛИ (сводимое к В). Нельзя: арифметические операции, функции, отрицания, неравенства над полями. Для ГДЕ/виртуальной таблицы индексировать поля основной таблицы выборки; для ПО ЛЕВОГО соединения — поля правой таблицы; для ПО ВНУТРЕННЕГО — поля таблицы с большим количеством записей. Требования можно не соблюдать, если в таблицах всегда мало данных (< 1000) или запрос выполняется очень редко. Иначе: преобразовать условия (п. 3–4), добавить заранее вычисляемые индексированные поля или пересмотреть архитектуру.

2. Оператор ИЛИ в основном условии допустим только для последнего/единственного поля индекса, когда ИЛИ можно заменить на В. Для одного поля: ГДЕ Поле = &Зн1 ИЛИ Поле = &Зн2 допустимо (эквивалентно В). Для разных полей: ГДЕ Поле1 = &Зн ИЛИ Поле2 = &Зн нельзя заменить на В, но можно переписать через ОБЪЕДИНИТЬ ВСЕ (каждое поле должно быть проиндексировано). В дополнительном условии ИЛИ допустим без ограничений.

3. ПОДОБНО в основном условии — только для последнего/единственного поля индекса. Функции со строками по возможности приводить к ПОДОБНО (например, ПОДСТРОКА(Поле, 1, 6) = "строка" → Поле ПОДОБНО "строка%"). Если литерал не с начала — добавить вычисляемое при записи поле и индексировать его.

4. МЕЖДУ в основном условии для последнего/единственного поля. Функции с датой приводить к МЕЖДУ (например, МЕСЯЦ(Поле) = 1 → Поле МЕЖДУ &НачалоМесяца И &КонецМесяца).

5. ВЫБОР допустим только в дополнительных условиях, не в основном.

6. Арифметические операции над полями — только в дополнительных условиях.

7. Если пользователю назначено несколько ролей с разным RLS, к запросу добавляются условия всех ролей через ИЛИ, что может приводить к неоптимальному выполнению. Рекомендуется: одна роль на объект (по чтению/записи); либо одинаковые условия RLS в разных ролях; либо одна «смешанная» роль с RLS без ИЛИ.

См. также: Типичные причины неоптимальной работы запросов (ИТС); Несоответствие индексов и условий запроса.

## Разрешение итогов для периодических регистров сведений

Рекомендуется разрешить итоги для периодического регистра сведений, если выполнены все условия: в регистре ожидается большой объем данных; в конфигурации предусмотрены частые запросы к срезам последних/первых на текущий момент (без периода в параметрах СрезПоследних/СрезПервых); остальные условия для СрезПервых/СрезПоследних задаются только на значения измерений (и разделителей в режиме Независимо/совместно); в RLS регистра используются только измерения (и такие разделители). В этом случае запросы к текущим ценам/значениям могут выполняться напрямую из дополнительных таблиц итогов и существенно ускоряются. Если запросы к срезам чаще всего на конкретный период (например, на дату документа) или в условиях виртуальных таблиц используются подзапросы и соединения («через точку»), итоги разрешать не следует. Рассмотреть пересмотр запросов так, чтобы условия для итогов выполнялись (например, при отсутствии записей с будущей датой выполнять запрос к срезу без даты). #std708

Пересчет итогов не требуется — актуализация выполняется автоматически при записи набора записей в регистр (исключение — принудительное отключение через УстановитьИспользованиеИтогов(Ложь)).

## Эффективное обращение к виртуальной таблице «Остатки»

#std733

Самым эффективным вложенным запросом для чтения остатков будет чтение хранимой таблицы текущих остатков без группировки по измерениям. Платформа сгенерирует такой запрос, если соблюдены все условия: получение остатков без указания даты; не используется разделение итогов (учитывать влияние на параллельность записи); внешний запрос использует все измерения виртуальной таблицы (в ВЫБРАТЬ или в условиях соединения).

Неправильно: передавать дату в Остатки(&Дата, ...) когда нужны текущие остатки; не использовать измерение (например, Склад) во внешнем запросе — тогда во вложенном запросе будет группировка по этому измерению.

Правильно: Остатки(, Склад = &Склад) без даты; во внешнем запросе в условии соединения использовать все измерения, например ПО ОстаткиТоваров.Номенклатура = СпрНоменклатура.Ссылка И ОстаткиТоваров.Склад = &Склад.

## Использование временных таблиц

#std777

1. В общем случае временные таблицы рекомендуется использовать для повышения производительности и стабильности запросов. Использование для других целей (например, архитектура кода) допустимо, но может в части случаев снижать производительность.

2. Не следует: создавать временные таблицы с очень большим объемом данных (сотни тысяч записей) — иначе снижение производительности и исчерпание места; помещать во временную таблицу больше данных или полей, чем требуется последующим запросам; создавать и удалять временные таблицы в цикле, если можно создать одну до цикла; копировать одну временную таблицу в другую только ради переименования — передавать имя таблицы.

3. Временные таблицы следует индексировать, когда это даёт прирост производительности: большая временная таблица участвует в соединении — в индекс поля из условия ПО; обращение к временной таблице в подзапросе В (...) — в индекс поля из списка выбора, соответствующие полям слева от В. Маленькие (< 1000 записей) индексировать не обязательно. При нескольких условиях выбора/соединения строить индекс для наиболее часто проверяемого условия. См. также: стандарт 652 «Несоответствие индексов и условий запроса».

См. также: Ограничения на соединения с вложенными запросами и виртуальными таблицами; Ограничения на вложенные запросы в условии соединения.
