# Глава 17. Интеграция с внешними системами

> **Agent ref:** Интеграция 1С с внешним миром: Web-сервисы (SOAP/WSDL), HTTP-сервисы (REST), стандартный OData, вызов внешних API.
> **Ключевые разделы:** 17.2 Web-сервисы | 17.3 HTTP-сервисы и HTTP API | 17.4 OData
> **API:** HTTPСоединение, HTTPЗапрос, HTTPОтвет; WSПрокси, WSОпределения; URL ws/…, hs/…, odata/standard.odata
> ---

## 17.1. Обзор механизмов интеграции

Интеграция: обмен данными и/или вызов функций между системами. В 1С для **функциональной интеграции** по сети используются:
- **Web-сервисы (SOAP)** — экспорт/импорт по WSDL, XDTO; клиент и сервер.
- **HTTP-сервисы (REST)** — клиент: HTTPСоединение/HTTPЗапрос/HTTPОтвет; сервер: объект HTTP-сервис, шаблоны URL, методы.
- **Стандартный интерфейс OData** — доступ к данным БД без кода (публикация + HTTP).

Дополнительно: обмен сообщениями через **Сервис интеграции** (AMQP, 1С:Шина); **внешние источники данных** (ODBC, OLAP/XMLA) — чтение/запись во внешние БД.

## 17.2. Web-сервисы

### 17.2.1. Стандарты и ограничения

**Поддерживается:** SOAP 1.1, WS-I Basic Profile 1.1, HTTP 1.1, TLS 1.1/1.2 (в т.ч. ГОСТ), MTOM; аутентификация Basic, NTLM/Negotiate.

**Не поддерживается:** WS-Policy, WS-Addressing, WS-Security; использование уже занятого targetNamespace.

### 17.2.2. URL и публикация

Адрес доступа: `http://host/base/ws/ИмяWebСервиса` или `http://host/base/ws/АдресWebСервиса` (Имя файла публикации). Разделители в URL при публикации — только через фрагменты URL или default.vrd; параметр Z не поддерживается.

Создание: объект конфигурации Web-сервис, операции и параметры (типы XDTO), модуль с процедурами-обработчиками. Публикация — через диалог публикации на веб-сервере (руководство администратора).

### 17.2.3. Вызов и сеанс

При вызове: из пула берётся/создаётся соединение с ИБ, создаётся сеанс (вызывается УстановкаПараметровСеанса в модуле сеанса), затем выполняется метод Web-сервиса. Не рекомендуется выполнять тяжёлые операции в УстановкаПараметровСеанса. Модуль Web-сервиса выполняется на сервере в обычном режиме.

Свойства операции: Тип возвращаемого значения (XDTO), Возможно пустое значение, В транзакции. Параметр: Тип значения (XDTO), Направление передачи (Входной/Выходной/Входной-Выходной).

### 17.2.4. Клиент: сторонние Web-сервисы

- **Статическая ссылка (WSСсылки):** импорт WSDL в конфигураторе; быстрее, т.к. описание один раз. Для смены адреса сервиса при том же WSDL — явно передать URL в СоздатьWSПрокси(…, АдресПространстваИмен, ИмяСервиса, ИмяПорта, , , , РеальныйURL).
- **Динамическая ссылка:** WSОпределения(WSDL_URL), Новый WSПрокси(Определение, ПространствоИмен, ИмяСервиса, ИмяПорта[, , , РеальныйURL]). Актуальное описание при каждом вызове.

Пример создания прокси и вызова (статическая ссылка):

```bsl
Прокси = WSСсылки.ИмяСервиса.СоздатьWSПрокси("http://example.com/ns", "ИмяСервиса", "ИмяСервисаSoap");
Результат = Прокси.ИмяОперации(Параметр);
```

Динамическая ссылка:

```bsl
Определение = Новый WSОпределения("http://host/path.1cws?wsdl");
Прокси = Новый WSПрокси(Определение, "http://example.com/ns", "ИмяСервиса", "ИмяСервисаSoap");
```

### 17.2.5. Ошибки WSDL при загрузке

При неверном WSDL платформа выдаёт исключение с диагностикой. Формат записи: `<Тип элемента>.<Имя> [ ... ] <Описание ошибки>`. Типичные сообщения: «Тип не найден», «Сообщение не найдено», «Привязка не найдена», «Схема должна содержать атрибут targetNamespace», «В document-literal стиле часть сообщения должна ссылаться на объявление элемента в схеме» (element, не type), «Транспорт в привязке должен быть задан как http://schemas.xmlsoap.org/soap/http». Рекомендации по исправлению — по спецификации Basic Profile 1.1 (правила R2xxx, R27xx).

## 17.3. HTTP-сервисы и работа с HTTP API

### 17.3.1. Клиент: объекты и режимы

- **HTTPСоединение** — установка соединения с хостом, таймаут, при необходимости — ЗащищенноеСоединениеOpenSSL/ЗащищенноеСоединениеКриптоПро (HTTPS).
- **HTTPЗапрос** — URI, Заголовки, тело (строка/двоичные/поток/файл): ПолучитьТелоКакСтроку/УстановитьТелоИзСтроки и аналоги для двоичных данных и потока.
- **HTTPОтвет** — КодСостояния, Заголовки, тело (ПолучитьТелоКакСтроку, ПолучитьТелоКакДвоичныеДанные, ПолучитьТелоКакПоток).

Синхронный вызов — только на сервере: ВызватьHTTPМетод(Метод, Запрос). Асинхронный — только на клиенте: ВызватьHTTPМетодАсинх(Метод, Запрос), возвращает Обещание; специализированные методы: ПолучитьАсинх, ОтправитьДляОбработкиАсинх (POST), ЗаписатьАсинх (PUT), ИзменитьАсинх (PATCH), УдалитьАсинх (DELETE) и др.

Пример (клиент, HTTPS, JSON):

```bsl
ЗащищенноеСоединение = Новый ЗащищенноеСоединениеOpenSSL(Неопределено, СертификатыУдостоверяющихЦентровОС);
Соединение = Новый HTTPСоединение("api.example.com", 443, , , , 30, ЗащищенноеСоединение);
Запрос = Новый HTTPЗапрос("/path");
Запрос.Заголовки.Вставить("Content-Type", "application/json");
ОбещанныйРезультат = Соединение.ВызватьHTTPМетодАсинх("GET", Запрос);
Ответ = Ждать ОбещанныйРезультат;
Если Ответ.КодСостояния = 200 Тогда
    Читатель = Новый ЧтениеJSON; Читатель.УстановитьСтроку(Ответ.ПолучитьТелоКакСтроку());
    Данные = ПрочитатьJSON(Читатель, Истина); Читатель.Закрыть();
КонецЕсли;
```

Параметры запроса — в URI после `?` (ключ=значение&…). Тело запроса — по контракту API (JSON, XML и т.д.).

### 17.3.2. JWT

Объект **ТокенДоступа**: Заголовки (обязательно alg, typ), ПолезнаяНагрузка; свойства Эмитент, Получатели, КлючСопоставленияПользователя, ВремяСоздания, ВремяЖизни и др. Подпись: Подписать(АлгоритмПодписи, КлючПодписи) или ПодписатьАсинх. Строковое представление — Строка(ТокенДоступа). В HTTP-запрос: заголовок Authorization с "Bearer " + ТекстТокена или Запрос.ДобавитьТокенДоступа(ТокенДоступа); либо параметр запроса (например, AccessToken=…).

### 17.3.3. Собственный HTTP-сервис (REST-сервер)

URL: `http://host/base/hs/корневойURL/относительныйURL`. Корневой URL задаётся в свойстве HTTP-сервиса; относительный — сопоставляется с **шаблонами URL**.

Шаблон: символы, `/`, параметризованные сегменты `{имя}`, в конце допускается `*`. Пример: `/query/{string}/*`. Сегменты из `{…}` попадают в Запрос.ПараметрыURL. Правила сопоставления: регистрозависимое; точное совпадение приоритетнее параметризованного; `*` — любое число сегментов; параметризованный приоритетнее `*`. В одном шаблоне не допускаются два параметризованных сегмента с одним именем; `*` только в конце.

Для шаблона задаются **методы** (GET, POST, PUT, DELETE, PATCH, MERGE, OPTIONS и др., или Любой). Обработчик — функция с одним параметром HTTPСервисЗапрос, возвращающая HTTPСервисОтвет. Имя обработчика формируется из имени шаблона и имени метода (например, ПримерGET). В запросе доступны: HTTPМетод, БазовыйURL, ОтносительныйURL, Заголовки, ПараметрыURL, ПараметрыЗапроса, тело (двоичные/строка). Ответ: КодСостояния, установка тела (УстановитьТелоИзСтроки, из двоичных данных или из файла). Если обработчик вернёт не HTTPСервисОтвет — клиенту 500. Отсутствие обработчика для метода — 405.

Публикация HTTP-сервисов — через диалог публикации на веб-сервере. Разделители — только URL или default.vrd; параметр Z не поддерживается.

## 17.4. Стандартный интерфейс OData

### 17.4.1. Назначение и протокол

Доступ к данным ИБ извне без кода: чтение, создание, обновление, удаление, проведение документа. Протокол OData v3. Форматы: atom-xml (по умолчанию), json ($format=json или Accept: application/json). Сервер — при публикации; клиент — через HTTPСоединение и формирование URL/тела запросов.

### 17.4.2. URL

База: `http://host/base`. Признак OData: `/odata/standard.odata`. Далее — имя ресурса и параметры запроса.

Имя ресурса: `Префикс_ИмяОбъектаКонфигурации_Суффикс`. Префиксы: Catalog (справочник), Document (документ), DocumentJournal, Constant, ChartOfAccounts, AccumulationRegister, CalculationRegister, AccountingRegister, BusinessProcess, Task, ExchangePlan, InformationRegister (виртуальные таблицы — через функцию, например SliceLast()). Суффикс: имя табличной части, имя реквизита ТЧ/набора записей, имя виртуальной таблицы, RowType, RecordType. Пример набора: `Catalog_Товары`. Пример сущности: `Catalog_Товары(guid'...')`. Регистр сведений с ключом: `InformationRegister_КурсыВалют(Period=datetime'...', Валюта_Key=guid'...')`.

Специальные ресурсы: `$metadata` — описание интерфейса (GET); упрощённый список сущностей — GET на `http://host/base/odata/standard.odata`.

### 17.4.3. Параметры запроса

- **$format** — atom (по умолчанию) или json.
- **$filter** — условие отбора. Операции: eq, ne, gt, ge, lt, le, and, or, not; add, sub, mul, div; функции: substringof, startswith, endswith, like, concat, substring; dateadd, datedifference, day, month, year и др.; cast(expr, type), isof(expr, type); лямбды any(d: условие), all(d: условие). Типы в отборе: guid'value', datetime'yyyy-mm-ddThh:mm:ss'. Не поддерживаются: length, indexof, replace, tolower, toupper, trim; сравнение с ХранилищеЗначения.
- **$select** — список свойств; * — все; ** — все кроме табличных частей. Представления: суффикс ____Presentation; для всех полей представлений: *____Presentation.
- **$top**, **$skip** — ограничение и пропуск записей; при $inlinecount=allpages параметры $skip и $top игнорируются.
- **$orderby** — свойства и asc/desc, через запятую; допускается вложенность (Контрагент/ИНН asc).
- **$expand** — раскрытие навигационных свойств (связанные сущности). Не поддерживается для табличных частей и при запросе одиночной сущности; для составного типа — указывать тип в пути (Договор/Document_ДоговорНайма/Заявление).
- **$count** — GET к ресурсу/$count?$filter=… — возврат количества.
- **$inlinecount** — allpages (вернуть количество в ответе) или none.
- **allowedOnly** — при true возвращаются только объекты, разрешённые правами/RLS; иначе возможна ошибка 401.

### 17.4.4. Типы данных и суффиксы свойств

Соответствие типов 1С и OData: Строка → Edm.String, Дата → Edm.DateTime, Число → Edm.Int16/Int32/Int64, Double, Булево → Edm.Boolean, Ссылка/УникальныйИдентификатор → Edm.Guid, Перечисление → Edm.String. ХранилищеЗначения: навигационное свойство (получение контента), _Base64Data, _Type. Ссылочный тип: значение _Key и навигационное свойство. Составной тип: значение + _Type (диспетчеризационное; для Неопределено — StandardODATA.Undefined).

### 17.4.5. HTTP-методы и типовые операции

- **GET** — чтение набора или сущности.
- **POST** — создание (URL набора сущностей, тело в atom или json).
- **PUT/PATCH** — обновление (канонический URL сущности). PATCH — только переданные свойства; PUT — все обязательные. Для ссылочных реквизитов в json: ИмяРеквизита@odata.bind с URI сущности.
- **DELETE** — удаление по каноническому URL (прямое удаление, не пометка).
- Проведение документа: POST к `Document_Имя(guid'...')/Post?PostingModeOperational=false`. Отмена проведения: Unpost.
- Оптимистическая блокировка: заголовок If-Match со значением DataVersion сущности при PATCH/DELETE; при несовпадении — 412.
- Режим загрузки данных: заголовок 1C_OData-DataLoadMode: true.

### 17.4.6. Ошибки (4XX/5XX)

В теле ответа может передаваться xml с внутренним кодом и сообщением. Коды: 0 — не поддерживается; 2 — неверный формат запроса; 6 — неверный URL; 7 — не хватает элемента ключа; 8 — тип сущности не найден; 9 — экземпляр не найден; 13 — создание строк ТЧ напрямую не поддерживается; 15 — сущность с таким ключом уже существует; 19 — HTTP-метод запрещён; 20 — ошибка прав доступа / RLS (рекомендуется allowedOnly=true); 21 — нереализованная функция; и др.

### 17.4.7. Публикация и состав

Публикация — диалог публикации на веб-сервере. Состав доступных объектов задаётся методом УстановитьСоставСтандартногоИнтерфейсаOData() (в режиме совместимости 8.3.4 и ниже не используется — доступны все поддерживаемые объекты). Разделители — только в URL или default.vrd.

## 17.5. Внешние источники данных (кратко)

Реляционные: объект ВнешниеИсточникиДанных, таблицы и поля; доступ через ODBC (строка соединения или DSN). Чтение и запись; СУБД: Microsoft SQL Server, IBM Db2, PostgreSQL, Oracle, MySQL. В запросе — один источник данных; обращение к таблице: ВнешнийИсточникДанных.ИмяИсточника.Таблица.ИмяТаблицы. Транзакции и блокировки — через менеджер внешнего источника (распределённые транзакции не поддерживаются).

Аналитические (OLAP): кубы, таблицы измерений, измерения, ресурсы; доступ по XMLA (HTTP). Только чтение. Строка соединения: URL поставщика XMLA + параметры ProviderName, DataSourceName, Catalog.

## 17.6. Повторное использование сеансов (интернет-сервисы)

Для Web-, HTTP-сервисов и OData: снижение задержек за счёт переиспользования сеансов. Варианты: не использовать; использовать (управление заголовком IBSession: start — создание сеанса, cookie ibsession; finish — завершение); использовать автоматически (пул сеансов по типу сервиса, имени, пользователю, разделителям, безопасному режиму). Настройки — в свойствах Web-сервиса/HTTP-сервиса (Повторное использование сеансов, Время жизни сеанса) и в default.vrd (приоритет над конфигурацией). OData по умолчанию — автоматическое использование, время жизни 20 с.

## 17.7. Сервисы интеграции (кратко)

Подключение к 1С:Шина (AMQP). Объект Сервис интеграции; каналы (входящие/исходящие), сообщения. Отправка: СоздатьСообщение(), заполнение тела (ПолучитьТелоКакПоток()), ОтправитьСообщение() по каналу. Сообщения хранятся в ИБ до подтверждения доставки. Получение — обработчик в модуле сервиса интеграции; флаг «В транзакции» определяет поведение при откате. Обработка очередей — ВыполнитьОбработку()/ОстановитьОбработку(); регламентное задание рекомендуется раз в минуту.
