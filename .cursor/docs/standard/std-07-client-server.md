## Поддержка толстого клиента, управляемое приложение, клиент-сервер

Область применения: управляемое приложение, обычное приложение. #std680 #std556

1. В управляемом режиме из-за ряда ограничений тонкого клиента может возникнуть необходимость поддержки запуска толстого клиента (в режиме управляемого приложения). Подробнее см. Функциональность обычного приложения, отсутствующая в управляемом приложении.

2. При этом разработка конфигураций, рассчитанных на режим управляемого приложения, как правило, ведется исходя из того, что в клиент-серверной архитектуре код следующих модулей компилируется и выполняется только на сервере:
   - модуль менеджера;
   - модуль объекта;
   - модуль сеанса.

В указанных модулях может встречаться обращение к общим модулям, доступным только на сервере. Однако в толстом клиенте, в режиме управляемого приложения (клиент-сервер), возможны ситуации, когда указанные модули могут начать компилироваться и выполняться на стороне клиента:
   - если объект (справочник, документ и т.п.) явно создается и вызывается в клиентском коде;
   - когда платформа 1С:Предприятие неявно обращается к модулям менеджеров и модулю сеанса для вызова их обработчиков событий на клиенте.

Компиляция и выполнение таких модулей на клиенте могут приводить к ошибкам. Для того чтобы избежать незапланированной компиляции и исполнения указанных модулей на клиенте, а также лишних сообщений режима проверки конфигурации, следует:

- полностью исключить из клиентского контекста код модулей объектов (наборов записей и т.п.), заключив его в инструкцию препроцессора и дополнив вызовом исключения:

```
#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда
…
#Иначе
ВызватьИсключение НСтр("ru = 'Недопустимый вызов объекта на клиенте.'");
#КонецЕсли
```

- полностью исключить из клиентского контекста код модуля сеанса, заключив его в инструкцию препроцессора:

```
#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда
…
#КонецЕсли
```

- полностью исключить из клиентского контекста код модулей менеджеров всех видов объектов метаданных, заключив его в инструкцию препроцессора:

```
#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда
…
#КонецЕсли
```

Ограничение: если представление объектов формируется обработчиками событий модуля менеджера ОбработкаПолученияПредставления и ОбработкаПолученияПолейПредставления, то в толстом клиенте (управляемое приложение, клиент-сервер) представление будет формироваться по умолчанию, без вызова этих обработчиков. (Обработчики ОбработкаПолученияДанныхВыбора и ОбработкаПолученияФормы вызываются всегда только на сервере.)

3. В тех случаях, когда требуется снять указанное ограничение, необходимо дополнительно обеспечить работу на клиенте:
   - обработчиков событий модулей менеджеров ОбработкаПолученияПредставления и ОбработкаПолученияПолейПредставления;
   - кода подписок на эти события.

Код перечисленных обработчиков следует вынести за инструкции препроцессора, указанные в п. 2, а обработчики подписок разместить в клиент-серверных модулях. При необходимости вызова серверных процедур (и функций) из клиентского кода следует размещать вызываемые процедуры в серверных общих модулях с признаком Вызов сервера. При этом в параметры процедур (и в возвращаемые значения функций) не передавать значения мутабельных типов (СправочникОбъект, ДокументОбъект и пр.). Не следует для этих целей всем общим модулям с признаком Сервер принудительно устанавливать флажок Вызов сервера.

Пример: обработчик ОбработкаПолученияПредставления вызывает общий модуль, недоступный на клиенте — правильно выполнить переход на сервер (не передавать на клиент значения мутабельных типов):

```
Процедура ОбработкаПолученияПредставления(Данные, Представление, СтандартнаяОбработка)
    ВзаимодействияВызовСервера.ОбработкаПолученияПредставления(Данные, Представление);
    СтандартнаяОбработка = Ложь;
КонецПроцедуры
```

## Использование модулей с повторным использованием возвращаемых значений

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std724 #std782

1. Общие модули с повторным использованием возвращаемых значений (кэш) предусмотрены для кэширования результатов работы функций — на время сеанса или на время вызова. Их следует применять для экономии вычислительных ресурсов сервера и для минимизации клиент-серверного взаимодействия.

2. Чрезмерное (неоправданное) применение таких модулей может приводить к излишнему потреблению памяти.
   - 2.1. Недопустимо создавать общие модули с повторным использованием, из которых возвращаются данные, вычисление которых выполняется быстрее, чем получение из кэша (например, строковые константы).
   - 2.2. Следует помещать в кэш только такие данные, к которым потом будут часто обращаться. Кэш не хранит данные вечно (значение удаляется через 20 минут после вычисления или через 6 минут после последнего использования, при нехватке памяти, при перезапуске рабочего процесса и при переключении клиента на другой рабочий процесс).
   - 2.3. Диапазон значений входных параметров функций, размещенных в общих модулях с повторным использованием, не должен быть широким.

См. также: раздел «Повторное использование возвращаемых значений» документации по платформе 1С:Предприятие; Использование значений, влияющих на поведение клиентского приложения.

Неправильно размещать в модуле с повторным использованием:

```
Функция ИмяПакетаУправления() Экспорт
    Возврат "ManagementPackage";
КонецФункции
```

Имеет смысл кэшировать данные, полученные из базы данных, внешних источников или путем сложных (ресурсоемких) вычислений.

3. Не следует изменять данные, полученные из кэша. В качестве возвращаемых значений рекомендуется использовать значения, состояние которых изменить нельзя: ФиксированныйМассив, ФиксированнаяСтруктура (кэш возвращает ссылку на один и тот же объект в памяти).

4. Если в модуле с повторным использованием несколько экспортных функций вызывают друг друга, результат «внутренних» вызовов не кэшируется. Чтобы возвращаемое значение всегда получалось из кэша, следует явно указывать имя модуля:

```
Функция ЭтоАвтономноеРабочееМесто() Экспорт
    Возврат ОбменДаннымиПовтИсп.АвтономнаяРаботаПоддерживается() И ...;
КонецФункции
```

5. Если у общего модуля свойство «Повторное использование возвращаемых значений» установлено в «На время сеанса», то в возвращаемых значениях нельзя использовать значения типа МенеджерВременныхТаблиц, Запрос, объекты базы данных (ДокументОбъект, ОтчетОбъект и т.п.) — ни непосредственно, ни в составе коллекций. Ограничение распространяется на использование значений во временном хранилище.

## Использование значений, влияющих на поведение клиентского приложения

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std459

Если поведение целого ряда форм или команд зависит от одних и тех же значений (параметров, настроек поведения клиентского приложения), для минимизации клиент-серверного взаимодействия рекомендуется применять общие модули с повторным использованием возвращаемых значений (свойство «Повторное использование возвращаемых значений» = «На время сеанса»). Применение допускается, если изменение соответствующего значения в течение сеанса не является критичным.

Функции такого общего модуля должны за один вызов возвращать сразу все значения, которые могут понадобиться в тех или иных обстоятельствах.

Пример: если при работе всех форм подсистемы регистрации занятости требуются параметры «Время занятости по умолчанию», «Время начала рабочего дня», «Время окончания рабочего дня», все эти параметры необходимо получать одним вызовом, возвращающим структуру:

```
Настройки = НастройкиПодсистемыРегистрации();
ВремяНачала = Настройки.ВремяНачалаРабочегоДня;
ВремяОкончания = Настройки.ВремяОкончанияРабочегоДня;
```

Данные, влияющие на поведение отдельных форм, рекомендуется размещать в реквизитах формы, получая их при создании формы на сервере. Общие модули с повторным использованием — только если от значения зависит поведение команд или большого числа форм.

Не рекомендуется использовать переменные модуля управляемого приложения и модуля обычного приложения для минимизации клиент-серверного взаимодействия.

## Получение предопределенных значений на клиенте

Область применения: управляемое приложение, мобильное приложение, обычное приложение.

Для получения ссылок на предопределенные элементы справочников, планов видов характеристик, планов счетов, планов видов расчета, ссылок на значения перечислений, системных перечислений и точек маршрута бизнес-процессов в коде, выполняемом на клиенте (тонкий клиент, веб-клиент), где недоступны СправочникМенеджер.\<ИмяСправочника\>, ПеречислениеМенеджер.\<ИмяПеречисления\> и т.п., предназначена функция глобального контекста ПредопределенноеЗначение. Например:

```
ЮрФизЛицо = ПредопределенноеЗначение("Перечисление.ЮридическоеФизическоеЛицо.ЮридическоеЛицо");
```

При использовании БСП 2.1.4 и выше рекомендуется использовать функцию ПредопределенныйЭлемент общего модуля ОбщегоНазначения или ОбщегоНазначенияКлиент (возвращает Неопределено для несуществующих в ИБ предопределенных элементов). Не следует реализовывать дополнительные механизмы кеширования на клиенте предопределенных значений: серверный вызов выполняется только при первом обращении, результат автоматически кешируется.

См. также: Использование предопределенных элементов; раздел «Работа с предопределенными значениями» в документации к платформе 1С:Предприятие (ИТС).

## Минимизация количества серверных вызовов

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std443 #std487

1.1. Разработку управляемого приложения необходимо вести с контролем количества вызовов серверных процедур и функций из клиентского кода (серверных вызовов), а в части случаев — также объема передаваемых данных (трафика). На каждое действие пользователя в клиентском коде не должно выполняться дополнительных вызовов на сервер. Исключения требуют обоснования.

1.2. Для режима с низкой скоростью соединения следует оптимизировать и количество вызовов, и объем передаваемых данных. Отладку рекомендуется проводить в режиме имитации задержек серверных вызовов.

### Запуск клиентского приложения

2.1. В простейшем случае код при запуске не должен приводить к обращениям на сервер. Когда необходимо получать данные с сервера: не вызывать серверные процедуры и функции напрямую из кода модуля приложения, модуля управляемого приложения и модуля внешнего соединения; правильно — за один вызов передавать на клиент все параметры, необходимые для запуска. Если параметры запрашиваются из разных мест клиентского кода, разместить такую функцию в общем серверном модуле с повторным использованием возвращаемых значений.

### Открытие управляемой формы

3.1. Открывать форму за один вызов методом ОткрытьФорму (ОткрытьФормуМодально). Параметры передавать через параметр Параметры этих методов.

3.2. При открытии формы не допускается выполнять обращений к серверу из обработчиков клиентских событий формы ПриОткрытии и ПриПовторномОткрытии. При необходимости обращения к серверным данным размещать эти данные в реквизитах формы в ПриСозданииНаСервере.

Неправильно: получать настройку на сервере и передавать в ОткрытьФорму. Правильно: открывать форму без параметра и получать значение константы в ПриСозданииНаСервере формы.

### Выполнение локальной команды управляемой формы

4.1. Выполнение локальной команды формы должно приводить не более чем к одному вызову сервера. Если команда выполняет только клиентские операции — все необходимые данные должны быть заранее переданы на клиент (рекомендуется в ПриСозданииНаСервере в реквизиты формы). Если команда связана с бизнес-логикой на сервере — вся она должна выполняться за один серверный вызов.

### Выбор из справочника

4.2. При выборе из справочника допускается только один серверный вызов из кода (вызов ОткрытьФорму / ОткрытьФормуМодально).

4.3. Если после выбора необходимо выполнить бизнес-логику только на сервере — допустим один дополнительный серверный вызов.

### Выполнение глобальной команды

5.1. При выполнении глобальной команды допускается только один серверный вызов. Если команда открывает форму — вызов должен выполняться при ОткрытьФорму / ОткрытьФормуМодально.

### Выполнение команды формирования отчета

6.1. Не допускается выполнять дополнительных серверных вызовов из кода конфигурации. При открытии формы отчета не допускается обращений к серверу из ПриОткрытии и ПриПовторномОткрытии. Правильно: открывать форму отчета с параметром СформироватьПриОткрытии = Истина либо устанавливать этот параметр в ПриСозданииНаСервере.

### Выполнение подбора элементов

7.1. Объем передаваемых данных между формой объекта и формой подбора может быть большим. Не рекомендуется передавать потенциально большой массив в качестве параметра формы подбора (тип ДанныеФормыКоллекция) — возможно дополнительное «дочитывание» данных с сервера.

7.2. Рекомендуется использовать временное хранилище (чтение и запись на сервере). Открытие формы подбора — не более двух обращений на сервер; закрытие — не более двух. При помещении данных во временное хранилище: либо помещать на время жизни формы с уникальным идентификатором формы и очищать после использования, либо предварительно инициализировать и переиспользовать хранилище (не удалять значение при переиспользовании).

См. также: Использование значений, влияющих на поведение клиентского приложения; Использование объекта ДанныеФормыКоллекция.

## Минимизация кода, выполняемого на клиенте

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std629

1.1. Необходимо минимизировать объем кода, выполняемого на стороне клиента. Не следует выполнять на клиенте сложные алгоритмы, требующие значительных ресурсов. Такие алгоритмы размещать в серверном коде с минимально необходимым числом обращений с клиента. См. также: Минимизация количества серверных вызовов.

Причины: клиентский компьютер, как правило, менее производительный; в веб-клиенте код выполняется интерпретатором заметно медленнее.

1.2. Рекомендуется оставлять на клиенте алгоритмы, скорость работы которых заведомо быстрее затрат на один вызов серверной функции (например, перерасчет доступности элементов при изменении данных пользователем).

2.1. Исключение — подсистемы для работы с ПО на клиентском компьютере (торговое оборудование, клиент-банк, печатные формы в офисные программы и т.п.).

2.2. Если подсистема работает с клиентским ПО только в определенных режимах, использовать директивы препроцессора. Например, для кода, недоступного в веб-клиенте:

```
#Если ВебКлиент Тогда
    Предупреждение(НСтр("ru = 'Загрузка адресного классификатора не доступна в веб-клиенте.'"));
#Иначе
    ОткрытьФорму("РегистрСведений.АдресныйКлассификатор.Форма.ФормаЗагрузкиАдресногоКлассификатора");
#КонецЕсли
```

## Доступ к файловой системе из кода конфигурации

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std542

1. При обращении к файлам и каталогам доступ ограничен правами пользователя ОС, от имени которого запущено приложение.
   - 1.1. На клиенте — права пользователя, от имени которого запущено приложение (тонкий, толстый или веб-клиент).
   - 1.2. На сервере: при клиент-серверной ИБ — права пользователя, от имени которого запущен сервер 1С:Предприятие; при файловой базе на веб-сервере — права пользователя веб-сервера.

2. Запрещается выполнять запись файлов в каталог исполняемых файлов (КаталогПрограммы). КаталогПрограммы допустимо использовать только для чтения или запуска файлов.

3. Для переносимости между ОС и кодировками:
   - 3.1. В автоматически формируемых именах файлов — только латиница и цифры; кодировка текстовых файлов — UTF-8.
   - 3.2. При вводе имени пользователем разрешить русскоязычные имена с возможностью транслитерации в латиницу; по умолчанию при возможности предлагать имя латиницей и UTF-8 для текстовых файлов. В БСП для транслитерации — СтроковыеФункцииКлиентСервер.СтрокаЛатиницей.

### Работа с временными файлами и каталогами

- Для имени временного файла использовать ПолучитьИмяВременногоФайла (исключение — веб-клиент, см. ниже). Иначе возможны конфликты в многопользовательском режиме, проблемы с правами и накопление не удаленных файлов.
- Для временного каталога также рекомендуется имя через ПолучитьИмяВременногоФайла (исключение — веб-клиент).
- В веб-клиенте ПолучитьИмяВременногоФайла недоступен: использовать КаталогВременныхФайлов и УникальныйИдентификатор; в БСП для создания временного каталога на клиенте — ФайловаяСистемаКлиент.СоздатьВременныйКаталог.
- После работы с временным файлом или каталогом удалять их самостоятельно.
- На сервере полностью завершать работу с временными файлами/каталогами в рамках одного серверного вызова (при следующем вызове код может выполниться на другом узле кластера). Для сохранения данных между вызовами в пределах сеанса — временное хранилище (ПоместитьВоВременноеХранилище, ПолучитьИзВременногоХранилища).
- Для передачи данных между сеансами во временных файлах — константа с общим путем, доступным со всех серверов кластера; регистр сведений для имен файлов и времени создания; удаление файлов и записей при штатном завершении; регламентное задание для очистки «зависших» записей и файлов.

### Передача файлов между клиентом и сервером

- При одновременной работе с файлом на клиенте и сервере использовать передачу через временное хранилище (ПоместитьФайлы, ПолучитьФайл, ПолучитьФайлы, НачатьПомещениеФайла, ПоместитьВоВременноеХранилище, ПолучитьИзВременногоХранилища). Клиент и серверы кластера — разные компьютеры с разной файловой системой и правами ОС.
- Для сохранения данных во временном хранилище между несколькими серверными вызовами при помещении использовать параметр УникальныйИдентификаторФормы метода ПоместитьФайл. При повторном помещении того же файла предыдущее значение удалять вручную (УдалитьИзВременногоХранилища).
- В БСП для помещения файлов — процедуры ЗагрузитьФайл и ЗагрузитьФайлы общего модуля ФайловаяСистемаКлиент; для сохранения между вызовами — свойство ИдентификаторФормы параметра ПараметрыЗагрузки.

См. также: Установка внешних компонент и расширений платформы; Общие требования к конфигурации.

## Оптимизация использования оперативной памяти

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std725

1. Не следует разрабатывать решения исходя из неограниченного объема оперативной памяти. Для многопользовательских систем неэффективное использование памяти может критически сказаться на работоспособности. Избегать формирования больших структур данных в памяти; при неограниченном объеме данных ограничивать обработку порциями и сохранять результаты в базу или файлы.

2. При потенциально неограниченных выборках получать данные порциями фиксированного размера. Неправильно: Выгрузить() весь результат запроса в таблицу значений или обходить РезультатЗапроса.Выбрать() без ограничения (результат при выполнении запроса все равно считывается в память целиком). Правильно: ограничивать выборку (например, ВЫБРАТЬ ПЕРВЫЕ 1000 … с условием по необработанным) или использовать Выборка = Справочники.Номенклатура.Выбрать(…).

3. Недопустимо загружать большие XML/HTML целиком в ДокументDOM, ТекстовыйДокумент, ДокументHTML или создавать XDTO-пакеты размером с весь файл. Использовать последовательное чтение/запись: ЧтениеXML, ЧтениеТекста, ЗаписьXML, ЗаписьТекста; при XDTO — читать XML последовательно (ЧтениеXML), десериализовывать отдельные фрагменты фабрикой XDTO.

4. Избегать циклических ссылок (утечки памяти). Разрывать (очищать) ссылки, когда объект не нужен. Для выявления утечек — технологический журнал (logcfg.xml, элемент \<leaks\>).

5. Чрезмерное применение общих модулей с повторным использованием возвращаемых значений может приводить к излишнему потреблению памяти.

## Таймауты при работе с внешними ресурсами

Область применения: управляемое приложение, обычное приложение. #std748

1. При работе с внешними ресурсами (WSОпределения, WSПрокси, HTTPСоединение, FTPСоединение, ИнтернетПочтовыйПрофиль) следует задавать таймаут — предельное время ожидания выполнения операции. Иначе при бесконечном ожидании программа может зависнуть или часть функционала станет недоступна. Таймаут защищает от нестабильного подключения, антивируса/брандмауэра, некорректного прокси, сбоев веб-сервера.

2. Время операции складывается из этапов: DNS Lookup, Connect, Send, Wait, Receive, Cache Read. При таймауте в 60 сек. все этапы должны успеть выполниться. Величину таймаута определять исходя из ожидаемого времени операции: для быстрых операций — небольшая; в общем случае не более 3 минут; для длительной передачи больших объемов (Send/Cache Read) — большой, но не более 12 часов.

3. Рекомендации по снижению таймаута и повышению отзывчивости:
   - 3.1. Для веб-сервисов с таймаутом более ~20 сек.: предусмотреть контрольную операцию Ping; предварительно получать прокси с таймаутом ~7 сек. и вызывать Ping; затем получать основной прокси. В БСП — функция WSПрокси общего модуля ОбщегоНазначения (поддержка Ping); для HTTP(S)/FTP(S) — подсистема «Получение файлов через Интернет».
   - 3.2. Для других внешних ресурсов применять аналог Ping (REST — тестовая команда с кодом 200; FTP/WebDAV — загрузка/отправка файла-пустышки).
   - 3.3. Веб-сервисы с объективно долгими операциями (этап Wait) переводить на асинхронный режим: фоновое задание для «тяжелой» операции; отдельные операции проверки готовности и получения результата.

4. Рекомендуемые величины таймаутов (секунд):

| Операция | Таймаут (сек.) |
|----------|----------------|
| Получение описания веб-сервиса | 7 |
| Проверка адреса, взаимодействие с менеджером сервиса (SaaS), «быстрые» операции | 10–20 |
| Сведения об одном контрагенте, обмен сообщениями, отправка SMS, удаленное администрирование ИБ (SaaS) | 60–120 (после Ping) |
| Передача сообщений обмена через веб-сервис, получение файлов до 1 Мб | 120–180 (после Ping) |
| Загрузка файлов более 1 Мб | Размер в Мб × 128 (при 64 кбит/с), иначе не более 43200 (12 ч) |

## Работа в разных часовых поясах

Область применения: управляемое приложение, обычное приложение. #std643

1. Конфигурации должны быть рассчитаны на работу, когда часовой пояс серверного компьютера не совпадает с реальным часовым поясом пользователей (например, сервер в Москве, пользователи во Владивостоке — операции по местному времени). Такой сценарий востребован в клиент-серверных ИБ и в решениях в модели сервиса (SaaS).

2. Серверный код:
   - 2.1. Во всех серверных процедурах и функциях вместо ТекущаяДата использовать ТекущаяДатаСеанса (приводит время сервера к часовому поясу сеанса пользователя).
   - 2.2. Когда нужна «универсальная» отметка времени, не зависящая от часового пояса сеанса, использовать УниверсальноеВремя (например, для кеша, времени последнего выполнения фонового задания).
   - 2.3. При методах платформы, возвращающих локальную дату сервера, приводить ее к УниверсальноеВремя или к времени сеанса (МестноеВремя(…, ЧасовойПоясСеанса())).

3. Клиентский код:
   - 3.1. ТекущаяДата на клиенте недопустима (время на клиенте и сервере не должно различаться). Вместо этого: передавать с сервера на клиент дату/время в часовом поясе сеанса; при работе с документами использовать дату документа.
   - 3.2. В алгоритме закрытия месяца передавать с клиента на сервер дату; определение закрываемого месяца — на сервере. Неправильно: вызывать ТекущаяДата() на клиенте для ТекДата. Правильно: ТекДата = Неопределено на клиенте, в форме обработки использовать НачалоМесяца(ТекущаяДатаСеанса()).
   - 3.3. При подборе номенклатуры передавать дату документа; вычисление даты расчетов — на сервере. При зачете авансов параметр КонецПериода вычислять по дате документа, а не по ТекущаяДата() на клиенте.
   - 3.4. В БСП в остальных случаях — функция ДатаСеанса общего модуля ОбщегоНазначенияКлиент.

4. Исключения из правил 2 и 3 возможны в обоснованных случаях (нужно действительно текущее время сервера); обоснование — в комментарии к вызову.

5. В одной процедуре (функции) избегать многократного обращения к ТекущаяДатаСеанса (ТекущаяДата) — значения могут отличаться. Правильно: один раз вычислить и использовать переменную для последующих выражений.
