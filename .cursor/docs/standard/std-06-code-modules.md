# Домен: Код и модули

> Стандарты и методики (ИТС v8std).

---

## Тексты модулей

1. Тексты модулей должны быть написаны на русском языке. Исключение составляют веб-сервисы, имена методов и параметров которых рекомендуется задавать на английском языке (например, метод TestConnection веб-сервиса EnterpriseDataExchange), а также различные идентификаторы сторонних информационных систем, например, названия полей «внешних» структур данных, которые программно обрабатываются в коде.

1.1. В текстах модулей не допускается использовать букву «ё». Исключения составляют интерфейсные тексты, выводимые пользователю в сообщениях, формах и справке, где употребление буквы «ё» в ряде случаев допустимо. Подробнее см. Тексты.

1.2. В текстах модулей не допускается использовать неразрывные пробелы и знак минус в других кодировках (короткое, длинное тире и т.п.). Такие символы часто оказываются в тексте при копировании из офисных документов и приводят к ряду сложностей при разработке (поиск, подсказки типов, расширенная проверка в 1С:EDT, синтаксические ошибки).

2. Программные модули не должны иметь неиспользуемых процедур и функций.

3. Программные модули не должны иметь закомментированных фрагментов кода, а также фрагментов, связанных с процессом разработки (отладочный код, служебные отметки TODO, MRG и т.п.) и с конкретными разработчиками. После завершения отладки или рефакторинга такие фрагменты удаляются.

**Неправильно:**
```bsl
Процедура ПередУдалением(Отказ)
    // Если Истина Тогда Сообщение("Для отладки"); КонецЕсли;
КонецПроцедуры
```

**Правильно:** удалить обработчик из кода после отладки.

4. Тексты модулей оформляются по принципу «один оператор в одной строке». Несколько операторов допускается только для однотипных операторов присваивания, например:
```bsl
НачальныйИндекс = 0;
Индекс = 0;
Результат = 0;
```

5. Текст модуля должен быть оформлен синтаксическим отступом. Для отступа используется табуляция (не пробелы). Размер табуляции — 4 символа.

5.1. С крайней левой позиции начинаются только: операторы Процедура, КонецПроцедуры, Функция, КонецФункции; предварительное объявление процедур и функций; заголовки процедур и функций; объявление переменных модуля; операторы раздела основной программы; директивы компилятора &НаКлиенте, &НаСервере и т.д. (#std456); инструкции препроцессора (#Область, #КонецОбласти).

5.2. Процедуры НачатьТранзакцию и ЗафиксироватьТранзакцию не являются операторными скобками, поэтому текст внутри них не сдвигается.

6. При длине строки более 120 символов следует использовать переносы. См. также: Перенос выражений.

7.1. Комментарии должны быть понятными, составляться по правилам русского языка, в деловом стиле. Между «//» и текстом — пробел.

7.2. Небольшие комментарии пишутся в конце строки.

7.3. Большие комментарии пишутся перед комментируемым кодом в отдельной строке, выравниваются по левой границе фрагмента.

8. Большие процедуры и функции можно разбивать на сворачиваемые области. Имена областей должны удовлетворять стандарту «Правила образования имен переменных».

## Структура модуля

#std455

1.1. В программном модуле (общие модули, модули объектов, менеджеров, форм, команд и т.п.) в общем случае присутствуют разделы в следующей последовательности:
- заголовок модуля;
- раздел описания переменных;
- экспортные процедуры и функции (программный интерфейс);
- обработчики событий объекта (формы);
- служебные процедуры и функции;
- раздел инициализации.

Некоторые разделы допустимы только в модулях определённого вида (например, обработчики событий элементов — только в модулях форм).

1.2. Объёмные разделы рекомендуется разбивать на подразделы по функциональному признаку.

1.3. Разделы и подразделы оформляются в виде областей. Имена областей — по стандарту «Правила образования имен переменных».

1.4. Шаблон разделов для общих модулей (русск./англ.):
- #Область ПрограммныйИнтерфейс / #Region Public
- #Область СлужебныйПрограммныйИнтерфейс / #Region Internal
- #Область СлужебныеПроцедурыИФункции / #Region Private

1.5. Шаблон для модулей объектов, менеджеров, наборов записей, обработок, отчётов:
- #Область ОписаниеПеременных / #Region Variables
- #Область ПрограммныйИнтерфейс / #Region Public
- #Область ОбработчикиСобытий / #Region EventHandlers
- #Область СлужебныйПрограммныйИнтерфейс / #Region Internal
- #Область СлужебныеПроцедурыИФункции / #Region Private
- #Область Инициализация / #Region Initialize

1.6. Шаблон для модулей форм: ОписаниеПеременных, ОбработчикиСобытийФормы, ОбработчикиСобытийЭлементовШапкиФормы, ОбработчикиСобытийЭлементовТаблицыФормы (для каждой таблицы — свой раздел), ОбработчикиКомандФормы, СлужебныеПроцедурыИФункции.

1.7. Шаблон для модулей команд: ОбработчикиСобытий, СлужебныеПроцедурыИФункции.

1.8. В модуле не должно быть пустых областей.

2.1. Заголовок модуля — комментарий в начале модуля с кратким описанием и условиями применения. Для модулей форм в заголовке рекомендуется описание параметров формы.

2.2. Переменные модуля — с комментарием о назначении, имена по правилам образования имен переменных.

2.3. Программный интерфейс размещается сразу после описания переменных.

2.4–2.6. Обработчики событий формы/объекта — после программного интерфейса; служебные процедуры и функции — следом; раздел инициализации — в конце.

## Имена процедур и функций

1. Правильный выбор имён процедур и функций важен для читаемости. Хорошее имя в сочетании с именами параметров часто избавляет от дополнительного описания. Сложности в выборе имени могут свидетельствовать о неправильной архитектуре. См. также: Описание процедур и функций.

2. Имена процедур, функций и формальных параметров следует образовывать от терминов предметной области так, чтобы из имени было понятно назначение («говорящие» имена).

**Неправильно:** `ВыполнитьПроверку(Параметр1, Рекв, ТЗ)`  
**Правильно:** `РеквизитОбъектаЗаданногоТипа(Объект, ИмяРеквизита, ТипЗначения)`.

3. Имена образуются удалением пробелов между словами; каждое слово с прописной буквы (предлоги и местоимения из одной буквы — тоже прописные).

4. Не рекомендуется в названиях процедур и функций описывать типы параметров и возвращаемых значений (за редким исключением). Например, правильно: `ИменаРолейСПравомДобавления()`, `ДополнительныеНастройки()` вместо `ПолучитьМассивРолей...`, `ПолучитьСтруктуру...`.

5. Имена процедур образуются от неопределённой формы глагола: правильно `ЗагрузитьКонтрагента()`, неправильно `ЗагрузкаКонтрагента()`.

6.1. Имена функций — от описания возвращаемого значения: `ПолноеИмяПользователя()`, `ДатаНачалаСеанса()`.

6.2. Для функций создания объекта в имени рекомендуется слово «Новый»: `НовоеПолеФормы()`, `НовыйЭлементСправочникаФайлы()`.

6.3. Для проверки условия — имя с «Это» или причастия: `ДокументПроведен()`, `РеквизитыДокументаИзменены()`.

6.4. Глаголы в неопределённой форме уместны, когда важен способ получения значения: `ВыбратьДанныеПоПравилу()`, `ПреобразоватьДанныеПоПравилу()`.

6.5. Если функция в первую очередь выполняет действие, а возврат значения не основная задача — имя от глагола: `РазрешитьРедактированиеРеквизитовОбъекта(Форма)`.

## Описание процедур и функций

1. Описание рекомендуется выполнять в виде комментария к процедуре/функции. В 1С:Предприятие 8.3 комментарий выводится в контекстной подсказке. В EDT комментарий используется для типизации параметров и возвращаемого значения.

2. Обязательно комментировать процедуры и функции программного интерфейса модулей.

3. Остальные процедуры и функции (в т.ч. обработчики событий) рекомендуется комментировать, если нужно пояснить назначение или неочевидные причины невыполнения действий.

4. Избегать избыточных комментариев, не дающих дополнительной информации.

5. Структура комментария: секции «Описание», «Параметры», «Возвращаемое значение» (для функций), «Пример», при необходимости «Варианты вызова». Описание не должно совпадать с именем; для процедур и функций секция должна начинаться с глагола. Для параметров: имя, тип (или список типов), при необходимости текстовое описание. Для структур и таблиц — описание свойств/колонок со звёздочками (*). Для массивов — тип элементов («из Тип»). Допустимы ссылки на функции-конструкторы в формате «см. ПутьКФункции».

6. При директиве компиляции сначала размещается комментарий, затем директива.

7. Код процедур и функций отделяется друг от друга пустыми строками.

## Параметры процедур и функций

#std640

1. Формальные параметры объявляются по общим правилам образования имен переменных; имена — от терминов предметной области.

2. Не следует использовать вместо параметров переменные модулей, реквизиты формы и т.п.

3. Параметры располагаются в логической последовательности, рекомендуется от общего к частному. Необязательные параметры (со значениями по умолчанию) — после обязательных.

4. Не рекомендуется объявлять много параметров (ориентир — не более семи), в том числе со значениями по умолчанию (ориентир — не более трёх). При большом числе параметров рекомендуется группировать их в структуры или пересмотреть логику (разбить на несколько функций).

5. При вызове избегать громоздких конструкций: не рекомендуется вкладывать вызовы других функций в параметры; вложенный конструктор `Новый Структура(...)` допустим только при небольшом (до трёх) количестве свойств. Обязательные параметры не пропускать (передавать явно, в т.ч. Неопределено, если допустимо).

6. См. также: передача параметров по ссылке и по значению; Особенности использования структур в качестве параметров; Использование объектов типа Структура.

## Особенности использования структур в качестве параметров процедур и функций

Для функций с параметрами типа Структура рекомендуется:

1. Определить функцию-конструктор параметров (без параметров), возвращающую структуру-заготовку со свойствами. Вызывающий код заполняет свойства и передаёт структуру в вызываемую функцию. Параметры со значениями по умолчанию инициализируются в структуре явно.

2. В вызывающем коде не инициализировать структуру параметров «с нуля» и не добавлять свойства, не определённые в функции-конструкторе; все допустимые параметры задаются в функции-конструкторе.

**Пример:** получить структуру через `ЦенообразованиеКлиентСервер.ПараметрыЗаполненияЦеныВСтрокеТЧ()`, заполнить свойства, передать в `ЦенообразованиеСервер.ЗаполнитьЦеныВСтрокеТЧ(...)`.

См. также: Использование объектов типа Структура.

## Правила образования имен переменных

1. Имена переменных образуются от терминов предметной области так, чтобы было понятно назначение.

2. Пробелы между словами удаляются; каждое слово с прописной буквы. Пример: `ДиалогРаботыСКаталогом`, `КоличествоПачекВКоробке`. Некорректно: `масРеквизитов`, `новСтр`.

3. Имена переменных запрещается начинать с подчёркивания.

4. Имена не должны состоять из одного символа (допускается только для счётчиков циклов).

5. Переменные-флаги называются по истинному значению: `ЕстьОшибки`, `ЭтоТоварТара`.

См. также: Использование переменных в программных модулях.

## Работа с параметром «Отказ» в обработчиках событий

#std686, #std454

1. В обработчиках событий с параметром Отказ (ПриЗаписи, ОбработкаПроверкиЗаполнения, ТоварыПередНачаломДобавления и т.п.) не следует присваивать параметру значение Ложь — к моменту проверки Отказ уже может быть Истина из другой подписки, и его сброс приведёт к ошибке.

**Неправильно:** `Отказ = ЕстьОшибкиЗаполнения();`  
**Правильно:** `Если ЕстьОшибкиЗаполнения() Тогда Отказ = Истина; КонецЕсли;` или `Отказ = Отказ Или ЕстьОшибкиЗаполнения();`

2. Аналогичные требования для параметров СтандартнаяОбработка, Выполнение и др.: не сбрасывать их в значение «выполнено», если уже установлено «не выполнено».

3. В событиях ПриЗаписи, ПередЗаписью, ПередУдалением, ОбработкаПроведения, ОбработкаПроверкиЗаполнения и т.п. неприемлемо устанавливать Отказ = Истина без информирования пользователя о причинах. Следует выводить сообщение или вызывать исключение (по п. 1.1 и 1.3 стандарта «Информирование пользователя»).

См. также: Информирование пользователя, Сообщения пользователю, Перехват исключений в коде.

## Перенос выражений

#std444

1. При длине строки более 120 символов использовать переносы. Строки длиннее 120 символов не рекомендуются, кроме случаев, когда перенос невозможен (например, длинная строковая константа для СообщениеПользователю).

2. Длинные арифметические выражения: при переносе знаки операции в начале новой строки; операнды с отступом или выравниванием по первому операнду.

3. Длинные строковые константы — перенос с помощью символа перевода строки. При конкатенации строк знак «+» рекомендуется в начале строки; для очень длинных строк допустим «+» в конце.

4. Параметры процедур и функций при переносе выравниваются по первому параметру или с стандартным отступом; закрывающая скобка и «;» в одной строке с последним параметром.

5. Сложные логические условия в Если…ИначеЕсли…КонецЕсли: каждое элементарное условие с новой строки при превышении 120 символов; операторы И, ИЛИ — в начале строки; условия с отступом.

6. Можно использовать автоформатирование (Текст — Блок — Форматировать) и приложенную обработку для расстановки переносов.

См. также: Тексты модулей.

## Использование дублирующего кода

1. Дублирование кода — копирование существующих фрагментов без изменений при одинаковой по смыслу функциональности. Оно создаёт риски: копирование ошибок, пропуск вхождений при исправлениях, усложнение сопровождения. Следует избегать дублирования; общие алгоритмы выносить в общие модули.

2. Дублирование оправданно, если развитие функционала может привести к существенному расхождению двух вариантов кода.

**Пример предотвращения дублирования:** при разной обработке на клиенте и сервере не использовать в клиент-серверном модуле инструкции препроцессора (#Если НЕ ТонкийКлиент и т.п.); разделить реализацию на серверную и клиентскую процедуры, общую часть вынести в клиент-серверный модуль и вызывать из обеих (#std440).

См. также: Разработка конфигураций с повторным использованием общего кода и объектов метаданных.

## Использование директив компиляции и инструкций препроцессора

#std439

1. Директивы компиляции &НаКлиенте, &НаСервере, &НаСервереБезКонтекста следует применять только в модулях управляемых форм и модулях команд. В остальных модулях — инструкции препроцессора. В серверных/клиентских общих модулях контекст очевиден; в клиент-серверных общих модулях директивы затрудняют понимание доступности процедур.

2. Не использовать в клиент-серверных общих модулях проверки контекста (#Если Сервер, #Если Клиент) для ветвления логики — контекст исполнения ненадёжно определяем. Процедуры с разным поведением на клиенте и сервере размещать в модулях с постфиксами Клиент и Сервер. Допустимо ветвление для режимов клиента (например, #Если ВебКлиент).

3. Не разрывать инструкциями препроцессора и областями грамматические конструкции, выражения, объявления и вызовы процедур и функций. EDT диагностирует такие ошибки. Правильно использовать инструкции без разрыва конструкций.

## Определение типа значения переменной

Определение типа выполнять путём сравнения с типом, а не иначе.

**Правильно:**
```bsl
Если ТипЗнч(Ссылка) = Тип("ДокументСсылка.ПоступлениеТоваровУслуг") Тогда
```

**Неправильно:** `Если Ссылка.Метаданные().Имя = "ПоступлениеТоваровУслуг" Тогда`

## Получение метаданных объектов

#std442, #std445

1. Если тип объекта метаданных известен (справочник, документ и т.п.), получать метаданные через метод Метаданные() этого объекта (или ссылки), а не через глобальный контекст Метаданные — второй способ существенно медленнее.

**Правильно:** `СправочникОбъект.Метаданные()`  
**Неправильно:** `Метаданные.Справочники[ИмяСправочника]`, `Метаданные.НайтиПоПолномуИмени("Справочник." + ИмяСправочника)`

2. Если тип заранее неизвестен — использовать метод НайтиПоТипу, например:  
`ИмяОбъектаМетаданного = Метаданные.НайтиПоТипу(ТипЗнч(Ссылка)).ПолноеИмя();`

## Использование переменных в программных модулях

#std639

1. В большинстве случаев вместо переменных модулей следует использовать более подходящие средства платформы (область видимости переменных модулей сложно контролировать, они часто становятся источником трудновоспроизводимых ошибок). Рекомендации по оформлению — в статье «Структура модуля».

2. В модулях объектов: для передачи параметров между подписками и из внешнего кода — свойство ДополнительныеСвойства, а не экспортные переменные модуля. Для внутренней передачи между обработчиками событий — неэкспортные переменные модуля. Для кодов возврата (ошибок) — строковые константы, а не числовые переменные модуля.

3. В модулях форм: для хранения и передачи промежуточных результатов между процедурами использовать параметры процедур/функций и реквизиты формы (серверные переменные формы не сохраняются между вызовами с клиента). Исключение — клиентские переменные в обработчиках ожидания, внешних событий и клиентских обработчиках элементов формы. Не кешировать в переменных формы статическую и легко вычисляемую информацию (например, предопределённые значения — для них на клиенте предназначен метод ПредопределенноеЗначение).

См. также: Использование параметров сеанса.

## Предварительная инициализация локальных переменных

#std494

В фрагментах, где значение локальной переменной вычисляется в ветвлениях (Если/ИначеЕсли/Иначе), рекомендуется явно инициализировать переменную до ветвления, чтобы избежать ситуации, когда переменная остаётся Неопределено, а последующий код ожидает определённый тип.

**Правильно:** перед блоком Если присвоить значение по умолчанию (например, 0), затем в ветвлениях при необходимости переприсвоить.

Рекомендация особенно полезна для больших блоков Если/ИначеЕсли/Иначе.

## Использование Журнала регистрации

Область применения: управляемое приложение, обычное приложение. #std498

1. Журнал регистрации предназначен для хранения событий при работе пользователей с ИБ; администратор анализирует их в различных разрезах.

2. Запись из встроенного языка рекомендуется в тех случаях, когда нужно сообщить дополнительную диагностическую информацию о событиях, не записываемых платформой (интерактивная работа и фоновые задания). Одна запись — одному событию; записи должны содержать обязательные атрибуты для анализа.

2.1. Строковый идентификатор типа события. Типы группировать по функциональному признаку, например «НазваниеГруппы.НазваниеСобытия». Текст типа — локализуем, задаётся основной язык конфигурации.

2.2. Уровень важности: Ошибка (критичные события), Предупреждения, Информация, Примечание.

2.3. Комментарий — текстовая информация о событии. Для ошибок — сведения для расследования. В одной записи не помещать несколько событий. Для исключений использовать: `ПодробноеПредставлениеОшибки(ИнформацияОбОшибке())`. Текст комментария локализуем.

3. Не использовать выборку из журнала регистрации в задачах, критичных к высокой скорости выполнения (при больших объёмах скорость падает). Рекомендуется отдельный регистр или специализированные объекты платформы.

См. также: Строки интерфейса в модулях конфигурации: требования по локализации.

## Перехват исключений в коде

#std499

1. В общем случае не рекомендуется перехватывать исключения; в частности не перехватывать только ради выдачи сообщения об ошибке (необработанное исключение и так будет показано пользователю и при необходимости записано в журнал регистрации на сервере).

2. При необходимости перехвата (например, чтобы уточнить текст ошибки для пользователя) необходимо фиксировать причину в журнале регистрации для администратора. Рекомендуется записывать подробное представление исключения, а краткое добавлять в сообщение пользователю.

3.1. Серверная бизнес-логика, вызываемая с клиента: не маскировать исходную проблему на клиенте общим сообщением; на сервере в Попытка/Исключение записывать в журнал ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()) и выполнять ВызватьИсключение; на клиенте показывать пользователю краткое представление. Не использовать ОписаниеОшибки (не возвращает стек).

3.2. Не анализировать текст исключений для интерпретации причины (текст может меняться при локализации). При необходимости различать типы ошибок — использовать коды возврата (строковые литералы: «Успешно», «НетМестаНаДиске» и т.п.), а не числовые коды.

3.3. При клиентской бизнес-логике при перехвате на клиенте рекомендуется выполнять серверный вызов для записи ошибки в журнал регистрации.

3.4. Недопустимо перехватывать любые исключения бесследно для администратора (без записи в журнал). Если перехват всех исключений обоснован — пояснить в комментарии и записать событие в журнал.

3.5. В событиях ПриЗаписи, ПередЗаписью, ОбработкаПроведения, ОбработкаПроверкиЗаполнения и т.п. не вызывать исключения для останавливающих предупреждений, которые пользователь должен отработать; устанавливать Отказ = Истина и выводить сообщения. Исключение допустимо, если пользователь не может отработать предупреждение в рамках операции или ситуация исключительная. См. Информирование пользователя.

3.6. Недопустимо проверять наличие реквизитов, методов, макетов и т.п. через перехват исключений. Использовать механизмы метаданных для явной проверки; при различиях из-за встраивания библиотек — описывать в переопределяемых модулях или пересмотреть логику (параметры, определяемые вызывающим кодом).

## Использование модуля объекта, модуля менеджера объекта и общих модулей

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std486

1. Модуль объекта — для реализации поведения отдельного экземпляра (СправочникОбъект, ДокументОбъект и т.п.). В нём размещаются процедуры и функции, работающие с данными объекта (ЭтотОбъект, переменные модуля), в т.ч. до записи в ИБ: обработчики событий объекта, процедуры заполнения. Вызов экспортных процедур/функций модуля объекта из других модулей может требовать предварительной загрузки объекта через ПолучитьОбъект (ресурсоёмко). См. также: Чтение отдельных реквизитов объекта из базы данных.

2. Модуль менеджера объекта — для «статической» функциональности, связанной с объектом метаданных, но не зависящей от конкретного экземпляра: функции для совокупности объектов (печать списка, общая информация, обновление ИБ); функции по ссылке на записанный объект (печатная форма, формирование движений по ссылке). Для выполнения не должен требоваться экземпляр объекта данных.

3. Если функциональность нельзя однозначно отнести к одному объекту метаданных — она общая для нескольких объектов; её следует размещать в общем модуле.

## Ограничения на использование экспортных процедур и функций

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std544

Не следует размещать экспортные процедуры и функции в модулях команд и общих команд. К этим модулям нет возможности обращаться из внешнего кода, поэтому экспорт в них не имеет смысла.

## Использование объектов типа Структура

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std693, #std407

Требования направлены на читаемость кода и упрощение изменений при коллективной разработке и доработках на внедрениях.

1. При создании Структуры не рекомендуется передавать в конструктор более трёх значений свойств. Использовать метод Вставить или присваивание свойств явно.

2. В конструкторе структуры не рекомендуется использовать конструкторы других объектов с параметрами; в частности не создавать вложенные структуры с объявлением свойств в одном вызове. Создать внешнюю структуру, заполнить свойства по отдельности, затем передать в функцию.

3. В конструкторе структуры не рекомендуется вызывать функции с большим (более трёх) количеством параметров. Создать структуру, затем присвоить свойствам результаты вызовов.

См. также: Параметры процедур и функций; Особенности использования структур в качестве параметров процедур и функций.

## Порядок записи движений документов

Область применения: управляемое приложение, мобильное приложение, обычное приложение. #std448, #std450

1. Не рекомендуется использовать явную запись наборов записей регистров (метод Записать) в процедурах обработки проведения документов. Запись должна производиться неявно системой при завершении процедуры проведения. Нарушение может приводить к взаимным блокировкам при параллельной работе пользователей.

2. Исключение — ситуация, когда данные, сохраняемые в регистрах, нужны в последующих алгоритмах до выхода из процедуры проведения.

См. также: Требования к проведению документов; Самодостаточность регистров; Использование активности движений.
