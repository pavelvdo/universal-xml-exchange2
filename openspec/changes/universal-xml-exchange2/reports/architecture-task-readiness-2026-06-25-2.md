# Task Readiness — universal-xml-exchange2 (после extend 2026-06-25, фокус S2/S3)

## Вердикт

**ГОТОВО С ЗАМЕЧАНИЯМИ** — S2 и S3 реализуемы силами writer (BSL) и пользователя (ручная конфигурация в Конфигураторе) по текущим артефактам. Блокирующих пробелов нет. Три замечания (criteria 3, 1/2, 7) — авто-исправимые уточнения постановки, не требующие пересмотра подхода.

Контекст проверки фактов по коду:
- Серверный модуль `рг_УниверсальныйОбменXMLСервер` сейчас — заглушка `ВыполнитьОбменПоПрофилю` (лог + возврат); ни одна из функций S2.7–S2.16 / S3.1–S3.6 ещё не реализована (ожидаемо для непринятых срезов).
- Веб-сервис `rgExchangeService` в `src/` отсутствует — это ручная конфигурация S2.3 (`[ ]`), ожидаемо.
- As-built форма обработки `УправляемаяФорма` (полная типовая «Универсальный обмен данными XML», ~2080 строк) **не содержит** выбора профиля-приёмника и команды «Выполнить обмен» (grep). Условие S2.2a «если отсутствует» разрешается фактом → элемент нужно создать.
- Движок `рг_УниверсальныйОбменДаннымиXML` — as-built в `cfe`, публичный API зафиксирован (S2.1 `[x]`, design § Engine Contract).

## Пробелы

### GAP-1 (criterion 3) — S2.2a сформулирована условно, условие уже разрешено фактом

- **Задача:** S2.2a (`tasks.md`).
- **Что отсутствует / неоднозначно:** формулировка «…команда «Выполнить обмен», **если отсутствует** на as-built форме» оставляет задачу с двумя путями. Механическая проверка as-built `УправляемаяФорма` (grep: нет ни выбора профиля-приёмника, ни команды «Выполнить обмен») детерминированно разрешает условие в «создать». Условную формулировку нужно зафиксировать как безусловную, иначе на apply возможен возврат «а есть ли уже?».
- **Рекомендация:** заменить условную формулировку на безусловную «создать выбор профиля-приёмника и команду «Выполнить обмен»» с явным именем целевой формы.
- **Авто-вставка (замена строки S2.2a):**
  `- [ ] S2.2a Ручное конфигурирование (Конфигуратор): на форме обработки \`УправляемаяФорма\` создать реквизит выбора профиля-приёмника (\`СправочникСсылка.рг_НастройкиОбменаXML\`, отбор \`Источник = Ложь\`) и команду «Выполнить обмен» (на as-built форме отсутствуют — проверено grep) (design D4, § Manual Configuration; S2.17)`

### GAP-2 (criterion 1) — S2.17 не называет целевую форму обработки

- **Задача:** S2.17 (`tasks.md`).
- **Что отсутствует / неоднозначно:** путь «модуль формы обработки `рг_УниверсальныйОбменДаннымиXML` (`Ext/Form/Module.bsl`)» неполон — у обработки две формы (`УправляемаяФорма`, `УправляемаяФормаНастройкиОтладкиОбработчиков`). Writer не определит целевую однозначно.
- **Рекомендация:** указать форму `УправляемаяФорма` и полный путь.
- **Авто-вставка (замена строки S2.17):**
  `- [ ] S2.17 В модуле формы обработки \`УправляемаяФорма\` (\`DataProcessors/рг_УниверсальныйОбменДаннымиXML/Forms/УправляемаяФорма/Ext/Form/Module.bsl\`): по команде «Выполнить обмен» для выбранного профиля-приёмника запускать серверный сеанс и показывать результат пользователю (spec exchange-import «Распаковка и загрузка XML»)`

## К сведению (не блокирует apply)

- **Behavior Contract, строка про ошибку:** для устранения размытости рекомендуется развести в design два сценария явно — «ошибка фазы 2 выгрузки на источнике → Ошибка» (D2) и «ошибка загрузки на приёмнике → источник остаётся Выполняется, confirm не вызывается» (Q1, S3.6). Содержательного противоречия нет, правка косметическая.
- **SUBOPTIMAL (criterion 7):** графт профильной команды в типовую форму `УправляемаяФорма`. Если в будущем потребуется изоляция — отдельная компактная форма/команда обработки для запуска обмена по профилю-приёмнику.

## Simplicity Check

- **Viable alternatives:**
  1. Встроенный движок (Option B, выбран) + сеанс в общем модуле `рг_УниверсальныйОбменXMLСервер` + тонкие обёртки WS + команда на типовой форме.
  2. Внешний epf-движок (Option A) — отклонён в extend (внешние обработки на сервере, админ-режим безопасности).
- **Selected simplest viable design:** вариант 1 — без внешних epf, без новых объектов сверх already-planned; вся серверная логика в одном общем модуле, WS — тонкие обёртки.
- **Why not simpler:** двухфазная модель (check-and-set под блокировкой + выгрузка вне транзакции + ConfirmDataReceipt) обязательна по Behavior Contract (инварианты атомарности и завершения сеанса); убрать фазы нельзя без нарушения spec.
- **Complexity budget:** новых объектов метаданных в коде — 0 (создаются вручную S1/S2); точек перехвата платформы — 0; новые процедуры/функции — серверные S2.7–S2.16 + S3.1–S3.6 (~12), 2 операции WS, 1 команда формы; условных ветвлений сверх контрактов статусов — нет.
