# Шаблон чата `/opsx:verify` — тон «архитектор — разработчику»

**Принцип.** В чат идёт **одно** финальное сообщение оркестратора после полного прогона (включая internal Repair Loop). Тон — архитектор, объясняющий разработчику 1С, что найдено в постановке. Chat Surface Contract — §2.6 `opsx-output-style.md`.

**Главный вопрос разработчика:** «могу ли я безопасно запустить `/opsx:apply`?». Ответ — одна из формулировок в первой строке (`templates/verdict-card.md`).

**Обязательный слот (HARD):** перед ссылкой на отчёт — **`**Следующий шаг:**`** с одной user-action командой или приглашением ответить в чате. Команда по `verify_mode` из snapshot / grep `tasks.md`:

| verify_mode | GO / silent_ok 1a |
|-------------|-------------------|
| `pre-apply` | `/opsx:apply <change-name>` |
| `post-apply` | `/opsx:archive <change-name>` |

**Запреты в чате** — см. `.cursor/rules/chat-output-budget.mdc` §7.

---

## Вариант 1 — «статус прежний» (silent_ok после фильтра новизны)

Прогон без новизны: те же `[x]`, артефакты не правились, challenge актуален. **Новый файл отчёта не создаётся.** Опора **только** на YAML `snapshot`, mtime и тело последнего отчёта — **не** на историю чата. `verify_mode` — из YAML snapshot последнего отчёта.

### 1a. Прошлый отчёт — «можно apply»

```
<change-name> — изменений в постановке с <дата прошлого отчёта> не было, статус прежний: можно запускать apply.

**Следующий шаг:** `/opsx:apply <change-name>`  *(post-apply: `/opsx:archive <change-name>`)*

Последний отчёт: openspec/changes/<name>/reports/verification-<дата>.md.
```

### 1b-decision. Прошлый отчёт — открыта развилка

Формат — как **3a-decision** ниже; текст из `## Решения до apply` последнего отчёта.

### 1b-repair. Прошлый отчёт — только repair-блокеры

**Не** выводить промежуточное сообщение. Запустить **Repair Loop** (internal repair-from-verify + re-verify). Финал — как после полного прогона (вариант 2 или 3a-decision, или terminal fail после 2 attempts).

## Вариант 2 — «можно запускать apply» (после прогона, без блокеров)

```
<change-name> — можно запускать apply. <Опционально: одна техническая фраза.>

<Опционально 2–3 предложения: что делает план в коде. Максимум 1 имя процедуры/модуля как анкор.>

<Подправил в постановке — только если были авто-правки, одной строкой простым языком.>

**Следующий шаг:** `/opsx:apply <change-name>`  *(post-apply: `/opsx:archive <change-name>`)*

Полный отчёт: openspec/changes/<name>/reports/verification-YYYY-MM-DD.md
```

### Вариант 2b — GO-saturated (decision cap, остаточный риск)

```
<change-name> — можно запускать apply. Остаточный риск <проза одной фразой> проверяется при приёмке <S<N>.accept / сценарий spec>.

<Опционально: что уже зафиксировано в постановке — 1 строка.>

**Следующий шаг:** `/opsx:apply <change-name>`  *(post-apply: `/opsx:archive <change-name>`)*

Полный отчёт: openspec/changes/<name>/reports/verification-YYYY-MM-DD.md
```

**Запрещено в чате:** «GO-saturated», «decision cap», agent-keys.

## Вариант 3a-decision — нужен выбор пользователя (максимум один блок)

```
<change-name> — до старта нужен ваш выбор по логике <тема>.

<Если decision_round >= 1 — блок «Уже зафиксировано» (2 строки прозой, без D0/D3/decision_id):>
Уже зафиксировано: <краткий перечень закрытых решений прозой>.
Новый вопрос: <что именно нового>; прежние решения не пересматриваются.

**Что решить: <заголовок-проблема одной строкой>**

<Связный абзац (2–3 предложения): что в постановке не закрыто и чем это аукнется для пользователя 1С / бизнеса — проблема и влияние сплетены, без подзаголовков. При необходимости — одно имя процедуры/модуля как анкор.>

- **A. <Вариант>** — <что получите, с честным «но»>. <Опционально одной фразой: «не отменяет <закрытое решение прозой>».>
- **B. <Вариант>** — <что получите, с честным «но»>.

**Следующий шаг:** ответьте в чате (A или B). После фиксации в постановке — снова `/opsx:verify <change-name>`.

Полный отчёт: openspec/changes/<name>/reports/verification-YYYY-MM-DD.md
```

**Supersedes (эскалация):** если classifier вернул `supersedes` — один абзац «Подтверждаете …, несмотря на …?» вместо A/B; слот «**Следующий шаг:** ответьте в чате: да / нет / уточните»; END TURN.

**END TURN.** Repair Loop **не** запускать до ответа.

## Вариант 4 — repair не удалось за N итераций

После 2 internal repair attempts блокеры repair-класса остались:

```
<change-name> — apply пока нельзя: не удалось автоматически дописать постановку за 2 итерации.

<2–3 предложения сути на языке эффекта: что именно не сходится, без списка файлов и internal-команд.>

**Следующий шаг:** опишите в чате, как поступить, или `/opsx:extend <change-name>`.

Полный отчёт: openspec/changes/<name>/reports/verification-YYYY-MM-DD.md
```

## Правила оформления (HARD)

- **Repair Loop — 0 промежуточных сообщений** (кроме progress marker «Дописываю постановку…»).
- **Запрещено в чате:** `/opsx:extend --from-verify`, «Запускаю extend», «Подтвердить?» на repair-path, перечень изменённых файлов, «ничего не требуется» без user-action команды на GO.
- **Максимум один блок decision** в сообщении.
- **Без таблиц** в чате.
- **Смешение «ничего не требуется» + «Подтвердить?»** — запрещено.
- **Decision 3a:** слот «**Следующий шаг:**» **заменяет** финальную строку «Напишите, какой вариант ближе…» (один CTA, не два).

## После решения пользователя (decision)

Пользователь отвечает свободным текстом. Оркестратор:

1. Одно предложение — перефраз выбора на языке кода 1С (без отдельного «Понял», см. §4 `chat-output-budget.mdc`): «Выбор: <пересказ>.»
2. Handoff на языке эффекта: «Зафиксирую выбор в постановке. Следующий шаг: `/opsx:verify <name>`.» — **без** internal-команд и путей.
3. Internal: user-extend `--from-verify` (lite confirm допустим).

Без перечисления правок в `design.md` / `tasks.md`.
