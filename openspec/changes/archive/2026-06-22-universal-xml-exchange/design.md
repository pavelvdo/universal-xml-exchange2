## Context

Расширение `рг_УниверсальныйОбменXML` на ЗУП КОРП 3.1 — первая площадка универсальной подсистемы обмена данными между ИБ 1С. Сейчас в выгрузке расширения только каркас (Configuration, роль, язык); объекты подсистемы отсутствуют.

**Топология:** база-источник публикует SOAP-веб-сервис и хранит макеты правил обмена; база-приёмник хранит профили подключения, инициирует сеанс, получает ZIP с XML и загружает данные обработкой `ргУниверсальныйОбменXML` расширения.

**Ограничения:** правки только в `src/ЗУП/cfe/рг_УниверсальныйОбменXML/`; эталон `src/ЗУП/cf/` — read-only; префикс расширения `обменXML_`.

## Goals / Non-Goals

**Goals:**

- Профили обмена в справочнике с безопасным хранением пароля и параметрами фильтрации
- SOAP `rgExchangeService.GetData` на источнике
- Выгрузка по правилам из макета с упаковкой в ZIP
- Загрузка на приёмнике через обработку `ргУниверсальныйОбменXML` расширения (единый движок на всех базах)
- Ручной и регламентный запуск обмена

**Non-Goals:**

- Автоматический поиск ссылок по примитивным параметрам (ответственность обработчиков правил КД 2.1)
- Замена типового обмена данными ЗУП / EnterpriseData
- Перенос на другие конфигурации в рамках этого change (заложена переносимость, реализация — только ЗУП)
- Двусторонняя синхронизация и очереди сообщений

## Existing Mechanisms

| Механизм | Контракт / назначение | Почему не используется как есть |
|----------|----------------------|----------------------------------|
| Подсистема «Обмен данными» ЗУП (узлы, планы обмена, EnterpriseData) | Штатная синхронизация между ИБ по планам обмена | Требуется pull-модель: приёмник инициирует выгрузку у источника по произвольным правилам из макета, без преднастроенного узла |
| Универсальный обмен данными (`УниверсальныйОбменДаннымиXML`, правила .xml, ПВД) | Выгрузка/загрузка по правилам конвертации КД 2.1 | **Используется** как эталон: обработка расширения `ргУниверсальныйОбменXML` создаётся на базе штатной cf с доработками по ТЗ |
| `ОбщегоНазначения.ЗаписатьДанныеВБезопасноеХранилище` / `ПрочитатьДанныеИзБезопасногоХранилища` (БСП) | Хранение паролей вне реквизитов | **Используется** для пароля профиля обмена |
| Регламентные задания БСП | Программное создание/обновление расписания | **Используется** для привязки профиля к `ргВыполнениеОбменаXML` |
| Типовые транспорты обмена (COM, EMAIL, WS-клиенты планов обмена) | Транспорт сообщений планов обмена | Другая модель данных и топология; нужен отдельный контракт GetData |

**Выбранный уровень Preference Hierarchy:** 3 (адаптация паттерна базы) — собственные объекты расширения, но движок и БСП-механизмы переиспользуются.

## Design Rationale

1. **Справочник профилей** — единая точка настройки для пользователя и регламентного задания; паттерн аналогичен регистрам настроек синхронизации в типовой конфигурации, но с доменной моделью «URL + макет + параметры».
2. **Веб-сервис на источнике** — единственная точка сетевого контракта; параметры передаются XDTO Structure (стандарт платформы), ответ — Base64 ZIP (сжатие трафика).
3. **Общий модуль `ргУниверсальныйОбменXMLСервер`** — серверная логика выгрузки/загрузки и обработчик регламента в одном месте; веб-сервис — тонкая обёртка.
4. **Обработка `ргУниверсальныйОбменXML`** — создаётся в расширении на базе штатной `УниверсальныйОбменДаннымиXML` (`src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML/`) с доработками по ТЗ; единый движок на источнике и приёмнике через `Обработки.ргУниверсальныйОбменXML.Создать()`, без внешнего epf и без прямого вызова типовой обработки cf в runtime.
5. **Параметры только примитивы** — упрощение сериализации XDTO; преобразование в ссылки — явно в правилах, без «магии» в ядре.

## Decisions

### D1. Хранение пароля

**Выбор:** БСП безопасное хранилище, владелец — ссылка элемента справочника, ключ `"Пароль"`.

**Альтернатива:** реквизит с пометкой «хранить в базе в зашифрованном виде» — отклонена (требование ТЗ).

### D2. Контракт GetData

**Выбор:** `GetData(ИмяМакетаПравил: String, Параметры: Structure) → Base64Binary`.

**Альтернатива:** передача параметров JSON-строкой — отклонена (нативный XDTO Structure).

### D3. Управление регламентными заданиями

**Выбор:** на каждый профиль — отдельный экземпляр регламентного задания `рг_ВыполнениеОбменаXML` с параметром `[СсылкаНаПрофиль]`; идентификатор задания хранится в реквизите `ИндентификаторРегламентногоЗадания` (UUID); расписание — в реквизите `Расписание` (ХранилищеЗначения) и синхронизируется в задание при записи профиля по флагу `ИспользоватьРасписание`. Создание/изменение/удаление задания — в модуле объекта справочника через `РегламентныеЗаданияСервер` в привилегированном режиме (паттерн `Справочники.РассылкиОтчетов`).

**Точка входа регламента:** `рг_УниверсальныйОбменXMLСервер.ВыполнитьОбменПоПрофилю(Профиль)` — при вызове из регламентного задания ссылка на профиль передаётся параметром задания.

**Права роли на регламентное задание не назначаются:** в платформе 1C тип `ScheduledJob` отсутствует в правах роли (подтверждено: 0 записей `ScheduledJob.*` в Rights.xml типовой ЗУП).

**Альтернатива:** одно общее задание со списком профилей — отклонена (независимые расписания на профиль).

### D4. Именование объектов (as-built выгрузка 2026-06-22)

**Объекты расширения:** `рг_НастройкиОбменаXML`, `рг_УниверсальныйОбменXMLСервер`, `рг_ВыполнениеОбменаXML`, `рг_ИспользованиеОбменаXML` (роль), `rgExchangeService`, `ргУниверсальныйОбменXML` (обработка S2 — ещё не выгружена).

**Реквизиты справочника** (без префикса `обменXML_`): `АдресСервиса`, `Логин`, `ИмяМакетаПравил`, `ИспользоватьРасписание`, `Расписание`, `ИндентификаторРегламентногоЗадания`; ТЧ `Параметры` (`ИмяПараметра`, `ЗначениеПараметра`); на форме — нехранимый `Пароль`.

### D5. Контракт WS-операции и аутентификация

**Выбор:** XDTO-пакет `http://v8.1c.ru/8.1/data/core`, тип параметра `Structure` (значения — строка, число, дата, булево); тип результата `base64Binary` (ZIP). Учётные данные — HTTP Basic на уровне WS-прокси (`ИмяПользователя`/`Пароль`), не параметры `GetData`.

**Альтернатива:** передача логина/пароля в теле GetData — отклонена (требование безопасности и типовой практики публикации).

### D6. Происхождение движка универсального обмена (verify 2026-06-22)

**Выбор:** обработка расширения `ргУниверсальныйОбменXML` создаётся на базе штатной `УниверсальныйОбменДаннымиXML` ЗУП (эталон `src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML/`) с доработками по ТЗ (pull-модель, серверный API, макеты правил, форма запуска по профилю). На всех базах участников обмена используется только `Обработки.ргУниверсальныйОбменXML.Создать()`.

**Альтернатива A (отклонена):** внешний `V8Exch82.epf` в макете `ДвижокОбмена` — риск несовместимости с платформой, дублирование штатного механизма cf.

**Альтернатива B (отклонена частично):** прямой runtime-вызов `Обработки.УниверсальныйОбменДаннымиXML` из cf — приёмник может не содержать типовую обработку; нет точки для доменных доработок по ТЗ.

## Решения verify (зафиксировано)

- **Движок обмена:** `ргУниверсальныйОбменXML` на базе штатной `УниверсальныйОбменДаннымиXML`, единая обработка расширения на всех базах (ответ пользователя 2026-06-22, decision `engine-origin`).
- **As-built имена метаданных:** артефакты синхронизированы с выгрузкой `src/ЗУП/cfe/рг_УниверсальныйОбменXML/` (verify repair 2026-06-22).
- **Права на регламентное задание:** не назначаются в роли; управление заданиями — `РегламентныеЗаданияСервер` + привилегированный режим в модуле объекта справочника (паттерн БСП).
- **Тип реквизита `Расписание` (объект):** только **ХранилищеЗначения** в метаданных справочника. **На форме** ХранилищеЗначения и `DataPath>Объект.Расписание` **недоступны** (ограничение платформы/Конфигуратора). Черновик сеанса — **реквизит формы `Расписание`** с произвольным типом (`<Type/>`, паттерн `Справочники.РассылкиОтчетов`); персистентность — `ТекущийОбъект.Расписание = Новый ХранилищеЗначения(Расписание)` + `ДополнительныеСвойства` в `ПередЗаписьюНаСерверe`. В BSL **запрещено** `Тип("РасписаниеРегламентногоЗадания")`; проверка — `РасписаниеЗаполнено()` (verify repair 2026-06-22, decisions `schedule-form-requisit-rassylki`, `schedule-no-type-name-check`).

## Behavior Contract

### Профиль обмена

- Пользователь создаёт элемент справочника с URL, логином, именем макета и строками параметров
- Пароль не отображается и не хранится в открытом реквизите после записи
- При включённом флаге регламентного обмена и заполненном расписании — создаётся/обновляется регламентное задание

### GetData (источник)

- При валидных учётных данных и существующем макете правил — возвращается Base64 ZIP с XML
- Параметры из Structure мапятся в параметры обработки обмена перед вызовом выгрузки
- Все ПВД из правил активируются; ядро не ищет ссылки по примитивам

### Сеанс обмена (приёмник)

- По выбранному профилю: чтение настроек, пароля из хранилища, вызов WS с HTTP Basic, распаковка ZIP во временный каталог на сервере, загрузка через `Обработки.ргУниверсальныйОбменXML.Создать()`
- Временные файлы и каталоги удаляются после сеанса (в `Попытка`/`Исключение`)
- Ошибки соединения и разбора ответа — сообщение пользователю / запись в журнал регистрации без «тихого» сбоя

### Форма обработки

- Ручной запуск обмена: форма `ргУниверсальныйОбменXML` — выбор профиля, команда «Выполнить обмен»

## Implementation Options

### Option A — монолитный общий модуль (выбрано)

Вся серверная логика в `ргУниверсальныйОбменXMLСервер`; форма обработки вызывает экспортные методы.

**Плюсы:** проще навигация в первой версии, один контур для регламента и UI.

### Option B — разделение модулей «Клиент» / «Сервер» / «ПовтИсп»

**Отклонено:** избыточно для объёма v1.0; возможен рефакторинг позже.

### Option C — транспорт: SOAP-веб-сервис vs HTTP-сервис (выбрано SOAP)

**Выбор:** SOAP `GetData` с XDTO `Structure` → `base64Binary` ZIP.

**Альтернатива HTTP-сервис:** бинарный ZIP телом ответа без +~33 % base64; проще крупные пакеты. Отклонена для v1: нет готового типизированного контракта `Structure` и WS-прокси с HTTP Basic из коробки; параметры и аутентификацию пришлось бы описывать вручную.

**Trade-off:** осознанно принимаем раздувание трафика base64 ради типобезопасности параметров и стандартной публикации SOAP на ЗУП 3.1 (D5, D7).

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| API штатной обработки отличается от ожидаемого при adopt | S2.6: static-сверка с `cf/.../УниверсальныйОбменДаннымиXML` до BSL S2.7; контракт в § Engine Contract |
| Большие выгрузки — таймаут SOAP | Лимит размера выгрузки документировать в § Engine Contract; при превышении — ошибка сеанса с сообщением пользователю/запись в журнал; таймаут публикации веб-сервиса настраивается администратором источника |
| Ошибки в правилах обмена (ссылки) | Явный контракт: ответственность правил; сообщение об ошибке из движка |
| Дублирование кода выгрузки/загрузки | Общие процедуры в серверном модуле |
| Обновление типовой ЗУП меняет эталон adopt | Версия расширения фиксирует базовую ревизию; при обновлении cf — регресс S2.accept |

## Assumptions

- На источнике опубликован веб-сервис с HTTP Basic, поддерживаемой WS-прокси платформы
- Расширение с обработкой `ргУниверсальныйОбменXML` установлено на источнике и приёмнике одинаковой версии
- Эталон adopt — `src/ЗУП/cf/DataProcessors/УниверсальныйОбменДаннымиXML/` (ЗУП 3.1.30.197); публичный API (имена методов выгрузки/загрузки, параметры) фиксируется в § Engine Contract при S2.6
- Правила обмена содержат обработчики преобразования примитивов в ссылки там, где нужны отборы
- Регистр БСП «БезопасноеХранилищеДанных» принимает ссылку справочника расширения как владельца

## Engine Contract

Заполняется задачей S2.6 (static-сверка с `cf/.../УниверсальныйОбменДаннымиXML/Ext/ObjectModule.bsl`):

- Способ создания: `Обработки.ргУниверсальныйОбменXML.Создать()`
- Обязательные макеты adopt: `ОбщиеПроцедурыФункции`, `ПараметрыОбработчиков` (из эталона cf)
- Методы выгрузки/загрузки, имена параметров обмена — по результату S2.6 (static-сверка cf)
- При превышении лимита размера выгрузки — ошибка сеанса, загрузка не выполняется
- Ядро `ргУниверсальныйОбменXMLСервер` не выполняет автоматический поиск ссылок по примитивам

## Slices

Декомпозиция — 2 вертикальных среза (Standard tier). Серверная выгрузка отдельным срезом не выделяется: наблюдаемый результат — end-to-end на приёмнике.

### Срез S1: Профили обмена и регламентные задания

**Сценарий:** администратор создаёт/редактирует профиль обмена (URL, логин, имя макета правил, параметры фильтрации), пароль сохраняется в безопасном хранилище, при включении регламента создаётся/обновляется/удаляется задание `ргВыполнениеОбменаXML`.
**Primary acceptance:** создать элемент `ргНастройкиОбменаXML` → заполнить URL, логин, имя макета, строки параметров, указать пароль → записать → переоткрыть: пароль не виден в реквизите, профиль сохранён.
**Приёмка:** ручной тест в ИБ с развёрнутым расширением.
**Связь со spec:** exchange-settings — все Requirements и Scenarios профиля и регламента.
**Зависимости:** нет.

### Срез S2: Сквозной обмен данными (источник + приёмник)

**Сценарий:** оператор на приёмнике выбирает профиль и запускает обмен; приёмник вызывает `GetData`; источник выгружает по правилам; приёмник загружает данные обработкой `ргУниверсальныйОбменXML`.
**Primary acceptance:** на источнике опубликован `rgExchangeService` → на приёмнике открыть обработку → выбрать профиль → запустить обмен → данные из источника появились в базе-приёмнике.
**Приёмка:** ручной end-to-end тест на двух ИБ.
**Связь со spec:** exchange-web-service, exchange-export, exchange-import; exchange-settings — Scenario «Использование пароля при обмене».
**Зависимости:** S1.

## Migration Plan

1. Установить расширение на источник и приёмник
2. Опубликовать `rgExchangeService` на источнике
3. Создать профиль на приёмнике, проверить ручной обмен
4. Настроить регламентное задание на профиле

**Rollback:** отключить регламентные задания, удалить профили; отключить публикацию сервиса на источнике.
