1.1. Конфигурируемость

1.2. Функционирование системы

Система «1С:Предприятие» состоит из платформы (среда разработки и исполнения), прикладных решений, методологии их создания и ИТ-поддержки. Конфигурируемость: набор объектов, структуры данных и алгоритмы задаются конфигурацией. Конфигурация создаётся и сопровождается штатными средствами платформы.

Функционирование делится на разработку (описание модели предметной области) и исполнение (обработка данных). На этапе разработки: формирование структуры информации; создание форм ввода и списков; организация хранения; отчёты и обработки; командные интерфейсы; пользователи и права. Результат — конфигурация как модель предметной области. В режиме Конфигуратор — создание и редактирование конфигураций, сравнение и объединение.

На этапе разработки система оперирует объектами: документ, журнал документов, справочник, реквизит, форма, регистр и др. На этапе исполнения — конкретными понятиями, описанными в конфигурации; обработка выполняется штатными средствами и алгоритмами конфигурации.

1.3. Основные понятия системы

1.3.1. Понятие «конфигурация»

Конфигурация — совокупность взаимосвязанных частей:
- подсистемы;
- структуры учётных данных, форм ввода, выбора, печати;
- механизмов учёта итогов и движений;
- отчётов и обработок;
- командного интерфейса;
- набора ролей (прав доступа);
- общих процедур и функций (модуль приложения, модуль управляемого приложения, модуль внешнего соединения, модуль сеанса, общие модули), макетов табличных документов;
- функциональных опций и их параметров;
- хранилищ настроек;
- средств работы с Web (Web-сервисы, WS-ссылки, HTTP-сервисы);
- вспомогательных объектов (картинки, шаблоны, стили).

Информационная структура проектируется на уровне типов объектов (константы, справочники, документы, регистры, перечисления и др.). Назначение ролей ограничивает доступ к конфиденциальной информации и к операциям (удаление, корректировка). Права назначаются только для существующих объектов конфигурации. Командный интерфейс и формы автоматически учитывают права доступа.

1.3.2. Объект конфигурации

1.3.2.1. Свойства объекта конфигурации

Объект конфигурации — формальное описание группы понятий со сходными характеристиками и назначением (например, тип Справочник — для списков однородных элементов). Объект описывает вид значения, а не конкретное значение: схема реквизитов и форм. Реализованный компьютерный аналог понятия предметной области — объект конфигурации.

Каждый объект обладает уникальным набором свойств, заданным на уровне системы. Главное свойство — имя (краткое наименование). При создании присваивается условное имя по виду и цифре (Реквизит1, Документ1). Имя не может быть длиннее 80 символов.

Часть свойств доступна для редактирования в конфигураторе; характер и пределы изменений заданы системой. Изменение свойств не меняет сущности объекта и не добавляет действий, не свойственных данному типу.

1.3.2.2. Основные виды объектов конфигурации

1.3.2.2.1. Константы

Для постоянной и условно постоянной информации. Значение вводится редко, используется часто. Количество констант неограниченно. Важные редактируемые свойства: имя, синоним, комментарий, тип данных, режим управления блокировкой, ссылка на модуль менеджера. Если данные зависят от времени — использовать регистр сведений без измерений, а не константу.

1.3.2.2.2. Справочники

Для постоянной информации с множеством значений. Настраиваемые свойства: длина и тип кода, количество уровней иерархии, уникальность кодов, набор реквизитов. Помимо кода и наименования — реквизиты и табличные части. Типы форм: элемента, группы, списка, выбора, выбора группы; по каждому типу — произвольное число форм. Подчинённые справочники: элемент «принадлежит» элементу справочника-владельца.

1.3.2.2.3. Перечисления

Постоянные наборы значений, не изменяемые при работе конфигурации. Значения задаются при конфигурировании и не меняются на этапе исполнения. Отличие от справочников: пользователь не может добавить новое значение перечисления.

1.3.2.2.4. Документы

Для отражения хозяйственных событий. Настраиваются: структура, номер, дата, время, длина номера, уникальность номеров. Несколько табличных частей — когда одним документом регистрируют разные по смыслу, но связанные события. Формы: ввода, выбора, списков, печатные — в неограниченном количестве. Все документы образуют единую хронологическую последовательность; внутри даты порядок задаётся временем документа. Проведение: непроводимый документ не влияет на учёт; проводимый изменяет состояние учёта и может формировать движения по регистрам.

1.3.2.2.5. Журналы документов

Для просмотра документов разных видов в одном списке. Один вид документа может входить в несколько журналов. Графы журнала отображают реквизиты документов разных видов (например, одна графа «Контрагент» для разных реквизитов разных документов). Неограниченное число форм и печатных форм.

1.3.2.2.6. Отчеты и обработки

Неограниченное число отчётов и обработок. Алгоритм отчёта — на встроенном языке или автоматически (система компоновки данных). Вывод — текстовый или табличный (макеты). Поддержка внешних обработок в отдельных файлах.

1.3.2.2.7. Планы видов характеристик

Для описания множеств однотипных объектов аналитического учёта.

1.3.2.2.8. Планы видов расчета

Для видов расчётов в механизмах периодических расчётов.

1.3.2.2.9. Планы счетов

Совокупность синтетических счетов для группировки информации о хозяйственной деятельности (бухгалтерский учёт).

1.3.2.2.10. Планы обмена

Организация обмена данными между информационными базами и внешними системами.

1.3.2.2.11. Бизнес-процессы и задачи

Формализованные последовательности работ; на их основе — списки задач по сотрудникам и этапам.

1.3.2.2.12. Регистры

Хранение и обработка информации, не имеющей объектной природы (изменения состояний, курсы валют, приход/расход и т.п.). Виды регистров: регистры сведений, регистры накопления, регистры расчётов, регистры бухгалтерии.

1.3.2.2.13. Специализированные объекты конфигурации (ветвь «Общие»)

Командный интерфейс, критерии отбора, права доступа; стили, библиотеки картинок; модуль приложения, общие модули; общие макеты и др.

1.3.2.3. Подчиненные группы объектов

Зависят от вида объекта: реквизиты, табличные части, формы, команды, макеты, графы (журналы), измерения и ресурсы (регистры). Реквизиты табличных частей — состав табличной части. Группы подчинённых объектов не удаляются и не имеют редактируемых свойств.

ВНИМАНИЕ! Число строк одной табличной части не может быть более 100 000.

Кратко по подчинённым: реквизиты — дополнительная информация в пределах объекта; табличные части — табличная дополнительная информация; формы — ввод, просмотр, редактирование, содержат модуль формы; команды — независимые и параметризуемые; макеты — табличные, HTML, текстовые (и двоичные, Active) для печатных форм; графы — графы журнала документов; измерения — объекты, данные о которых учитываются в регистре; ресурсы — данные, учитываемые в регистре.

1.3.2.4. Типизированные и типообразующие объекты

Тип данных объекта задаётся при создании/редактировании в конфигураторе. Примитивные типы: Число, Строка, Дата, Булево, NULL, Неопределено, Тип. Объекты конфигурации, образующие типы значений, — типообразующие. Типообразующие: справочники, документы, планы видов характеристик, планы счетов, планы видов расчёта, планы обмена, бизнес-процессы, задачи, перечисления. Для каждого такого объекта появляются три вида типов: Ссылка, Объект, Список (например, СправочникСсылка.<Имя>, СправочникОбъект.<Имя>, СправочникСписок.<Имя>). Эти типы можно присваивать типизированным объектам конфигурации.

Составной тип: в окне редактирования типа — флажок «Составной тип данных» и перечень допустимых типов; допускается ЛюбаяСсылка. Наборы типов (ЛюбаяСсылка, СправочникСсылка, Характеристика.<имя> и др.) формируются системой по метаданным. При запуске прикладного решения набор типов преобразуется, как правило, в составной тип. Если в наборе единственный тип — преобразование в этот тип (например, план видов характеристик с единственным типом в ТипЗначенияХарактеристик: Характеристика.Свойства становится одним типом, а не составным). Важно для проверки на заполненность: при составном типе проверять на Неопределено, при единственном — на значение по умолчанию типа.

1.3.3. Командный интерфейс

Средство навигации по функциональности. Строится на основе подсистем; прикладные объекты включаются в подсистемы. Система автоматически строит командный интерфейс (иерархия подсистем, команды доступа к спискам, отчётам, обработкам). Разработчик может редактировать порядок и видимость команд (редактор командного интерфейса). Команды по умолчанию создаются системой; разработчик может добавлять свои.

1.3.4. Форма

Форма — совокупность экранного диалога, модуля, реквизитов и команд. Состав: экранный диалог для ввода и редактирования; модуль формы (встроенный язык для контроля, расчётов); список реквизитов; команды. У объекта может быть несколько форм. Редактор форм в конфигураторе редактирует все компоненты формы во взаимосвязи.

1.3.5. Модуль

Модуль — программа на встроенном языке, располагается в заданных точках конфигурации и вызывается в известные моменты работы системы. Виды модулей:
- модуль управляемого приложения;
- модуль обычного приложения;
- модуль внешнего соединения;
- модуль сеанса;
- общие модули;
- модули форм;
- модули объектов конфигурации (менеджеров констант, справочников, документов, ПВХ, планов счетов, ПВР, планов обмена, БП, задач, отчётов, обработок, наборов записей регистров);
- модули менеджеров объектов (справочников, документов, ПВХ, планов счетов, ПВР, планов обмена, БП, задач, отчётов, обработок, регистров сведений, накопления, бухгалтерии, расчёта, перечислений, журналов документов, хранилищ настроек);
- модули наборов записей (регистров сведений, накопления, бухгалтерии, расчёта);
- модули команд.

Доступ к модулю: контекстное меню объекта — «Открыть модуль…». Корневые объекты: модуль управляемого приложения, сеанса, внешнего соединения, обычного приложения. Константы, журналы документов и некоторые объекты не имеют модуля.

В модулях объектов — переменные, процедуры и функции, доступные извне; процедуры событий (например, ввод на основании) и процедуры действий над объектом (печать и т.д.). Модуль менеджера расширяет функциональность менеджера; в нём нельзя объявлять переменные и тело модуля. Экспортные процедуры и функции модуля менеджера вызываются как методы объекта конфигурации.

Пример вызова: `Дебиторы = Справочники.Контрагент.ПолучитьСписокДебиторов();`

1.3.6. Макет

Объект конфигурации для формирования печатных форм. Общие макеты — в ветви «Общие» → «Общие макеты». Макеты объектов — в подчинённых «Макеты» или во внешних файлах (свойство табличного документа «Макет»). Типы макетов:
- Табличный документ;
- Текстовый документ;
- Двоичные данные;
- ActiveDocument (OLE Active document; документы Microsoft Office OpenXML не поддерживаются);
- HTML-документ;
- Географическая схема;
- Графическая схема;
- Схема компоновки данных;
- Макет оформления компоновки данных;
- Внешняя компонента (мобильное или обычное приложение).

1.4. Варианты работы, способы взаимодействия и виды клиентских приложений

1.4.1. Компоненты системы

- Информационная база — логически целостная система: минимум две конфигурации (основная и конфигурация БД), база данных, данные для администрирования.
- Клиентское приложение — программа, взаимодействующая с базой данных для получения данных; несколько видов и технологий доступа.
- Сервер «1С:Предприятие» (кластер серверов) — промежуточный слой между клиентом и БД.
- СУБД — управление физическими таблицами, целостность, согласованность, язык работы с данными.
- Веб-сервер — приём HTTP-запросов, передача обработчикам; при запросах от клиента 1С — взаимодействие с ИБ.

1.4.2. Варианты работы

1.4.2.1. Общая информация

Два варианта: файловый и клиент-серверный. Прикладные решения работают идентично. Файловый — для персонального использования и малых рабочих групп; клиент-серверный — для рабочих групп и предприятия.

1.4.2.2. Файловый вариант

Все данные ИБ (конфигурация, БД, админ. информация) в одном файле. Не требуются дополнительные ПО; достаточны ОС и «1С:Предприятие». Высокая целостность, простое резервное копирование (копирование файла ИБ). Имеет ограничения по масштабу и нагрузке; при доступе через веб-сервер рекомендуется малое число пользователей и отсутствие длительных операций.

1.4.2.3. Клиент-серверный вариант

Трёхуровневая архитектура: клиент — сервер 1С — СУБД. Объёмные операции выполняются на сервере; клиент получает выборки. Кластер серверов позволяет распределять нагрузку. Администрирование: пользователь указывает имя кластера и ИБ, имя и пароль; доступ только через сервер. Сервер 1С и СУБД могут работать на разных ОС (Windows, Linux). Поведение прикладного решения стремится к одинаковому для всех поддерживаемых СУБД; отличия описаны в документации по СУБД.

1.4.3. Виды взаимодействия компонентов

1. Прямое подключение — клиент подключается к файлу БД (файловый вариант) или к серверу «1С:Предприятие».
2. Подключение через веб-сервер — клиент подключается к веб-серверу; расширение веб-сервера обращается к БД или к серверу приложений.
3. Мобильная платформа — версия «1С:Предприятие» для мобильных ОС.

Возможно совмещение видов доступа. В клиент-серверном варианте виды подключения задаются конфигурацией соединений; в файловом — те же типы клиентов (толстый, тонкий, веб-клиент).

1.4.4. Виды клиентских приложений

- Тонкий клиент — ограниченный набор действий на клиенте; основная нагрузка на сервере; чёткое разделение клиентского и серверного кода.
- Веб-клиент — работа в браузере; больше ограничений из-за модели безопасности браузеров.
- Мобильный клиент — Android, iOS, Windows; доступ к камере, геолокации, уведомлениям.
- Мобильная платформа — исполнение на устройстве; ИБ может быть на устройстве; приложение на мобильной платформе может работать без постоянного канала связи.
- Мобильный клиент с автономным режимом — при неустойчивом или отсутствующем соединении с веб-сервером; требуется дополнительная настройка.
- Толстый клиент — выполнение практически всех действий прикладного решения; граница между клиентом и сервером размыта.

Самостоятельные приложения: тонкий клиент, мобильный клиент, приложение на мобильной платформе, толстый клиент. Веб-клиент работает только в среде браузера.

1.4.5. Интерфейс

Управляемый интерфейс — декларативное описание; задаётся расположение относительно других элементов и выравнивание. Прикладное решение с управляемым интерфейсом — управляемое приложение. Поддерживаются режимы отображения текста: слева-направо (LTR) и справа-налево (RTL); переключение по языку локализации или пользователя.

1.4.6. Обычное приложение

Обычный интерфейс — размещение с точностью до пиксела; поведение при изменении размеров и размещение в меню задаются вручную. Обычное приложение — только на ПК, только толстый клиент; недоступно для веб-браузера и мобильных устройств. Разработка новых решений на обычном интерфейсе не рекомендуется; поддержка — для совместимости с унаследованными решениями.

1.4.7. Сводная информация

Управляемое приложение: файловый и клиент-серверный доступ; ОС: Android, iOS, Linux, macOS, Windows (обычная и мобильная); доступ через веб-сервер (тонкий, веб-, мобильный клиент); работа на мобильных устройствах. Обычное приложение: файловый и клиент-серверный доступ; ОС: Linux, macOS, Windows (только обычная); без веб-сервера и мобильных устройств. Для новых решений рекомендуется управляемый интерфейс; обычный интерфейс и толстый клиент — для совместимости.

1.5. Технологические средства разработки

Встроенный язык — предметно-ориентированный, поддерживает типы данных, определяемые конфигурацией; объектная работа с данными; мягкая типизация; жёсткий контроль синтаксиса. Событийно-зависимое использование модулей в заданных точках конфигурации. Механизм запросов опирается на структуру ИБ. Текстовый редактор: подсветка синтаксиса, группировка конструкций, контекстная подсказка, шаблоны. Редактор форм — настройка форм ввода и просмотра. Редактор табличных документов — единый формат выходных документов; именованные области (в т.ч. вертикальные); интеграция с системой компоновки данных; табличный документ как элемент формы. Редакторы картинок и HTML-документов. Конструкторы — для стандартных элементов (формы, печатные формы, движения регистров и др.). Командный интерфейс настраивается и автоматически учитывает права доступа. Подсистемы — выделение разделов конфигурации; объект может входить в несколько подсистем; иерархия подсистем определяет интерфейс. Роли — наборы прав по должностям/видам деятельности; права на поля и записи объектов БД; пользователю назначаются роли, основной интерфейс и язык. Отладчик — пошаговое исполнение, замер времени, просмотр переменных. Хранилище конфигураций — групповая разработка, разграничение прав на изменение объектов. Поддержка конфигураций — файлы поставки, комплекты поставки, обновление типовых конфигураций.
