## Context

Расширение `рг_УниверсальныйОбменXML` на ЗУП КОРП 3.1: в выгрузке уже есть справочник `рг_НастройкиОбменаXML` (S1 архивного change), общий модуль, регламентное задание; веб-сервис и обработка — не завершены. Main specs и ADR-0003 зафиксировали pull-модель с именем макета правил и движком на базе штатной `УниверсальныйОбменДаннымиXML`.

**Новая постановка (v2):** единый справочник на обеих базах с ролью `Источник`; правила — в `ПравилаОбменаДанными`; жизненный цикл сеанса — перечисление `рг_СтатусыОбмена`; двухфазное завершение `GetData` → загрузка → `ConfirmDataReceipt`; движок — встроенная обработка `рг_УниверсальныйОбменДаннымиXML` в расширении (extend 2026-06-25).

**Ограничения:** правки только в `src/ЗУП/cfe/рг_УниверсальныйОбменXML/`; эталон `src/ЗУП/cf/` — read-only.

## Goals / Non-Goals

**Goals:**

- Статусная модель обмена на профиле-источнике (Новое → Выполняется → Выполнено / Ошибка)
- `GetData` только из «Новое» с атомарным переводом в «Выполняется»
- ZIP с XML правил и XML данных
- `ConfirmDataReceipt` на источнике после успешной загрузки на приёмнике
- Правила обмена в хранилище профиля; проверка при записи источника
- Префиксы для идентификации сеанса (`Префикс`, `ПрефиксБазыПриемника`)
- Журнал регистрации переходов статусов и ошибок
- Контроль «зависших» обменов: видимость «Выполняется», журнал, ручной сброс (авто-сброс — FU1, out of scope)
- БСП: безопасное хранилище пароля, регламентные задания (сохранить из S1)

**Non-Goals:**

- Автоматический поиск ссылок по примитивам (остаётся в обработчиках правил)
- Очереди сообщений, двусторонняя синхронизация
- Перенос на другие конфигурации в рамках этого change
- Автоматический сброс «зависших» «Выполняется» без отдельного регламента (только фиксация в журнале + ручной сброс)

## Design Rationale

1. **Статус на профиле-источнике** — единая точка истины для «готов к выгрузке / в процессе / завершён»; исключает повторный `GetData` без явного сброса в «Новое».
2. **Правила в хранилище** — источник не зависит от состава макетов конфигурации; пользователь загружает актуальный XML правил в профиль.
3. **ConfirmDataReceipt** — приёмник подтверждает успешную загрузку; источник не остаётся в «Выполняется» после доставки.
4. **Префиксы** — `GetData`/`ConfirmDataReceipt` находят профиль-источник по паре `Префикс` + `ПрефиксБазыПриемника` без передачи правил в каждом запросе.
5. **Встроенная `рг_УниверсальныйОбменДаннымиXML`** — as-built в расширении; серверный код создаёт экземпляр через менеджер обработки, без `ВнешниеОбработки` и макета epf.

## Behavior Contract

### Жизненный цикл профиля-источника

| Статус | Допустимые действия | Следующий статус |
|--------|---------------------|------------------|
| _(начало)_ | Пользователь загрузил правила, заполнил параметры, установил «Новое», записал | Новое |
| Новое | Вызов `GetData` с совпадающими префиксами | Выполняется (в той же транзакции, до выгрузки) |
| Выполняется | Успешный `ConfirmDataReceipt` | Выполнено |
| Выполняется | Повторный `ConfirmDataReceipt` (потеря ответа сети) | Выполнено (no-op, успех) |
| Выполняется | Ошибка выгрузки / таймаут без confirm | Ошибка (или остаётся Выполняется — см. Open Questions) |
| Любой | Ошибка с явной фиксацией | Ошибка |
| Выполнено / Ошибка | Пользователь вручную → «Новое» | Новое |

**Инварианты:**

- `GetData` MUST NOT выполнять выгрузку, если статус ≠ «Новое»
- Перевод «Новое» → «Выполняется» MUST быть атомарным check-and-set в короткой транзакции под управляемой блокировкой профиля; выгрузка движком обработки выполняется **вне** этой транзакции
- При ошибке выгрузки после перевода в «Выполняется» — отдельная запись: `Статус → Ошибка` + журнал
- `ConfirmDataReceipt` MUST переводить только из «Выполняется» в «Выполнено»
- Если профиль по префиксам не найден или найдено более одного — ошибка клиенту, запись в журнал, статус существующих профилей не меняется
- Профиль с `Источник = Истина` MUST иметь непустое `ПравилаОбменаДанными` при записи
- Профиль с `Источник = Ложь` MUST иметь `АдресСервиса` и `Логин`; пароль — в БСП

### Состав ZIP

ZIP MUST содержать минимум: файл XML правил обмена и файл XML данных выгрузки (имена файлов — соглашение в tasks, не фиксируем в spec).

### Журнал регистрации

При каждом изменении `Статус` и при ошибках обмена — запись в журнал с событием подсистемы обмена, префиксом профиля и текстом ошибки.

## Decisions

### D1. Перечисление `рг_СтатусыОбмена`

Значения: `Новое`, `Выполняется`, `Выполнено`, `Ошибка`. Хранится в реквизите `Статус` профиля-источника.

### D2. Контракт GetData (v2)

**Вход:** `Префикс` (String 9), `ПрефиксБазыПриемника` (String 9) — поиск единственного профиля-источника с `Источник = Истина` и совпадающими префиксами.

**Предусловие:** `Статус = Новое`.

**Действие (две фазы):**
1. Под управляемой блокировкой профиля: проверка `Статус = Новое` → `Статус → Выполняется` → commit.
2. Вне транзакции: загрузка правил из `ПравилаОбменаДанными`; подстановка параметров из ТЧ; выгрузка через `рг_УниверсальныйОбменДаннымиXML`; ZIP (правила + данные) → Base64Binary.

**При ошибке фазы 2:** отдельная запись `Статус → Ошибка`, журнал, исключение клиенту.
**При не найден / >1 профиля:** ошибка клиенту, журнал, без смены статусов.

### D3. Контракт ConfirmDataReceipt

**Вход:** те же префиксы.

**Предусловие:** `Статус = Выполняется`.

**Действие:** `Статус → Выполнено`, запись в журнал.

**Идемпотентность:** если профиль уже в «Выполнено» с теми же префиксами — вернуть успех (no-op), не ошибку (повтор после потери ответа сети).

### D4. Движок `рг_УниверсальныйОбменДаннымиXML`

Встроенная обработка расширения (as-built в `src/ЗУП/cfe/рг_УниверсальныйОбменXML/DataProcessors/рг_УниверсальныйОбменДаннымиXML/`). Серверный код создаёт экземпляр: `Обработки.рг_УниверсальныйОбменДаннымиXML.Создать()` и вызывает экспортные методы объектного модуля для выгрузки и загрузки. Макет `ДвижокОбмена` и `ВнешниеОбработки.Создать()` **не** используются.

### D5. As-built имена метаданных

Сохраняем префикс `рг_` в именах объектов выгрузки: `рг_НастройкиОбменаXML`, `рг_УниверсальныйОбменXMLСервер`, `рг_ВыполнениеОбменаXML`. Новое перечисление: `рг_СтатусыОбмена`.

### D6. Роль профиля

`Источник = Истина` — сервер (правила, параметры, статус, публикация WS); регламентное задание **не** создаётся. `Источник = Ложь` — клиент (адрес, логин, пароль, префиксы сеанса, инициация GetData); регламентное задание создаётся по существующему паттерну as-built.

Проверка правил в хранилище, уникальности пары префиксов среди профилей с `Источник = Истина`, отказные проверки роли — в существующий `ПередЗаписью` **до** работы с регламентным заданием (после early-return при `ОбменДанными.Загрузка`): (1) отказные проверки → (2) рег. задание только при `Источник = Ложь`.

Форма элемента: видимость групп реквизитов переключается по `Источник` (обработчик `ИсточникПриИзменении` в модуле формы).

## Risks / Trade-offs

- **[Risk] Конфликт ADR-0003** → Снят частично: API тот же, объект — в расширении; ADR при archive уточнить supersede
- **[Risk] Drift движка от cf `УниверсальныйОбменДаннымиXML`** → Обновление обработки в расширении при смене версии cf; проверка в Primary accept
- **[Risk] Зависший «Выполняется» без Confirm** → Журнал + ручной сброс; опционально регламент (out of scope)
- **[Risk] Дубли префиксов** → Уникальность пары префиксов на ИБ (проверка при записи)

## Migration Plan

1. Создать перечисление и новые реквизиты (Configurator)
2. Для существующих профилей-приёмников: заполнить `Источник = Ложь`, перенести URL/логин
3. Для новых профилей-источников: загрузить XML правил в хранилище, задать префиксы
4. Обновить BSL веб-сервиса и серверного модуля
5. Убедиться, что `рг_УниверсальныйОбменДаннымиXML` и веб-сервис выгружены в cfe

## Open Questions

2. Нужен ли регламентный job для сброса «зависших» «Выполняется» старше N часов? (out of scope v2, зафиксировать в follow-up)

### Публичный API движка (verified)

Реквизиты объекта (не параметры методов): `ИмяФайлаПравилОбмена`, `ИмяФайлаОбмена`, `ТаблицаНастройкиПараметров`.

| Операция | Метод | Вход | Выход |
|----------|-------|------|-------|
| Загрузка правил | `ЗагрузитьПравилаОбмена(Источник="", ТипИсточника="XMLФайл")` | `Движок.ИмяФайлаПравилОбмена` = путь к XML из профиля (временный файл) | Правила активированы в движке |
| Инициализация параметров | `ИнициализироватьПервоначальныеЗначенияПараметров()` | — | Значения по умолчанию из правил |
| Подстановка параметров | `УстановитьЗначениеПараметраВТаблице(ИмяПараметра, ЗначениеПараметра)` | строки ТЧ `Параметры` профиля | Обновлённая таблица параметров |
| Выгрузка данных | `ВыполнитьВыгрузку()` | после загрузки правил и параметров; `Движок.ИмяФайлаОбмена` = путь к XML данных | XML данных в `ИмяФайлаОбмена` |
| Подготовка загрузки | `ОткрытьФайлЗагрузки(ТолькоПрочитатьШапку=Ложь)` | `Движок.ИмяФайлаОбмена` = путь к XML данных; при необходимости `ИмяФайлаПравилОбмена` | Готовность к чтению |
| Загрузка данных | `ВыполнитьЗагрузку()` | после `ОткрытьФайлЗагрузки` / загрузки правил | Данные в ИБ приёмника; ошибки в протоколе движка |

**Последовательность выгрузки (фаза 2 GetData):** записать правила во временный XML → `ИмяФайлаПравилОбмена` → `ЗагрузитьПравилаОбмена()` → `ИнициализироватьПервоначальныеЗначенияПараметров()` → цикл `УстановитьЗначениеПараметраВТаблице` по ТЧ → `ИмяФайлаОбмена` → `ВыполнитьВыгрузку()`.

**Последовательность загрузки (приёмник):** серверный модуль `рг_УниверсальныйОбменXMLСервер` распаковывает ZIP во временный каталог → два файла (XML правил, XML данных) → `Движок.ИмяФайлаПравилОбмена` и `Движок.ИмяФайлаОбмена` = пути к **распакованным XML** (не `.zip`, чтобы не срабатывала встроенная `РаспаковатьZipФайл` в движке) → `ЗагрузитьПравилаОбмена()` → `ОткрытьФайлЗагрузки()` → `ВыполнитьЗагрузку()`. Удаление временных файлов — ответственность серверного модуля после сеанса.

**ZIP:** серверный модуль `рг_УниверсальныйОбменXMLСервер` упаковывает XML правил (из профиля) + XML данных (результат `ВыполнитьВыгрузку`); движок ZIP не формирует и на приёмнике ZIP не распаковывает.
