# BSL Anti-pattern Registry

Проектный реестр антипаттернов BSL. Каждая карточка — обобщённый принцип, не привязанный к конкретному инциденту.

Краткий индекс для агентов: `.cursor/rules/bsl-antipatterns.mdc`.

## Формат карточки

| Поле | Назначение |
|------|-----------|
| ID | AP-NNN (стабильная ссылка) |
| Название | Обобщённое, без привязки к конкретному кейсу |
| Категория | context, contract, performance, extension, ... |
| Severity | CRITICAL / HIGH / MEDIUM / LOW |
| Правило | Обобщённый принцип (1-2 предложения) |
| Симптом | Что видит проверяющий (ошибка, линтер, поведение) |
| Детектирование | Как искать: паттерн в коде, grep, условие |
| Пример BAD / GOOD | Минимальный, абстрактный |
| Ссылки | На вендорный стандарт, rule в coding-standards, ADR |
| Дата добавления | ГГГГ-ММ-ДД |
| Источник | change/debug, где обнаружено |

## AP-001: Серверная область видимости по умолчанию в модулях форм

**Категория:** context
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** mchd-auto-attach-send (debug — замечания статической проверки); 2026-04-19 — усиление: двуконтекстное использование одной `Перем` (change `do2-cf-partial-repeat-params-do3-ui`)

### Правило

В модуле управляемой формы `Перем`, `Процедура`, `Функция` без явной директивы компиляции (`&НаКлиенте` / `&НаСервере` / `&НаСервереБезКонтекста`) принадлежат серверному контексту. Клиентский код не имеет к ним доступа. Аналогично, обращение к встроенному типу как к идентификатору (`Тип(Структура)`) в тонком клиенте трактуется как переменная, а не тип — использовать строковую форму `Тип("Структура")`.

**Дополнение (двуконтекст):** одна и та же модульная `Перем` **не может** выступать «общей памятью» для серверного и клиентского контекста — это две разные области компиляции модуля формы. «Общий» контракт выражается явно: реквизит формы, параметры процедур, либо отдельная `&НаКлиенте Перем` с инициализацией из тех же входных данных, что и на сервере (без подмены одной переменной обоими концами).

### Симптом

**Статический (тонкий клиент):** **«Переменная не определена»** для:
- переменной модуля, объявленной без `&НаКлиенте`, при обращении из `&НаКлиенте`-процедуры;
- имени типа (например `Структура`) при использовании `Тип(ИмяТипа)` вместо `Тип("ИмяТипа")` в клиентском контексте.

**Семантический (граф использования):** модульная `Перем` без директивы читается или записывается **и** в процедурах с серверной директивой (`&НаСервере`, `&НаСервереБезКонтекста` и т.п.), **и** в процедурах с `&НаКлиенте` / `&НаКлиентеНаСервереБезКонтекста` — нарушение HIGH, даже если статическая проверка в конкретной среде не сработала.

**Маркеры false-negative для ревьювера** (не основание принять код, а триггер проверить граф): комментарии у объявления вроде «общий для сервера и клиента», «shared», «общая переменная модуля», «единое состояние формы».

### Детектирование

Проверять **оба** пути; достаточно одного срабатывания для флага HIGH по переменной модуля.

**Не HIGH по AP-001:** модульная `Перем` без директивы используется **только** в серверных процедурах/функциях этого же модуля формы (ни одного вхождения под `&НаКлиенте` / смешанной клиентской директивой) — конфликт контекстов отсутствует; отдельно по желанию ревью — рекомендации [ИТС 639](https://its.1c.ru/db/v8std/content/639/hdoc) (стиль, не severity этой карточки).

1. **Статический:** в `Forms/*/Ext/Form/Module.bsl` найти `Перем` без предшествующей директивы `&НаКлиенте` / `&НаСервере` / `&НаСервереБезКонтекста`; если имя используется в теле процедуры с `&НаКлиенте` (или клиентской части смешанной директивы) — HIGH.
2. **Семантический (граф):** для каждой такой `Перем` собрать все обращения по модулю; для каждого вхождения определить директиву **ближайшей охватывающей** процедуры/функции (без директивы в модуле формы = сервер, см. AP-020). Если среди вхождений есть и серверный, и клиентский контексты — HIGH (**двуконтекстная переменная**).
3. В клиентских процедурах формы: `Тип(ИмяБезКавычек)` — должно быть `Тип("ИмяТипа")`.
4. В клиентских процедурах: обращение к серверным общим модулям (например `ОбщегоНазначения` вместо `ОбщегоНазначенияКлиент`).

### Ремедиация

- **Реквизит формы** (предпочтительно, если значение должно быть доступно на клиенте после `ПриСозданииНаСервере` или переживать цепочку вызовов): заполнение на сервере, чтение на клиенте через `Объект` / реквизиты формы.
- **`&НаКлиенте Перем`** + явная инициализация на клиенте из тех же входных данных, что и на сервере (например `Параметры` в `ПриОткрытии`), если нужен клиентский кэш; серверная логика не опирается на «ту же» переменную без передачи значения.
- **Параметр процедуры** — если флаг нужен только в одной процедуре и её вызываемых локально цепочках (передавать явно).

### Контр-сигналы (не оправдание)

- Комментарий автора про «общую» переменную для двух контекстов.
- «У нас работает в толстом клиенте» — не гарантирует тонкий / веб-клиент.
- Фраза «на сервере не хранится между вызовами» — про **жизненный цикл вызова**, не про область видимости; не отменяет запрет двуконтекстной одной `Перем`.

### Пример BAD

```bsl
// Модуль формы — Перем без директивы = серверная
Перем МойКонтекст;

&НаКлиенте
Процедура ПриОткрытии(Отказ)
    МойКонтекст = ИнициализироватьКонтекст();  // «Переменная не определена» (Тонкий клиент)
КонецПроцедуры

&НаКлиенте
Процедура Обработать(Команда)
    Если ТипЗнч(Значение) = Тип(Структура) Тогда  // «Переменная не определена (Структура)»
        // ...
    КонецЕсли;
КонецПроцедуры
```

**Двуконтекст + комментарий-обманка (тот же AP-001, семантическое детектирование):**

```bsl
// Неверно: одна Перем не «общая» для сервера и клиента
Перем РежимСохранения;

&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    РежимСохранения = Истина; // серверная область
КонецПроцедуры

&НаКлиенте
Процедура ДействиеНаКлиенте()
    Если РежимСохранения = Ложь Тогда // клиент не видит серверную Перем — HIGH по графу
        Возврат;
    КонецЕсли;
КонецПроцедуры
```

### Пример GOOD

```bsl
#Область ОписаниеПеременных

&НаКлиенте
Перем МойКонтекст;

#КонецОбласти

&НаКлиенте
Процедура ПриОткрытии(Отказ)
    МойКонтекст = ИнициализироватьКонтекст();
КонецПроцедуры

&НаКлиенте
Процедура Обработать(Команда)
    Если ТипЗнч(Значение) = Тип("Структура") Тогда
        // ...
    КонецЕсли;
КонецПроцедуры
```

## AP-002: Клиентские методы в серверном контексте

**Категория:** context
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10

### Правило

`Оповестить()`, `ОповеститьОбИзменении()` — клиентские методы платформы. Вызов из серверного контекста (`&НаСервере`, `&НаСервереБезКонтекста`, серверный общий модуль) — ошибка компиляции или runtime. Также: имя метода с суффиксом «НаКлиенте» при директиве `&НаСервере` (и наоборот) — признак ошибки контекста.

### Симптом

- Runtime/статическая ошибка: «Метод не найден» при вызове `Оповестить()` на сервере.
- Логическая ошибка: оповещение не доходит до клиента.
- Имя метода противоречит директиве компиляции.

### Детектирование

1. В `&НаСервере`/`&НаСервереБезКонтекста`-процедурах: найти `Оповестить(`, `ОповеститьОбИзменении(`.
2. В серверных общих модулях: аналогично.
3. Имя метода содержит «НаКлиенте», но директива `&НаСервере` (и наоборот).

### Пример BAD

```bsl
&НаСервере
Процедура ОбновитьДанныеНаСервере()
    // ... обновление данных ...
    ОповеститьОбИзменении(Тип("СправочникСсылка.Клиенты"));  // клиентский метод на сервере
КонецПроцедуры
```

### Пример GOOD

```bsl
&НаКлиенте
Процедура ОбновитьДанные(Команда)
    ОбновитьДанныеНаСервере();
    ОповеститьОбИзменении(Тип("СправочникСсылка.Клиенты"));
КонецПроцедуры

&НаСервере
Процедура ОбновитьДанныеНаСервере()
    // ... обновление данных ...
КонецПроцедуры
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 19 (Оповестить только на клиенте)

## AP-003: ЭтаФорма вместо ЭтотОбъект в callback-ах

**Категория:** context
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10

### Правило

В `ОписаниеОповещения` второй параметр — объект-владелец callback-процедуры. В модуле формы это `ЭтотОбъект` (модуль формы как объект), не `ЭтаФорма` (управляемая форма как UI-объект). `ЭтаФорма` не сериализуется корректно при передаче контекста.

### Симптом

- Callback-процедура не находится при вызове.
- Ошибка сериализации при передаче `ЭтаФорма`.

### Детектирование

В модулях форм: `Новый ОписаниеОповещения(... , ЭтаФорма` — должно быть `ЭтотОбъект`.

### Пример BAD

```bsl
&НаКлиенте
Процедура ЗадатьВопрос(Команда)
    Оповещение = Новый ОписаниеОповещения("ПослеВопроса", ЭтаФорма);
    ПоказатьВопрос(Оповещение, "Продолжить?", РежимДиалогаВопрос.ДаНет);
КонецПроцедуры
```

### Пример GOOD

```bsl
&НаКлиенте
Процедура ЗадатьВопрос(Команда)
    Оповещение = Новый ОписаниеОповещения("ПослеВопроса", ЭтотОбъект);
    ПоказатьВопрос(Оповещение, "Продолжить?", РежимДиалогаВопрос.ДаНет);
КонецПроцедуры
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, секция «Клиент-Сервер / Асинхронные методы»

## AP-004: Defensive check на фиксированном контракте

**Категория:** contract
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10 + .cursor/docs/1c-coding-standards.md rule 14

### Правило

`ТипЗнч()`, `Свойство()`, `ЗначениеЗаполнено()`, `Колонки.Найти("Имя")`, `ЕстьРеквизитИлиСвойствоОбъекта()` и аналогичные проверки наличия поля/колонки — не добавлять, если контракт гарантирует тип/структуру/наличие колонки: документированный возврат функции, табличная часть, результат запроса с явным списком полей, параметр с документированным типом, реквизит формы (структура задана Form.xml). Проверка вместо знания типа — антипаттерн «компенсация незнания»: не знаешь тип — выясни по метаданным/документации, не добавляй проверку.

Отдельный случай: контракт не выяснен (ни код функции, ни документация не проверены, пользователь не опрошен), а проверка добавлена «на всякий случай». Даже если контракт формально неизвестен — добавлять проверку без попытки его установить = компенсация незнания. Сначала уточнить, потом решать.

### Симптом

- Избыточный код: проверка, которая всегда истинна (контракт гарантирует).
- Маскировка бага: если контракт нарушен — проверка скроет причину.

### Детектирование

1. `ТипЗнч(X) = Тип("...")` где X — возврат функции с документированным типом.
2. `Свойство("Имя")` на значении из ТЧ или запроса с явным полем.
3. `ЗначениеЗаполнено(X)` как guard на поле, гарантированно заполненном по контракту/метаданным (допустимо как бизнес-проверка).
4. `Колонки.Найти("Имя")` (или проверка результата >= 0) для реквизита формы, структура которого задана Form.xml того же объекта, или для ТЧ/результата запроса с явным списком колонок.

### Пример BAD

```bsl
Реквизиты = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(Ссылка, "ИНН, Наименование");
Если ТипЗнч(Реквизиты) = Тип("Структура") Тогда
    ИНН = Реквизиты.ИНН;
КонецЕсли;
```

Реквизит формы с колонкой «ВключитьВЗагрузку» в Form.xml — проверка наличия колонки избыточна:

```bsl
ЕстьКолонкаВключитьВЗагрузку = (ОрганизацииДиадок.Колонки.Найти("ВключитьВЗагрузку") >= 0);
Если ЕстьКолонкаВключитьВЗагрузку Тогда
    Строка.ВключитьВЗагрузку = СтрокаОрг.ВключитьВЗагрузку;
КонецЕсли;
```

### Пример GOOD

```bsl
Реквизиты = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(Ссылка, "ИНН, Наименование");
ИНН = Реквизиты.ИНН;
```

Для реквизита формы с фиксированной структурой (Form.xml) — обращение к колонке напрямую, без проверки Колонки.Найти:

```bsl
Строка.ВключитьВЗагрузку = СтрокаОрг.ВключитьВЗагрузку;
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 14 (контракт источника данных и защитные проверки)

## AP-005: Свойство() на не-Структуре

**Категория:** contract
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10 + .cursor/docs/1c-coding-standards.md rule 14

### Правило

Метод `.Свойство()` существует ТОЛЬКО у `Структура` и `ФиксированнаяСтруктура`. На ссылочном типе, объекте, строке ТЗ/ТЧ, форме, ОбъектXDTO — runtime-ошибка «Метод объекта не обнаружен». Для проверки наличия реквизита у формы/объекта/строки — `ОбщегоНазначенияКлиентСервер.ЕстьРеквизитИлиСвойствоОбъекта()`.

### Симптом

Runtime-ошибка «Метод объекта не обнаружен (Свойство)».

### Детектирование

`.Свойство("...")` на переменной, тип которой не `Структура`/`ФиксированнаяСтруктура` (строка ТЧ, СправочникСсылка, ДокументОбъект, форма и т.д.).

### Пример BAD

```bsl
Для Каждого СтрокаИспол Из Объект.Исполнители Цикл
    Если СтрокаИспол.Свойство("ПорядокСогласования") Тогда  // runtime-ошибка
        // ...
    КонецЕсли;
КонецЦикла;
```

### Пример GOOD

```bsl
Для Каждого СтрокаИспол Из Объект.Исполнители Цикл
    Порядок = СтрокаИспол.ПорядокСогласования;
КонецЦикла;
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 14 (дерево решений проверки наличия поля)

## AP-006: Defensive cake (стек избыточных проверок)

**Категория:** contract
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10 + .cursor/docs/1c-coding-standards.md rule 14

### Правило

2+ защитных проверок на одном значении, где хотя бы одна избыточна (поглощается другой или контракт фиксирован). Три типа:

- **fixed-contract cake** — все проверки избыточны (контракт гарантирует структуру).
- **redundant-layer cake** — динамический контракт, проверки перекрываются (одна поглощает другую).
- **cross-boundary cake** — caller проверяет/ловит то, что callee уже гарантирует контрактом или обрабатывает (INTEGRATION_CONTRACT_GATE, 1c-agent-patterns). Включает: Попытка на caller + Попытка на callee для одного внешнего фактора; ЗначениеЗаполнено() на caller при фиксированном возврате callee.

### Симптом

Цепочка `Если ТипЗнч(...) И Свойство(...) И ЗначениеЗаполнено(...)` — при фиксированном контракте все три лишние; при динамическом — `ЗначениеЗаполнено` поглощается проверкой типа. Cross-boundary: вызывающий код дублирует проверку или обработку исключения, уже выполняемую вызываемой функцией.

### Детектирование

- В условии `Если`: 2+ проверки из набора {`ТипЗнч`, `Свойство`, `ЗначениеЗаполнено`, `ЭтоСсылка`} на одном значении — проверить, поглощается ли одна другой.
- Cross-boundary: caller оборачивает вызов в Попытка и при этом вызываемая функция/процедура сама содержит Попытка для того же внешнего фактора; или caller проверяет возвращаемое значение (ЗначениеЗаполнено, ТипЗнч), хотя контракт callee гарантирует тип/заполненность.

### Пример BAD

```bsl
Если ТипЗнч(Описание) <> Тип("Структура")
    Или Не Описание.Свойство("BoxID")
    Или Не ЗначениеЗаполнено(Описание.BoxID) Тогда
    Возврат;
КонецЕсли;
```

### Пример GOOD

```bsl
Если Не ЗначениеЗаполнено(Описание.BoxID) Тогда
    Возврат;
КонецЕсли;
```

### Пример BAD (cross-boundary cake)

Caller и callee оба ловят один и тот же внешний фактор — избыточное наслоение Попыток:

```bsl
Функция ПодготовитьДанные()
    Попытка
        Возврат ВнешнийВызов();
    Исключение
        ЗаписьЖурналаРегистрации(...);
        Возврат Неопределено;
    КонецПопытки;
КонецФункции

Процедура ВыполнитьДействие()
    Попытка
        Данные = ПодготовитьДанные();
        Если Данные = Неопределено Тогда Возврат; КонецЕсли;
    Исключение
        СообщитьПользователю("Ошибка");
    КонецПопытки;
КонецПроцедуры
```

### Пример BAD (cross-boundary cake — вариант с суррогатом результата)

Callee возвращает структуру «Успех/Данные» вместо проброса исключения; caller тоже оборачивает вызов в Попытка. Две Попытки на один внешний фактор — исключение маскируется суррогатом:

```bsl
Функция Подготовить()
    Попытка
        Данные = ВнешнийВызов();
        Возврат Новый Структура("Успех", Истина, "Данные", Данные);
    Исключение
        ЗаписьЖурналаРегистрации(...);
        Возврат Новый Структура("Успех", Ложь);
    КонецПопытки;
КонецФункции

Процедура Выполнить()
    Результат = Подготовить();
    Если Не Результат.Успех Тогда ... КонецЕсли;
КонецПроцедуры
```

### Пример GOOD (одна Попытка на границе)

```bsl
Функция Подготовить()
    Возврат ВнешнийВызов();
КонецФункции

Процедура Выполнить()
    Попытка
        Данные = Подготовить();
    Исключение
        ЗаписатьОшибку(ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
        ОбщегоНазначения.СообщитьПользователю("...");
    КонецПопытки;
КонецПроцедуры
```

## AP-007: Перезапись параметра

**Категория:** contract
**Severity:** HIGH (overwrite), MEDIUM (mutation)
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.4 + .cursor/docs/1c-coding-standards.md rule 21

### Правило

Параметр функции — входной контракт, не переменная для переиспользования. `Параметр = НовоеЗначение` внутри тела — значение от caller выброшено. `.Очистить()`/`.Удалить()`/`.Добавить()` на коллекции-параметре без документированного `[out]` — мутация без контракта. Исключения: `Отказ = Истина`, `ПроверяемыеРеквизиты`, параметры с `[out]` в заголовке.

### Симптом

- Caller передаёт значение, но процедура его перезаписывает — значение потеряно.
- Caller обнаруживает, что его коллекция очищена/изменена неожиданно.

### Детектирование

1. Параметр на LEFT side присваивания: `Параметр = ...` внутри тела (не `Отказ`, не `[out]`).
2. Модифицирующие методы на параметре: `.Очистить()`, `.Удалить()`, `.Добавить()` без `[out]` в документации.

### Пример BAD

```bsl
Процедура Обработать(Данные, Источник = Неопределено)
    Если ЗначениеЗаполнено(Источник) Тогда
        Данные = ПолучитьДанныеОтИсточника(Источник);
    КонецЕсли;
КонецПроцедуры
```

### Пример GOOD

```bsl
Процедура ОбработатьДанные(Данные)
    // Данные используются как есть
КонецПроцедуры

Процедура ОбработатьДанныеОтИсточника(Источник)
    Данные = ПолучитьДанныеОтИсточника(Источник);
    ОбработатьДанные(Данные);
КонецПроцедуры
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 21 (Parameter Integrity)

## AP-008: Попытка вокруг детерминированной операции

**Категория:** exception
**Severity:** CRITICAL
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10 + .cursor/docs/1c-coding-standards.md rule 20

### Правило

`Попытка/Исключение` ставится только при наличии внешнего фактора: сеть/HTTP/FTP, файловая система, конкурентный доступ к данным, COM-объект, чужая конфигурация. Конвертация (hex, base64, числа), арифметика, доступ к метаданным текущей конфигурации, обращение к полю с фиксированным контрактом — внешнего фактора нет, Попытка не нужна.

### Симптом

`Попытка` оборачивает код, который может упасть только при баге — баг маскируется вместо исправления.

### Детектирование

`Попытка/Исключение` без внешнего фактора в теле: конвертация строк, арифметика, `Новый ДвоичныеДанные(...)`, `Base64Строка(...)`, доступ к метаданным, обращение к гарантированному полю.

### Пример BAD

```bsl
Попытка
    ДвоичныеДанные = Новый ДвоичныеДанные(ОтпечатокHex);
    Результат = Base64Строка(ДвоичныеДанные);
    Возврат Результат;
Исключение
    Возврат ОтпечатокHex;
КонецПопытки;
```

### Пример GOOD

```bsl
Если НЕ ЭтоВалидныйHex(ОтпечатокHex) Тогда
    ВызватьИсключение "Некорректный hex-формат отпечатка";
КонецЕсли;
ДвоичныеДанные = ПолучитьДвоичныеДанныеИзHexСтроки(ОтпечатокHex);
Возврат Base64Строка(ДвоичныеДанные);
```

- .cursor/docs/1c-coding-standards.md, rule 20 (Попытка Justification Gate)

## AP-009: Silent degradation в Исключение

**Категория:** exception
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.10 + .cursor/docs/1c-coding-standards.md rule 20

### Правило

Fallback в блоке `Исключение` возвращает/присваивает значение, неотличимое от успеха для caller. Caller продолжает работу с некорректными данными — ложноотрицательный результат ниже по стеку. Если fallback некорректен — re-raise после логирования.

### Симптом

- Функция возвращает входное значение без преобразования (ошибка скрыта).
- Запрос по некорректному значению не находит данных — caller считает «данных нет», а не «ошибка конвертации».

### Детектирование

В блоке `Исключение`: `Возврат <входной параметр>`, `Возврат Неопределено`, `Результат = <default>` — где caller не отличит ошибку от «не найдено».

### Пример BAD

```bsl
Функция ПреобразоватьОтпечаток(Отпечаток)
    Попытка
        Возврат КонвертироватьВBase64(Отпечаток);
    Исключение
        Возврат Отпечаток;  // caller получает hex вместо base64, запрос ничего не найдёт
    КонецПопытки;
КонецФункции
```

### Пример GOOD

```bsl
Функция ПреобразоватьОтпечаток(Отпечаток)
    Попытка
        Возврат КонвертироватьВBase64(Отпечаток);
    Исключение
        ЗаписьЖурналаРегистрации("Сертификаты.КонвертацияОтпечатка",
            УровеньЖурналаРегистрации.Ошибка,,, ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
        ВызватьИсключение;
    КонецПопытки;
КонецФункции
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 20 (Попытка Justification Gate — silent degradation)


## AP-013: Запрос в цикле (N+1)

**Категория:** performance
**Severity:** HIGH
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.2 + .cursor/docs/1c-coding-standards.md

### Правило

Запрос к базе данных или вызов функции, содержащей запрос, внутри цикла (`Пока Выборка.Следующий()`, `Для Каждого`) — классическая N+1 проблема. Каждая итерация = отдельный запрос к СУБД. Заменить на JOIN, временную таблицу или пакетный запрос.

### Симптом

Медленная работа при большом количестве записей. Количество запросов пропорционально количеству строк.

### Детектирование

Внутри `Пока ... Цикл` / `Для Каждого ... Цикл`: `Запрос.Выполнить()`, вызов функции с запросом, `ОбщегоНазначения.ЗначениеРеквизитаОбъекта()` (для нескольких объектов — использовать пакетный вариант).

### Пример BAD

```bsl
Выборка = Запрос.Выполнить().Выбрать();
Пока Выборка.Следующий() Цикл
    Данные = ПолучитьДанныеКлиента(Выборка.Клиент);
КонецЦикла;
```

### Пример GOOD

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|    Клиенты.Ссылка КАК Клиент,
|    Данные.Поле1 КАК Поле1
|ИЗ
|    Справочник.Клиенты КАК Клиенты
|    ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.ДанныеКлиентов КАК Данные
|    ПО Клиенты.Ссылка = Данные.Клиент";
```

## AP-015: Транзакция без паттерна безопасности

**Категория:** concurrency
**Severity:** CRITICAL
**Дата добавления:** 2026-03-12
**Источник:** извлечено из onec-code-reviewer.md cat.13

### Правило

`НачатьТранзакцию()` обязательно в связке: `Попытка` + `ЗафиксироватьТранзакцию()` в теле Попытки + `ОтменитьТранзакцию()` + `ВызватьИсключение` в блоке Исключение. Любое нарушение паттерна — зависшая транзакция при ошибке.

### Симптом

При ошибке внутри транзакции — транзакция не закрывается, блокировки не снимаются, данные недоступны другим пользователям.

### Детектирование

1. `НачатьТранзакцию()` без `ЗафиксироватьТранзакцию()`/`ОтменитьТранзакцию()` в том же scope.

3. `Исключение` блок транзакции без `ОтменитьТранзакцию()`.

### Пример BAD

```bsl
НачатьТранзакцию();
Объект.Записать();
ЗафиксироватьТранзакцию();
```

### Пример GOOD

```bsl
НачатьТранзакцию();
Попытка
    Объект.Записать();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

## AP-017: Инвертированный ранний выход

**Категория:** design
**Severity:** MEDIUM
**Дата добавления:** 2026-03-12
**Источник:** извлечено из .cursor/docs/1c-coding-standards.md rule 23

### Правило

Когда вся оставшаяся логика процедуры — одно действие или компактный блок, условие должно быть привязано к действию (`Если <условие> Тогда <действие>`), а не к выходу (`Если НЕ <условие> Тогда Возврат`). Ранний выход допустим только как guard clause — предусловие в начале процедуры, после которого идёт основная логика из нескольких действий.

### Симптом

`Возврат` непосредственно перед единственным оставшимся действием — инверсия условия ради `Возврат`.

### Детектирование

`Если НЕ <условие> Тогда Возврат; КонецЕсли;` за которым следует единственное действие (1-3 строки) до конца процедуры.

### Пример BAD

```bsl
РезультатЗапроса = Запрос.Выполнить();
Если РезультатЗапроса.Пустой() Тогда
    Возврат;
КонецЕсли;
УстановитьРезультат(Действие, Результат);
```

### Пример GOOD

```bsl
РезультатЗапроса = Запрос.Выполнить();
Если НЕ РезультатЗапроса.Пустой() Тогда
    УстановитьРезультат(Действие, Результат);
КонецЕсли;
```

## AP-018: ТекущаяСтрока() у реквизита формы ТаблицаЗначений

**Категория:** api
**Severity:** HIGH
**Дата добавления:** 2026-03-13
**Источник:** mchd-auto-attach-send (debug — «Метод объекта не обнаружен (ТекущаяСтрока)»)

### Правило

В управляемой форме реквизит типа «Таблица значений» не имеет метода `ТекущаяСтрока()`. Этот метод есть у объектов выборки (результат запроса), не у таблицы значений. Текущая строка табличного поля на форме задаётся элементом формы (Table); в коде её получают через **Элементы.&lt;ИмяЭлементаТаблицы&gt;.ТекущиеДанные**.

### Симптом

Runtime: **«Метод объекта не обнаружен (ТекущаяСтрока)»** при активизации строки таблицы, при нажатии кнопки «Записать»/«Удалить» и т.п., если в коде вызывается `РеквизитТаблицы.ТекущаяСтрока()`.

### Детектирование

1. В модуле формы (`Forms/*/Ext/Form/Module.bsl`): grep по `\.ТекущаяСтрока\s*\(` — вызов метода с круглыми скобками.
2. Проверить, что слева от `.ТекущаяСтрока()` — имя реквизита формы (реквизит-таблица), а не переменная выборки. Если это реквизит формы (часто совпадает с именем из DataPath таблицы на форме) — заменить на `Элементы.<ИмяЭлементаТаблицы>.ТекущиеДанные` (имя элемента — из Form.xml, атрибут `name` у `Table` с данным `DataPath`).

### Пример BAD

```bsl
&НаКлиенте
Процедура ТаблицаОрганизацийПриАктивизацииСтроки(Элемент)
    ТекущаяСтрока = Организации.ТекущаяСтрока();  // реквизит формы — метода нет
    Если ТекущаяСтрока = Неопределено Тогда Возврат; КонецЕсли;
    ПриСменеОрганизацииНаСервере(ТекущаяСтрока.АдресЯщика);
КонецПроцедуры

&НаСервере
Процедура УдалитьЗаписьНаСервере()
    ТекущаяСтрока = СписокЗаписейРС.ТекущаяСтрока();  // реквизит формы — метода нет
    Если ТекущаяСтрока = Неопределено Тогда Возврат; КонецЕсли;
    УдалитьЗапись(ТекущаяСтрока.Ключ);
КонецПроцедуры
```

### Пример GOOD

```bsl
&НаКлиенте
Процедура ТаблицаОрганизацийПриАктивизацииСтроки(Элемент)
    ТекущаяСтрока = Элементы.ТаблицаОрганизаций.ТекущиеДанные;
    Если ТекущаяСтрока = Неопределено Тогда Возврат; КонецЕсли;
    ПриСменеОрганизацииНаСервере(ТекущаяСтрока.АдресЯщика);
КонецПроцедуры

&НаСервере
Процедура УдалитьЗаписьНаСервере()
    ТекущаяСтрока = Элементы.ТаблицаСписокЗаписейРС.ТекущиеДанные;
    Если ТекущаяСтрока = Неопределено Тогда Возврат; КонецЕсли;
    УдалитьЗапись(ТекущаяСтрока.Ключ);
КонецПроцедуры
```

## AP-019: Новый Тип(...) вместо Тип(...) для дескриптора типа

**Категория:** api
**Severity:** HIGH
**Дата добавления:** 2026-03-13
**Источник:** /opsx-debug — замечание пользователя (ОбщаяФорма.НастройкиЭлектроннойПодписиИШифрования)

### Правило

Дескриптор типа в 1С получают встроенной функцией **Тип("ИмяТипа")**. Оператор **Новый** предназначен для создания экземпляров объектов. Выражение `Новый Тип("...")` платформа трактует как некорректную попытку создания значения и выдаёт ошибку «С помощью оператора Новый (New) не могут создаваться значения примитивных типов» — для любых имён типов (ссылки справочников/документов, примитивы вроде Число, Строка и т.д.).

### Симптом

- Ошибка при выполнении или статическая проверка: **«С помощью оператора Новый (New) не могут создаваться значения примитивных типов»** в строке, содержащей `Новый Тип("...")`.
- Типичные места: присваивание переменной типа «Тип», добавление колонки в таблицу значений с указанием типа.

### Детектирование

1. Grep по `.bsl`: `Новый\s*Тип\s*\(` — любое вхождение «Новый» непосредственно перед вызовом `Тип(`.
2. Заменить на вызов без `Новый`: оставить только `Тип("ИмяТипа")`.

### Пример BAD

```bsl
ТипСсылкиПрограммы = Новый Тип("СправочникСсылка.ПрограммыЭлектроннойПодписиИШифрования");
ТипСсылкиЭкземпляра = Новый Тип("СправочникСсылка.ргЭкземплярыСервераCryptoAPI");
ТаблицаВыборки.Колонки.Добавить("ТипРазмещения", Новый Тип("Число"), 0);
```

### Пример GOOD

```bsl
ТипСсылкиПрограммы = Тип("СправочникСсылка.ПрограммыЭлектроннойПодписиИШифрования");
ТипСсылкиЭкземпляра = Тип("СправочникСсылка.ргЭкземплярыСервераCryptoAPI");
ТаблицаВыборки.Колонки.Добавить("ТипРазмещения", Тип("Число"), 0);
```

## AP-020: Отсутствие явной директивы у процедуры/функции в модуле формы

**Категория:** context
**Severity:** HIGH
**Дата добавления:** 2026-03-13
**Источник:** /opsx-debug — замечание пользователя (модуль формы, скриншот)

### Правило

В модуле управляемой формы **контекст по умолчанию — &НаСервере**. Процедура или функция **без явной директивы** непосредственно перед собой выполняется на сервере. Если такая процедура должна выполняться на клиенте (например, вызывает `ЭлектроннаяПодписьСлужебныйКлиент.*` или другие клиентские методы), отсутствие `&НаКлиенте` приводит к ошибке «Метод не найден» или к неверному поведению. Явная директива (`&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста`) перед каждой процедурой/функцией устраняет неоднозначность и делает контекст однозначным.

### Симптом

- Процедура без директивы вызывает клиентский метод → **«Метод не найден»** или неверная логика.
- При чтении кода неочевидно, на клиенте или на сервере выполняется процедура (по умолчанию — сервер).

### Детектирование

1. В модуле формы (`Forms/*/Ext/Form/Module.bsl`): для каждой строки с объявлением `Процедура` или `Функция` проверить, что непосредственно над ней (с учётом пустых строк и комментариев) есть строка с одной из директив: `&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста`.
2. Первая процедура/функция в файле без директивы выше — по умолчанию выполняется на сервере; если по смыслу это клиентский код — антипаттерн.

### Пример BAD

```bsl
&НаКлиенте
Процедура ДобавитьИзУстановленныхНаКомпьютере(Команда)
    // ...
КонецПроцедуры

Процедура ДобавитьТолькоДляШифрованияИзфайлов(Команда)   // без директивы = НаСервере по умолчанию
    ПараметрыСоздания = Новый Структура;
    ЭлектроннаяПодписьСлужебныйКлиент.ДобавитьСертификатПослеВыбораНазначе...  // клиентский метод → ошибка на сервере
КонецПроцедуры
```

### Пример GOOD

```bsl
&НаКлиенте
Процедура ДобавитьИзУстановленныхНаКомпьютере(Команда)
    // ...
КонецПроцедуры

&НаКлиенте
Процедура ДобавитьТолькоДляШифрованияИзфайлов(Команда)
    ПараметрыСоздания = Новый Структура;
    ЭлектроннаяПодписьСлужебныйКлиент.ДобавитьСертификатПослеВыбораНазначе...
КонецПроцедуры
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, секция «Клиент-Сервер / Директивы компиляции»
- AP-001: серверная область по умолчанию (переменные); здесь — та же семантика для процедур/функций

## AP-021: Тихий пропуск при невыполнении структурного предусловия (Fail-fast violation)

**Категория:** design
**Severity:** HIGH
**Дата добавления:** 2026-03-14
**Источник:** .cursor/docs/1c-coding-standards.md rule 16, onec-code-reviewer Phase 2 (P2.1)

### Правило

При неудаче структурной проверки (тип, свойство, размер коллекции, ожидаемый формат) код обязан явно сигнализировать об ошибке — `ВызватьИсключение` с понятным текстом. Молчаливый пропуск запрещён: ни `Продолжить` в цикле, ни тихий `Возврат` из функции, ни пустая ветка `Если` без `Иначе`. Бизнес-фильтрация (по Статусу, типу документа и т.п.) — допустима; это не маскировка структурной ошибки.

### Симптом

Структурный баг (неверный тип, отсутствующее свойство, несовпадение размеров) не проявляется сразу — скрыт молчаливым пропуском элемента или возвратом без ошибки.

### Детектирование

1. В цикле: `Если НЕ Элемент.Свойство("Поле") Тогда Продолжить;` или `Если ТипЗнч(Значение) <> Тип("Структура") Тогда Продолжить;` — проверка типа/свойства, но пропуск вместо исключения.
2. В процедуре/функции: `Если НЕ ДанныеКорректны(Входные) Тогда Возврат; КонецЕсли;` — вызывающий код не узнает, что обработка не произошла.
3. Ветка без Иначе: `Если ТипЗнч(Значение) = Тип("Структура") Тогда ОбработатьСтруктуру(Значение); КонецЕсли;` — при не-Структура молча ничего не произошло.

### Пример BAD

```bsl
Если НЕ ЭлементРезультата.Свойство("Данные") Тогда Продолжить;
Если ТипЗнч(Элемент) <> Тип("Структура") Тогда Продолжить;

Если НЕ ДанныеКорректны(Входные) Тогда Возврат; КонецЕсли;
```

### Пример GOOD

```bsl
Если НЕ ОбщегоНазначенияКлиентСервер.ЕстьРеквизитИлиСвойствоОбъекта(Элемент, "Данные") Тогда
    ВызватьИсключение "Ожидалось наличие реквизита Данные";
КонецЕсли;

Если ТипЗнч(Значение) = Тип("Структура") Тогда
    ОбработатьСтруктуру(Значение);
Иначе
    ВызватьИсключение "Ожидалась Структура, получен " + ТипЗнч(Значение);
КонецЕсли;
```

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 16 (Fail-fast вместо тихого пропуска)

## AP-022: &Перед/&После у Функции (не Процедуры)

**Категория:** extension
**Severity:** CRITICAL
**Дата добавления:** 2026-03-14
**Источник:** onec-code-reviewer category 5 (5.1), .cursor/docs/1c-coding-standards.md rule 15, 1c-extensions SKILL

### Правило

Аннотации `&Перед` и `&После` применимы **исключительно к процедурам** (`Процедура`). Для перехвата или дополнения **функции** (`Функция`) в расширении необходимо использовать `&Вместо` (с вызовом `ПродолжитьВызов()` при необходимости) или `&ИзменениеИКонтроль`. Использование `&Перед`/`&После` перед `Функция` — ошибка платформы расширений.

### Симптом

Ошибка при компиляции или при применении расширения; неожиданное поведение при вызове перехваченной функции.

### Детектирование

В модулях расширения (cfe): grep по `&Перед` или `&После`, затем проверить, что следующее объявление — `Процедура`, а не `Функция`. Строка вида `&Перед`/`&После` непосредственно перед `Функция` — антипаттерн.

### Пример BAD

```bsl
&Перед
Функция ПолучитьДанные(Параметр)
    // ...
КонецФункции
```

### Пример GOOD

```bsl
&Вместо
Функция ПолучитьДанные(Параметр)
    ДополнительнаяПроверка(Параметр);
    Возврат ПродолжитьВызов(Параметр);
КонецФункции
```

## AP-023: Пользовательское взаимодействие внутри транзакции

**Категория:** concurrency
**Severity:** HIGH
**Дата добавления:** 2026-03-14
**Источник:** .cursor/docs/1c-coding-standards.md rule 18, onec-code-reviewer category 13 (13.1)

### Правило

Вызовы `ПоказатьВопрос()`, `Предупреждение()`, `Сообщить()` между `НачатьТранзакцию()` и `ЗафиксироватьТранзакцию()`/`ОтменитьТранзакцию()` блокируют поток сервера до ответа пользователя. Транзакция удерживает блокировки на время ожидания — другие пользователи не могут получить доступ к заблокированным данным. Вендорный стандарт: пользовательское взаимодействие не допускается внутри транзакции.

### Симптом

Долгая блокировка данных при ожидании ответа пользователя; деградация производительности многопользовательской системы.

### Детектирование

Найти блок кода между `НачатьТранзакцию()` и парой `ЗафиксироватьТранзакцию()`/`ОтменитьТранзакцию()` (в том же scope). Внутри этого блока не должно быть вызовов `ПоказатьВопрос(`, `Предупреждение(`, `Сообщить(`.

### Пример BAD

```bsl
НачатьТранзакцию();
Попытка
    ЗаписатьОбъект(Объект);
    Если НужноПодтверждение Тогда
        ПоказатьВопрос(..., "Подтвердить?");  // блокирует поток и транзакцию
    КонецЕсли;
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

### Пример GOOD

```bsl
Если НужноПодтверждение Тогда
    Результат = ПоказатьВопрос(..., "Подтвердить?");
    Если Результат <> КодВозвратаДиалога.Да Тогда Возврат; КонецЕсли;
КонецЕсли;
НачатьТранзакцию();
Попытка
    ЗаписатьОбъект(Объект);
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

- .cursor/docs/1c-coding-standards.md, rule 18 (транзакционный паттерн)
- Вендорный стандарт: std-02-locks-transactions

## AP-024: Stub/placeholder-код

**Категория:** design
**Severity:** HIGH
**Дата добавления:** 2026-03-14
**Источник:** onec-code-reviewer category 4 (4.1), category 12

### Правило

В релизном коде не допускаются заглушки и плейсхолдеры: пустой отпечаток/идентификатор, захардкоженный TODO/FIXME, всегда ложное условие (`Если Ложь Тогда`), возврат тестовых или мок-данных без реальной реализации. Код должен быть полной реализацией или явно помечен как устаревший/экспериментальный по правилам проекта.

### Симптом

Функциональность не работает или возвращает пустое/тестовое значение; пользователь или интеграция получают невалидные данные.

### Детектирование

1. `Возврат ""` или `Возврат Неопределено` в сочетании с комментарием TODO/FIXME/placeholder.
2. `Если Ложь Тогда` — ветка никогда не выполняется (заглушка).
3. Строковые литералы-заглушки: "test", "placeholder", "FIXME", "TODO", пустая строка как возврат идентификатора/отпечатка.
4. Пустой отпечаток сертификата или аналогичный технический идентификатор без получения реального значения.

### Пример BAD

```bsl
Функция ПолучитьОтпечатокСертификата()
    // TODO: реализовать получение отпечатка
    Возврат "";
КонецФункции

Если Ложь Тогда  // временно отключено
    ВыполнитьПроверку();
КонецЕсли;
```

### Пример GOOD

```bsl
Функция ПолучитьОтпечатокСертификата()
    Сертификат = ПолучитьТекущийСертификат();
    Возврат Сертификат.Отпечаток;
КонецФункции
```

- onec-code-reviewer.md, category 4 (Code Quality), category 12 (Release Readiness)

## AP-025: Строка для пользователя без НСтр()

**Категория:** api
**Severity:** MEDIUM
**Дата добавления:** 2026-03-14
**Источник:** onec-code-reviewer category 10, вендорный стандарт (НСтр для пользовательских строк)

### Правило

Текст, отображаемый пользователю (в сообщениях, предупреждениях, вопросах, исключениях с пользовательским текстом), должен быть обёрнут в `НСтр("ru = '...'")` для поддержки локализации. Вендорный стандарт 1С: все пользовательские строки — через НСтр().

### Симптом

Строка не локализуется; при смене языка интерфейса текст остаётся на одном языке.

### Детектирование

Строковый литерал (не ключ структуры, не формат) в первом/основном аргументе вызовов: `СообщитьПользователю(`, `ПоказатьПредупреждение(`, `ПоказатьВопрос(`, `ОбщегоНазначения.СообщитьПользователю(` — без обёртки `НСтр(`. Исключения: технические сообщения (логи, отладка), ключи, имена метаданных.

### Пример BAD

```bsl
ОбщегоНазначения.СообщитьПользователю("Ошибка при обработке данных");
ПоказатьПредупреждение(, "Документ не проведён");
```

### Пример GOOD

```bsl
ОбщегоНазначения.СообщитьПользователю(НСтр("ru = 'Ошибка при обработке данных'"));
ПоказатьПредупреждение(, НСтр("ru = 'Документ не проведён'"));
```

## AP-026: Некорректное использование API ХранилищеЗначения

**Категория:** api
**Severity:** CRITICAL
**Дата добавления:** 2026-03-14
**Источник:** mchd-auto-attach-send (debug — КД_НастройкаМЧД, строки 193, 257)

### Правило

`ХранилищеЗначения` — immutable-обёртка с минимальным API: чтение — `.Получить()` (или БСП `ОбщегоНазначения.ИзвлечьИзХранилищаЗначения()`); создание — `Новый ХранилищеЗначения(Значение[, Сжатие])`. Других методов нет: `.Размер()`, `.Количество()`, `.Установить()` отсутствуют. Функций `ПоместитьВХранилищеЗначения` / `ОбщегоНазначения.ПоместитьВХранилищеЗначения` не существует ни в платформе, ни в БСП.

### Симптом

Runtime: «Метод объекта не обнаружен» / «Процедура или функция не определена».

### Детектирование

- Любой вызов метода кроме `.Получить()` на переменной, прошедшей `ТипЗнч(...) = Тип("ХранилищеЗначения")`.
- Вызов `ПоместитьВХранилищеЗначения(...)` без или с любым префиксом модуля.
- Любой вызов метода `.Размер()` / `.Количество()` на переменной типа ХранилищеЗначения.

### Пример BAD

```bsl
Если ТипЗнч(Хранилище) <> Тип("ХранилищеЗначения") Или Не Хранилище.Размер() Тогда
    Возврат Неопределено;
КонецЕсли;
Результат = ПоместитьВХранилищеЗначения(Данные);
```

### Пример GOOD

```bsl
Если ТипЗнч(Хранилище) <> Тип("ХранилищеЗначения") Тогда
    Возврат Неопределено;
КонецЕсли;
Значение = Хранилище.Получить();
Если Значение = Неопределено Тогда
    Возврат Неопределено;
КонецЕсли;
Результат = Новый ХранилищеЗначения(Данные);
```

### Ссылки

- AP-005 (Свойство на не-Структуре), AP-018 (ТекущаяСтрока на ТЗ), AP-019 (Новый Тип вместо Тип)
- Платформенная документация: ХранилищеЗначения

## AP-027: Guard-then-catch (Попытка после предусловия)

**Категория:** exception
**Severity:** HIGH
**Дата добавления:** 2026-03-14
**Источник:** семантический антипаттерн (flow-centric), аналог AP-008 без хардкода API

### Правило

Код устанавливает предусловие (guard clause), утверждая «я знаю контракт и гарантирую безопасность», затем оборачивает операцию над проверенным значением в `Попытка` — противоречие. Либо guard достаточен (внешнего фактора нет) — тогда Попытка лишняя; либо внешний фактор есть (TOCTOU) — тогда guard бесполезен, убрать guard.

### Симптом

- Маскировка бага: при корректном коде операция не должна падать после валидного guard.
- Дублирование: guard и Попытка защищают одно и то же.

### Детектирование

`Попытка` блоку **непосредственно** предшествует guard clause (`Если ... Возврат` / `Если ... Продолжить`), валидирующий **то же значение**, которое используется внутри `Попытка`.

### Пример BAD

```bsl
Если ТипЗнч(КонтекстСеансаИзАвторизации) <> Тип("ХранилищеЗначения") Тогда
    Возврат Неопределено;
КонецЕсли;
Попытка
    ДанныеКонтекста = ОбщегоНазначения.ИзвлечьИзХранилищаЗначения(КонтекстСеансаИзАвторизации);
Исключение
    ЗаписьЖурналаРегистрации(...);
    Возврат Неопределено;
КонецПопытки;
```

### Пример GOOD

```bsl
Если ТипЗнч(КонтекстСеансаИзАвторизации) <> Тип("ХранилищеЗначения") Тогда
    Возврат Неопределено;
КонецЕсли;
ДанныеКонтекста = ОбщегоНазначения.ИзвлечьИзХранилищаЗначения(КонтекстСеансаИзАвторизации);
```

### Remediation

(a) Guard достаточен, внешнего фактора нет — убрать `Попытка`. (b) Внешний фактор есть (TOCTOU, race) — убрать guard, оставить `Попытка`.

### Ссылки

- AP-008 (Попытка вокруг детерминированной операции), AP-009 (часто сопутствует — silent degradation в fallback)
- .cursor/docs/1c-coding-standards.md, rule 20 (Попытка Justification Gate)

## AP-028: Check-after-establish (проверка свойства после установления типа)

**Категория:** contract
**Severity:** HIGH
**Дата добавления:** 2026-03-14
**Источник:** семантический антипаттерн (flow-centric), аналог AP-004 без хардкода контрактов

### Правило

Код устанавливает тип или состав значения (guard, конструктор, запрос, итерация ТЧ), затем проверяет свойство/реквизит, гарантированный установленным типом — избыточная проверка. Доверяй установленному контракту: обращаться к полю напрямую.

### Симптом

- Избыточный код: проверка всегда истинна при корректном коде.
- Компенсация незнания: не знаешь состав — выясни по метаданным/запросу, не добавляй проверку после установления типа.

### Детектирование

1. **Type guard → property check:** `Если ТипЗнч(X) = Тип("Структура") Тогда ... X.Свойство("Ключ")` — тип установлен, ключ при неопределённой структуре нужно проверять только если ключи опциональны; при фиксированной структуре — избыточно.
2. **Constructor → property check:** `X = Новый Структура("A, B") ... X.Свойство("A")` — ключи заданы конструктором.
3. **Query → column check:** выборка из запроса с явным списком полей, затем `.Свойство("Поле")` на строке выборки — столбцы заданы текстом запроса.
4. **Known collection → field check:** `Для Каждого Стр Из Объект.Товары Цикл ... Стр.Свойство("Номенклатура")` — поля ТЧ заданы метаданными; для строки ТЧ использовать прямой доступ к полю, не `.Свойство()`.

### Пример BAD

```bsl
Реквизиты = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(Ссылка, "ИНН, Наименование");
Если ТипЗнч(Реквизиты) = Тип("Структура") И Реквизиты.Свойство("ИНН") Тогда
    ИНН = Реквизиты.ИНН;
КонецЕсли;
```

### Пример GOOD

```bsl
Реквизиты = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(Ссылка, "ИНН, Наименование");
ИНН = Реквизиты.ИНН;
```

### Remediation

Убрать проверку, обращаться к полю напрямую. Если контракт действительно неопределён (опциональные ключи) — расширить guard одной проверкой ключа, не дублировать ТипЗнч + Свойство.

### Ссылки

- AP-004 (Defensive check на фиксированном контракте), AP-005 (Свойство на не-Структуре), AP-006 (defensive cake)
- .cursor/docs/1c-coding-standards.md, rule 14 (контракт источника данных и защитные проверки)

## AP-029: Defense stack (Попытка + Свойство как компенсация незнания контракта)

**Категория:** contract/exception (гибрид)
**Severity:** HIGH (defense stack), CRITICAL (defense stack + phantom field)
**Дата добавления:** 2026-03-14
**Источник:** CPA instrument — главный дефект AI-генерированного кода (mchd-auto-attach-send)

### Правило

Writer не знает контракт внешнего API и компенсирует незнание оборонительным кодом: оборачивает обращения к полям в `Свойство()`, а весь блок — в `Попытка`. Ни одна операция внутри Попытка (Свойство(), ТипЗнч(), присваивание, сравнение) не может выбросить исключение при корректных данных — внешнего фактора нет. Паттерн «2+ слоя защиты» на одном обращении к полю данных (Попытка + Свойство, или Попытка + ТипЗнч + Свойство + проверка на Неопределено) = defense stack. Phantom field — имя поля в Свойство("ИмяПоля") не обнаружено в существующем коде расширения для того же источника данных.

### Симптом

- Попытка оборачивает только чтение полей через Свойство() и присваивания — маскировка незнания контракта.
- Имя поля придумано (не верифицировано по API/трассе/существующему коду) и «защищено» Свойство().

### Детектирование

1. Для каждого блока Попытка/Исключение: перечислить все операции внутри Попытка. Если все операции — Свойство(), ТипЗнч(), присваивания, сравнения (ни одна не обращается к сети, ФС, COM, БД) → defense stack, AP-029 HIGH.
2. Для каждого Свойство("ИмяПоля") к данным из внешнего/чужого источника: Grep "ИмяПоля" в .bsl расширения. Если обращение к этому полю на том же источнике не найдено → phantom field WARNING; если phantom field + defense stack → AP-029 CRITICAL.
3. «Данные из внешнего API» не является внешним фактором — данные уже в памяти; внешний фактор = операция, обращающаяся к внешнему ресурсу прямо сейчас.

### Пример BAD

```bsl
Попытка
    ЗначВерх = Неопределено;
    Если КонтрактМЧД.Свойство("ЭтоДействующаяМЧД", ЗначВерх) Тогда
        ЭтоДействующаяМЧД = (ЗначВерх = Истина);
    Иначе
        Если КонтрактМЧД.Свойство("СтатусПроверки", СтатусПроверки) И СтатусПроверки <> Неопределено Тогда
            Если СтатусПроверки.Свойство("СтруктураОшибок", СтруктураОшибок) И СтруктураОшибок.Свойство("ЭтоДействующаяМЧД", ЗначВлож) Тогда
                ЭтоДействующаяМЧД = (ЗначВлож = Истина);
            КонецЕсли;
        КонецЕсли;
    КонецЕсли;
Исключение
    ЭтоДействующаяМЧД = Ложь;
КонецПопытки;
// КонтрактМЧД уже в памяти; Свойство() не бросает; Попытка лишняя; верхний уровень "ЭтоДействующаяМЧД" может быть phantom field.
```

### Пример GOOD

```bsl
// Контракт верифицирован: поле только по пути СтатусПроверки.СтруктураОшибок.ЭтоДействующаяМЧД
Если КонтрактМЧД.Свойство("СтатусПроверки", СтатусПроверки) И СтатусПроверки <> Неопределено Тогда
    Если СтатусПроверки.Свойство("СтруктураОшибок", СтруктураОшибок) И СтруктураОшибок <> Неопределено Тогда
        ЗначВлож = Неопределено;
        ЭтоДействующаяМЧД = СтруктураОшибок.Свойство("ЭтоДействующаяМЧД", ЗначВлож) И (ЗначВлож = Истина);
    КонецЕсли;
КонецЕсли;
// Без Попытка; одно место чтения по верифицированному пути.
```

### Remediation

Убрать Попытка. Для каждого Свойство("Поле") верифицировать имя поля по контракту источника (существующий код, трасса, документация API). Если поле подтверждено и контракт фиксирован — обращаться напрямую. Если контракт динамический — одна проверка Свойство() без Попытка.

### Ссылки

- AP-008 (Попытка вокруг детерминированной операции), AP-004 (defensive check), AP-027 (guard-then-catch)
- .cursor/docs/1c-coding-standards.md, rule 20 (Попытка Justification Gate), rule 14 (контракт источника данных)

## AP-030: Скрытый частичный результат (Попытка + Продолжить/Возврат без обратной связи)

**Категория:** exception/ux
**Severity:** CRITICAL
**Дата добавления:** 2026-03-15
**Источник:** принцип «явный сбой по умолчанию» (fail-fast over hidden partial)

### Правило

Попытка с лог + Продолжить или тихим Возврат, но без обратной связи пользователю = скрытый частичный результат. Пользователь выполнил действие — визуально всё ок, но часть работы не сделана. Расследование ложится на поддержку; при тестировании проблемы проходят мимо.

### Симптом

- Пользователь не видит ошибку; при тестировании сбои не воспроизводятся явно.
- Поддержка находит ошибку только в ЖР.
- «Нажал кнопку — ничего не произошло» при частичном или полном сбое.

### Детектирование

Попытка + в блоке Исключение: `Продолжить`, `Возврат` (без последующего сообщения пользователю в вызывающем коде), или тихое присвоение — при этом отсутствует вызов `ОбщегоНазначения.СообщитьПользователю` / `ОбщегоНазначенияКлиент.СообщитьПользователю` или `ВызватьИсключение`.

Даже при наличии `СообщитьПользователю` — если частичный результат не задокументирован в design.md/spec как явное требование → AP-030 (ревьювер проверяет ссылку на документ).

### Пример BAD

```bsl
// В цикле: пользователь не узнает, что часть элементов пропущена
Попытка
    НоваяСтрока = СписокМЧД.Добавить();
    НоваяСтрока.РегНомерМЧД = КонтрактМЧД.Идентификатор;
    // ...
Исключение
    ЗаписьЖурналаРегистрации(...);
    Продолжить;
КонецПопытки;
```

```bsl
// В функции: caller не показывает ошибку пользователю
Попытка
    ВыполнитьОперацию(Данные);
    Возврат Истина;
Исключение
    ЗаписьЖурналаРегистрации(...);
    Возврат Ложь;  // вызывающий код не вызывает СообщитьПользователю
КонецПопытки;
```

### Пример GOOD

**Вариант 1 (fail-fast):** убрать Попытку — пусть исключение всплывёт (стандартное окно ошибки 1С).

**Вариант 2 (явная обратная связь):** в блоке Исключение или после возврата — сообщить пользователю:

```bsl
Попытка
    ВыполнитьОперацию(Данные);
    Возврат Истина;
Исключение
    ЗаписьЖурналаРегистрации(...);
    ОбщегоНазначения.СообщитьПользователю(НСтр("ru = 'Ошибка при выполнении операции. Подробности в журнале регистрации.'"));
    Возврат Ложь;
КонецПопытки;
```

Для цикла: после цикла проверить счётчик ошибок и показать сообщение («Обработано с ошибками: N. Подробности в ЖР.»).

**Вариант 3 (проверка контракта вместо Попытки):** если Попытка существует из-за неизвестного контракта (обращение к полям данных из внешнего API/ядра) — убрать Попытку и защититься проверкой типа/свойств перед обращением:

```bsl
Для Каждого КлючИЗначение Из ДанныеИзAPI Цикл
    Элемент = КлючИЗначение.Значение;
    Если ТипЗнч(Элемент) <> Тип("Структура") Тогда
        ЗаписьЖурналаРегистрации(..., "Неожиданный тип: " + ТипЗнч(Элемент));
        Продолжить;
    КонецЕсли;
    Если Не Элемент.Свойство("Идентификатор") Тогда
        ЗаписьЖурналаРегистрации(..., "Отсутствует поле Идентификатор");
        Продолжить;
    КонецЕсли;
    // Контракт проверен — обращаемся к полям напрямую
    НоваяСтрока = Таблица.Добавить();
    НоваяСтрока.Идентификатор = Элемент.Идентификатор;
КонецЦикла;
```

При root cause = contract-uncertainty (Phase 2.5) ремедиация — Вариант 3, не Вариант 2 (добавить сообщение).

### Ссылки

- .cursor/docs/1c-coding-standards.md, rule 20 (Попытка Justification Gate), rule 24 (Partial Execution Gate: фиксация в design обязательна)
- AP-009 (Silent degradation), AP-010 (бесследное подавление)

## AP-031: Мета-имя из постановки/оркестрации

**Категория:** naming  
**Severity:** MEDIUM (экспортные процедуры и функции — HIGH)  
**Дата добавления:** 2026-03-20  
**Источник:** ревью продуктивного кода; имена из языка ЗНИ/агента вместо домена

### Правило

Идентификаторы в продуктивном коде (переменные, параметры, процедуры, функции) должны отражать сущности и процессы **предметной области**, а не терминологию постановки задачи, оркестрации разработки или внутренних артефактов (tasks, phase gates, «механика» задачи).

**Первичный критерий (обязателен к проверке):** если разработчик, знакомый с предметной областью конфигурации, но **не** читавший ЗНИ/TЗ, не может одной фразой объяснить, что обозначает имя, — имя считается мета-именем или непрозрачным. Список маркеров ниже — **вспомогательный**, не исчерпывающий.

### Симптом

Код читается как перенос чеклиста или псевдокода постановки; смысл сущности в терминах конфигурации не угадывается из имени.

### Детектирование

Три критерия применяются последовательно; любой сработавший — нарушение.

1. **Первичный: доменный тест** (см. Правило). Коллега без ЗНИ не может одной фразой объяснить, что обозначает имя → нарушение.

2. **Вторичный: тест уровня абстракции.** Имя описывает роль данных в реализации (контейнер, промежуточная коллекция, накопитель, этап обработки), а не доменную сущность, которую данные представляют. Операциональный тест: мысленно убрать реализационное слово (Массив, Новый, Добавляемые, Временный…) — если остаток не несёт доменного смысла, имя провалило тест. Пример: `МассивРеквизитовМеханики` → убрать «Массив» и «Механики» → «Реквизиты» — пустышка; `МассивРеквизитовПризнакаИсключения` → «ПризнакИсключения» — домен, ок.

3. **Третичный: контрастный тест.** В одном scope есть доменно названные данные и безликий контейнер для них → контейнер провалил тест. Пример: рядом `ВключеноИсключениеУчастниковПоУсловию` (домен) и `МассивРеквизитовМеханики` (контейнер для того же реквизита без отсылки к домену).

4. **Не считать нарушением:** доменные термины с квалификатором (`РежимНалогообложения`, `ШагМаршрута`, `ФазаСогласования`), имена, совпадающие с реквизитами/перечислениями метаданных.

#### Калибровочные примеры (не исчерпывающий ориентир)

Вспомогательные маркеры, помогающие заметить подозрительные имена:
- **Номер ЗНИ/задачи в идентификаторе:** `74261`, `pav74261_`, `ПрограммныйИнтерфейс74261`, любой цифровой хвост или встроенный ticket из постановки.
- **Kebab-токены change-name в идентификаторе:** фрагменты slug ЗНИ (`proyden`, `usloviya` из `proyden-usloviya-74261`) — не домен конфигурации.
- **Jargon оркестрации (English):** `Fallback`, `PostWrite`, `PreWrite`, `Guard`, `Guard\d*`, `Mechanics`, `Wrapper`, `Orchestrat`, `Temp`, `Tmp`.
- Термины оркестрации/ЗНИ: Гейт, Gate, Механика, Mechanics, ДействиеДля…, Задача как абстракция без домена.
- Фаза, Шаг, Этап, Phase, Step — если не устоявшийся доменный термин.
- Режим, РежимЗадачи без уточнения предметного режима.
- Флаг как часть имени, когда тип уже булев (`ФлагОбработки` вместо `ОбработкаВыполнена`).
- Пустышки: Результат, Данные, Состояние, Контекст, Параметры без уточняющего существительного домена.
- Контейнерные слова без домена: МассивДобавляемыхРеквизитов, НовыйРеквизит, МассивРеквизитовМеханики.
- Нумерация без домена: Шаг1, Этап2, Вариант3.
- Англоязычные аналоги: Workflow, Pipeline, Orchestration в именах, если не термин предметки.

### Пример BAD

```bsl
МеханикаВключена = ПолучитьНастройку();
ДействиеДляГейта = ОпределитьДействие();
Результат = Новый Структура;
Шаг1 = ВыполнитьПервыйШаг();

// implementation-role names (тест уровня абстракции)

НовыйРеквизит = Новый РеквизитФормы("ВключеноИсключениеУчастниковПоУсловию", ...);
```

### Пример GOOD

## AP-032: Рассогласование персистентного состояния

**Категория:** exception/integrity
**Severity:** CRITICAL
**Дата добавления:** 2026-03-21
**Источник:** ревью ДействияСервер (pav_ИсключениеУчастниковПоУсловию) — подавление исключения записи действия в цикле синхронизации; downstream-создание задач по неконсистентным данным

### Правило

Попытка ловит исключение от операции с персистентными побочными эффектами (запись в БД, файловую систему, внешнюю систему), не пробрасывает, и выполнение продолжается в контексте, который предполагает успешное завершение этой операции. Результат: downstream-логика оперирует неконсистентным состоянием (часть данных обновлена, часть — нет), что ведёт к некорректным бизнес-операциям (создание лишних задач, виз, документов; потеря данных).

Отличие от AP-030 (скрытый частичный результат): AP-030 — пользователь не видит ошибку (UX-слой); добавление СообщитьПользователю решает AP-030. AP-032 — данные в БД рассогласованы и downstream-логика на них опирается (integrity-слой); добавление сообщения **не** решает AP-032. Оба AP могут срабатывать одновременно.

Отличие от AP-009 (silent degradation): AP-009 — return value неотличимо от успеха. AP-032 — процедура может не возвращать значения (Процедура, не Функция), но персистентное состояние после вызова неконсистентно.

### Симптом

- После пакетной обработки часть элементов обновлена в БД, часть — нет; пользователь и downstream-код не различают обработанные и необработанные.
- Создание задач / виз / документов по «старым» данным элемента, который не удалось записать.
- Ошибка видна только в ЖР; при тестировании обычно не воспроизводится (срабатывает при конкурентном доступе, таймауте, нехватке прав).

### Детектирование

1. Попытка без ВызватьИсключение в блоке Исключение.
2. Операция внутри Попытка имеет персистентные побочные эффекты:
   - Прямые: `Записать()`, `ЗафиксироватьТранзакцию()`, `УдалитьФайлы()`, HTTP-запрос с side effect.
   - Через callee (1 уровень): вызываемая процедура содержит `Записать()` / `НачатьТранзакцию()` / `ЗафиксироватьТранзакцию()`.
3. Downstream-зависимость (любое из):
   - Блок внутри цикла (Для Каждого / Для / Пока) — гарантированный partial batch.
   - После КонецПопытки (или после цикла) код читает/использует те же данные из БД.
   - Экспортная Процедура без re-raise — caller продолжает без сигнала о сбое.
   - Документированный контракт процедуры подразумевает «все элементы обработаны» / «состояние согласовано».

### Пример BAD

```bsl
// В цикле: часть действий записана, часть — нет.
// После цикла ОбновитьИсключенныхВДеревеИзБД читает из БД и показывает
// «актуальное» состояние — но для незаписанных действий оно устаревшее.
Для Каждого ЭлементДействия Из ЭлементыДействия Цикл
    Если ... Тогда
        Попытка
            СинхронизироватьИзБП(ЭлементДействия.Действие); // внутри → Записать()
        Исключение
            ЗаписьЖурналаРегистрации(..., ОписаниеОшибки());
        КонецПопытки;
    КонецЕсли;
КонецЦикла;
ОбновитьОтображениеИзБД(Дерево); // читает все действия — в т.ч. незаписанные
```

```bsl
// Последовательный код: запись настройки подавлена, применение читает из БД
Попытка
    ЗаписатьНастройку(НовоеЗначение); // Записать()
Исключение
    ЗаписьЖурналаРегистрации(...);
КонецПопытки;
ПрименитьНастройкуИзБД(); // читает старое значение
```

### Пример GOOD

**Вариант 1 (re-raise — атомарность):** пробросить исключение; caller/пользователь увидит ошибку, downstream не выполнится на неконсистентных данных:

```bsl
Для Каждого ЭлементДействия Из ЭлементыДействия Цикл
    СинхронизироватьИзБП(ЭлементДействия.Действие);
КонецЦикла;
ОбновитьОтображениеИзБД(Дерево);
```

**Вариант 2 (accumulate + signal — управляемый partial batch):** накопить сбои, передать caller, заблокировать downstream для необработанных элементов:

```bsl
СбойныеДействия = Новый Массив;
Для Каждого ЭлементДействия Из ЭлементыДействия Цикл
    Попытка
        СинхронизироватьИзБП(ЭлементДействия.Действие);
    Исключение
        ЗаписьЖурналаРегистрации(..., Строка(ЭлементДействия.Действие) + ": " + ОписаниеОшибки());
        СбойныеДействия.Добавить(ЭлементДействия.Действие);
    КонецПопытки;
КонецЦикла;
Если СбойныеДействия.Количество() > 0 Тогда
    ТекстОшибки = СтрШаблон(НСтр("ru = 'Не удалось синхронизировать %1 из %2 действий. Подробности в журнале регистрации.'"),
        СбойныеДействия.Количество(), ЭлементыДействия.Количество());
    ВызватьИсключение ТекстОшибки;
КонецЕсли;
ОбновитьОтображениеИзБД(Дерево);
```

- AP-030 (скрытый частичный результат — UX-слой; AP-032 — integrity-слой)

## AP-033: Экспортная процедура/функция в модуле формы (Form-as-Service)

**Категория:** architecture
**Severity:** HIGH
**Дата добавления:** 2026-04-01
**Источник:** усиление контроля ревью — соблюдение вендорного стандарта ITS «Правила создания модулей форм» (границы модуля формы)

### Правило

Модуль формы предназначен для логики, которая нужна **только** этой форме и выполняется в контексте формы (UI-оркестрация, обработчики событий формы, взаимодействие с пользователем, получение данных для отображения).

Экспортная процедура/функция (`Экспорт`) в модуле формы расширяет API поверхности формы и делает форму «сервисом» для вызовов извне, нарушая границу ответственности. Переиспользуемую бизнес-логику и методы, которые требуется вызывать из других модулей, следует размещать в модуле объекта/менеджера или в общем модуле.

### Симптом

- В модуле формы (`Forms/*/Ext/Form/Module.bsl`) присутствуют `Процедура ... Экспорт` или `Функция ... Экспорт`.
- Другие модули вызывают методы формы как внешний API (форма становится точкой интеграции), либо экспорт используется как «удобный» способ переиспользовать бизнес-логику из UI.
- Поведение экспортного метода неустойчиво из‑за зависимости от контекста формы (`ЭтаФорма`, `Элементы`, состояние UI).

### Детектирование

1. В файле `Forms/*/Ext/Form/Module.bsl` найти объявления `Процедура ... Экспорт` и `Функция ... Экспорт`.
2. Для каждого совпадения проверить исключения:
   - **Подключаемые команды БСП:** процедуры вида `Подключаемый_*` (для подсистемы «Подключаемые команды»/команды доп. отчётов и обработок) допускаются.
   - **Callback-процедуры платформы:** экспортные процедуры, передаваемые по имени в `ОписаниеОповещения(..., ЭтотОбъект)` и вызываемые как callbacks (при отсутствии альтернативы) допускаются.
3. Любой экспорт в модуле формы, не подпадающий под исключения, — **нарушение AP-033**.

### Пример BAD

```bsl
// Модуль формы: экспорт превращает форму в сервис.
// Внутри — бизнес-логика (запросы/запись/расчёты).

&НаКлиенте
Функция РассчитатьПоказатели(Параметры) Экспорт
    Результат = РассчитатьПоказателиНаСервере(Параметры);
    Возврат Результат;
КонецФункции

&НаСервере
Функция РассчитатьПоказателиНаСервере(Параметры)
    // запросы, расчёт, запись в регистр
    // ...
КонецФункции
```

### Пример GOOD

```bsl
// Модуль формы: только обработчик UI и вызов переиспользуемой логики.

&НаКлиенте
Процедура Рассчитать(Команда)
    Параметры = СформироватьПараметрыРасчёта();
    Результат = ПоказателиСервис.РассчитатьПоказатели(Параметры);
    ПоказатьРезультат(Результат);
КонецПроцедуры
```

### Remediation

- Перенести переиспользуемую бизнес-логику из модуля формы в модуль объекта/менеджера или общий модуль.
- В модуле формы оставить тонкие обработчики UI (сбор параметров, вызов логики, отображение результата).
- Если экспорт обязателен по исключению — зафиксировать обоснование (комментарий у процедуры/в design) и минимизировать зависимость метода от состояния формы.

### Ссылки

- Вендорный стандарт ITS: «Правила создания модулей форм», п.1.1 (модуль формы — только логика формы; экспортные методы — не в модуле формы)
- `onec-code-reviewer.md`, Category 6 (Module Structure) и Category 16 (Anti-pattern registry) — AP-033

## AP-039: ОбработчикСобытия у программно созданного элемента формы (вместо УстановитьДействие)

**Категория:** api  
**Severity:** HIGH  
**Дата добавления:** 2026-04-19  
**Источник:** do2-cf-partial-repeat-params-do3-ui — программное создание колонки таблицы (`Элементы.Добавить(..., Тип("ПолеФормы"), ...)`)

### Правило

Для **элементов формы**, полученных из коллекции `Элементы` (`ПолеФормы`, `КнопкаФормы`, `ГруппаФормы` и другие подтипы `ЭлементФормы`), привязка обработчика события выполняется методом **`УстановитьДействие(ИмяСобытия, ИмяПроцедуры)`**. Метода **`ОбработчикСобытия(...)`** у этих объектов **нет** — при выполнении платформа выдаёт **«Метод объекта не обнаружен (ОбработчикСобытия)»**, форма может не открыться.

### Симптом

- Runtime: **«Метод объекта не обнаружен (ОбработчикСобытия)»** в строке вида `ЭлементФормы.ОбработчикСобытия("ПриИзменении", "ИмяПроцедуры")` после `Элементы.Добавить(...)` / `Элементы.Вставить(...)`.

### Детектирование

1. Grep по `*.bsl` в модулях форм: `\.ОбработчикСобытия\s*\(` — вызов как **метода** с точкой и скобкой.  
2. Легитимные идентификаторы вида `ОбработчикСобытияВыбор`, `Функция ОбработчикСобытияБизнесПроцесса(...)` под этот паттерн **не** попадают (нет `.ОбработчикСобытия(`).

### Пример BAD

```bsl
Поле = ЭтаФорма.Элементы.Добавить("ИмяПоля", Тип("ПолеФормы"), ЭтаФорма.Элементы.ТаблицаИсполнителей);
Поле.ОбработчикСобытия("ПриИзменении", "ИмяПоляПриИзменении");
```

### Пример GOOD

```bsl
Поле = ЭтаФорма.Элементы.Добавить("ИмяПоля", Тип("ПолеФормы"), ЭтаФорма.Элементы.ТаблицаИсполнителей);
Поле.УстановитьДействие("ПриИзменении", "ИмяПоляПриИзменении");
```

### Ссылки

- AP-018 (другой несуществующий API у контекста формы/таблицы), AP-019 (неверный вызов встроенной функции типа)
- Примеры в проекте: `src/ДО2/cf/BusinessProcesses/Согласование/Forms/ФормаБизнесПроцесса/Ext/Form/Module.bsl` (`УстановитьДействие` у `Элементы.Исполнители`); `src/ДО2/cf/Tasks/ЗадачаИсполнителя/Forms/ЗадачиМне/Ext/Form/Module.bsl`; `src/ДО2/cf/DataProcessors/РасширенныйВводКонтактнойИнформации/Forms/ВводАдреса/Ext/Form/Module.bsl`

## AP-040: Changelog/work-artefact metadata в комментариях

**Категория:** release-hygiene  
**Severity:** MEDIUM  
**Дата добавления:** 2026-04-21  
**Источник:** openspec/project.md  
**Область:** все строки комментария `//`, включая **шапку процедуры/функции (JSDoc)** над `Процедура` / `Функция`.

### Правило

Комментарии в коде не должны содержать артефакты процесса разработки: ссылки на OpenSpec/design (kebab-case имена change, пути `reports/`…, `openspec/`…), номера заявок, закомментированный старый код с метками постановки, а также «ИИ-типографику».

**Исключение:** маркеры `// +++ …` / `// --- …` и `{cfMarkerPrefix} …` из whitelist [`openspec/project.md` §Форматы и соглашения по комментариям BSL] и пайплайна `/opsx:apply` — **не** классифицировать как AP-040 **удаление** (границы блока по ЗНИ). Process-only **содержимое** — **AP-053** (rewrite, не delete).

### Детектирование

1. Сноска на документацию процесса в `//`: `(см. <имя-kebab-change> …)`, `см. <kebab-case>/…`, строки содержащие `reports/` или `openspec/` или `temp/reports/` как путь артефакта, упоминание файла `*.md` как источника рассуждения.
2. Kebab-case имена change и process-лексика (ЗНИ, ADR, design, proposal) в комментарии без доменного смысла для читателя кода на ИБ.
3. Устаревшие маркеры: `// НАЧАЛО Изменения внес:`, `Заявка №`, `Подрядчик:`, date-author stamps (вне whitelist-маркеров ЗНИ).
4. Ссылки на постановку: `// Design §3`, `// D11`, `// По design Decision N`, номера задач (`п. 3.1`).
5. `AI-typography markers`: стрелки (`→ ➜ ➔ ❯ ❮ ⇒`), em/en-dash (`— –`), типографические кавычки (`« » „ " " ' '`), декораторы (`••• ★ ✓ ✔ ✗ ✖ …`), 3+ подряд `*` или `=` вне шапки метода.

### Пример BAD

```bsl
// Делает X.
// (см. do2-feature-name / reports/architecture-2026-05-07.md).
```

```bsl
// → [legacy-id]
// *** НАЧАЛО ИЗМЕНЕНИЙ ***
// «Заявка №12345»
// п. 3.1 ТЗ
```

## AP-046: Hook-scope early return

**Категория:** extension/architecture  
**Severity:** HIGH; CRITICAL для подкейса `&Вместо` без `ПродолжитьВызов`  
**Дата добавления:** 2026-04-29  
**Источник:** review hardening: ранний `Возврат` в перехваченной процедуре расширения

### Правило

Перехваченная процедура расширения (`&Перед`, `&После`, `&Вместо`, `&ИзменениеИКонтроль`) является общей точкой композиции доработок. Feature-flag или предметный фильтр должен ограничивать только блок своей доработки, а не всё тело hook-процедуры. Иначе одна фича неявно отключает последующие независимые доработки, добавленные в ту же точку расширения.

Для `&Вместо` ранний `Возврат` без `ПродолжитьВызов` в альтернативной ветке является критичным: расширение молча снимает типовое поведение целиком.

### Симптом

В начале hook-процедуры стоит `Если ... Тогда Возврат; КонецЕсли;`, после которого остаётся длинное тело или несколько смысловых блоков. Условие не является структурным предусловием метода, а относится к отдельной фиче, настройке или предметному фильтру.

### Детектирование

1. Файл относится к расширению, процедура помечена `&Перед`, `&После`, `&Вместо` или `&ИзменениеИКонтроль`.
2. В первых 1-3 операторах процедуры есть `Если ... Тогда Возврат; КонецЕсли;`.
3. Условие является неструктурным: feature-flag (`Использовать*`, `Включена*`, булева настройка), предметный фильтр, проверка режима или вида объекта.
4. После guard-а остаётся тело ≥ 5 строк или ≥ 2 смысловых блока.
5. Подкейс `&Вместо`: если ранний `Возврат` находится до любого `ПродолжитьВызов` и ни одна альтернативная ветка не вызывает `ПродолжитьВызов`, severity = CRITICAL.

### Контр-сигналы

- Структурный fail-fast по неверному типу, отсутствующему обязательному параметру или нарушенному контракту относится к AP-021, а не к AP-046.
- Если после guard-а остаётся единственное действие на 1-3 строки, это AP-017.
- Если процедура состоит только из одного блока одной фичи и не является точкой композиции нескольких доработок, AP-046 не применять.
- Для `&Вместо` допустим `Возврат` после `ПродолжитьВызов`, если типовое поведение уже выполнено или результат явно возвращён вызывающему коду.

### Ремедиация

Сузить область действия guard-а: обернуть feature-flag или предметный фильтр вокруг конкретного блока доработки. Остальные блоки hook-процедуры должны выполняться независимо.

Для `&Вместо` альтернативная ветка должна явно вызывать `ПродолжитьВызов(...)`, если расширение не выполняет собственную полную замену с архитектурным обоснованием.

### Пример BAD (feature-flag в &После)

```bsl
&После("ЗаполнитьЗадачуСогласовать")
Процедура pav_ЗаполнитьЗадачуСогласоватьПосле(Задача, Строка)

    Если НЕ pavДополнительныеФункцииВызовСервера.ИспользоватьРолиДляАвтоподстановок() Тогда
        Возврат;
    КонецЕсли;

    ЗаполнитьИсполнителяПоРоли(Задача);
    ОбновитьДополнительныеРеквизитыЗадачи(Задача);

КонецПроцедуры
```

### Пример GOOD (feature-flag в &После)

```bsl
&После("ЗаполнитьЗадачуСогласовать")
Процедура pav_ЗаполнитьЗадачуСогласоватьПосле(Задача, Строка)

    Если pavДополнительныеФункцииВызовСервера.ИспользоватьРолиДляАвтоподстановок() Тогда
        ЗаполнитьИсполнителяПоРоли(Задача);
    КонецЕсли;

    ОбновитьДополнительныеРеквизитыЗадачи(Задача);

КонецПроцедуры
```

### Пример BAD (&Вместо без ПродолжитьВызов)

```bsl

Процедура pav_ВыполнитьДействие(Параметр)

    Если НЕ pavНастройки.ВключенаНоваяОбработка() Тогда
        Возврат;
    КонецЕсли;

    ВыполнитьНовуюОбработку(Параметр);

КонецПроцедуры
```

### Пример GOOD (&Вместо с явным продолжением)

```bsl
        ВыполнитьНовуюОбработку(Параметр);
    Иначе
        ПродолжитьВызов(Параметр);
    КонецЕсли;

```

## AP-047: Substituted Authority (локальная подмена владельца поведения)

**Категория:** architecture  
**Severity:** HIGH  
**Дата добавления:** 2026-04-29  
**Источник:** обобщение класса дефектов `existing-mechanism-priority.mdc` — Substituted Authority

### Правило

Если код должен сохранить поведение, владельцем которого является другой слой (базовая конфигурация, БСП, платформа, общий модуль, внешний контракт), он должен делегировать владельцу поведения, а не воспроизводить это поведение локальным эквивалентом.

Локальная копия становится вторым источником истины: сегодня она может совпадать с владельцем, но после обновления базы, БСП или контракта владелец изменится, а копия останется прежней. Это нарушение Preference Hierarchy из `.cursor/rules/existing-mechanism-priority.mdc`: переход от делегирования к альтернативному механизму требует явного обоснования.

### Симптом

- Метод или функция с именем вроде `Типовое*`, `Стандартное*`, `Базовое*`, `Исходное*` вручную повторяет алгоритм другого слоя.
- Off-ветка feature-flag возвращает результат локальной копии вместо вызова владельца поведения.
- В design / tasks / комментариях есть формулировки «эквивалентно типовой», «повторяет тело», «как в базе», но в коде нет делегирования (`ПродолжитьВызов(...)`, штатного API, БСП-обёртки или существующего общего модуля).
- Локальный код повторяет запрос, форматирование результата, ветвления или правила другого метода ради «эквивалентности».

### Детектирование

Применять семантически, не только через grep. Нарушение есть, если выполняются все три условия:

1. В коде или surrounding context заявлена цель сохранить чужое поведение: база, БСП, платформа, общий модуль, внешний контракт.
2. Есть локальный алгоритм, структурно подобный алгоритму владельца: совпадает запрос, последовательность шагов, набор полей, форматирование результата или правила ветвления.
3. Есть доступный механизм делегирования владельцу поведения, но он не используется: `ПродолжитьВызов(...)` для `&Вместо`, штатная экспортная функция, БСП-API, существующий общий модуль или точечное `&ИзменениеИКонтроль` через `#Вставка` / `#Удаление`.

Дополнительные grep-сигналы для ручной проверки: `Типовое`, `Стандартное`, `Базовое`, `Исходное`, `эквивалент`, `повторяет тело`, `как в базе`, `как типовой`.

### Контр-сигналы

- Локальная функция не повторяет владельца, а реализует новое предметное поведение с другим контрактом, набором данных или правилами.
- Делегирование невозможно технически или функционально, и это зафиксировано в `design.md`, `architecture-*.md` или ADR: описаны владелец поведения, почему уровни 1-3 Preference Hierarchy не подходят, drift-риск и план верификации.
- Локальный код является адаптацией паттерна базы, а не копией конкретного поведения владельца; это уровень 3 Preference Hierarchy и он явно обоснован.

### Ремедиация

1. Делегировать владельцу поведения: вызвать штатный API, БСП-обёртку, общий модуль или `ПродолжитьВызов(...)`.
2. Для расширений с feature-flag fallback на стандартное поведение предпочесть `&Вместо` + `ПродолжитьВызов(...)` в off-ветке, если задача не требует точечного `#Вставка` / `#Удаление` внутри типового тела.
3. Для `&ИзменениеИКонтроль` сузить изменение до собственных блоков внутри `#Вставка` / `#Удаление`; не создавать локальную копию всего базового алгоритма как fallback.
4. Если локальная реализация должна стать новым источником истины — оформить архитектурное обоснование уровня 4, ADR или design-секцию с drift-риском и проверкой актуальности копии.

### Пример BAD

```bsl
&ИзменениеИКонтроль("ПолучитьПредставление")
Функция ext_ПолучитьПредставление(Ссылка)

    #Вставка
    Если НЕ Настройки.ИспользоватьНовоеПредставление() Тогда
        Возврат ext_СтандартноеПредставление(Ссылка);
    КонецЕсли;
    #КонецВставки

    // типовое тело с точечной доработкой
    Возврат СформироватьНовоеПредставление(Ссылка);

КонецФункции

Функция ext_СтандартноеПредставление(Ссылка)
    // Локальная копия алгоритма базового ПолучитьПредставление:
    // тот же запрос, те же поля, то же форматирование результата.
    Возврат ВыполнитьКопиюТиповогоАлгоритма(Ссылка);
КонецФункции
```

### Пример GOOD (делегирование владельцу)

```bsl

    Возврат ПродолжитьВызов(Ссылка);

КонецФункции
```

### Пример GOOD (&ИзменениеИКонтроль с узкой доработкой)

## AP-048: Ручная сериализация ссылки/GUID в обмене

**Категория:** integration  
**Severity:** HIGH  
**Дата добавления:** 2026-04-29  
**Источник:** `.cursor/docs/standard/std-12-integration-exchange.md`

### Правило

В JSON, HTTP/RMQ-сообщениях и других внешних обменах ссылку 1С передавать через `XMLСтрока(<Ссылка>)`.

Не извлекать GUID вручную через `УникальныйИдентификатор()`, не приводить его к строке и не оборачивать результат `УникальныйИдентификатор()` в `XMLСтрока()`. Это дублирует платформенный механизм и часто приводит к лишним проверкам пустой ссылки.

### Симптом

- В подготовке структуры сообщения встречается `Ссылка.УникальныйИдентификатор()`.
- GUID собирается через `Строка(...)` или `"" + ...`.
- Код вызывает `XMLСтрока(Ссылка.УникальныйИдентификатор())` вместо `XMLСтрока(Ссылка)`.
- Перед сериализацией пустой ссылки вручную подставляется `00000000-0000-0000-0000-000000000000`.

### Детектирование

Применять в коде подготовки JSON/обмена: рядом есть `ЗаписатьJSON`, `ЗаписьJSON`, `СтруктураСообщения`, `HTTP`, `RMQ`, `Вставить("GUID"`, `Выгруз`, `Загруз`, `Обмен`.

Grep-сигналы:

1. `УникальныйИдентификатор()` внутри подготовки структуры сообщения.
2. `Строка(.*УникальныйИдентификатор())`.
3. `XMLСтрока(.*УникальныйИдентификатор())`.
4. `"" + .*УникальныйИдентификатор()`.

### Контр-сигналы

- Внешний контракт требует именно значение типа `УникальныйИдентификатор`, а не строковый GUID, и это явно описано в коде/дизайне.
- Код не готовит внешний обмен, а работает с внутренним объектом платформы, где нужен именно тип `УникальныйИдентификатор`.

### Ремедиация

1. Передавать ссылку напрямую: `XMLСтрока(<Ссылка>)`.
2. Удалить ручную обработку пустой ссылки, если она только подставляет нулевой GUID.
3. Сохранить имя поля внешнего контракта (`GUID`, `ContractorGUID` и т.п.), если контракт уже закреплён.

### Пример BAD

```bsl
СтруктураСообщения.Вставить("ContractorGUID", Строка(Контрагент.УникальныйИдентификатор()));
СтруктураСообщения.Вставить("OrderGUID", "" + ЗаказПокупателя.УникальныйИдентификатор());
СтруктураСообщения.Вставить("ReasonGUID", XMLСтрока(ПричинаОтмены.УникальныйИдентификатор()));
```

### Пример GOOD

```bsl
СтруктураСообщения.Вставить("ContractorGUID", XMLСтрока(Контрагент));
СтруктураСообщения.Вставить("OrderGUID", XMLСтрока(ЗаказПокупателя));
СтруктураСообщения.Вставить("ReasonGUID", XMLСтрока(ПричинаОтмены));
```

## AP-052: Фильтрация строк формы через условное оформление

**Категория:** api  
**Severity:** HIGH  
**Дата добавления:** 2026-05-03  
**Источник:** запрос пользователя 2026-05-03; пример из `src/ДО2/cf/BusinessProcesses/Согласование/Forms/ФормаИзменениеПараметров/Ext/Form/Module.bsl`:1409-1459

### Правило

Условное оформление управляемой формы управляет оформлением элементов (видимость колонок/полей, цвет, доступность, шрифт), а не составом строк таблицы. Чтобы скрыть/отфильтровать строки таблицы формы, использовать `Элементы.<ИмяТаблицыНаФорме>.ОтборСтрок` (`Новый ФиксированнаяСтруктура(...)` или настройку существующих элементов отбора).

`Оформление.УстановитьЗначениеПараметра("Видимость", Ложь)` с полем = именем таблицы и отбором по полям строки не скрывает строки: он скрывает оформление ячеек, строки остаются видимы как пустые.

### Симптом

Разработчик заявляет "скрытие строк по условию", но при выполнении строки таблицы остаются на форме (часто как пустые), потому что `"Видимость" = Ложь` применяется к полям/колонкам, а не к строкам.

### Детектирование (семантически)

1. В модуле формы `Forms/*/Ext/Form/Module.bsl` найти блок (обычно отдельная серверная процедура), где одновременно:
   - `УсловноеОформление.Элементы.Добавить()`;
   - `Поля.Добавить().Поле = Новый ПолеКомпоновкиДанных("<Имя>")`, где `<Имя>` — имя реквизита формы типа Таблица (или элемента таблицы), а не отдельной колонки/группы;
   - `Отбор.Элементы` содержит условие по полю строки этой таблицы (`Объект.<Таблица>.<Реквизит>` или иной путь к реквизиту строки);
   - `Оформление.УстановитьЗначениеПараметра("Видимость", Ложь)`.
2. Усилители: имя процедуры/комментарий содержит `Скрытия`, `Скрыть`, `Фильтр строк`, `Видимость строк`, `Исключённых`; в том же сценарии нет присваивания `Элементы.<Таблица>.ОтборСтрок` на ту же таблицу.
3. Не считать срабатыванием:
   - `"Видимость"` применяется к колонке/полю внутри строки (легитимное скрытие ячеек/колонок);
   - используются параметры оформления `Цвет`, `ЦветТекста`, `Шрифт`, `ВидВыделения`, `Доступность` (легитимное оформление, не фильтрация);
   - отбор и оформление применены к скалярному реквизиту формы или элементу вне таблицы.

### Пример BAD

```bsl
&НаСервере
Процедура pav_УстановитьУсловноеОформлениеСкрытияИсключённыхПоУсловию()
    ЭлементОформления = УсловноеОформление.Элементы.Добавить();
    Поля = ЭлементОформления.Поля.Элементы;
    Поля.Добавить().Поле = Новый ПолеКомпоновкиДанных("Исполнители");

    ОтборЭлемента = ЭлементОформления.Отбор.Элементы.Добавить(Тип("ЭлементОтбораКомпоновкиДанных"));
    ОтборЭлемента.ЛевоеЗначение = Новый ПолеКомпоновкиДанных("Объект.Исполнители.pav_ИсключёнПоУсловию");
    ОтборЭлемента.ВидСравнения = ВидСравненияКомпоновкиДанных.Равно;
    ОтборЭлемента.ПравоеЗначение = Истина;

    ЭлементОформления.Оформление.УстановитьЗначениеПараметра("Видимость", Ложь);
КонецПроцедуры
```

### Пример GOOD

```bsl
&НаКлиенте
Процедура pav_УстановитьВидимостьИсключённыхПоУсловию()
    Если pav_ПоказыватьИсключённыхПоУсловию Тогда
        Элементы.Исполнители.ОтборСтрок = Неопределено;
    Иначе
        Элементы.Исполнители.ОтборСтрок = Новый ФиксированнаяСтруктура("pav_ИсключёнПоУсловию", Ложь);
    КонецЕсли;
КонецПроцедуры
```

Признак строки `pav_ИсключёнПоУсловию` рассчитывается заранее на сервере при загрузке/изменении данных.

### Ремедиация

1. Удалить процедуру установки УО, скрывающую строки таблицы через `"Видимость" = Ложь`.
2. Обеспечить, что в строке таблицы есть реквизит-признак отбора (вычислить при загрузке/изменении источника данных).
3. Применить `Элементы.<ИмяТаблицыНаФорме>.ОтборСтрок` как `ФиксированнаяСтруктура` или через настройку существующих элементов отбора (`Элементы.<Таблица>.ОтборСтрок.<Поле>.Использование/ВидСравнения/Значение`); пересчитывать при изменении управляющего флажка/исходных данных.
4. Для подсветки оставшихся строк (цвет/шрифт) — отдельный элемент `УсловноеОформление` с параметрами оформления, не `Видимость`.

### Связанные AP

- AP-018 (другой неверный API контекста формы/таблицы)
- AP-039 (другой неверный API программно созданного элемента формы)
- AP-016 (band-aid: подмена корректного механизма "похожим" внешне поведением)

### Ссылки

- Rule: `.cursor/rules/bsl-antipatterns.mdc`
- Reviewer: `.cursor/agents/onec-code-reviewer.md`, Phase 2 step 16 (Anti-pattern registry)

## AP-051: Несжатые смежные changelog-маркеры одного ЗНИ

**Категория:** release-hygiene
**Severity:** MEDIUM
**Дата добавления:** 2026-05-03
**Источник:** `openspec/project.md` → `MARKER-MERGE-001`

### Правило

Whitelisted-маркеры ЗНИ `// +++ ... [ID#NNN]` / `// --- ... [ID#NNN]` должны оформлять один цельный изменённый фрагмент, а не каждый соседний оператор отдельно.

Если подряд идут несколько блоков с одним `[ID#NNN]` **или** с одним domain_label + developer + семантикой изменения (канон без ID — см. `openspec/project.md` MARKER-MERGE-001), их нужно объединить в один внешний блок: один открывающий `// +++` перед всем фрагментом и один закрывающий `// ---` после него.

Если у объединяемых открывающих маркеров разные даты, в результирующем маркере оставляется самая поздняя дата. Автор берётся из последнего открывающего маркера.

### Детектирование

Семантическая проверка reviewer-а, не regex-механика:

1. В коде подряд идут пары `// +++ ... [ID#NNN]` ... `// --- ... [ID#NNN]`.
2. Следующий блок имеет тот же `[ID#NNN]`.
3. Смысл изменения один и тот же: тот же ЗНИ, та же задача, соседний фрагмент одного изменения.
4. Между блоками нет кода или комментариев чужого ЗНИ.

Если ID разные, изменения относятся к разным ЗНИ или между блоками есть чужой код/комментарий, объединение не применять.

### Ремедиация

1. Удалить внутреннюю пару `// --- ... [ID#NNN]` / `// +++ ... [ID#NNN]`.
2. Оставить один внешний блок вокруг всего соседнего фрагмента.
3. Если даты отличаются, в открывающем маркере оставить самую позднюю дату.
4. Не использовать это правило как замену AP-040: если маркеры не попадают в whitelist проекта, их нужно удалить, а не объединять.

### Пример BAD

```bsl
// +++ <ФИО> 02.05.2026 Учет условия для невыполненных задачи в форме «Повтор согласования»
ДобавленныйРеквизитФормы = Новый РеквизитФормы("pav_ПоказыватьИсключённыхПоУсловию", Новый ОписаниеТипов("Булево"));
МассивДобавляемыхРеквизитов.Добавить(ДобавленныйРеквизитФормы);
// --- <ФИО>
// +++ <ФИО> 02.05.2026 Учет условия для невыполненных задачи в форме «Повтор согласования»
ДобавленныйРеквизитФормы = Новый РеквизитФормы("pav_ИсключёнПоУсловиюУчастника",
    Новый ОписаниеТипов("Булево"), "Объект.Исполнители");
МассивДобавляемыхРеквизитов.Добавить(ДобавленныйРеквизитФормы);
// --- <ФИО>
```

### Пример GOOD

```bsl
// +++ <ФИО> 02.05.2026 Учет условия для невыполненных задачи в форме «Повтор согласования»
ДобавленныйРеквизитФормы = Новый РеквизитФормы("pav_ПоказыватьИсключённыхПоУсловию", Новый ОписаниеТипов("Булево"));
МассивДобавляемыхРеквизитов.Добавить(ДобавленныйРеквизитФормы);
ДобавленныйРеквизитФормы = Новый РеквизитФормы("pav_ИсключёнПоУсловиюУчастника",
    Новый ОписаниеТипов("Булево"), "Объект.Исполнители");
МассивДобавляемыхРеквизитов.Добавить(ДобавленныйРеквизитФормы);
// --- <ФИО>
```

- Rule: `.cursor/rules/bsl-antipatterns.mdc`
- Project convention: `openspec/project.md` → `MARKER-MERGE-001`

## AP-053: Пустое или process-only содержимое whitelisted-маркера

**Категория:** release-hygiene  
**Severity:** MEDIUM  
**Дата добавления:** 2026-06-09  
**Источник:** `openspec/project.md` § «Канон маркеров (domain_label)»

### Правило

Whitelist exempt — **удаление** пары (AP-040). **Содержимое** открывающих whitelist-строк и однострочных cf-маркеров (`// {cfMarkerPrefix} …` без `+++`) должно быть осмысленным **domain_label** на русском, без process-меток.

Запреты domain_label — SSOT в `openspec/project.md` § Канон domain_label (new/apply/reviewer ссылаются, не дублируют).

### Область

- cfe: open `// +++ …` в scope whitelist (PAO, ДО3)
- cf: open `// {cfMarkerPrefix} … +++` и однострочные `// {cfMarkerPrefix} {domain_label}`

Исключение: documented `marker_style: minimal` в Metadata change — пустой suffix допустим.

### Детектирование

1. Нет domain_label после автора/даты/префикса (кроме `minimal`).
2. Process-лексика из списка запретов project.md без доменного смысла.
3. Kebab-case имя change как основное содержимое маркера.

### Ремедиация

**Переписать текст** открывающего/однострочного маркера; отчёты и артефакты — в `reports/`, не в BSL. **Не** удалять пару как AP-040.

### Пример BAD

```bsl
// +++ <ФИО> 09.06.2026 устранение findings release-review ЭДО ПАО cfe
```

### Пример GOOD

```bsl
// +++ <ФИО> 09.06.2026 при 429 и 5xx не снимать операцию с опроса — повтор на следующем цикле
// {cfMarkerPrefix} флаг «Не устанавливать подписанта в документ» +++
```

### Ссылки

- AP-040 (граница: removal vs rewrite)
- `.cursor/docs/marker-layers-guide.md`

## AP-049: Числовая строка через Строка() + удаление НПП

**Категория:** integration  
**Severity:** MEDIUM  
**Дата добавления:** 2026-04-29  
**Источник:** `.cursor/docs/standard/std-12-integration-exchange.md`

### Правило

Если внешнему контракту нужна строка с числом без разделителей групп, использовать `Формат(<Число>, "ЧГ=0")`.

Не получать такую строку через `Строка(<Число>)` с последующим удалением `Символы.НПП` или обычных пробелов.

### Симптом

- `СтрЗаменить(Строка(<Число>), Символы.НПП, "")`.
- `СтрЗаменить(СтрЗаменить(Строка(<Число>), ...), ...)` ради очистки пробелов.
- Комментарий или имя переменной указывает на подготовку числового поля для JSON/обмена, а форматирование сделано через строковую чистку.

### Детектирование

Grep-сигналы:

1. `СтрЗаменить(Строка(`.
2. `Символы.НПП` рядом с числовым полем внешнего обмена.
3. `Строка(Сумма|Количество|Цена|Итог|Total|Amount)` в подготовке JSON без явного `Формат`.

### Контр-сигналы

- Код намеренно формирует человекочитаемое представление для интерфейса, а не машинный контракт.
- Формат строки задан внешним контрактом иначе и явно описан рядом с кодом.

### Ремедиация

Заменить строковую очистку на форматирование:

```bsl
СтроковоеЧисло = Формат(Сумма, "ЧГ=0");
```

При наличии требований к десятичному разделителю или точности указать их в формате явно.

### Пример BAD

```bsl
ДанныеСтроки.Вставить("TotalSumm", СтрЗаменить(Строка(СуммаДокумента), Символы.НПП, ""));
```

### Пример GOOD

```bsl
ДанныеСтроки.Вставить("TotalSumm", Формат(СуммаДокумента, "ЧГ=0"));
```

### Ссылки

- Rule: `.cursor/rules/bsl-antipatterns.mdc`
- Standard: `.cursor/docs/standard/std-12-integration-exchange.md`

## AP-050: Неинформативное блокирующее сообщение

**Категория:** ux  
**Severity:** MEDIUM  
**Дата добавления:** 2026-04-29  
**Источник:** `.cursor/docs/standard/std-12-integration-exchange.md`

### Правило

Если сообщение блокирует пользовательский сценарий, запись, проведение или обмен, оно должно объяснять, что именно не так и что ожидается.

Минимальный состав полезного сообщения:

1. Что произошло.
2. Какое фактическое значение мешает операции.
3. Какое значение или действие ожидается.

### Симптом

- Рядом с `Отказ = Истина`, `ВызватьИсключение`, `СообщитьПользователю`, `ПоказатьПредупреждение` сообщение формата «некорректный статус», «заполните данные», «операция невозможна».
- Код проверяет конкретный статус, реквизит, право или условие обмена, но сообщение не сообщает фактическое значение и ожидаемое условие.
- Пользователь не может понять из сообщения, что исправить без обращения к разработчику.

### Детектирование

Это семантическая эвристика reviewer-а, не массовая regex-проверка всех строк.

Проверять только блокирующие сообщения:

1. Ветка устанавливает `Отказ = Истина`.
2. Ветка вызывает `ВызватьИсключение`.
3. Ветка показывает предупреждение и сразу прекращает сценарий.
4. Ветка сообщает пользователю об ошибке обмена/проведения/записи.

Если сообщение просто информирует о завершении операции или логирует техническую диагностику для администратора, AP-050 не применять.

### Контр-сигналы

- Фактическое и ожидаемое состояние уже очевидны из контекста формы и текста сообщения.
- Сообщение сопровождается выделением конкретного поля формы через БСП-механизм, и пользователь видит, что заполнить.
- Детали намеренно не раскрываются по требованиям безопасности, но сообщение даёт безопасное действие для пользователя.

### Ремедиация

1. Добавить фактическое значение: текущий статус, документ, реквизит, контрагент, период.
2. Добавить ожидаемое условие: нужный статус, заполненный реквизит, допустимый период, требуемое право.
3. Для пользовательских строк соблюдать локализацию проекта (`НСтр`, БСП-сообщения), если она требуется окружающим кодом.

### Пример BAD

```bsl
Если НЕ ЗаказИмеетКорректныйСтатус(Заказ) Тогда
    ВызватьИсключение "Заказ имеет некорректный статус для обмена";
КонецЕсли;
```

### Пример GOOD

```bsl
Если НЕ ЗаказИмеетКорректныйСтатус(Заказ) Тогда
    ВызватьИсключение СтрШаблон(
        "Нельзя отправить заказ %1: текущий статус %2, ожидается %3.",
        Заказ,
        ТекущийСтатус,
        ОжидаемыйСтатус);
КонецЕсли;
```

### Severity / Action

Default action: `REFACTOR`.

Повысить до `MUST_FIX`, если сообщение блокирует пользовательский сценарий и без уточнения пользователь не может восстановиться: неизвестно, какой статус нужен, какое поле заполнить, какой объект исправить или почему обмен остановлен.

### Ссылки

- Rule: `.cursor/rules/bsl-antipatterns.mdc`
- Standard: `.cursor/docs/standard/std-12-integration-exchange.md`
