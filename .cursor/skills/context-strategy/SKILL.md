Plan how to analyze files **before** reading them. Choose between direct reads, partial reads, and subagent delegation to keep the main agent's context lean and focused.

## Entry Protocol (MANDATORY — STOP-GATE)

**СТОП. Не читать файлы из задачи.**

### Шаг 1 — Инвентаризация

Для каждого файла из задачи пользователя собрать информацию **без чтения содержимого**:

1. Проверить существование: `Glob` по пути.
2. Оценить размер: из метаданных IDE (`total lines` в recently viewed), `wc -l` через Shell, или Read первых 5 строк для ориентира.
3. Определить тип: код (`.bsl`), данные (`.xml`, `.json`, `.csv`), отчёт (`.htm`, `.html`), документ (`.md`).

Результат — таблица:

```
| # | Файл                          | Строк | Тип     |
|---|-------------------------------|-------|---------|
| 1 | ObjectModule.bsl              | 2519  | Код     |
| 2 | Файл выгрузки.xml             | ~800  | Данные  |
| 3 | Диадок документы.htm           | ~200  | Отчёт   |
```

### Шаг 2 — Классификация связей

Определить по контексту задачи пользователя (не по содержимому файлов):

- **Независимые** — файлы не ссылаются друг на друга, каждый отвечает на свой вопрос
- **Цепочка вызовов** — модуль A вызывает модуль B, который вызывает C
- **Кросс-ссылки** — данные из файла X нужно наложить на алгоритм из файла Y (синтез на финальном этапе)

### Шаг 3 — Выбор стратегии (Decision Matrix)

| Ситуация | Стратегия | Агент/режим |
|----------|-----------|-------------|
| 1 файл < 150 строк | Читать напрямую (Read) | — |
| 1 файл 150–500 строк, конкретный аспект | Читать фрагмент (Read offset/limit) | — |
| 1 файл 500+ строк | Субагент с фокусированным вопросом | 1С-контент (`src/**`, `cfe/**`, `.bsl`, 1С XML) → `onec-code-explorer`; прочие → `explore` |
| Файл данных (XML, HTML, CSV, JSON) любого размера | Субагент для извлечения структуры | 1С XML/metadata → `onec-code-explorer`; прочие данные → `explore` |
| 2–3 `.bsl` с цепочкой вызовов | Один субагент на всю цепочку | `onec-code-explorer` |
| 3+ независимых файлов | Параллельные субагенты (до 4) | Комбинация по типам |
| Смешанный набор (код + данные) | Комбинация стратегий | Для 1С-контента использовать только 1С-агентов; generic `explore` допустим только для не-1С данных |

**Пограничные случаи:**
- Файл данных < 50 строк (маленький JSON/CSV) → читать напрямую
- Файл 150–500 строк, нужен целиком → субагент (контекст дороже одного Read)
- Неизвестный размер файла → оценить через Shell `wc -l`, затем решить

### Шаг 4 — Формулировка вопросов субагентам

Для каждого субагента сформулировать **конкретный вопрос** до запуска. Использовать шаблоны ниже.

### Шаг 5 — Вывод плана (кратко)

Показать пользователю краткий план (2–5 строк):

```
Стратегия анализа:
- ObjectModule.bsl (2519 строк) → субагент onec-code-explorer: алгоритм сопоставления
- Файл выгрузки.xml → субагент explore/fast: структура и ключевые записи
- Диадок документы.htm + ДокументыВАктивнойОбработке.htm → субагент explore/fast: таблицы документов
Затем — синтез результатов.
```

Если план очевидно прост (1–2 файла, оба маленькие) — пропустить вывод, перейти к выполнению.

### Шаг 6 — Выполнение

Запустить субагентов / прямое чтение согласно плану. Независимые субагенты — параллельно (до 4). Перейти к протоколу синтеза.

## Шаблоны вопросов субагентам

### Код `.bsl` — анализ алгоритма

```
В файле [путь] найди и опиши [алгоритм / паттерн / процедуру].

Верни:
- Назначение (1–2 предложения)
- Входные параметры и их контракты
- Ключевые ветвления и условия
- Выходной результат
- Ограничения и edge cases
- Вызываемые зависимости (какие модули/функции использует)

Формат: структурированное описание. Не возвращай полный исходный код —
только ключевые фрагменты (до 30 строк суммарно) с номерами строк.
```

### Код `.bsl` — поиск вызовов / зависимостей

```
В файле [путь] найди все вызовы [функция / модуль / паттерн].

Для каждого вызова верни:
- Строку вызова (номер + контекст ±2 строки)
- Из какой процедуры/функции вызывается
- С какими параметрами

Формат: нумерованный список.
```

### Данные XML / JSON — структура

```
Разбери файл [путь]. Извлеки:
- Корневую структуру (вложенность элементов, глубина 2–3 уровня)
- Количество записей / элементов верхнего уровня
- Ключевые поля каждой записи (имена, типы если можно определить)
- 3–5 примеров записей (сокращённо — ключевые поля)

Формат: схема структуры + таблица примеров.
Не возвращай файл целиком.
```

### Отчёт HTML — извлечение данных

```
Из файла [путь] извлеки все [сущности: документы / записи / строки таблицы].

Для каждой сущности верни поля: [перечислить ожидаемые поля, напр.: номер, дата, контрагент, сумма].

Формат: таблица. Если записей > 20 — первые 10 + последние 5 + общее количество.
```

### Произвольный файл — разведка

Использовать когда неизвестно, что в файле:

```
Прочитай файл [путь]. Определи:
- Тип содержимого (код, данные, отчёт, лог, конфигурация)
- Общий размер и структура
- Ключевая информация (первые 30 строк + выводы)

Формат: краткая сводка (до 15 строк).
```

## Протокол синтеза

После получения результатов от всех субагентов / прямых чтений:

1. **Проверить полноту**: все файлы обработаны? Все вопросы получили ответ?
2. **Кросс-анализ**: наложить результаты друг на друга в контексте исходного вопроса пользователя. Что в данных соответствует / противоречит алгоритму? Где пересечения?
3. **Точечное дочитывание** (при пробелах): если синтез выявил неясность — прочитать конкретный фрагмент (Read offset/limit, 10–30 строк), не целый файл.
4. **Сформировать ответ** на исходный вопрос пользователя, используя результаты субагентов как доказательную базу.

## Исключения (Bypass)

Скилл **не нужен**, если:

- Пользователь просит «прочитай файл» без аналитической задачи
- Задача о конкретной строке / фрагменте одного файла («что в строке 140?»)
- 1–2 файла < 150 строк каждый и вопрос простой
- Файл уже загружен в контекст (recently viewed, < 200 строк)
- Однострочная правка, опечатка

## Guardrails

- **Не читать файлы до завершения Entry Protocol** (шаги 1–5)
- **Не запускать субагентов без сформулированного вопроса** — размытый вопрос → размытый ответ
- **Максимум 4 параллельных субагента** (ограничение платформы)
- **Субагент возвращает структурированную выжимку**, не сырые данные — указывать это в промпте
- **При неизвестном контексте** (не понятно, что спросить у субагента) — прочитать первые 20–30 строк файла для ориентации, затем сформулировать вопрос
- **Не перезапрашивать полный файл** после субагента — если нужно уточнение, читать конкретный фрагмент (offset/limit)
